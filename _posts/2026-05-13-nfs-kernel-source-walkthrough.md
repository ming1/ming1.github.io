# NFS in the Linux Kernel: Source Code Walkthrough

The Linux NFS implementation spans roughly **202,000 lines of code** across
the client, server, shared utilities, and RPC transport layer. This post
walks through the kernel source layout, explaining what each directory and
file group does, how they relate to each other, and where to look when
studying or debugging NFS.

All paths are relative to the kernel source root. Code statistics are
approximate and based on the current mainline kernel.

## 1. The Big Picture: NFS Source Organization

```
  Linux Kernel NFS Source Layout
  ──────────────────────────────

  fs/nfs/                          NFS Client (~85,000 LOC, 88 files)
  ├── Core files                   Inode, dir, file, superblock, mount
  ├── Protocol files               NFSv2/v3/v4 XDR, procedures
  ├── NFSv4 state management       Opens, locks, delegations, sessions
  ├── pNFS layout drivers          Block, Files, Flex Files
  ├── Callback handling            Server → client callbacks
  └── Caching & tracing            fscache, tracepoints

  fs/nfsd/                         NFS Server (~52,000 LOC, 56 files)
  ├── Core files                   Service, VFS ops, exports, file handles
  ├── Protocol files               NFSv2/v3/v4 XDR, procedures
  ├── NFSv4 state management       Client tracking, recovery, leases
  ├── pNFS layout support          Block, Flex Files (server side)
  └── Admin & monitoring           debugfs, netlink, statistics

  fs/nfs_common/                   Shared Code (~1,300 LOC, 7 files)
  ├── grace.c                      Grace period coordination
  ├── nfsacl.c                     ACL protocol helpers
  ├── nfslocalio.c                 Local I/O bypass
  └── nfs_ssc.c                    Server-side copy support

  net/sunrpc/                      RPC Transport (~51,000 LOC, 64 files)
  ├── Core RPC                     Client, server, scheduling, XDR
  ├── xprtrdma/                    RDMA transport (InfiniBand/RoCE)
  └── auth_gss/                    Kerberos/GSSAPI authentication

  include/linux/nfs*.h             Kernel-internal NFS headers
  include/uapi/linux/nfs*.h        User-space visible NFS definitions
  include/linux/sunrpc/            RPC API headers
```

The architecture separates four concerns:

| Layer | Directory | Purpose |
|-------|-----------|---------|
| **NFS Client** | `fs/nfs/` | Mounts remote NFS servers, reads/writes files |
| **NFS Server** | `fs/nfsd/` | Exports local filesystems via NFS protocol |
| **Shared** | `fs/nfs_common/` | Code used by both client and server |
| **RPC Transport** | `net/sunrpc/` | Sun RPC protocol, XDR, TCP/UDP/RDMA transport |

The client and server are independent kernel modules — a machine can run
one, both, or neither. They communicate over the network via Sun RPC
(even when they are on the same machine, unless local I/O bypass is
enabled).

## 2. fs/nfs/ — The NFS Client

The NFS client is the larger of the two, at ~85,000 lines. It implements
a Linux filesystem that makes remote NFS servers appear as local
directories.

### 2.1 Core Client Operations

These files implement the fundamental filesystem interface — the VFS
operations that the kernel calls when applications do `open()`, `read()`,
`stat()`, `readdir()`, etc. on NFS-mounted files.

```
  Application
      │
      │  open("/mnt/nfs/file.txt", O_RDONLY)
      ▼
  VFS (Virtual File System)
      │
      │  Dispatches to NFS filesystem operations
      ▼
  ┌──────────────────────────────────────────────────┐
  │  fs/nfs/ Core Files                              │
  │                                                  │
  │  super.c (~1,460 LOC)                             │
  │    Superblock ops, mount/unmount, filesystem      │
  │    registration (nfs_fs_type, nfs4_fs_type)       │
  │                                                  │
  │  client.c (~1,530 LOC)                            │
  │    NFS client lifecycle — struct nfs_client       │
  │    creation, sharing, destruction. Manages        │
  │    connections to NFS servers (one nfs_client     │
  │    per server, shared across mounts)              │
  │                                                  │
  │  inode.c (~2,800 LOC)                             │
  │    Inode management — attribute caching,          │
  │    revalidation (checks if cached attrs are       │
  │    stale), change attribute tracking              │
  │                                                  │
  │  dir.c (~3,430 LOC)                               │
  │    Directory operations — lookup, readdir,        │
  │    create, mkdir, unlink, rename. Calls           │
  │    nfs_sillyrename() from unlink.c when           │
  │    deleting open files                            │
  │                                                  │
  │  file.c (~970 LOC)                                │
  │    File operations — open, release, mmap,         │
  │    fsync, flock                                   │
  │                                                  │
  │  namespace.c                                     │
  │    Filesystem namespace, automount of referrals   │
  │                                                  │
  │  getroot.c                                       │
  │    Root dentry acquisition during mount            │
  │                                                  │
  │  nfsroot.c                                       │
  │    NFS root filesystem (booting over NFS)          │
  └──────────────────────────────────────────────────┘
```

**Key data structures:**

- `struct nfs_client` (`include/linux/nfs_fs_sb.h`) — represents a
  connection to one NFS server. Shared across all mounts from that server.
  Holds the RPC client handle, protocol version, and session info.

- `struct nfs_server` (`include/linux/nfs_fs_sb.h`) — per-mount server
  info. Contains mount options (rsize, wsize, acregmin, etc.), capability
  flags, and a pointer to the `nfs_client`.

- `struct nfs_inode` (`include/linux/nfs_fs.h`) — NFS-specific inode data,
  embedded in the VFS inode via `container_of()`. Holds cached attributes,
  delegation state, page cache state, and the NFSv4 open state list.

### 2.2 I/O Operations

These files handle the actual data path — reading and writing file
contents over NFS.

|------|-----|---------|

| `direct.c` | 900 | O_DIRECT I/O (bypasses page cache) |
| `io.c` | — | I/O helper functions |

```
  Read path:                         Write path:
  ──────────                         ───────────

  nfs_file_read()                    nfs_file_write()
      │                                  │
      ▼                                  ▼
  generic_file_read_iter()           nfs_file_write_iter()
      │                                  │
      ▼ (page cache miss)               ▼
  nfs_readahead()                    nfs_writepages() / nfs_write_folio()
      │                                  │
      ▼                                  ▼

      │                                  │
      ▼                                  ▼

```

The I/O path uses **pagelist.c** as the central engine that collects pages
into batches, applies layout driver logic (for pNFS), and dispatches RPC
calls. The `struct nfs_pageio_descriptor` manages this batching.

### 2.3 Protocol & XDR

Each NFS protocol version has its own set of files for XDR (External Data
Representation) encoding/decoding and procedure implementations.

```
  Protocol Version Files
  ──────────────────────

  NFSv2:                  NFSv3:                  NFSv4:
  ┌──────────────┐       ┌──────────────┐       ┌──────────────────────┐
  │ nfs2xdr.c    │       │ nfs3xdr.c    │       │ nfs4xdr.c (~7,790 LOC)│
  │ (~1,090 LOC) │       │ (~2,510 LOC) │       │ nfs4proc.c (~10,750) │
  │ proc.c       │       │ nfs3proc.c   │       │ nfs4state.c (~2,710) │
  │ (~780 LOC)   │       │ (~1,130 LOC) │       │ nfs4session.c (~650) │
  └──────────────┘       │ nfs3client.c │       │ nfs4client.c (~1,200)│
                         │ (~145 LOC)   │       │ nfs4file.c (~460)    │
                         └──────────────┘       │ nfs4super.c          │
                                                │ nfs4namespace.c      │
                                                │ nfs4idmap.c          │
                                                │ nfs4renewd.c         │
                                                │ nfs42proc.c (~1,680) │
                                                │ nfs42xdr.c (~1,830)  │
                                                │ nfs42xattr.c (~1,070)│
                                                │ nfs40proc.c (~395)   │
                                                │ nfs40client.c (~245) │
                                                └──────────────────────┘
```

The XDR files are the **protocol boundary** — they translate between
the kernel's internal data structures and the NFS wire format. Every
NFS operation (LOOKUP, READ, WRITE, GETATTR, etc.) has an XDR encoder
(for the request) and an XDR decoder (for the reply).

**`nfs4xdr.c`** is the largest single file at ~7,790 LOC because NFSv4
uses **compound operations** — a single RPC can contain multiple
operations (SEQUENCE + PUTFH + READ), and each compound element needs
its own encoder/decoder.

**`nfs4proc.c`** is the largest procedure file at ~10,750 LOC because
NFSv4 has far more operations than v2/v3 (opens with stateids,
delegations, sessions, LAYOUTGET, COPY, etc.).

### 2.4 NFSv4 State Management

NFSv4 is a **stateful** protocol (unlike v2/v3 which are stateless). The
server tracks which files clients have open, what locks they hold, and
what delegations they've been granted. This state must survive server
reboots and network partitions.

| File | LOC | Purpose |
|------|-----|---------|
| `nfs4state.c` | 2,200 | Client-side state machine: open recovery, lock recovery, lease renewal |
| `nfs4session.c` | 520 | Session slot tables: tracks in-flight RPCs, enforces exactly-once semantics |
| `nfs4renewd.c` | ~160 | Lease renewal daemon: sends RENEW/SEQUENCE to prevent server from reclaiming state |
| `nfs4idmap.c` | 615 | UID/GID ↔ name mapping: NFSv4 uses string names ("user@domain"), not numeric IDs |

```
  NFSv4 State Machine (client side)
  ─────────────────────────────────

  Normal operation:
  ┌─────────┐    OPEN     ┌──────────┐    LOCK    ┌──────────┐
  │  Idle   │ ──────────► │  Open    │ ─────────► │  Locked  │
  └─────────┘             │ (stateid)│            │ (lock    │
                          └──────────┘            │  stateid)│
                               │                  └──────────┘
                               │ Delegation
                               ▼
                          ┌──────────┐
                          │Delegated │  Client can read/write
                          │          │  without contacting server
                          └──────────┘

  Server reboots → client must recover state:
  ┌──────────────────────────────────────────────────────────┐
  │  nfs4state.c: state recovery loop                        │
  │                                                          │
  │  1. Detect server reboot (NFS4ERR_STALE_CLIENTID)        │
  │  2. nfs4_discover_server_trunking() — re-establish       │
  │  3. nfs4_reclaim_open_state() — re-open all files        │
  │  4. nfs4_reclaim_lock_state() — re-acquire all locks     │
  │  5. Resume normal I/O                                    │
  └──────────────────────────────────────────────────────────┘
```

### 2.5 Delegation & Callback

NFS delegations allow the server to grant a client exclusive (or read)
access to a file, eliminating the need for round-trips on every
operation. The callback channel allows the server to revoke delegations
when another client needs access.

| File | LOC | Purpose |
|------|-----|---------|
| `delegation.c` | 1,700 | Delegation lifecycle: grant, return, conflict resolution |
| `callback.c` | 265 | Callback RPC service setup (server → client channel) |
| `callback_proc.c` | 620 | Callback handlers: `CB_RECALL`, `CB_LAYOUTRECALL`, `CB_NOTIFY` |
| `callback_xdr.c` | 865 | Callback XDR encoding/decoding |

```
  Delegation flow:
  ────────────────

  Client                          Server
    │                               │
    │  OPEN(file) ─────────────────►│
    │◄── reply + delegation ────────│  Server grants read/write delegation
    │                               │
    │  READ/WRITE (local, no RPC)   │  Client operates from cache
    │  GETATTR (local, no RPC)      │  No server round-trips needed
    │                               │
    │  ← CB_RECALL(delegation) ─────│  Another client wants the file
    │                               │
    │  flush dirty data ────────────►│
    │  DELEGRETURN ─────────────────►│  Client returns delegation
    │                               │
    │  (back to normal RPC per op)  │
```

### 2.6 pNFS Layout Drivers

pNFS (Parallel NFS) allows clients to do data I/O directly to storage
devices, bypassing the MDS. Each layout type has its own driver in a
subdirectory.

```
  fs/nfs/
  ├── pnfs.c (~3,540 LOC)              Core pNFS framework
  │   pnfs.h (~930 LOC)                  Layout driver API
  │   pnfs_dev.c (~380 LOC)              Device ID management
  │   pnfs_nfs.c (~1,230 LOC)              NFS-specific pNFS helpers
  │
  ├── filelayout/                      Files Layout (ID=1)
  │   ├── filelayout.c (~1,150 LOC)         NFSv4.1 I/O to data servers
  │   ├── filelayout.h                   Stripe computation
  │   └── filelayoutdev.c (~305 LOC)      Device resolution
  │
  ├── flexfilelayout/                  Flex Files Layout (ID=4)
  │   ├── flexfilelayout.c (~3,040 LOC)   NFSv3/v4 I/O, mirroring, stats
  │   ├── flexfilelayout.h                Mirror structures
  │   └── flexfilelayoutdev.c (~640 LOC)  Multi-version device resolution
  │
  └── blocklayout/                     Block + SCSI Layouts (ID=3,5)
      ├── blocklayout.c (~1,050 LOC)        Block I/O via submit_bio()
      ├── blocklayout.h                   Extent tree, device structures
      ├── dev.c (~590 LOC)                 Block device open/close, SCSI PR
      ├── extent_tree.c (~730 LOC)         Red-black tree: file offset → LBA
      └── rpc_pipefs.c (~270 LOC)          blkmapd communication
```

The **Flex Files driver** is the largest (2,460 LOC in the main file)
because it handles mirroring (write to all mirrors, read load-balancing),
multiple DS protocol versions, I/O statistics tracking (`LAYOUTSTATS`),
and error reporting (`LAYOUTERROR`).

The core framework (`pnfs.c`) manages layout lifecycle: `LAYOUTGET`,
caching, recall (`CB_LAYOUTRECALL`), and `LAYOUTRETURN`. Layout drivers
register via `pnfs_register_layoutdriver()` and provide callbacks for
I/O (`read_pagelist`, `write_pagelist`) and device management.

## 3. fs/nfsd/ — The NFS Server (knfsd)

The kernel NFS server (knfsd) exports local filesystems to NFS clients.
It runs as kernel threads that process incoming RPC requests.

### 3.1 Core Server Operations

```
  Incoming NFS RPC request
      │
      ▼
  net/sunrpc/svc.c: svc_process()     RPC layer decodes, dispatches

  │  fs/nfsd/ Core Files                              │
  │                                                  │
  │  nfssvc.c (~1,080 LOC)                              │

  │                                                  │
```

Each NFS version has its own procedure handler file that implements the

| File | LOC | Operations |

### 3.3 NFSv4 State Management (Server Side)

The server's state management is the most complex subsystem, centered on
a single massive file:

**`nfs4state.c` (~9,560 LOC)** — the largest file in the entire NFS
codebase. It tracks:

- **Client sessions** — which clients are connected, their session slots
- **Open state** — which files are opened by which clients, with what
  access modes
- **Lock state** — byte-range locks held by clients
- **Delegations** — which clients hold read/write delegations
- **Lease management** — server leases grant clients state for a time
  window; if a client doesn't renew, the server reclaims its state

```
  Server state hierarchy:
  ───────────────────────

  struct nfs4_client          (per client connection)
      │
      ├── struct nfsd4_session[]  (NFSv4.1+ sessions)
      │       └── slot_table      (exactly-once semantics)
      │
      ├── struct nfs4_openowner[] (per open-owner, maps to process)
      │       └── struct nfs4_ol_stateid[]  (per-file open state)
      │               └── struct nfs4_lockowner[]
      │                       └── struct nfs4_ol_stateid[] (lock state)
      │
      └── struct nfs4_delegation[]  (delegated files)
```

Supporting files:

| File | LOC | Purpose |
|------|-----|---------|
| `nfs4recover.c` | 1,480 | State recovery after server restart — persists client state to disk so it can be reclaimed during grace period |
| `nfs4callback.c` | 1,460 | Server → client callback invocation: `CB_RECALL`, `CB_LAYOUTRECALL` |
| `nfs4layouts.c` | ~970 | pNFS layout grants, recalls, and lifecycle on the server side |

### 3.4 ACL Support

| File | LOC | Purpose |
|------|-----|---------|
| `nfs4acl.c` | ~880 | NFSv4 ACL ↔ POSIX ACL translation |
| `nfs2acl.c` | ~390 | NFSv2 NFSACL sideband protocol |
| `nfs3acl.c` | ~280 | NFSv3 NFSACL sideband protocol |

NFSv4 ACLs are richer than POSIX ACLs (they have DENY entries, inheritance
flags, etc.). `nfs4acl.c` translates between the two models since Linux
filesystems use POSIX ACLs internally.

### 3.5 pNFS Server Support

The server side of pNFS is relatively thin — it manages layout grants
and coordinates with the underlying filesystem to produce layouts.

| File | LOC | Purpose |
|------|-----|---------|
| `nfs4layouts.c` | ~970 | Core: layout stateid tracking, `LAYOUTGET`/`LAYOUTRETURN` processing, `CB_LAYOUTRECALL` |
| `blocklayout.c` | ~530 | Block layout: calls `s_export_op->map_blocks()` to map files to block extents |
| `blocklayoutxdr.c` | ~295 | Block layout XDR encoding |
| `flexfilelayout.c` | ~144 | Flex Files layout: minimal server-side support (most logic is client-side) |

Note: the Flex Files server-side code is tiny (115 LOC) because Flex Files
is designed so that the data servers are unmodified NFS servers — the MDS
just tells the client where to find the data. The client does all the
heavy lifting.

## 4. fs/nfs_common/ — Shared Code

Small but critical — code used by both the NFS client and server.

| File | LOC | Purpose |
|------|-----|---------|
| `common.c` | 135 | `nfs_stat_to_errno()` — translates NFS status codes to Linux errno values |
| `grace.c` | 125 | Grace period management — during server restart, only reclaim operations are allowed |
| `nfsacl.c` | 400 | ACL XDR encoding/decoding for the NFSACL sideband protocol (v2/v3) |
| `nfslocalio.c` | 800 | Local I/O bypass — when the NFS client and server are in the same kernel, skip the network stack entirely |
| `nfs_ssc.c` | 200 | Server-side copy helpers — allows the NFS server to act as a copy source for inter-server COPY operations |

**Local I/O (`nfslocalio.c`)** is a notable optimization: if the NFS client
detects that the server is running on the same machine, it can bypass the
entire network stack and RPC layer, performing I/O directly through the
VFS. This avoids the serialization/deserialization overhead and network
round-trip.

## 5. net/sunrpc/ — The RPC Transport Layer

NFS doesn't use sockets directly — it uses Sun RPC (ONC RPC), which
provides procedure-call semantics, XDR marshaling, and authentication.
The entire RPC layer lives in `net/sunrpc/`.

### 5.1 Core RPC

| File | LOC | Purpose |
|------|-----|---------|
| `clnt.c` | — | RPC client: creates calls, waits for replies, handles retransmit |
| `svc.c` | — | RPC server: receives calls, dispatches to procedures, sends replies |
| `sched.c` | — | RPC task scheduler: manages async RPC operations, sleep/wake |
| `xdr.c` | — | XDR encoding/decoding primitives |
| `xprtsock.c` | — | TCP and UDP transport |
| `auth.c` | — | Authentication framework (AUTH_SYS, RPCSEC_GSS) |
| `cache.c` | — | Kernel RPC cache (for export lookups, id mapping) |
| `stats.c` | — | `/proc/net/rpc/` statistics |

```
  RPC Call Flow (client side):
  ────────────────────────────

  NFS client calls nfs4_proc_read()
      │
      ▼
  rpc_call_sync() or rpc_call_async()     (clnt.c)
      │
      ├── XDR encode request               (xdr.c + nfs4xdr.c)
      ├── Select transport                  (xprt.c)
      │       ├── TCP socket                (xprtsock.c)
      │       ├── RDMA                      (xprtrdma/)
      │       └── Local                     (xprtlocal.c)
      ├── Send on wire
      ├── Wait for reply (or async callback)
      ├── XDR decode reply                  (xdr.c + nfs4xdr.c)
      └── Return result to NFS client
```

### 5.2 RDMA Transport (xprtrdma/)

NFS-over-RDMA (RFC 8267) provides high-performance transport using
InfiniBand or RoCE. The `xprtrdma/` subdirectory implements both client
and server RDMA transports:

| File | Purpose |
|------|---------|
| `rpc_rdma.c` | RDMA-specific RPC marshaling |
| `transport.c` | Client RDMA transport |
| `svc_rdma_transport.c` | Server RDMA transport |
| `verbs.c` | RDMA verbs (connection management, memory registration) |
| `frwr_ops.c` | Fast Registration Work Requests (FRWR) for memory registration |

### 5.3 GSSAPI / Kerberos Authentication (auth_gss/)

The `auth_gss/` subdirectory implements RPCSEC_GSS (RFC 2203), which
provides Kerberos-based authentication and optional data integrity /
privacy (encryption) for NFS:

| File | Purpose |
|------|---------|
| `auth_gss.c` | GSSAPI authentication mechanism |
| `gss_krb5_mech.c` | Kerberos 5 mechanism |
| `gss_krb5_crypto.c` | Kerberos encryption/integrity |
| `svcauth_gss.c` | Server-side GSS authentication |

There are three security levels:
- `krb5` — authentication only (verify client identity)
- `krb5i` — authentication + integrity (detect tampering)
- `krb5p` — authentication + integrity + privacy (encrypt data)

## 7. How It All Fits Together

```
  Application: read("/mnt/nfs/file.txt", buf, 4096)
      │
      ▼
  VFS: calls nfs_file_read()                        [fs/nfs/file.c]
      │
      ▼
  Page cache miss: nfs_readahead()                   [fs/nfs/read.c]
      │
      ├── pNFS? → layout driver (flexfilelayout.c)   [fs/nfs/flexfilelayout/]
      │   └── Direct I/O to data server
      │
      └── No pNFS → standard NFS read
          │
          ▼
      nfs4_proc_read()                               [fs/nfs/nfs4proc.c]
          │
          ▼
      XDR encode: SEQUENCE + PUTFH + READ            [fs/nfs/nfs4xdr.c]
          │
          ▼
      rpc_call_async()                               [net/sunrpc/clnt.c]
          │
          ├── auth: krb5 sign/encrypt                [net/sunrpc/auth_gss/]
          │
          ├── transport: TCP send                    [net/sunrpc/xprtsock.c]
          │   (or RDMA send)                         [net/sunrpc/xprtrdma/]
          │
          ▼  ════════════ network ════════════
          │
      svc_process()                                  [net/sunrpc/svc.c]
          │
          ▼
      XDR decode request                             [fs/nfsd/nfs4xdr.c]
          │
          ▼
      nfsd4_read()                                   [fs/nfsd/nfs4proc.c]
          │
          ▼
      nfsd_read() → vfs_read()                       [fs/nfsd/vfs.c]
          │
          ▼
      Local filesystem (ext4, XFS, etc.)
          │
          ▼  ════════════ reply ════════════
          │
      XDR encode reply + data                        [fs/nfsd/nfs4xdr.c]
          │
          ▼
      Back to client → XDR decode → page cache → app gets data
```

## 8. Code Volume Summary

| Component | Files | ~LOC | Largest File |
|-----------|-------|------|-------------|
| NFS Client (`fs/nfs/`) | 88 | 85,000 | `nfs4proc.c` (~10,750) |
| NFS Server (`fs/nfsd/`) | 56 | 52,000 | `nfs4state.c` (~9,560) |
| NFS Common (`fs/nfs_common/`) | 7 | 1,300 | `nfsacl.c` (~420) |
| RPC Transport (`net/sunrpc/`) | 64 | 51,000 | — |
| Headers (`include/`) | ~60 | ~12,900 | `nfs4trace.h` (~2,870) |
| **Total** | **~275** | **~202,000** | — |

The two largest individual files in the NFS ecosystem tell you where the
complexity lives:

- **`nfs4proc.c`** (~10,750 LOC, client) — NFSv4 has ~60 operations, each
  needing request construction, error handling, state recovery, and
  delegation interaction.

- **`nfs4state.c`** (~9,560 LOC, server) — The server must track every open
  file, lock, and delegation across all clients, handle lease expiry,
  process grace period reclaims, and deal with network partitions. This
  is the heart of NFSv4's stateful design.
