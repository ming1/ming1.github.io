## 1. What Is memfd?

A **memfd** (memory file descriptor) is an anonymous, RAM-backed file created
via the `memfd_create()` syscall. It behaves like a regular file — you can
`read()`, `write()`, `mmap()`, `ftruncate()`, and pass it over unix sockets —
but it lives entirely in memory with no link in any filesystem namespace.

```
memfd_create("my-buffer", MFD_CLOEXEC)
    → returns fd
    → visible as /proc/<pid>/fd/<N> → "memfd:my-buffer"
    → backed by tmpfs (shmem) pages
    → no path on any mounted filesystem
```

The key differentiator from other shared memory mechanisms is **file sealing**:
the ability to irrevocably restrict operations on the file, enabling safe sharing
between mutually untrusted processes.

## 2. Motivation & History

### The Problem with Traditional Shared Memory

| Mechanism | Issue |
|-----------|-------|
| `shm_open("/name")` | Global namespace — any process can open/modify by name |
| `tmpfile()` / `O_TMPFILE` | Requires a mounted filesystem; may be disk-backed; no sealing |
| `mmap(MAP_SHARED\|MAP_ANONYMOUS)` | Only inheritable via `fork()`, cannot pass to unrelated processes |
| `/tmp` files | On-disk (unless tmpfs), namespace pollution, TOCTOU races |

All of these share a fundamental limitation: **no mechanism to guarantee
immutability**. If you pass a file descriptor to another process, nothing
prevents that process (or any other process with access) from modifying the
contents. This is a problem for:

- **Graphics compositors** receiving pixel buffers from untrusted clients
- **IPC frameworks** passing serialized messages between sandboxed processes
- **Container runtimes** sharing configuration or binary data

### memfd Solves This

memfd was introduced in **Linux 3.17 (2014)** by David Herrmann, originally
motivated by the kdbus IPC and graphics buffer sharing use cases. The design
goals were:

1. **Anonymous** — no filesystem path, no namespace pollution
2. **FD-based sharing** — pass via `SCM_RIGHTS` over unix sockets, `fork()`, or `pidfd_getfd()`
3. **Sealing** — irrevocably restrict operations for safe sharing between mutually untrusted parties

## 3. How memfd Works

### Backing Store

memfd is backed by **shmem (tmpfs)** — the same in-memory filesystem that
backs `SYSV` shared memory, `/dev/shm`, and `MAP_ANONYMOUS|MAP_SHARED` pages.
With the `MFD_HUGETLB` flag, it uses **hugetlbfs** instead.

```
                  memfd_create("buf", flags)
                            │
                  flags & MFD_HUGETLB?
                     ┌──────┴──────┐
                     │             │
                    Yes            No
                     │             │
           hugetlb_file_setup()  shmem_file_setup()
                     │             │
                 hugetlbfs       tmpfs
              anonymous inode  anonymous inode
```

Key properties:

- **No disk I/O** — pages live in RAM (swappable to swap space, but never
  written to a filesystem on disk)
- **Page-cache backed** — uses the kernel page cache for the shmem address space
- **Swappable** — under memory pressure, shmem pages can be swapped out just
  like anonymous memory
- **Name is cosmetic** — the string passed to `memfd_create()` appears only in
  `/proc/<pid>/fd/` as `memfd:<name>` for debugging; it has no uniqueness
  requirements and does not create any filesystem entry

The entire memfd implementation lives in `mm/memfd.c` (~524 lines), with seal
enforcement spread across `mm/shmem.c` and mmap checks in `memfd_check_seals_mmap()`.

## 4. API Reference

### Syscall

```c
#include <sys/mman.h>

int memfd_create(const char *name, unsigned int flags);
```

Returns a file descriptor on success, -1 on error. The file starts with size 0;
use `ftruncate()` to set the size before `mmap()`.

### Flags

```c
/* include/uapi/linux/memfd.h */

#define MFD_CLOEXEC         0x0001U
#define MFD_ALLOW_SEALING   0x0002U
#define MFD_HUGETLB         0x0004U
#define MFD_NOEXEC_SEAL     0x0008U
#define MFD_EXEC            0x0010U
```

| Flag | Value | Description |
|------|-------|-------------|
| `MFD_CLOEXEC` | 0x0001 | Set close-on-exec (`O_CLOEXEC`) on the fd |
| `MFD_ALLOW_SEALING` | 0x0002 | Allow `F_ADD_SEALS` / `F_GET_SEALS` via `fcntl()` |
| `MFD_HUGETLB` | 0x0004 | Use hugetlbfs backing instead of tmpfs |
| `MFD_NOEXEC_SEAL` | 0x0008 | Create non-executable; seal exec bits (`F_SEAL_EXEC`). Implies `MFD_ALLOW_SEALING` |
| `MFD_EXEC` | 0x0010 | Create with execute permission. Mutually exclusive with `MFD_NOEXEC_SEAL` |
| `MFD_HUGE_*` | varies | Huge page size encoding (OR'd with `MFD_HUGETLB`). E.g., `MFD_HUGE_2MB`, `MFD_HUGE_1GB` |

Notes:
- `MFD_EXEC` and `MFD_NOEXEC_SEAL` are mutually exclusive — setting both returns `-EINVAL`
- If neither `MFD_EXEC` nor `MFD_NOEXEC_SEAL` is set, the behavior depends on the `vm.memfd_noexec` sysctl (see Section 8)
- Without `MFD_ALLOW_SEALING` (and without `MFD_NOEXEC_SEAL`), seals default to `F_SEAL_SEAL` — preventing any seal changes

### Available Huge Page Sizes

```c
/* include/uapi/linux/memfd.h */

#define MFD_HUGE_64KB    /* ... */
#define MFD_HUGE_512KB
#define MFD_HUGE_1MB
#define MFD_HUGE_2MB
#define MFD_HUGE_8MB
#define MFD_HUGE_16MB
#define MFD_HUGE_32MB
#define MFD_HUGE_256MB
#define MFD_HUGE_512MB
#define MFD_HUGE_1GB
#define MFD_HUGE_2GB
#define MFD_HUGE_16GB
```

Usage: `memfd_create("buf", MFD_HUGETLB | MFD_HUGE_2MB)`.

## 5. File Sealing

File sealing is memfd's key differentiator. Seals are **per-inode, additive, and
irreversible** restrictions on file operations. Once a seal is set, it can never
be removed — not by the process that set it, not by root, not by anyone.

### Why Sealing Matters: The Security Gap in Unix

Traditional Unix file permissions answer the question "**who** can access this
file?" — owner, group, and others each get read/write/execute bits. But
permissions say nothing about **what operations are permanently forbidden**. Any
process with write permission can modify a shared file at any time, and
permissions can be changed by the owner or root.

This creates a fundamental problem for shared memory: when process A passes a
file descriptor to process B, there is no way for B to **verify** that A (or
anyone else with access) won't modify the contents later. The only safe option
is for B to copy the data into its own private buffer — which defeats the
purpose of zero-copy sharing.

File sealing fills this gap by restricting **what** can be done to a file,
permanently:

| Unix Permissions | File Sealing |
|-----------------|--------------|
| Controls **who** can access | Controls **what** operations are allowed |
| Can be changed by owner/root | **Irreversible** — cannot be removed by anyone |
| Per-user/group granularity | Per-inode, affects all accessors equally |
| No immutability guarantee | Kernel-enforced immutability via `F_SEAL_WRITE` |

The combination is powerful: permissions control access, sealing controls
behavior. A consumer receiving a sealed memfd has a **kernel-enforced guarantee**
about the file's properties — not a promise from the sender, but an invariant
enforced by the kernel itself.

### The Trust Model

Sealing enables safe zero-copy sharing between **mutually untrusted** processes.
Neither side needs to trust the other:

- The **consumer** verifies seals before using the data — if the expected seals
  are present, the kernel guarantees immutability regardless of what the producer
  does afterward
- The **producer** seals the data before sharing — once `F_SEAL_SEAL` is set,
  the consumer cannot add or remove seals either

```
  Producer                              Consumer
  ────────                              ────────
  1. fd = memfd_create("msg",
         MFD_ALLOW_SEALING)
  2. ftruncate(fd, size)
  3. write(fd, data, size)
  4. fcntl(fd, F_ADD_SEALS,
         F_SEAL_SHRINK |
         F_SEAL_GROW |
         F_SEAL_WRITE |
         F_SEAL_SEAL)
  5. sendmsg(fd) ──────────────────────►
                                        6. Receive fd via unix socket
                                        7. seals = fcntl(fd, F_GET_SEALS)
                                        8. Verify: seals has SHRINK|GROW|WRITE
                                           → data is guaranteed immutable
                                        9. ptr = mmap(fd, ...)
                                           → safe to use without copying
```

The consumer at step 8 doesn't need to trust the producer — the seal check is a
query to the kernel, and the kernel enforces the seals regardless of what any
process attempts afterward. If the seals are present, the data **cannot** be
modified. This is what makes zero-copy safe: the consumer can `mmap()` and use
the data directly without copying it first.

Without sealing, the consumer would need to copy the data into its own buffer
to prevent the producer from modifying it after sharing — defeating zero-copy.

### All Seal Types

```c
/* include/uapi/linux/fcntl.h */

#define F_SEAL_SEAL          0x0001  /* prevent further seals */
#define F_SEAL_SHRINK        0x0002  /* prevent file from shrinking */
#define F_SEAL_GROW          0x0004  /* prevent file from growing */
#define F_SEAL_WRITE         0x0008  /* prevent writes */
#define F_SEAL_FUTURE_WRITE  0x0010  /* prevent future writes while mapped */
#define F_SEAL_EXEC          0x0020  /* prevent chmod modifying exec bits */
```

| Seal | Prevents | Blocks |
|------|----------|--------|
| `F_SEAL_SEAL` | Adding any further seals | `fcntl(F_ADD_SEALS)` |
| `F_SEAL_SHRINK` | Reducing file size | `ftruncate()` to smaller size, `fallocate(FALLOC_FL_PUNCH_HOLE)` |
| `F_SEAL_GROW` | Increasing file size | `ftruncate()` to larger size, `write()` beyond EOF, `fallocate()` past EOF |
| `F_SEAL_WRITE` | All writes and modifications | `write()`, `pwrite()`, `mmap(PROT_WRITE, MAP_SHARED)`, `fallocate()` |
| `F_SEAL_FUTURE_WRITE` | New writable access | New `mmap(PROT_WRITE, MAP_SHARED)`, `write()` — but existing writable mmaps continue working |
| `F_SEAL_EXEC` | Changing execute permission | `chmod()` / `fchmod()` that modifies the execute bits |

### F_SEAL_WRITE vs F_SEAL_FUTURE_WRITE

These two write seals have an important behavioral difference:

**`F_SEAL_WRITE`** is the strict seal:
- Blocks all new writable access (`write()`, new writable `mmap()`)
- Waits for all existing GUP (Get User Pages) pins to be released
- Returns `-EBUSY` if pins cannot be released within ~150ms
- After success, no process can modify the file contents

**`F_SEAL_FUTURE_WRITE`** is the cooperative seal:
- Blocks `write()` syscalls and new writable `mmap()` calls
- Does **not** affect existing `MAP_SHARED` writable mappings
- Existing mappings can still write through memory-mapped access
- Use case: Android's `ASharedMemory` — the creator can still write through
  its existing mapping while preventing the receiver from opening new writable
  access

### Using Seals via fcntl

```c
#include <fcntl.h>

/* Add seals (additive — never removes existing seals) */
int ret = fcntl(fd, F_ADD_SEALS,
                F_SEAL_SHRINK | F_SEAL_GROW | F_SEAL_WRITE);

/* Query current seals */
int seals = fcntl(fd, F_GET_SEALS);
if (seals & F_SEAL_WRITE)
    printf("File is write-sealed\n");
```

## 6. Sealing Implementation

### Data Structure

Seals are stored as a bitmask in the filesystem inode's private data:

```c
/* include/linux/shmem_fs.h */
struct shmem_inode_info {
    spinlock_t      lock;
    unsigned int    seals;      /* shmem seals */
    unsigned long   flags;
    unsigned long   alloced;    /* data pages alloced to file */
    unsigned long   swapped;    /* subtotal assigned to swap */
    /* ... */
    struct inode    vfs_inode;
};
```

For hugetlbfs memfds, seals are stored in `hugetlbfs_inode_info.seals`.

### Initialization

```c
/* mm/shmem.c:shmem_get_inode() */
info->seals = F_SEAL_SEAL;   /* default: sealing is sealed (disabled) */
```

By default, a new shmem inode has `F_SEAL_SEAL` set — meaning no seals can be
added. When `MFD_ALLOW_SEALING` or `MFD_NOEXEC_SEAL` is passed, this initial
seal is cleared:

```c
/* mm/memfd.c:memfd_alloc_file() */
if (flags & MFD_NOEXEC_SEAL) {
    inode->i_mode &= ~0111;             /* remove execute bits */
    *file_seals &= ~F_SEAL_SEAL;        /* allow sealing */
    *file_seals |= F_SEAL_EXEC;         /* seal exec bits */
} else if (flags & MFD_ALLOW_SEALING) {
    *file_seals &= ~F_SEAL_SEAL;        /* allow sealing */
}
```

### Seal Enforcement Points

Seals are checked at multiple points throughout the VFS and shmem layers:

```c
/* mm/shmem.c:shmem_write_begin() — blocks write() syscall */
if (info->seals & (F_SEAL_WRITE | F_SEAL_FUTURE_WRITE))
    return -EPERM;
if ((info->seals & F_SEAL_GROW) && pos + len > inode->i_size)
    return -EPERM;

/* mm/shmem.c:shmem_setattr() — blocks ftruncate() */
if ((newsize < oldsize && (info->seals & F_SEAL_SHRINK)) ||
    (newsize > oldsize && (info->seals & F_SEAL_GROW)))
    return -EPERM;

/* mm/shmem.c:shmem_setattr() — blocks chmod on exec bits */
if ((info->seals & F_SEAL_EXEC) &&
    ((inode->i_mode ^ attr->ia_mode) & 0111))
    return -EPERM;

/* mm/shmem.c:shmem_fallocate() — blocks fallocate */
if (info->seals & (F_SEAL_WRITE | F_SEAL_FUTURE_WRITE))
    return -EPERM;

/* mm/memfd.c:memfd_check_seals_mmap() — blocks writable mmap */
if (seals & (F_SEAL_WRITE | F_SEAL_FUTURE_WRITE)) {
    if (vm_flags & VM_SHARED && vm_flags & VM_WRITE)
        return -EPERM;
    *vm_flags_ptr &= ~VM_MAYWRITE;  /* prevent mprotect upgrade */
}
```

### The F_SEAL_WRITE Challenge: Waiting for GUP Pins

Setting `F_SEAL_WRITE` is more complex than other seals because drivers may
hold references to folios via `get_user_pages()` (GUP) for pending I/O, even
without any user-space writable mapping.

The kernel must wait for these pins to be released:

```c
/* mm/memfd.c:memfd_add_seals() */
if ((seals & F_SEAL_WRITE) && !(*file_seals & F_SEAL_WRITE)) {
    error = mapping_deny_writable(file->f_mapping);
    if (error)
        goto unlock;

    error = memfd_wait_for_pins(file->f_mapping);
    if (error) {
        mapping_allow_writable(file->f_mapping);
        goto unlock;
    }
}
```

`memfd_wait_for_pins()` works in multiple scan passes:

```
                    memfd_wait_for_pins()
                           │
              ┌────────────┴────────────┐
              │  memfd_tag_pins()       │
              │  Tag all folios with    │
              │  elevated refcount      │
              │  using MEMFD_TAG_PINNED │
              └────────────┬────────────┘
                           │
              ┌────────────┴────────────┐
              │  Scan loop (up to 5x)   │
              │  ~150ms total           │
              │                         │
              │  Each pass:             │
              │   - Check tagged folios │
              │   - Clear tag if refs   │
              │     dropped to expected │
              │   - Exponential backoff │
              │     between scans       │
              └────────────┬────────────┘
                           │
                ┌──────────┴──────────┐
                │                     │
           All cleared          Still pinned
                │                     │
            return 0            return -EBUSY
```

The `MEMFD_TAG_PINNED` tag reuses `PAGECACHE_TAG_TOWRITE` — safe because tmpfs
and hugetlbfs never perform writeback and thus never set this tag.

## 7. Hugetlb memfd

### Overview

The `MFD_HUGETLB` flag creates a memfd backed by hugetlbfs instead of tmpfs.
This gives the benefits of huge pages (fewer TLB misses, contiguous physical
memory) combined with memfd's anonymous nature and sealing capabilities.

```c
/* 2MB huge pages */
int fd = memfd_create("huge", MFD_HUGETLB | MFD_HUGE_2MB);
ftruncate(fd, 2 * 1024 * 1024);
void *p = mmap(NULL, 2 * 1024 * 1024, PROT_READ | PROT_WRITE,
               MAP_SHARED, fd, 0);

/* 1GB huge pages */
int fd = memfd_create("giga", MFD_HUGETLB | MFD_HUGE_1GB);
```

### Use Cases

- **KVM guest memory** — QEMU uses hugetlb memfd for VM RAM, passed via
  fd to the KVM subsystem
- **DPDK** — large contiguous DMA buffers for network packet processing
- **DMA buffer sharing** — `udmabuf` can wrap hugetlb memfd pages into
  a dma-buf for GPU/accelerator access
- **Database shared buffers** — large shared memory regions with minimal TLB
  overhead

### Folio Allocation

When `memfd_pin_folios()` (in `mm/gup.c`) encounters a page that hasn't been
faulted in yet, it calls `memfd_alloc_folio()`:

```c
/* mm/memfd.c:memfd_alloc_folio() */
struct folio *memfd_alloc_folio(struct file *memfd, pgoff_t idx)
{
    if (is_file_hugepages(memfd)) {
        /* Allocate from hugetlb pool:
         * 1. Reserve huge page
         * 2. Allocate folio (non-HIGHMEM, non-MOVABLE for DMA)
         * 3. Zero the folio to prevent information leaks
         * 4. Add to page cache under hugetlb fault mutex
         */
        /* ... */
    }
    /* Regular shmem: */
    return shmem_read_folio(memfd->f_mapping, idx);
}
```

## 8. Noexec Security (MFD_NOEXEC_SEAL)

### The Problem: Confused Deputy Attack

Historically, memfds were always created with execute permission. This enables
a class of attacks where a privileged process creates a memfd for data sharing,
but an attacker overwrites its contents and executes it:

```
Privileged process              Attacker
──────────────────              ────────
1. fd = memfd_create("buf", 0)
   → has execute bit set
2. write(fd, data, len)
3. sendmsg(fd) ────────────────► 4. Receive fd
                                 5. ftruncate(fd, 0)
                                 6. write(fd, exploit_code, len)
                                 7. fexecve(fd, ...)
                                    → executes attacker's code
                                      with elevated privileges
```

This was documented in multiple ChromeOS vulnerability reports (VRP bug
[crbug.com/1305267](https://crbug.com/1305267)).

### Solution: MFD_NOEXEC_SEAL

`MFD_NOEXEC_SEAL` (Linux 6.3, 2023) creates the memfd without execute
permission and seals the exec bits so they cannot be re-added:

```c
/* mm/memfd.c:memfd_alloc_file() */
if (flags & MFD_NOEXEC_SEAL) {
    inode->i_mode &= ~0111;      /* clear all execute bits */
    *file_seals &= ~F_SEAL_SEAL; /* allow adding more seals */
    *file_seals |= F_SEAL_EXEC;  /* prevent changing exec bits */
}
```

Note: `MFD_NOEXEC_SEAL` clears execute bits *before* setting `F_SEAL_EXEC`, so
the file is non-executable with its exec bits sealed. It does **not**
automatically add write seals.

### W^X Enforcement for Executable memfds

The W^X (write XOR execute) enforcement applies when `F_SEAL_EXEC` is added
via `fcntl(F_ADD_SEALS)` to an **already-executable** memfd (one created with
`MFD_EXEC | MFD_ALLOW_SEALING`). In that case, sealing the exec bits also
implies all write seals — ensuring the executable content cannot be modified:

```c
/* mm/memfd.c:memfd_add_seals() — only fires when exec bits are set */
if (seals & F_SEAL_EXEC && inode->i_mode & 0111)
    seals |= F_SEAL_SHRINK | F_SEAL_GROW |
             F_SEAL_WRITE  | F_SEAL_FUTURE_WRITE;
```

This is how `runc` uses memfd: it creates an executable memfd with
`MFD_EXEC | MFD_ALLOW_SEALING`, copies a binary into it, seals it with
`F_SEAL_EXEC | F_SEAL_SEAL` (which triggers W^X, adding all write seals),
and then calls `fexecve()` — guaranteeing the binary cannot be tampered with
between copy and exec.

### vm.memfd_noexec Sysctl

A per-pid-namespace sysctl controls the default behavior when neither
`MFD_EXEC` nor `MFD_NOEXEC_SEAL` is specified:

| Value | Name | Behavior |
|-------|------|----------|
| 0 | `MEMFD_NOEXEC_SCOPE_EXEC` | Default. Acts as if `MFD_EXEC` was set (executable, for backward compatibility) |
| 1 | `MEMFD_NOEXEC_SCOPE_NOEXEC_SEAL` | Acts as if `MFD_NOEXEC_SEAL` was set (non-executable by default) |
| 2 | `MEMFD_NOEXEC_SCOPE_NOEXEC_ENFORCED` | Rejects `memfd_create()` unless `MFD_NOEXEC_SEAL` is explicitly set |

The setting is **hierarchical**: child pid namespaces inherit the parent's value
at creation time, and the kernel enforces the **most restrictive** setting in the
namespace hierarchy.

```c
/* mm/memfd.c:check_sysctl_memfd_noexec() */
struct pid_namespace *ns = task_active_pid_ns(current);
int sysctl = pidns_memfd_noexec_scope(ns);

if (!(*flags & (MFD_EXEC | MFD_NOEXEC_SEAL))) {
    if (sysctl >= MEMFD_NOEXEC_SCOPE_NOEXEC_SEAL)
        *flags |= MFD_NOEXEC_SEAL;
    else
        *flags |= MFD_EXEC;
}

if (!(*flags & MFD_NOEXEC_SEAL) &&
    sysctl >= MEMFD_NOEXEC_SCOPE_NOEXEC_ENFORCED)
    return -EACCES;
```

This allows gradual migration: containers can set `vm.memfd_noexec=1` to make
old software default to non-executable memfds, while new software can still
explicitly request `MFD_EXEC` when needed.

## 9. Use Cases & Examples

### 9.1 Basic: Create, Write, mmap

```c
#include <stdio.h>
#include <string.h>
#include <sys/mman.h>
#include <unistd.h>

int main(void)
{
    /* Create memfd */
    int fd = memfd_create("example", MFD_CLOEXEC);

    /* Set size */
    ftruncate(fd, 4096);

    /* Write data */
    const char *msg = "Hello from memfd!";
    write(fd, msg, strlen(msg));

    /* Memory-map */
    char *ptr = mmap(NULL, 4096, PROT_READ, MAP_SHARED, fd, 0);
    printf("Read: %s\n", ptr);

    munmap(ptr, 4096);
    close(fd);
    return 0;
}
```

### 9.2 Share via Unix Socket

```c
/* Producer */
int fd = memfd_create("shared-data", MFD_CLOEXEC | MFD_ALLOW_SEALING);
ftruncate(fd, size);
void *ptr = mmap(NULL, size, PROT_READ | PROT_WRITE, MAP_SHARED, fd, 0);
memcpy(ptr, data, size);
munmap(ptr, size);

/* Seal: immutable from this point */
fcntl(fd, F_ADD_SEALS,
      F_SEAL_SHRINK | F_SEAL_GROW | F_SEAL_WRITE | F_SEAL_SEAL);

/* Send fd via SCM_RIGHTS */
struct msghdr msg = { ... };
struct cmsghdr *cmsg = CMSG_FIRSTHDR(&msg);
cmsg->cmsg_level = SOL_SOCKET;
cmsg->cmsg_type = SCM_RIGHTS;
memcpy(CMSG_DATA(cmsg), &fd, sizeof(fd));
sendmsg(sock, &msg, 0);
```

```c
/* Consumer */
int fd = /* receive via recvmsg SCM_RIGHTS */;

/* Verify seals before trusting the data */
int seals = fcntl(fd, F_GET_SEALS);
if (!(seals & F_SEAL_WRITE) || !(seals & F_SEAL_SHRINK)) {
    fprintf(stderr, "Refusing unsealed memfd\n");
    close(fd);
    return -1;
}

/* Safe to mmap — contents are guaranteed immutable */
struct stat st;
fstat(fd, &st);
void *ptr = mmap(NULL, st.st_size, PROT_READ, MAP_SHARED, fd, 0);
/* Use data directly, no copy needed */
```

### 9.3 Hugetlb memfd

```c
#include <sys/mman.h>

int fd = memfd_create("huge-buf",
                      MFD_HUGETLB | MFD_HUGE_2MB | MFD_CLOEXEC);
if (fd < 0) {
    perror("memfd_create");
    return 1;
}

size_t size = 2 * 1024 * 1024;  /* must be multiple of huge page size */
ftruncate(fd, size);

void *ptr = mmap(NULL, size, PROT_READ | PROT_WRITE,
                 MAP_SHARED | MAP_POPULATE, fd, 0);
if (ptr == MAP_FAILED) {
    perror("mmap");
    return 1;
}

/* Use ptr — backed by a single 2MB physical page */
memset(ptr, 0x42, size);

munmap(ptr, size);
close(fd);
```

### 9.4 Zero-Copy I/O with ublk SHMEM_ZC

memfd combined with `O_DIRECT` enables true zero-copy I/O. For example, the
ublk (userspace block device) shared memory zero-copy pattern:

```c
/* ublk server */
int memfd = memfd_create("ublk_buf", MFD_ALLOW_SEALING);
ftruncate(memfd, buf_size);
void *server_va = mmap(NULL, buf_size, PROT_READ | PROT_WRITE,
                       MAP_SHARED | MAP_POPULATE, memfd, 0);

/* Register buffer VA with ublk kernel driver */
ublk_ctrl_cmd(UBLK_U_CMD_REG_BUF, server_va, buf_size);

/* Share memfd with client via SCM_RIGHTS */
sendmsg(client_sock, &msg_with_memfd, 0);
```

```c
/* Client application */
int memfd = /* received via recvmsg */;
void *client_va = mmap(NULL, buf_size, PROT_READ | PROT_WRITE,
                       MAP_SHARED, memfd, 0);

/* O_DIRECT write using the shared buffer */
int blkfd = open("/dev/ublkb0", O_RDWR | O_DIRECT);
write(blkfd, client_va, 4096);
/* Kernel matches PFNs → zero-copy, no data movement */
```

Both processes map the same physical pages. When the client does O_DIRECT I/O,
the kernel bio contains the exact same page frames, enabling PFN matching
without any data copy.

## 10. memfd vs Alternatives

| Feature | `memfd_create()` | `shm_open()` | `tmpfile()` | `mmap(MAP_SHARED\|MAP_ANON)` |
|---------|-----------------|-------------|------------|---------------------------|
| Anonymous (no path) | Yes | No (named in `/dev/shm`) | No (unlinked but was on FS) | Yes |
| Cross-process sharing | Via fd passing | By name | Via fd passing | Only via `fork()` |
| File sealing | Yes | No | No | No |
| Hugetlb support | `MFD_HUGETLB` | No | No | `MAP_HUGETLB` (but no fd) |
| Noexec control | `MFD_NOEXEC_SEAL` | No | No | No |
| `read()`/`write()` | Yes | Yes | Yes | No (no fd) |
| `mmap()` | Yes | Yes | Yes | Already mapped |
| Survives process death | Fd refcount | Name persists | No | No |
| Backed by | shmem/hugetlbfs | shmem | Filesystem | Anonymous pages |

**When to use memfd:**
- Sharing data between processes where immutability guarantees are needed (sealing)
- Creating anonymous shared memory that can be passed as a file descriptor
- Allocating huge-page-backed buffers without a hugetlbfs mount
- Userspace block device zero-copy I/O (ublk shmem_zc)
- Wrapping into dma-buf via udmabuf for GPU/accelerator access

### Syscall Entry Point

```c
/* mm/memfd.c */
SYSCALL_DEFINE2(memfd_create,
        const char __user *, uname,
        unsigned int, flags)
{
    error = sanitize_flags(&flags);     /* validate flags, apply noexec sysctl */
    name = alloc_name(uname);           /* prepend "memfd:" prefix */
    fd_flags = (flags & MFD_CLOEXEC) ? O_CLOEXEC : 0;
    return FD_ADD(fd_flags, memfd_alloc_file(name, flags));
}
```

### memfd_alloc_file() — File Creation

```c
/* mm/memfd.c */
struct file *memfd_alloc_file(const char *name, unsigned int flags)
{
    if (flags & MFD_HUGETLB)
        file = hugetlb_file_setup(name, 0, mk_vma_flags(VMA_NORESERVE_BIT), ...);
    else
        file = shmem_file_setup(name, 0, mk_vma_flags(VMA_NORESERVE_BIT));

    /* LSM security check */
    security_inode_init_security_anon(inode, ...);

    /* Enable seeking, reading, writing, and large file support */
    file->f_mode |= FMODE_LSEEK | FMODE_PREAD | FMODE_PWRITE;
    file->f_flags |= O_LARGEFILE;

    /* Configure initial seals based on flags */
    if (flags & MFD_NOEXEC_SEAL) {
        inode->i_mode &= ~0111;          /* strip execute */
        *file_seals &= ~F_SEAL_SEAL;     /* allow sealing */
        *file_seals |= F_SEAL_EXEC;      /* lock exec bits */
    } else if (flags & MFD_ALLOW_SEALING) {
        *file_seals &= ~F_SEAL_SEAL;     /* allow sealing */
    }
    return file;
}
```

### memfd_pin_folios() — Driver Folio Pinning

```c
/* mm/gup.c */
long memfd_pin_folios(struct file *memfd, loff_t start, loff_t end,
                      struct folio **folios, unsigned int max_folios,
                      pgoff_t *offset)
{
    /* Only works for shmem or hugetlbfs files */
    if (!shmem_file(memfd) && !is_file_hugepages(memfd))
        return -EINVAL;

    do {
        /* Look up folios in page cache */
        nr_found = filemap_get_folios_contig(memfd->f_mapping, ...);

        /* Pin each folio with FOLL_PIN semantics */
        for (i = 0; i < nr_found; i++) {
            try_grab_folio(folio, 1, FOLL_PIN);
            folios[nr_folios++] = folio;
        }

        /* If page not in cache, allocate via memfd_alloc_folio() */
        if (!nr_found) {
            folio = memfd_alloc_folio(memfd, start_idx);
        }

        /* Migrate movable folios if needed for FOLL_LONGTERM */
        ret = check_and_migrate_movable_folios(nr_folios, folios);
    } while (ret == -EAGAIN);

    return nr_folios;
}
EXPORT_SYMBOL_GPL(memfd_pin_folios);
```

This function is used by drivers like `udmabuf` to pin memfd folios for DMA
access, creating a bridge between memfd and the DMA-BUF subsystem.

## 12. Summary

| Property | Detail |
|----------|--------|
| **What** | Anonymous RAM-backed file, no filesystem path |
| **Backing** | shmem (tmpfs) or hugetlbfs |
| **Created by** | `memfd_create(name, flags)` syscall |
| **Key feature** | File sealing — irrevocable, additive restrictions |
| **Seals** | `F_SEAL_SEAL`, `SHRINK`, `GROW`, `WRITE`, `FUTURE_WRITE`, `EXEC` |
| **Security** | `MFD_NOEXEC_SEAL` prevents confused deputy attacks; `vm.memfd_noexec` sysctl for gradual migration |
| **Sharing** | Pass fd via `SCM_RIGHTS`, `fork()`, or `pidfd_getfd()` |
| **Huge pages** | `MFD_HUGETLB \| MFD_HUGE_2MB` (or `1GB`, etc.) |
| **Driver use** | `memfd_pin_folios()` for DMA; `udmabuf` wraps memfd into dma-buf |
| **Zero-copy** | Combined with `O_DIRECT` for ublk shmem_zc pattern |
| **Introduced** | Linux 3.17 (2014), by David Herrmann |
| **Source** | `mm/memfd.c` (~524 lines) |
