## 1. Top View & Architecture

### 1.1 What Are BPF Open-Coded Iterators?

BPF open-coded iterators are a **kfunc-based looping mechanism** that lets
BPF programs iterate over kernel data structures (tasks, cgroups, VMAs,
numbers, etc.) with **verifier-guaranteed termination** and no fixed
instruction count limit.

Think of them as **kernel-provided `for` loops** — the iterator handles
traversal, the BPF program handles per-element logic:

```c
/* Iterate over all tasks in the system */
struct task_struct *task;

bpf_for_each(task, task, NULL, BPF_TASK_ITER_ALL_PROCS) {
    bpf_printk("pid=%d comm=%s", task->pid, task->comm);
    if (task->pid == target_pid)
        break;
}
```

The verifier knows this loop terminates because the iterator's `_next()`
function returns NULL when exhausted.

### 1.2 The KF_ITER Pattern

Every open-coded iterator follows the same three-function pattern:

```
  ┌────────────────────────────────────────────────────────┐
  │                   KF_ITER Pattern                       │
  ├────────────────────────────────────────────────────────┤
  │                                                        │
  │  1. bpf_iter_<type>_new(iter, args...)   [KF_ITER_NEW] │
  │     └─ Initialize iterator on BPF stack                │
  │     └─ May take extra args (filter, range, etc.)       │
  │     └─ Returns 0 on success, negative on error         │
  │                                                        │
  │  2. bpf_iter_<type>_next(iter)          [KF_ITER_NEXT] │
  │     └─ Advance to next element                         │
  │     └─ Returns pointer to element, or NULL when done   │
  │     └─ MUST have exactly 1 argument (the iterator)     │
  │                                                        │
  │  3. bpf_iter_<type>_destroy(iter)    [KF_ITER_DESTROY] │
  │     └─ Clean up iterator state                         │
  │     └─ Returns void                                    │
  │     └─ MUST have exactly 1 argument                    │
  │                                                        │
  └────────────────────────────────────────────────────────┘
```

**Source:** [`include/linux/btf.h:73-75`](https://github.com/torvalds/linux/blob/master/include/linux/btf.h),
[`kernel/bpf/btf.c:8555-8619`](https://github.com/torvalds/linux/blob/master/kernel/bpf/btf.c)

### 1.3 High-Level Data Flow

```
  BPF Program (on BPF stack)          Kernel
  ──────────────────────────          ──────
  struct bpf_iter_<type> it;
       │
       │ bpf_iter_<type>_new(&it, ...)
       ▼
  ┌──────────┐  kfunc call   ┌──────────────────┐
  │ Iterator │ ─────────────→│ Initialize kernel │
  │ state    │               │ traversal state   │
  │ (opaque) │               │ (inside __opaque) │
  └──────────┘               └──────────────────┘
       │
       │ loop { bpf_iter_<type>_next(&it) }
       ▼
  ┌──────────┐  kfunc call   ┌──────────────────┐
  │ Returns  │ ←─────────────│ Advance internal  │
  │ element  │               │ cursor, return    │
  │ or NULL  │               │ next element ptr  │
  └──────────┘               └──────────────────┘
       │
       │ bpf_iter_<type>_destroy(&it)
       ▼
  ┌──────────┐  kfunc call   ┌──────────────────┐
  │ Cleaned  │ ─────────────→│ Release resources │
  │ up       │               │ (locks, refs)     │
  └──────────┘               └──────────────────┘
```

## 2. Background & Motivation

### 2.1 The Problem Before Open-Coded Iterators

BPF programs had two fundamental looping limitations:

#### Problem 1: Bounded loop verification is expensive

The BPF verifier must prove every program terminates. For loops, it does
this by exploring every possible execution path. With a bounded for loop:

```c
for (i = 0; i < 1000; i++) {
    /* verifier explores EACH iteration as a separate state */
}
```

The verifier explores up to `1000 × branches_per_iteration` states. This
quickly hits the **1,000,000 instruction verification limit** for complex
loop bodies.

#### Problem 2: No way to iterate kernel data structures

Before open-coded iterators, BPF programs could not walk kernel data
structures like the task list, VMA tree, or cgroup hierarchy. The older
**seq_file-based BPF iterators** existed but had major limitations:

```
 Seq_file iterators (older, still exist):
 ┌─────────────────────────────────────────────────────────┐
 │  • Run as separate programs attached to /sys/fs/bpf/    │
 │  • Each element triggers a BPF program invocation       │
 │  • Can't break early — must process all elements        │
 │  • Can't be used inline within another BPF program      │
 │  • Output via seq_file (text), not programmable logic   │
 └─────────────────────────────────────────────────────────┘

 Open-coded iterators (newer, this document):
 ┌─────────────────────────────────────────────────────────┐
 │  • Used inline within any BPF program                   │
 │  • Support break/continue like normal for loops         │
 │  • Verifier proves termination via iterator protocol    │
 │  • Zero-overhead when done (no seq_file layer)          │
 │  • Can nest iterators (e.g., tasks within cgroups)      │
 └─────────────────────────────────────────────────────────┘
```

### 2.2 How Open-Coded Iterators Solve These Problems

The key insight: **the verifier doesn't need to unroll iterator loops**.
Instead, it verifies the loop body **once** with two states:

1. `_next()` returns non-NULL → loop body executes (ACTIVE state)
2. `_next()` returns NULL → loop exits (DRAINED state)

```
  Verifier state exploration:

  Traditional for loop:             Open-coded iterator:
  ┌─────────────────────┐          ┌─────────────────────┐
  │ i=0: verify body    │          │ State 1: non-NULL    │
  │ i=1: verify body    │          │   → verify body once │
  │ i=2: verify body    │          │ State 2: NULL         │
  │ ...                 │          │   → verify exit path │
  │ i=999: verify body  │          │                       │
  │ = 1000 iterations!  │          │ = 2 states only!      │
  └─────────────────────┘          └─────────────────────┘
```

This means open-coded iterators have **O(1) verification cost** regardless
of how many elements are iterated at runtime.

### 2.3 Design Goals

1. **Verifier-friendly**: O(1) verification cost per loop
2. **Ergonomic**: `bpf_for_each()` / `bpf_for()` macros look like normal loops
3. **Safe**: Iterator protocol guarantees termination and cleanup
4. **Extensible**: New iterator types are just three kfuncs
5. **Composable**: Iterators can be nested

## 3. Core Infrastructure

### 3.1 KF_ITER Kfunc Flags

Three flags in `include/linux/btf.h` mark kfuncs as iterator functions:

```c
#define KF_ITER_NEW     (1 << 8)   /* constructor */
#define KF_ITER_NEXT    (1 << 9)   /* advance + return element */
#define KF_ITER_DESTROY (1 << 10)  /* destructor */
```

These flags trigger special validation in `btf_check_iter_kfuncs()`:

| Check | Enforced by |
|-------|-------------|
| Naming: `bpf_iter_<type>_{new,next,destroy}` | `btf.c:8596-8598` |
| First arg: `struct bpf_iter_<type> *` | `btf.c:8569-8576` |
| Struct size: multiple of 8 bytes | `btf.c:8582-8583` |
| `_next` and `_destroy`: exactly 1 argument | `btf.c:8601-8602` |
| `_next`: must return pointer type | `btf.c:8604-8608` |
| `_destroy`: must return void | `btf.c:8611-8616` |

**Source:** [`kernel/bpf/btf.c:8555-8619`](https://github.com/torvalds/linux/blob/master/kernel/bpf/btf.c)

### 3.2 Iterator State on BPF Stack

Each iterator type defines an opaque struct that lives on the BPF stack:

```c
/* What BPF program sees (opaque) */
struct bpf_iter_num {
    __u64 __opaque[1];   /* 8 bytes */
} __aligned(8);

/* What the kernel stores inside (internal) */
struct bpf_iter_num_kern {
    int cur;   /* current value */
    int end;   /* end value (exclusive) */
};
/* sizeof(bpf_iter_num_kern) == sizeof(bpf_iter_num) == 8 */
```

The verifier marks the stack slots occupied by the iterator as
`STACK_ITER`, preventing the BPF program from corrupting them.

### 3.3 Verifier Iterator State Machine

The verifier tracks each iterator's lifecycle:

```
                    bpf_iter_<type>_new()
                           │
                           ▼
                   ┌───────────────┐
                   │ BPF_ITER_STATE│
                   │   _ACTIVE     │
                   └───────┬───────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
              ▼            ▼            ▼
        _next()!=NULL  _next()==NULL  _destroy()
              │            │            │
              ▼            ▼            ▼
         ┌────────┐  ┌──────────┐  ┌─────────┐
         │ ACTIVE │  │ DRAINED  │  │ (freed) │
         │ (loop) │  │ (done)   │  │         │
         └────┬───┘  └────┬─────┘  └─────────┘
              │            │
              └────────────┘
                    │
                    ▼
              _destroy()
                    │
                    ▼
             ┌─────────┐
             │ (freed) │
             └─────────┘
```

The verifier enforces:
- `_next()` can only be called on ACTIVE iterators
- `_destroy()` MUST be called before the program returns
- After `_next()` returns NULL (DRAINED), further `_next()` also returns NULL

**Source:** [`include/linux/bpf_verifier.h:32-35`](https://github.com/torvalds/linux/blob/master/include/linux/bpf_verifier.h)

## 4. Convenience Macros

The raw three-function pattern is verbose. Libbpf provides ergonomic macros:

### 4.1 `bpf_for_each(type, cur, args...)`

Generic iterator loop — works with any iterator type:

```c
struct task_struct *task;

bpf_for_each(task, task, NULL, BPF_TASK_ITER_ALL_PROCS) {
    /* task points to each process */
    bpf_printk("pid=%d", task->pid);
}
```

Expands to:
```c
for (struct bpf_iter_task ___it __cleanup(bpf_iter_task_destroy),
     *___p = (bpf_iter_task_new(&___it, NULL, BPF_TASK_ITER_ALL_PROCS), NULL);
     (task = bpf_iter_task_next(&___it));
    )
```

Key features:
- Auto-calls `_new()` once at initialization
- Auto-calls `_destroy()` via `__attribute__((cleanup(...)))`
- Loop condition is `_next()` — breaks on NULL
- Supports `break` and `continue`

### 4.2 `bpf_for(i, start, end)`

Numeric iterator — like `for (i = start; i < end; i++)`:

```c
int i;
bpf_for(i, 0, 100) {
    /* i goes from 0 to 99 */
    array[i] = i * 2;
}
```

The verifier **proves** `i ∈ [start, end)`, so array accesses within
this range don't need extra bounds checks.

### 4.3 `bpf_repeat(N)`

Run a loop body N times without exposing the counter:

```c
bpf_repeat(1000) {
    /* executed 1000 times */
    do_work();
}
```

**Source:** [`tools/lib/bpf/bpf_helpers.h:349-445`](https://github.com/torvalds/linux/blob/master/tools/lib/bpf/bpf_helpers.h)

## 5. All Built-In Iterator Types

### 5.1 Overview

| Iterator | Type iterated | File | Constructor args |
|----------|--------------|------|-----------------|
| `bpf_iter_num` | Integer range [start, end) | `bpf_iter.c` | `start`, `end` |
| `bpf_iter_task` | Tasks (processes/threads) | `task_iter.c` | `task`, `flags` |
| `bpf_iter_task_vma` | Virtual memory areas | `task_iter.c` | `task`, `addr`, `flags` |
| `bpf_iter_css` | Cgroup subsystem states | `cgroup_iter.c` | `start`, `flags` |
| `bpf_iter_css_task` | Tasks in a cgroup | `cgroup_iter.c` | `css`, `flags` |
| `bpf_iter_bits` | Set bits in a bitmask | `helpers.c` | `unsafe_ptr`, `nr_bits` |
| `bpf_iter_kmem_cache` | Slab allocator caches | `helpers.c` | (none) |
| `bpf_iter_dmabuf` | DMA-buf objects | `helpers.c` | (none) |

Custom iterators can be added by any subsystem (e.g., `bpf_iter_uring_buf`
in io_uring).

### 5.2 Reference Example: `bpf_iter_num` (Simplest)

The number iterator is the simplest — it iterates integers [start, end):

```c
/* kernel/bpf/bpf_iter.c:770-825 */

struct bpf_iter_num_kern {
    int cur;    /* current value (starts at start-1) */
    int end;    /* end value (exclusive) */
};

__bpf_kfunc int bpf_iter_num_new(struct bpf_iter_num *it, int start, int end)
{
    struct bpf_iter_num_kern *s = (void *)it;

    if (start > end) {
        s->cur = s->end = 0;
        return -EINVAL;
    }
    s->cur = start - 1;  /* pre-decrement so first _next returns start */
    s->end = end;
    return 0;
}

__bpf_kfunc int *bpf_iter_num_next(struct bpf_iter_num *it)
{
    struct bpf_iter_num_kern *s = (void *)it;

    if ((s64)(s->cur + 1) >= s->end) {
        s->cur = s->end = 0;
        return NULL;    /* iteration done */
    }
    s->cur++;
    return &s->cur;     /* return pointer to current value */
}

__bpf_kfunc void bpf_iter_num_destroy(struct bpf_iter_num *it)
{
    struct bpf_iter_num_kern *s = (void *)it;
    s->cur = s->end = 0;   /* cleanup */
}
```

**Source:** [`kernel/bpf/bpf_iter.c:770-825`](https://github.com/torvalds/linux/blob/master/kernel/bpf/bpf_iter.c)

### 5.3 Complex Example: `bpf_iter_task` (Process Iterator)

The task iterator walks all processes or threads:

```c
/* kernel/bpf/task_iter.c:965-1028 */

__bpf_kfunc int bpf_iter_task_new(struct bpf_iter_task *it,
        struct task_struct *task__nullable, unsigned int flags)
{
    struct bpf_iter_task_kern *kit = (void *)it;

    kit->pos = NULL;

    switch (flags) {
    case BPF_TASK_ITER_ALL_THREADS:   /* all threads system-wide */
    case BPF_TASK_ITER_ALL_PROCS:     /* all processes (group leaders) */
        kit->task = &init_task;       /* start from init */
        break;
    case BPF_TASK_ITER_PROC_THREADS:  /* threads of a specific process */
        if (!task__nullable)
            return -EINVAL;
        kit->task = task__nullable;
        break;
    default:
        return -EINVAL;
    }

    kit->pos = kit->task;
    kit->flags = flags;
    return 0;
}

__bpf_kfunc struct task_struct *bpf_iter_task_next(struct bpf_iter_task *it)
{
    struct bpf_iter_task_kern *kit = (void *)it;
    struct task_struct *pos = kit->pos;

    if (!pos) return NULL;         /* exhausted */

    if (kit->flags == BPF_TASK_ITER_ALL_PROCS)
        goto get_next_task;

    kit->pos = __next_thread(kit->pos);   /* next thread */
    if (kit->pos || kit->flags == BPF_TASK_ITER_PROC_THREADS)
        return pos;

get_next_task:
    kit->task = next_task(kit->task);     /* next process */
    if (kit->task == &init_task)
        kit->pos = NULL;                  /* wrapped around */
    else
        kit->pos = kit->task;

    return pos;
}

__bpf_kfunc void bpf_iter_task_destroy(struct bpf_iter_task *it)
{
    /* nothing to clean up — task list traversal is stateless */
}
```

**Source:** [`kernel/bpf/task_iter.c:965-1028`](https://github.com/torvalds/linux/blob/master/kernel/bpf/task_iter.c)

## 6. Verifier Deep Dive

### 6.1 Stack Slot Tracking

When `_new()` is called, the verifier marks the iterator's stack slots as
`STACK_ITER`:

```
  BPF Stack (example: bpf_iter_num, 8 bytes = 1 slot)
  ┌─────────────────┐
  │   STACK_ITER     │  iter.state = BPF_ITER_STATE_ACTIVE
  │   (8 bytes)      │  iter.btf_id = <bpf_iter_num>
  │   iter.depth = 0 │
  └─────────────────┘
```

For larger iterators (e.g., `bpf_iter_task` with 16 bytes = 2 slots),
multiple consecutive stack slots are marked.

### 6.2 Loop Verification Strategy

When the verifier encounters a `_next()` call, it explores **two branches**:

```
         bpf_iter_task_next(&it)
                │
        ┌───────┴───────┐
        │               │
    non-NULL           NULL
    (ACTIVE)         (DRAINED)
        │               │
        ▼               ▼
  Execute loop      Exit loop
  body once         (state pruning:
  (verifier          equivalent to
   explores          any prior NULL
   this path)        state)
```

The verifier uses **state pruning**: once it has verified the loop body
with a non-NULL return, it doesn't re-verify for subsequent iterations
because the register state is equivalent.

### 6.3 Enforced Rules

| Rule | Error if violated |
|------|------------------|
| `_new()` before `_next()` | "uninitialized iterator" |
| `_destroy()` before program exit | "leaked iterator" |
| Cannot write to STACK_ITER slots directly | "corrupted iterator state" |
| `_next()` and `_destroy()` must have exactly 1 arg | `-EINVAL` from `btf_check_iter_kfuncs` |
| Iterator struct size must be 8-byte aligned | `-EINVAL` |
| Naming must match `bpf_iter_<type>_{new,next,destroy}` | `-EINVAL` |

## 7. Real-World Example: Counting Processes per Cgroup

This example shows a BPF program that counts how many processes belong to
each cgroup in a subtree — using nested iterators:

```c
SEC("raw_tp/sys_enter")
int count_cgroup_tasks(void *ctx)
{
    struct cgroup_subsys_state *css;
    struct task_struct *task;
    struct cgroup *root_cg = /* ... */;

    /* Outer loop: iterate cgroup subtree */
    bpf_for_each(css, css, root_cg->self, BPF_CGROUP_ITER_DESCENDANTS_POST) {
        int count = 0;

        /* Inner loop: iterate tasks in this cgroup */
        bpf_for_each(css_task, task, css, CSS_TASK_ITER_PROCS) {
            count++;
        }

        bpf_printk("cgroup %llu has %d procs",
                   css->cgroup->kn->id, count);
    }

    return 0;
}
```

**What this demonstrates:**
1. **Nested iterators** — cgroup tree walk contains task walk
2. **`break`/`continue` support** — both loops support early exit
3. **O(1) verification** — even with nesting, the verifier explores each
   loop body once (2 states per level = 4 total states)
4. **Ergonomic macros** — `bpf_for_each` makes it look like a normal loop

### 7.1 Without Open-Coded Iterators

Before open-coded iterators, this was **impossible** to express in a
single BPF program. You would need:

1. A seq_file iterator program that outputs cgroup info
2. A separate seq_file iterator program for tasks per cgroup
3. Userspace glue to combine the outputs
4. No ability to break early or maintain state between iterations

## 8. How to Implement a Custom Iterator

Adding a new iterator type requires three steps:

### Step 1: Define the opaque and kernel structs

```c
/* In your header (visible to BPF programs) */
struct bpf_iter_my_type {
    __u64 __opaque[2];  /* must be 8-byte aligned, multiple of 8 */
} __aligned(8);

/* In your .c file (kernel-internal) */
struct bpf_iter_my_type_kern {
    struct my_type *pos;
    int flags;
    /* must fit in sizeof(bpf_iter_my_type) */
};
```

### Step 2: Implement the three kfuncs

```c
__bpf_kfunc int bpf_iter_my_type_new(struct bpf_iter_my_type *it, /* args */)
{
    struct bpf_iter_my_type_kern *k = (void *)it;
    BUILD_BUG_ON(sizeof(*k) > sizeof(*it));

    /* initialize traversal state */
    k->pos = first_element();
    return 0;
}

__bpf_kfunc struct my_type *bpf_iter_my_type_next(struct bpf_iter_my_type *it)
{
    struct bpf_iter_my_type_kern *k = (void *)it;
    struct my_type *ret = k->pos;

    if (!ret) return NULL;
    k->pos = next_element(k->pos);
    return ret;
}

__bpf_kfunc void bpf_iter_my_type_destroy(struct bpf_iter_my_type *it)
{
    /* release any held resources */
}
```

### Step 3: Register the kfuncs

```c
BTF_KFUNCS_START(my_kfuncs)
BTF_ID_FLAGS(func, bpf_iter_my_type_new, KF_ITER_NEW)
BTF_ID_FLAGS(func, bpf_iter_my_type_next, KF_ITER_NEXT | KF_RET_NULL)
BTF_ID_FLAGS(func, bpf_iter_my_type_destroy, KF_ITER_DESTROY)
BTF_KFUNCS_END(my_kfuncs)
```

The verifier automatically validates the naming convention, argument counts,
and return types.

## 9. Open-Coded vs Seq_file Iterators

Both exist in the kernel. Here's when to use each:

| Feature | Seq_file iterators | Open-coded iterators |
|---------|-------------------|---------------------|
| Introduced | v5.8 (2020) | v6.4 (2023) |
| Usage | Separate program, attached to BPF link | Inline within any BPF program |
| `break` support | No — must process all elements | Yes |
| Nesting | Not supported | Supported |
| State between elements | Separate program invocations | Same function scope |
| Verification cost | Per-element program verification | O(1) regardless of count |
| Output mechanism | seq_file (text-based) | Any (maps, ringbuf, printk) |
| Use case | Dumping `/proc`-like info | Complex in-program logic |

Seq_file iterators are still used for `/sys/fs/bpf/` inspection tools.
Open-coded iterators are preferred for program logic.

```c
void                          bpf_iter_css_destroy(struct bpf_iter_css *it);
```

### 11.4 Convenience Macros

```c
bpf_for_each(type, cur, args...)   /* generic iterator loop */
bpf_for(i, start, end)             /* numeric loop with bounds proof */
bpf_repeat(N)                      /* repeat N times */
```

---

## 12. Summary

BPF open-coded iterators are a **kfunc-based looping pattern** that solves
two fundamental BPF limitations:

1. **Verification scalability** — O(1) verification cost per loop, regardless
   of iteration count, by exploring only two states (non-NULL / NULL)
2. **Kernel data structure traversal** — safely walk tasks, cgroups, VMAs,
   and custom data from within BPF programs

Key design properties:

- **Three-function protocol**: `_new()` / `_next()` / `_destroy()`
- **Stack-resident state**: iterator lives on BPF stack (no heap allocation)
- **Verifier-enforced lifecycle**: `STACK_ITER` marking, mandatory cleanup
- **Composable**: iterators can be nested
- **Extensible**: any subsystem can add iterators with three kfuncs
- **Ergonomic**: `bpf_for_each()`, `bpf_for()`, `bpf_repeat()` macros
