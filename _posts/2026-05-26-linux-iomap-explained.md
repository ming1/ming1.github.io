# Overview

iomap is the in-kernel framework that lets a filesystem implement
read, write, direct I/O, writeback, fiemap, seek-hole/data, and
swap-on-file in terms of **extent mappings** rather than per-block
`buffer_head`s. A filesystem supplies one small vtable
(`struct iomap_ops` with `iomap_begin` and `iomap_end`); iomap calls
back into it to translate file ranges to disk addresses, then drives
the page cache, the bio submission path, and the writeback ioend
lifecycle on the filesystem's behalf. This post walks the subsystem
outside-in: boundary, use cases, exported interfaces, core data
structures, neighbouring-subsystem contracts, and live tracing.

## 1.3 What iomap is NOT

iomap is constantly confused with neighbours:

- **Not `buffer_head`.** The old per-block `struct buffer_head` chain
  is replaced by `struct iomap_folio_state` attached to
  `folio->private` (see §4.4). A filesystem on iomap does not call
  `__block_write_begin`, `block_read_full_folio`, or grow buffers; it
  calls `iomap_file_buffered_write` and supplies `iomap_ops`. ext4
  uses iomap only for DIO/DAX/zero/fiemap — its main buffered path is
  still `buffer_head`-based.
- **Not `aops->write_begin` / `write_end`.** iomap replaces these in
  one shot: a filesystem registers `iomap_write_ops` (with optional
  `get_folio`/`put_folio` hooks for COW) and calls
  `iomap_file_buffered_write` from `.write_iter`.
- **Not the legacy `fs/direct-io.c`.** That file (the old
  `__blockdev_direct_IO`) is a separate, `buffer_head`-based DIO
  implementation. iomap's DIO is `fs/iomap/direct-io.c` only.
- **Not the writeback subsystem.** Dirty-page accounting, writeback
  scheduling, and bdi_writeback live in `mm/page-writeback.c` and
  `fs/fs-writeback.c` — see the
  [writeback explainer](/tech/2026/05/24/linux-writeback-explained.html).
  iomap is what the filesystem's `writepages` aop *does* once the
  writeback thread invokes it. iomap builds the bios; writeback
  decides when.

# 2. Use Cases and Task Contexts

iomap exists so filesystems can share one implementation of the
page-cache, DIO, and writeback machinery. Concrete consumers in
mainline:

| Filesystem | What it uses iomap for |
|---|---|
| **XFS** | Read, readahead, buffered write, writepages, DIO, fiemap, seek hole/data, swapfile, page_mkwrite, truncate page (full adoption). |
| **exfat** | Read, readahead, buffered write, writepages, DIO, zero_range, page_mkwrite, seek hole/data (near-full adoption). |
| **erofs** | Read, readahead, DIO, fiemap, seek hole/data (read-only filesystem). |
| **fuse** | Buffered write, read_folio, readahead, writepages. |
| **GFS2** | Buffered write, writepages, DIO, fiemap, seek hole/data (no iomap read path). |
| **zonefs** | Buffered write, writepages, DIO, page_mkwrite, swapfile. |
| **ext4** | DIO, fiemap, seek hole/data, swapfile. Buffered I/O still uses `buffer_head`. |
| **ext2** | DIO, fiemap. |
| **f2fs**, **btrfs** | DIO only. |
| **hpfs** | fiemap only. |

The task contexts iomap runs in are inherited from its caller:

| Path | Task context |
|---|---|
| `read(2)` / `write(2)` → `iomap_file_buffered_write` / `iomap_read_folio` | Syscall context (current → `vfs_write` → `aops` / iomap). |
| `pwritev2(RWF_DSYNC)` + DIO | Syscall, may sleep in completion if `wait_for_completion`. |
| AIO / io_uring DIO completion | Bio end-io → `iomap_dio_bio_end_io` → `iomap_dio_done()` picks one of: wake the sync waiter; queue `iomap_dio_complete_work` to `s_dio_done_wq`; or, when called with `inline_completion` (io_uring polling), run completion in the submitter's task context. |
| Writeback flusher | `bdi_writeback` kthread → `do_writepages` → `iomap_writepages`. |
| `iomap_finish_ioends` | Softirq (REQ_FUA fast path) **or** per-superblock `s_dio_done_wq` workqueue. |
| `iomap_swapfile_activate` | `sys_swapon` syscall context, once at activation. |
| `fiemap` / `lseek(SEEK_HOLE)` | Syscall context. |

A representative end-to-end walkthrough — a buffered write on XFS:

```
write(2)
  └─ xfs_file_write_iter
       └─ iomap_file_buffered_write(..., &xfs_buffered_write_iomap_ops, ...)
            └─ iomap_iter(iter, ops)                  [loop]
                 ├─ ops->iomap_begin → xfs_buffered_write_iomap_begin
                 │     (reserves delalloc, returns struct iomap)
                 ├─ iomap_write_iter
                 │     ├─ __iomap_write_begin (alloc folio, iomap_folio_state)
                 │     ├─ copy_folio_from_iter_atomic
                 │     └─ iomap_set_range_dirty (bitmap, no buffer_head)
                 └─ ops->iomap_end → xfs_buffered_write_iomap_end
```

Writeback later picks up the dirty folio, calls
`xfs_vm_writepages` → `iomap_writepages` → `iomap_writeback_folio` →
`iomap_add_to_ioend`, which appends to the current
`struct iomap_ioend` bio chain. On bio completion,
`iomap_finish_ioends` retires the batch (see §4.5).

# 3. Interfaces

This section is grouped by purpose, not by file. Every name below is
an `EXPORT_SYMBOL_GPL` from `fs/iomap/*` (37 of them total) or a
struct in `include/linux/iomap.h`.

## 3.1 The `iomap_ops` contract

Filesystems implement two callbacks:

```c
struct iomap_ops {
    int (*iomap_begin)(struct inode *inode, loff_t pos, loff_t length,
                       unsigned flags, struct iomap *iomap,
                       struct iomap *srcmap);
    int (*iomap_end)(struct inode *inode, loff_t pos, loff_t length,
                     ssize_t written, unsigned flags, struct iomap *iomap);
};
```

- `iomap_begin` looks up (or reserves, for writes) the mapping at
  `pos`, fills `*iomap`, may shorten the range via `iomap->length`.
  For COW, the *source* mapping goes into `srcmap` and the
  destination into `iomap`.
- `iomap_end` is called once per iter step. For writes it commits or
  unreserves; for COW it remaps; for reads it is usually a no-op.

`flags` (`IOMAP_WRITE`, `IOMAP_ZERO`, `IOMAP_DIRECT`, `IOMAP_NOWAIT`,
`IOMAP_FAULT`, `IOMAP_REPORT`, `IOMAP_DAX`, `IOMAP_ATOMIC`,
`IOMAP_DONTCACHE`, ...) tell the filesystem what operation is in
progress and what the caller can tolerate (blocking, allocation,
etc.).

## 3.2 The iteration primitive

```c
int iomap_iter(struct iomap_iter *iter, const struct iomap_ops *ops);
int iomap_iter_advance(struct iomap_iter *iter, u64 count);
```

`iomap_iter` is the canonical iomap control-flow primitive — every
single operation in §3.3–3.6 ultimately rides on it. The pattern:

```c
struct iomap_iter iter = { .inode = inode, .pos = pos, .len = len, .flags = ... };
while ((ret = iomap_iter(&iter, ops)) > 0) {
    /* per-step body, e.g. iomap_write_iter(&iter, ...) */
    iter.status = step(&iter);
}
return iter.status ? iter.status : ret;
```

On each call, `iomap_iter` first invokes `ops->iomap_end` for the
previous step (if any), passing it the number of bytes the body
consumed (`advanced = iter->pos - iter->iter_start_pos`), then
invokes `ops->iomap_begin` for the next mapping. The body itself
advances `iter.pos` / `iter.len` by calling `iomap_iter_advance(iter,
count)` for each chunk it consumes within a mapping. The loop
terminates when `iter.len` is zero, when the body returns an error in
`iter.status`, or when the body made no progress (and the mapping
isn't marked stale). **Fan-in in `fs/iomap/`: 12** — every top-level
operation calls it.

# 4. Core Data Structures

iomap is a small subsystem with a small type vocabulary. Five
load-bearing structs:

## 4.1 `struct iomap` — one extent mapping

```c
struct iomap {
    u64                  addr;       /* disk offset, bytes; or special */
    loff_t               offset;     /* file offset, bytes */
    u64                  length;     /* bytes */
    u16                  type;       /* HOLE/DELALLOC/MAPPED/UNWRITTEN/INLINE */
    u16                  flags;      /* IOMAP_F_* */
    struct block_device *bdev;
    struct dax_device   *dax_dev;
    void                *inline_data;
    void                *private;    /* filesystem private */
    u64                  validity_cookie;
};
```

**Invariant**: every byte in `[offset, offset+length)` of the file
maps to the same `type` and the same contiguous disk region (or has
the same special status, for `HOLE`/`DELALLOC`/`UNWRITTEN`). If the
filesystem cannot describe a longer range as one mapping, it returns
a shorter `length` and iomap re-asks for the next chunk.

The flag space is large but each flag is purposeful. A few worth
calling out:

- `IOMAP_F_NEW` — the mapping was freshly allocated this call; the
  in-cache folio range needs zeroing before user data is copied in.
- `IOMAP_F_SHARED` — reflink/COW source; a write must break sharing.
- `IOMAP_F_STALE` — set by the filesystem in the body of an iter
  step when it discovers the mapping was invalidated under it (e.g.
  a competing truncate). iomap then re-issues `iomap_begin` instead
  of advancing. This is iomap's retry-on-invalidation mechanism.
- `IOMAP_F_ANON_WRITE` — disk block isn't assigned yet; the bio
  submission hook will assign and possibly split.
- `IOMAP_F_ATOMIC_BIO` — submit with `REQ_ATOMIC` (torn-write
  protection).

`validity_cookie` plus the filesystem's optional `iomap_valid` hook
let iomap detect mid-operation invalidation without holding the
filesystem extent lock across the whole I/O.

## 4.2 `struct iomap_ops` — the contract

Already shown in §3.1. The key design decision: **two callbacks,
not one per operation**. Buffered read, buffered write, DIO, fiemap,
seek and swap all funnel through the same `iomap_begin` / `iomap_end`
pair. The filesystem distinguishes by `flags` (`IOMAP_WRITE`,
`IOMAP_DIRECT`, `IOMAP_REPORT`, ...) and by the operation having
called the appropriate top-level entry point. XFS, for example, has
*different* `iomap_ops` instances for buffered-read, buffered-write,
DIO-read, DIO-write, COW, and write-back — each preset with the
behaviour it needs and the right `iomap_begin` implementation.

## 4.3 `struct iomap_iter` — loop state

```c
struct iomap_iter {
    struct inode      *inode;
    loff_t             pos;
    u64                len;
    loff_t             iter_start_pos;   /* for partial advance */
    int                status;
    unsigned           flags;
    struct iomap       iomap;            /* destination map */
    struct iomap       srcmap;           /* COW source map */
    struct folio_batch *fbatch;
    void              *private;
};
```

**Lifetime**: stack-allocated by the caller of the top-level
operation; lives exactly as long as the `while (iomap_iter(...) > 0)`
loop. The body sees `iter.iomap` / `iter.srcmap` as read-only
descriptions of the current step, calls `iomap_iter_advance()` to
report bytes consumed, and stores any error in `iter.status`.
`iomap_iter` itself runs `iomap_end` for the previous step before
invoking the next `iomap_begin`.

## 4.4 `struct iomap_folio_state` — the `buffer_head` killer

```c
struct iomap_folio_state {
    spinlock_t   state_lock;
    unsigned int read_bytes_pending;
    atomic_t     write_bytes_pending;
    unsigned long state[];  /* 2 bits/block: [uptodate | dirty] */
};
```

This is the single biggest design change iomap introduces. The legacy
buffer_head world tracked sub-page state with one `struct buffer_head`
per block, chained via `page->private`. iomap collapses that to:

- A **single allocation** per folio, sized as a flexible array based
  on `blocks_per_folio`.
- A **2-bit-per-block bitmap**: bits `[0, b_p_f)` are uptodate,
  bits `[b_p_f, 2*b_p_f)` are dirty.
- Two counters that gate completion: `read_bytes_pending` (reads in
  flight to this folio) and `write_bytes_pending` (writes).
- One spinlock guarding the bitmap.

**Lifetime**: attached to `folio->private` via `folio_attach_private`
inside `ifs_alloc()` (lazy — allocated on first sub-block operation;
single-block folios skip it). Detached by `iomap_release_folio` /
`iomap_invalidate_folio` via `folio_detach_private`. Replaces an
entire `buffer_head` chain — no `bh_lru`, no `try_to_free_buffers`,
no `mark_buffer_dirty` per block. Sub-folio I/O works by inspecting
the bitmap to decide which sub-ranges need bio submission.

## 4.5 `struct iomap_ioend` — writeback batch

The unit of writeback completion. An ioend wraps one bio chain plus
metadata the filesystem needs at completion time:

```c
struct iomap_ioend {
    struct list_head      io_list;
    u16                   io_type;       /* mapping type at submit */
    u16                   io_flags;
    u32                   io_folios;     /* folio count */
    struct inode         *io_inode;
    size_t                io_size;
    loff_t                io_offset;
    sector_t              io_sector;
    struct bio            io_bio;        /* embedded — head of chain */
    /* + private filesystem area */
};
```

**Lifetime**: allocated from `iomap_ioend_bioset` by
`iomap_init_ioend` at the start of a writeback range. Grown by
`iomap_add_to_ioend` (appends to the bio chain; spills to a new ioend
when the bio cannot grow). Submitted by
`iomap_ioend_writeback_submit`. Completed by `iomap_finish_ioends`
once **all** chained bios end — driven from the bio end-io callback.

Completion path is dual:

- **REQ_FUA fast path / non-error**: completed inline from softirq
  (`iomap_finish_ioends` directly from `io_bio.bi_end_io`).
- **Error or non-trivial unwritten/COW conversion**: punted to the
  per-superblock `s_dio_done_wq` workqueue (`iomap_finish_ioend_work`)
  so the filesystem's end-io hook can sleep on the log / transaction
  layer.

`iomap_sort_ioends` and `iomap_ioend_try_merge` exist so a filesystem
(XFS in particular) can collect a list of ioends across many bios,
sort them by file offset, merge adjacent ones with the same
type/flags, and then complete them in order — important for journal
ordering on writeback of allocation/unwritten conversion.

## 4.6 The DIO state machine — `struct iomap_dio`

DIO deserves its own walkthrough.

```c
struct iomap_dio {
    struct kiocb            *iocb;
    const struct iomap_dio_ops *dops;
    loff_t                   i_size;
    atomic_t                 ref;       /* outstanding bios + holders */
    unsigned                 flags;
    int                      error;
    size_t                   done_before;
    union {
        struct {                      /* sync */
            struct task_struct *waiter;
        } submit;
        struct {                      /* async */
            struct work_struct  work;
        } aio;
    };
    /* fs private area */
};
```

The flow inside `__iomap_dio_rw` (`fs/iomap/direct-io.c:803`):

1. Build a `struct iomap_dio` and stash `iocb`, `dops`, the current
   `i_size`, and flags (`IOMAP_DIO_WRITE`, `IOMAP_DIO_DIRTY`, ...).
2. If async and the superblock has no `s_dio_done_wq` yet,
   `sb_init_dio_done_wq()`.
3. `inode_dio_begin()` to pin the inode against truncate.
4. `blk_start_plug()` + the iter loop:

   ```c
   while ((ret = iomap_iter(&iomi, ops)) > 0)
       iomi.status = iomap_dio_iter(&iomi, dio);
   ```

   `iomap_dio_iter` dispatches by mapping type to one of three
   per-mapping handlers:
   - **`iomap_dio_bio_iter`** — the common case: build bios directly
     from the user `iov_iter` via `bio_iov_iter_get_pages`, set
     `REQ_FUA` if the caller asked for durable + the bdev supports
     it, optionally compute integrity metadata, submit via
     `dops->submit_io` (or `submit_bio` by default), then
     `iomap_iter_advance` for the bytes actually iov'd in.
   - **`iomap_dio_hole_iter`** — read of a hole: zero the user buffer
     and advance.
   - **`iomap_dio_inline_iter`** — inline data extent: memcpy and
     advance.
5. `blk_finish_plug()`.
6. Completion. When the last in-flight bio drops `dio->ref` to zero,
   `iomap_dio_done()` decides which of three modes runs:
   - **Sync** (`wait_for_completion == true`): wakes the submitting
     task; the submitter then runs `iomap_dio_complete` itself before
     returning from the syscall.
   - **Async via workqueue**: `iomap_dio_done()` sets
     `IOMAP_DIO_COMP_WORK` (forced for errors, and for writes whose
     mapping has cached pages that need invalidation), queues
     `iomap_dio_complete_work` onto the per-superblock
     `s_dio_done_wq`, and returns; the worker then runs
     `iomap_dio_complete` and `iocb->ki_complete`.
   - **Async inline**: neither of the above — `iomap_dio_complete_work`
     runs directly from bio end-io context. The caller-completion
     path used by io_uring polling reaches this by passing
     `inline_completion = true` into `__iomap_dio_bio_end_io`, which
     clears `IOMAP_DIO_COMP_WORK` (since the caller is in task
     context and can sleep) so completion finishes inline on the
     submitter's CPU — no workqueue hop, no extra context switch.

`REQ_FUA` is set on the bio when the filesystem opts into write-through
(`IOMAP_DIO_WRITE_THROUGH`), the underlying bdev supports FUA, no
completion-time work is needed, and the mapping is clean — letting
`iomap_dio_rw` complete an O_DSYNC write without a separate
post-write cache flush. Fallback to buffered I/O on `-ENOTBLK`,
`IOMAP_DIO_PARTIAL` partial-result recovery on `-EFAULT`, and the
in-flight bounce-buffer path (`IOMAP_DIO_BOUNCE` plus
`bio_iov_iter_unbounce`) for callers that need a stable kernel copy
of the iovec are all handled by the same state machine.

# 5. Relations with Other Subsystems

iomap is the connective tissue between filesystems and several core
subsystems. The contracts:

| Neighbour | Contract |
|---|---|
| **Block layer** | iomap allocates bios from a `bio_set` (its own `iomap_ioend_bioset` for writeback; bio cache for DIO/reads), fills them via `bio_iov_iter_get_pages`, submits via `submit_bio` or the filesystem's `submit_io` hook, and consumes `bio_end_io` to drive completion. iomap never inspects request queues. |
| **Page cache + folio API** | iomap calls `__filemap_get_folio`, `folio_lock`, `folio_mark_dirty`, `folio_start_writeback`, `folio_end_writeback`, `folio_attach_private`, etc. The folio-only API (no `struct page` for new code) is *the* prerequisite that made `struct iomap_folio_state` possible. |
| **VFS** | Provides `aops` implementations (`read_folio`, `readahead`, `writepages`, `dirty_folio`, ...) for a filesystem to plug into; supplies the `lseek(SEEK_HOLE|SEEK_DATA)` and `FS_IOC_FIEMAP` helpers. Does not implement `read_iter`/`write_iter` — the filesystem still owns the `file_operations`. |
| **MM** | `iomap_swapfile_activate` walks the file's extents into `swap_info_struct->swap_extent_root` so `swap_writepage`/`swap_readpage` can issue I/O directly to the block device. `iomap_page_mkwrite` is the filesystem's `vm_ops->page_mkwrite`. |
| **Writeback** | iomap is what `do_writepages` invokes via `aops->writepages`; it is *not* the writeback scheduler. The dirty-throttle math, bdi management, and writeback wb selection happen in `mm/page-writeback.c` and `fs/fs-writeback.c` (see the [writeback explainer](/tech/2026/05/24/linux-writeback-explained.html)). |
| **DAX** | `IOMAP_DAX` short-circuits bio submission entirely; the `dax_dev` in `struct iomap` is used by `fs/dax.c` to copy via DAX direct mappings. |
| **Block-integrity** | When `IOMAP_F_INTEGRITY` is set on a read iomap and the bdev integrity profile is present, iomap (via `iomap_bio_read_ops`) lets the filesystem attach a protection-information payload to read bios. For writes, the same path is reached through the writeback ioend. |

# 6. Dynamic Observation with bpftrace

iomap exposes a small, stable tracepoint surface in
`fs/iomap/trace.h`. Enumerate before instrumenting:

```bash
$ sudo bpftrace -l 'tracepoint:iomap:*'
tracepoint:iomap:iomap_readpage
tracepoint:iomap:iomap_readahead
tracepoint:iomap:iomap_writeback_folio
tracepoint:iomap:iomap_release_folio
tracepoint:iomap:iomap_invalidate_folio
tracepoint:iomap:iomap_dio_invalidate_fail
tracepoint:iomap:iomap_dio_rw_queued
tracepoint:iomap:iomap_zero_iter
tracepoint:iomap:iomap_iter_dstmap
tracepoint:iomap:iomap_iter_srcmap
tracepoint:iomap:iomap_add_to_ioend
tracepoint:iomap:iomap_iter
tracepoint:iomap:iomap_dio_rw_begin
tracepoint:iomap:iomap_dio_complete
```

Three patterns from the recipe — boundary count, latency histogram,
involved tasks.

**Boundary count — what's actually using iomap right now**

```bash
$ sudo bpftrace -e '
  tracepoint:iomap:iomap_iter {
    @[args->ino, kstack(2)] = count();
  }'
```

Each event is one iter step. Bucketing by `ino` plus the top two
stack frames tells you which inodes are active and via which entry
point (`iomap_file_buffered_write`, `__iomap_dio_rw`, ...).

**DIO latency histogram across `iomap_dio_rw_begin` /
`iomap_dio_complete`**

```bash
$ sudo bpftrace -e '
  tracepoint:iomap:iomap_dio_rw_begin {
    @start[tid] = nsecs;
  }
  tracepoint:iomap:iomap_dio_complete /@start[tid]/ {
    @us = hist((nsecs - @start[tid]) / 1000);
    delete(@start[tid]);
  }'
```

These two tracepoints are bracketing — `_begin` fires at the top of
`__iomap_dio_rw` and `_complete` fires inside `iomap_dio_complete`,
which runs on *whichever* task ends up completing the I/O (sync
waiter, AIO workqueue, or io_uring caller-completion). The histogram
therefore reflects end-to-end DIO latency including any deferred
completion hop.

**Involved tasks — who calls `iomap_writeback_folio`?**

```bash
$ sudo bpftrace -e '
  tracepoint:iomap:iomap_writeback_folio {
    @[comm, kstack] = count();
  }'
```

You should see writeback kthreads (`kworker/u*:*-writeback`,
`flush-<dev>`) dominate; if a syscall task (`fsync`, an application)
shows up, that's a synchronous writeback path (`__filemap_fdatawrite_range`).

Traces above were collected on Linux mainline at commit `d387b06f7c15`
on a Fedora 42 host with a writing fio workload over XFS on NVMe.

# Summary

**Main advantages**

- **One primitive, many ops.** Every iomap operation is
  `iomap_iter()` + `iomap_ops` + a per-step body. Adding a new
  operation (seek, swap, atomic-write, ...) costs a hundred lines, not
  a thousand.
- **Folio-native and `buffer_head`-free.** `iomap_folio_state` is one
  allocation with two bitmaps per folio, replacing a chain of
  `struct buffer_head`s. Sub-folio I/O works without `bh_lru`,
  `mark_buffer_dirty`, or per-block locking.
- **Extent-mapped, not block-mapped.** A single `struct iomap`
  describes a contiguous run of any length; the filesystem can return
  large mappings and amortise its lookup. `IOMAP_F_MERGED` /
  `IOMAP_F_BOUNDARY` flags compose cleanly.
- **Clean DIO/buffered/writeback split.** The three paths are sibling
  consumers of the same iter primitive, not interleaved in one
  function. `iomap_dio` supports inline (caller-context) completion
  for io_uring polling without disturbing the AIO workqueue path.
- **Invalidation-aware.** `IOMAP_F_STALE` + `validity_cookie` let a
  step body discover that the mapping has gone stale (truncate, COW
  remap) and re-issue `iomap_begin` without holding the FS extent
  lock across the whole I/O.

**Main problems / limitations**

- **Only filesystems that supply `iomap_ops` benefit.** ext4's
  default buffered path is still `buffer_head`; the rewrite is partial
  and gated on filesystem maintainer effort.
- **No `buffer_head` co-existence on the same inode.** A filesystem
  switches an inode to iomap or stays on `buffer_head`s — there is no
  per-range mix.
- **`iomap_begin` ordering is on the filesystem.** Race-free mapping
  between iomap and concurrent truncate / fallocate / COW remap is
  the filesystem's problem; iomap supplies `IOMAP_F_STALE` /
  `validity_cookie` but does not serialise on its own.
- **Writeback ordering is per-filesystem.** `iomap_sort_ioends` and
  `iomap_ioend_try_merge` provide the building blocks; XFS uses them
  to drive journal ordering, but other filesystems are on their own
  to compose them correctly.
- **Folio-only.** New iomap code paths assume the folio API. Older
  in-tree filesystems still on `struct page` or `buffer_head` cannot
  partially adopt — adoption is per-operation but each operation is
  folio-only.
- **DIO is intricate.** `direct-io.c` at 904 lines is the single
  densest file in the subsystem; the three completion modes (sync /
  workqueue / inline) plus `REQ_FUA` write-through, `IOMAP_DIO_BOUNCE`,
  `-ENOTBLK` fallback to buffered, and `IOMAP_DIO_PARTIAL` recovery
  make it the area most likely to surprise.
