## 1. Top View & Architecture

### 1.1 What Is a Dynptr?

A **dynptr** (dynamic pointer) is a BPF abstraction that lets BPF programs
safely access **variable-length, possibly non-contiguous memory** — such as
network packets, ring buffer entries, or file contents — through a single,
unified interface.

Think of dynptr as a **smart pointer with runtime bounds checking**, enforced
by the BPF verifier at load time and by the kernel at run time.

```
 ┌───────────────────────────────────────────────────────┐
 │                   BPF Program                         │
 │                                                       │
 │   struct bpf_dynptr ptr;      /* opaque, 16 bytes */  │
 │                                                       │
 │   bpf_dynptr_from_skb(skb, 0, &ptr);                 │
 │   bpf_dynptr_read(buf, len, &ptr, offset, 0);        │
 │   bpf_dynptr_slice(&ptr, offset, buf, sizeof(buf));   │
 │                                                       │
 └──────────────────────┬────────────────────────────────┘
                        │  helper / kfunc call
                        ▼
 ┌───────────────────────────────────────────────────────┐
 │               Kernel (helpers.c / filter.c)           │
 │                                                       │
 │   struct bpf_dynptr_kern {                            │
 │       void *data;     /* backing object (skb, etc) */ │
 │       u32   size;     /* bits 0-23: size              │
 │                          bits 28-30: type             │
 │                          bit 31: read-only flag */    │
 │       u32   offset;   /* current offset */            │
 │   };                                                  │
 │                                                       │
 │   Type-specific dispatch:                             │
 │     LOCAL / RINGBUF  →  direct memmove                │
 │     SKB              →  skb_header_pointer / load     │
 │     XDP              →  bpf_xdp_pointer / copy_buf   │
 │     SKB_META         →  bpf_skb_meta_pointer          │
 │     FILE             →  freader_fetch                 │
 └───────────────────────────────────────────────────────┘
```

**Source:** [`include/linux/bpf.h:1406-1435`](https://github.com/torvalds/linux/blob/master/include/linux/bpf.h)
 / [`kernel/bpf/helpers.c:1738-2990`](https://github.com/torvalds/linux/blob/master/kernel/bpf/helpers.c)

### 1.2 Dynptr Type Hierarchy

All six dynptr types share the same 16-byte opaque `struct bpf_dynptr` on the
BPF stack, but point to different backing objects:

```
                      bpf_dynptr
                          │
          ┌───────────────┼───────────────────────┐
          │               │                       │
     Memory-backed    Network-packet          File-backed
          │               │                       │
    ┌─────┴─────┐   ┌─────┼──────┐               │
    │           │   │     │      │               │
  LOCAL    RINGBUF  SKB  XDP  SKB_META         FILE
    │           │   │     │      │               │
 (direct    (direct (skb  (xdp  (skb          (freader
  memory)   ringbuf) frag  frag  metadata)     file I/O)
                    aware) aware)
```

| Type | Backing Object | Contiguous? | Writable? | Introduced |
|------|---------------|-------------|-----------|------------|
| `LOCAL` | Program memory / map value | Yes | Yes | v5.19 (2022-05) |
| `RINGBUF` | Ring buffer entry | Yes | Yes | v5.19 (2022-05) |
| `SKB` | `struct sk_buff` | **No** (fragments) | Depends on prog type | v6.3 (2023-03) |
| `XDP` | `struct xdp_buff` | **No** (multi-buffer) | Yes | v6.3 (2023-03) |
| `SKB_META` | skb metadata area | Yes | Yes | 2025-08 (`89d912e494f7`) |
| `FILE` | File contents | No (paged) | **No** (read-only) | 2025-10 (`8d8771dc03e4`) |

## 2. Background & Motivation

### 2.1 The Problem Before Dynptr

BPF programs had several fundamental limitations:

#### Problem 1: Fixed-size reservations in ring buffer

`bpf_ringbuf_reserve()` gives zero-copy access to ring buffer memory, but
requires a **compile-time constant** size (`ARG_CONST_ALLOC_SIZE_OR_ZERO`).
`bpf_ringbuf_output()` accepts dynamic sizes, but always **copies** the data:

```c
/* Zero-copy but size must be a compile-time constant */
void *sample = bpf_ringbuf_reserve(&ringbuf, 256, 0);
/* ↑ 256 is constant, verifier is happy */

/* This does NOT compile — size must be a constant */
void *sample = bpf_ringbuf_reserve(&ringbuf, dynamic_len, 0);

/* Dynamic size works, but always copies (extra memcpy) */
bpf_ringbuf_output(&ringbuf, buf, dynamic_len, 0);
```

#### Problem 2: Manual bounds checking for packets

Network packet parsing required tedious, error-prone manual bounds checking
against `data_end`:

```c
/* BEFORE: manual bounds checks everywhere */
void *data     = (void *)(long)skb->data;
void *data_end = (void *)(long)skb->data_end;

struct ethhdr *eth = data;
if ((void *)(eth + 1) > data_end)       /* check #1 */
    return TC_ACT_SHOT;

struct iphdr *ip = (void *)(eth + 1);
if ((void *)(ip + 1) > data_end)        /* check #2 */
    return TC_ACT_SHOT;

struct tcphdr *tcp = (void *)ip + ip->ihl * 4;
if ((void *)(tcp + 1) > data_end)       /* check #3 */
    return TC_ACT_SHOT;
```

#### Problem 3: Non-linear (fragmented) packet data inaccessible

SKB packet data can be split across the linear head area and paged
fragments. XDP multi-buffer packets span multiple fragments. Before dynptr,
BPF programs could **only access the linear portion** — the fragmented data
was invisible:

```
         sk_buff layout
 ┌──────────────────────┐
 │  Linear head area    │ ← ctx->data to ctx->data_end
 │  (headers, maybe     │   BPF can access this
 │   some payload)      │
 ├──────────────────────┤
 │  Paged fragments     │ ← BPF CANNOT access this!
 │  (rest of payload)   │   (non-linear data)
 │  frag[0], frag[1]... │
 └──────────────────────┘
```

### 2.2 How Dynptr Solves These Problems

Dynptr addresses all three problems with one abstraction:

| Problem | Dynptr Solution |
|---------|----------------|
| Fixed-size reserve or memcpy | `bpf_ringbuf_reserve_dynptr` allows dynamic size with zero-copy access |
| Manual bounds checks | `bpf_dynptr_read/slice` do bounds checking internally |
| Non-linear data invisible | `bpf_dynptr_slice` transparently copies fragmented data to a temp buffer |

### 2.3 Design Goals

The dynptr design had these core goals:

1. **Verifier-safe**: The verifier can prove memory safety without knowing
   the runtime size at load time
2. **Unified API**: Same `read/write/slice` API works for all buffer types
3. **Zero-copy when possible**: Direct pointers for contiguous data, copy
   only when necessary (fragmented data)
4. **Opaque to BPF programs**: Programs cannot corrupt dynptr internals
   (the `data`, `size`, `offset` fields are hidden)

## 3. Core Data Structures

### 3.1 User-facing Structure (UAPI)

What the BPF program sees — completely opaque:

```c
/* include/uapi/linux/bpf.h */
struct bpf_dynptr {
    __u64 __opaque[2];            /* 16 bytes, contents hidden */
} __attribute__((aligned(8)));
```

**Source:** [`include/uapi/linux/bpf.h:7482-7484`](https://github.com/torvalds/linux/blob/master/include/uapi/linux/bpf.h)

### 3.2 Kernel Implementation Structure

What the kernel actually stores inside those 16 bytes:

```c
/* include/linux/bpf.h:1406-1419 */
struct bpf_dynptr_kern {
    void *data;       /* pointer to backing object */
    u32   size;       /* see bit layout below */
    u32   offset;     /* current byte offset into data */
} __aligned(8);
```

The `size` field packs three pieces of information into 32 bits:

```
 Bit layout of bpf_dynptr_kern.size:
 ┌────┬─────────┬──────────┬───────────────────────────────┐
 │ 31 │ 30 - 28 │ 27 - 24  │        23  ...  0             │
 ├────┼─────────┼──────────┼───────────────────────────────┤
 │ RO │  type   │ reserved │  usable size (max 16MB - 1)   │
 └────┴─────────┴──────────┴───────────────────────────────┘
   │      │                          │
   │      │                          └─ DYNPTR_SIZE_MASK  = 0xFFFFFF
   │      └──────────────────────────── DYNPTR_TYPE_SHIFT = 28
   └─────────────────────────────────── DYNPTR_RDONLY_BIT  = BIT(31)
```

> **Note:** The kernel header comment says "Bits 24-30 = dynptr type", reserving
> the full range for future expansion. Currently only bits 28-30 are used
> (3 bits, enough for 7 types). Bits 24-27 are zero.

**Source:** [`kernel/bpf/helpers.c:1745-1751`](https://github.com/torvalds/linux/blob/master/kernel/bpf/helpers.c)

Constants:
```c
#define DYNPTR_MAX_SIZE     ((1UL << 24) - 1)   /* 16,777,215 bytes (~16 MB) */
#define DYNPTR_TYPE_SHIFT   28
#define DYNPTR_SIZE_MASK    0xFFFFFF
#define DYNPTR_RDONLY_BIT   BIT(31)
```

/* include/linux/bpf.h:1421-1435 */
enum bpf_dynptr_type {

    BPF_DYNPTR_TYPE_XDP,         /* 4 - xdp_buff packet */

    BPF_DYNPTR_TYPE_FILE,        /* 6 - file contents */

```

### 3.4 How Offset and Size Work Together

Dynptr uses an `offset` + `size` window to let you "zoom in" on a subset of the data:

```
 Backing object (e.g., packet data, 1500 bytes)
 ┌───────────────────────────────────────────────────────┐
 │aaaaa│▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓│bbbbbbbbb│
 └───────────────────────────────────────────────────────┘
       ↑                                     ↑
       offset                          offset + size
       (start of accessible region)    (end of accessible region)

 Initial state:   offset = 0,  size = total_length  (e.g., 1500)
 After adjust(5, 1491):  offset += 5 → 5,  size = 1491-5 = 1486
     (trimmed 5 bytes from head, 9 bytes from tail)
```

The `bpf_dynptr_adjust(start, end)` kfunc lets you narrow this window —
useful for skipping headers or trimming trailers. Note that `adjust()`
**advances** the offset by `start` and sets the new size to `end - start`.
It cannot expand the window beyond the original bounds.

#### `bpf_dynptr_from_mem()` — Create LOCAL dynptr

```c
/* kernel/bpf/helpers.c:1846-1869 */
BPF_CALL_4(bpf_dynptr_from_mem, void *, data, u64, size,
           u64, flags, struct bpf_dynptr_kern *, ptr)
```

| Parameter | Description |
|-----------|------------|
| `data` | Pointer to BPF program memory or map value |
| `size` | Number of bytes |
| `flags` | Must be 0 (reserved for future use) |
| `ptr` | Output: uninitialized dynptr to populate |
| **Returns** | 0 on success, `-E2BIG` if size > 16MB, `-EINVAL` if flags ≠ 0 |

#### `bpf_dynptr_from_skb()` — Create SKB dynptr

```c
/* net/core/filter.c:12135-12149 */
__bpf_kfunc int bpf_dynptr_from_skb(struct __sk_buff *s, u64 flags,
                                    struct bpf_dynptr *ptr__uninit)
```

Creates a dynptr wrapping the entire packet data (including fragments).
The dynptr size equals `skb->len` (not just the linear head).

> **Verifier trick:** For program types that cannot write packet data
> (e.g., `sched_cls` without direct packet access), the verifier silently
> **replaces** `bpf_dynptr_from_skb` with `bpf_dynptr_from_skb_rdonly`
> at fixup time ([`verifier.c:23134-23144`](https://github.com/torvalds/linux/blob/master/kernel/bpf/verifier.c)).
> This wrapper calls `bpf_dynptr_from_skb()` then sets the read-only bit,
> preventing writes through `bpf_dynptr_write()` or `bpf_dynptr_slice_rdwr()`.

**Source:** [`net/core/filter.c:12135`](https://github.com/torvalds/linux/blob/master/net/core/filter.c)

#### Bounds Checking: `bpf_dynptr_check_off_len()`

Every data access function (`read`, `write`, `data`, `slice`) calls this
inline function first to validate bounds:

```c
/* include/linux/bpf.h:1447-1455 */
static inline int bpf_dynptr_check_off_len(
    const struct bpf_dynptr_kern *ptr, u64 offset, u64 len)
{
    u64 size = __bpf_dynptr_size(ptr);

    if (len > size || offset > size - len)
        return -E2BIG;

    return 0;
}
```

This check ensures `offset + len <= size` without integer overflow
(by rearranging to `offset > size - len`). The `size` is the **usable
size** of the dynptr (already accounting for any prior `adjust()` calls),
not the total backing buffer size.

```
  Dynptr accessible region (after adjust):
  ┌──────────────────────────────┐
  │          size bytes          │
  │     ┌──────────┐            │
  │     │ len bytes│            │  ✓ OK: offset + len ≤ size
  │     └──────────┘            │
  │     ↑                       │
  │   offset                    │
  └──────────────────────────────┘

  ┌──────────────────────────────┐
  │          size bytes          │
  │              ┌───────────────┼──┐
  │              │  len bytes    │  │  ✗ FAIL: offset + len > size
  │              └───────────────┼──┘
  │              ↑               │
  │            offset            │
  └──────────────────────────────┘
```

#### `bpf_dynptr_read()` — Copy data out from dynptr

```c
/* kernel/bpf/helpers.c:1881-1924 */
BPF_CALL_5(bpf_dynptr_read, void *, dst, u64, len,
           const struct bpf_dynptr_kern *, src, u64, offset, u64, flags)
```

| Parameter | Description |
|-----------|------------|
| `dst` | Destination buffer (program memory) |
| `len` | Number of bytes to read |
| `src` | Source dynptr |
| `offset` | Byte offset into dynptr data |
| `flags` | Must be 0 |
| **Returns** | 0 on success, negative errno on failure |

Internally dispatches by type:
- **LOCAL/RINGBUF**: `memmove(dst, data + offset, len)` — simple copy
- **SKB**: `__bpf_skb_load_bytes()` — handles non-linear skb data
- **XDP**: `__bpf_xdp_load_bytes()` — handles XDP multi-buffer
- **SKB_META**: `memmove()` via `bpf_skb_meta_pointer()` — metadata area
- **FILE**: `bpf_file_fetch_bytes()` — reads from file via `freader`

**Source:** [`kernel/bpf/helpers.c:1881`](https://github.com/torvalds/linux/blob/master/kernel/bpf/helpers.c)

#### `bpf_dynptr_write()` — Copy data into dynptr

```c
/* kernel/bpf/helpers.c:1937-1983 */
BPF_CALL_5(bpf_dynptr_write, const struct bpf_dynptr_kern *, dst,
           u64, offset, void *, src, u64, len, u64, flags)
```

Same dispatch pattern as `read`, but:
- Fails with `-EINVAL` if dynptr is **read-only** (e.g., SKB in certain program types, FILE)
- SKB write uses `__bpf_skb_store_bytes()` with flags support (e.g., `BPF_F_RECOMPUTE_CSUM`)

#### `bpf_dynptr_data()` — Get direct pointer (LOCAL/RINGBUF only)

```c
/* kernel/bpf/helpers.c:1996-2026 */
BPF_CALL_3(bpf_dynptr_data, const struct bpf_dynptr_kern *, ptr,
           u64, offset, u64, len)
```

Returns a **direct pointer** to the data. Only works for LOCAL and RINGBUF
types (where data is guaranteed contiguous). Returns `NULL` for:
- **SKB/XDP/SKB_META**: use `bpf_dynptr_slice()` instead
- **Read-only dynptrs**: even LOCAL/RINGBUF return NULL if read-only

#### `bpf_dynptr_slice()` — Get pointer with optional buffer fallback

```c
/* kernel/bpf/helpers.c:2807-2853 */
__bpf_kfunc void *bpf_dynptr_slice(const struct bpf_dynptr *p, u64 offset,
                                   void *buffer__nullable, u64 buffer__szk)
```

This is the **most versatile** data access function. It returns a pointer to the data
at `offset`, but the pointer might point to:

1. **Directly into the backing data** (if data is contiguous at that offset) — zero copy
2. **Into the user-provided `buffer`** (if data is fragmented and had to be copied)

> **`__szk` = constant size:** The `buffer__szk` parameter **must be a
> compile-time constant** (e.g. `sizeof(struct ethhdr)`). The `__szk` suffix
> tells the verifier to enforce `tnum_is_const()` on this argument
> ([`verifier.c:13760`](https://github.com/torvalds/linux/blob/master/kernel/bpf/verifier.c)).
> This is required because the verifier must know the exact returned pointer
> bounds at load time to track memory safety for all subsequent accesses.
>
> ```c
> // OK — sizeof is a compile-time constant
> eth = bpf_dynptr_slice(&ptr, off, buf, sizeof(*eth));
>
> // REJECTED — "R5 must be a known constant"
> eth = bpf_dynptr_slice(&ptr, off, buf, dynamic_len);
> ```
>
> For variable-length reads, use `bpf_dynptr_read()` instead — its `len`
> argument (`ARG_CONST_SIZE_OR_ZERO`) allows runtime values as long as the
> destination buffer is large enough.

```
 Case 1: Data is contiguous (LOCAL, RINGBUF, or linear SKB head)
 ┌──────────────────────────────┐
 │         backing data         │
 │     ▲                        │
 │     │ returned pointer       │
 │     │ points here directly   │
 └──────────────────────────────┘

 Case 2: Data is fragmented (SKB paged area, XDP fragments)
 ┌──────────────┐ ┌──────────────┐
 │  SKB head    │ │  fragment    │
 │              │ │              │
 └──────────────┘ └──────┬───────┘
                         │ copied to buffer
                         ▼
              ┌──────────────────┐
              │  user buffer     │  ← returned pointer
              └──────────────────┘
```

**Source:** [`kernel/bpf/helpers.c:2807`](https://github.com/torvalds/linux/blob/master/kernel/bpf/helpers.c)

#### `bpf_dynptr_slice_rdwr()` — Writable version of slice

```c
/* kernel/bpf/helpers.c:2897-2928 */
__bpf_kfunc void *bpf_dynptr_slice_rdwr(const struct bpf_dynptr *p, u64 offset,
                                        void *buffer__nullable, u64 buffer__szk)
```

Same as `bpf_dynptr_slice()` but returns a **writable** pointer. Key usage pattern:

```c
struct ethhdr *eth = bpf_dynptr_slice_rdwr(&dynptr, 0, buf, sizeof(buf));
if (!eth)
    return TC_ACT_SHOT;

/* modify eth header */
eth->h_proto = htons(ETH_P_IP);

/* If data was copied to buffer, persist changes */
if (eth == buf)
    bpf_dynptr_write(&dynptr, 0, buf, sizeof(buf), 0);
```

#### When to Use `bpf_dynptr_data()` vs `bpf_dynptr_slice()`

```
  ┌───────────────┐      LOCAL or RINGBUF?       ┌───────────────────┐
  │  Need direct   │ ──── Yes ─────────────────→ │ bpf_dynptr_data() │
  │  pointer to    │                              │  Always direct    │
  │  dynptr data   │                              │  pointer, writable│
  │                │                              └───────────────────┘
  │                │ ──── No (SKB/XDP/FILE) ──→  ┌───────────────────┐
  │                │                              │ bpf_dynptr_slice()│
  │                │                              │  May copy to buf  │
  └───────────────┘                              └───────────────────┘
```

- Use `bpf_dynptr_data()` for LOCAL/RINGBUF — it's a classic BPF helper
  (not a kfunc), returns a direct writable pointer, and is the simplest API.
- Use `bpf_dynptr_slice()` for any type — it's a kfunc that works
  universally. For LOCAL/RINGBUF it returns a direct pointer (same as
  `bpf_dynptr_data`). For SKB/XDP/FILE it may copy to the user-provided
  buffer.

### 4.4 Ring Buffer Functions

```
  bpf_ringbuf_reserve_dynptr(map, size, flags, &ptr)
       │
       ├──→ Reserves `size` bytes in ring buffer
       ├──→ Initializes dynptr with type = RINGBUF
       └──→ Must be followed by EXACTLY ONE of:

  bpf_ringbuf_submit_dynptr(&ptr, flags)     ← commit to consumer
  bpf_ringbuf_discard_dynptr(&ptr, flags)    ← discard reservation
```

The verifier **enforces** that every reserved dynptr is either submitted or
discarded before the program exits (tracked via reference counting).

**Source:** [`kernel/bpf/ringbuf.c:670-739`](https://github.com/torvalds/linux/blob/master/kernel/bpf/ringbuf.c)

### 4.5 Tracing / Probe-Read Kfuncs

BPF tracing programs can copy data from user/kernel space into a dynptr.
These kfuncs chunk the copy in 256-byte blocks, writing into the dynptr
at a given offset:

| Kfunc | Source | Sleepable? |
|-------|--------|------------|
| `bpf_probe_read_user_dynptr` | User memory (nofault) | No |
| `bpf_probe_read_kernel_dynptr` | Kernel memory (nofault) | No |
| `bpf_probe_read_user_str_dynptr` | User string (nofault) | No |
| `bpf_probe_read_kernel_str_dynptr` | Kernel string (nofault) | No |
| `bpf_copy_from_user_dynptr` | User memory | Yes |
| `bpf_copy_from_user_str_dynptr` | User string | Yes |
| `bpf_copy_from_user_task_dynptr` | Other task's user memory | Yes |

**Source:** [`kernel/trace/bpf_trace.c:3538-3594`](https://github.com/torvalds/linux/blob/master/kernel/trace/bpf_trace.c)

These are useful in tracing/LSM programs to safely read variable-length data
(e.g. filenames, command arguments) into a dynptr-backed buffer.

### 4.6 File Dynptr Variants

The FILE dynptr has both non-sleepable and sleepable initialization kfuncs:

```c
/* kernel/bpf/helpers.c:4433-4441 */
__bpf_kfunc int bpf_dynptr_from_file(struct file *file, u32 flags,
                                     struct bpf_dynptr *ptr__uninit);

__bpf_kfunc int bpf_dynptr_from_file_sleepable(struct file *file, u32 flags,
                                               struct bpf_dynptr *ptr__uninit);
```

> **Verifier trick:** Similar to SKB dynptrs, the verifier automatically
> replaces `bpf_dynptr_from_file` with `bpf_dynptr_from_file_sleepable`
> when the program is in a sleepable context
> ([`verifier.c:23151-23153`](https://github.com/torvalds/linux/blob/master/kernel/bpf/verifier.c)).
> BPF programs always call `bpf_dynptr_from_file()` — the verifier picks
> the right variant.

FILE dynptrs are always **read-only** and must be released with
`bpf_dynptr_file_discard()`. Internally they use a `struct bpf_dynptr_file_impl`
with 64-bit offset and size (overriding the 24-bit limit of normal dynptrs),
enabling access to files larger than 16 MB.

> **Implementation detail:** For most dynptr types, `read()/slice()` compute
> the final offset as `ptr->offset + user_offset`. For FILE dynptrs, the
> offset is stored inside `bpf_dynptr_file_impl->offset` and handled by
> `bpf_file_fetch_bytes()` internally — the `read()` call passes only the
> user offset, not `ptr->offset + user_offset`.

```c
/* kernel/bpf/helpers.c:1738-1743 */
struct bpf_dynptr_file_impl {
    struct freader freader;  /* kernel file reader */
    u64 offset;              /* 64-bit file offset */
    u64 size;                /* 64-bit file size */
};
```

#### Example: Reading a Process's Executable File from an LSM Hook

This example from the kernel selftests
([`tools/testing/selftests/bpf/progs/file_reader.c`](https://github.com/torvalds/linux/blob/master/tools/testing/selftests/bpf/progs/file_reader.c))
shows a BPF LSM program that reads a process's executable binary via
FILE dynptr and verifies its contents against a userspace copy.

**The problem:** An LSM hook wants to inspect the contents of the file
being executed — for example, to compute a hash for integrity checking
or to scan for signatures. Before FILE dynptr, BPF programs had no way
to read file contents.

**The solution:** Use `bpf_get_task_exe_file()` to get the executable's
`struct file *`, wrap it with `bpf_dynptr_from_file()`, then read
contents with `bpf_dynptr_read()`.

```c
SEC("lsm/file_open")
int on_open_validate_file_read(void *c)
{
    struct file *file;
    struct bpf_dynptr dynptr;
    char buf[4096];

    /* Get the current process's executable file */
    file = bpf_get_task_exe_file(bpf_get_current_task_btf());
    if (!file)
        return 0;

    /* Create a read-only FILE dynptr wrapping the file contents.
     * The verifier auto-selects the sleepable variant when the
     * program runs in a sleepable context (e.g., task_work callback).
     */
    if (bpf_dynptr_from_file(file, 0, &dynptr))
        goto out;

    /* Read 4096 bytes from offset 0 — the ELF header and beyond.
     * bpf_dynptr_read() internally uses freader_fetch() to read
     * from the file's page cache, bounded by the dynptr's size.
     */
    if (bpf_dynptr_read(buf, sizeof(buf), &dynptr, 0, 0))
        goto cleanup;

    /* Inspect buf[] — e.g., check ELF magic, compute hash, etc. */
    if (buf[0] != 0x7f || buf[1] != 'E' || buf[2] != 'L' || buf[3] != 'F')
        goto cleanup;  /* not an ELF file */

    /* Read from a different offset (e.g., skip to section headers) */
    if (bpf_dynptr_read(buf, 64, &dynptr, 4096, 0))
        goto cleanup;

    /* Use bpf_dynptr_adjust() to narrow the window, then read again */
    bpf_dynptr_adjust(&dynptr, 100, 200);  /* bytes 100-199 only */
    bpf_dynptr_read(buf, 100, &dynptr, 0, 0);  /* reads file[100..199] */

cleanup:
    /* Must release — FILE dynptrs are ref-counted */
    bpf_dynptr_file_discard(&dynptr);
out:
    bpf_put_file(file);
    return 0;
}
```

**Key points:**

```
  bpf_get_task_exe_file()
       │
       ▼
  struct file *
       │
       │ bpf_dynptr_from_file(file, 0, &dynptr)
       ▼
  ┌────────────────────────────────────────────┐
  │  FILE dynptr                               │
  │  data → bpf_dynptr_file_impl               │
  │    └─ freader (kernel file reader)         │
  │    └─ offset = 0                           │
  │    └─ size = U64_MAX (unbounded)           │
  │  read-only, ref-counted                    │
  └────────────────┬───────────────────────────┘
                   │
    bpf_dynptr_read(buf, len, &dynptr, offset, 0)
                   │
                   ▼
  ┌────────────────────────────────────────────┐
  │  freader_fetch(&freader, offset, len)      │
  │    → reads from file's page cache          │
  │    → copies to BPF program's buf           │
  └────────────────────────────────────────────┘
                   │
    bpf_dynptr_file_discard(&dynptr)
                   │
                   ▼
  ┌────────────────────────────────────────────┐
  │  freader_cleanup() + bpf_mem_free()        │
  │  Release file reader resources             │
  └────────────────────────────────────────────┘
```

**Lifecycle rules:**
- `bpf_dynptr_from_file()` allocates internal state (`bpf_dynptr_file_impl`)
  via `bpf_mem_alloc` — this is why FILE dynptrs are **ref-counted**
- `bpf_dynptr_file_discard()` **must** be called before the program returns
  — the verifier enforces this, just like `bpf_ringbuf_discard_dynptr()`
- The file is NOT pinned by the dynptr — the caller must hold a reference
  (e.g., via `bpf_get_task_exe_file()`) for the duration of the dynptr's lifetime
- `bpf_dynptr_read()` and `bpf_dynptr_slice()` both work; `bpf_dynptr_write()`
  is rejected (FILE dynptrs are read-only)

**Source:** [`tools/testing/selftests/bpf/progs/file_reader.c`](https://github.com/torvalds/linux/blob/master/tools/testing/selftests/bpf/progs/file_reader.c)

### 4.7 Program Type Compatibility

Not every dynptr type is available in every BPF program type. The kernel
registers dynptr kfuncs per program type
([`net/core/filter.c:12502-12515`](https://github.com/torvalds/linux/blob/master/net/core/filter.c)):

| BPF Program Type | SKB | XDP | SKB_META | LOCAL | RINGBUF |
|-----------------|-----|-----|----------|-------|---------|
| `SCHED_CLS` (TC) | Yes | — | Yes | Yes | Yes |
| `SCHED_ACT` | Yes | — | Yes | Yes | Yes |
| `XDP` | — | Yes | — | Yes | Yes |
| `SK_SKB` | Yes | — | — | Yes | Yes |
| `SOCKET_FILTER` | Yes | — | — | Yes | Yes |
| `CGROUP_SKB` | Yes | — | — | Yes | Yes |
| `LWT_OUT/IN/XMIT/SEG6LOCAL` | Yes | — | — | Yes | Yes |
| `NETFILTER` | Yes | — | — | Yes | Yes |
| `TRACING` | Yes | — | — | Yes | Yes |

> **Note:** `LOCAL` and `RINGBUF` dynptrs are available in all program types

> `FILE` dynptrs require sleepable program context for the sleepable variant.
>
> For SKB dynptrs, the verifier automatically sets the **read-only** flag
> for program types that cannot write packet data (determined by
> `may_access_direct_pkt_data()`).

## 5. Verifier Integration

The BPF verifier plays a crucial role in dynptr safety. It tracks dynptr state
on the BPF stack and enforces type safety and lifetime rules.

### 5.1 Stack Slot Tracking

A dynptr occupies **2 consecutive 8-byte stack slots** (16 bytes total).
The verifier marks these slots as `STACK_DYNPTR`:

```
  BPF Stack
  ┌───────────────┐  slot N
  │  STACK_DYNPTR │  first_slot = true
  │  (8 bytes)    │  dynptr type, dynptr_id
  ├───────────────┤  slot N+1
  │  STACK_DYNPTR │  first_slot = false
  │  (8 bytes)    │  same dynptr_id
  └───────────────┘
```

**Key verifier functions:**

| Function | File:Line | Purpose |
|----------|-----------|---------|
| `mark_stack_slots_dynptr()` | `verifier.c:762` | Mark 2 stack slots for a new dynptr |
| `is_dynptr_reg_valid_init()` | `verifier.c:985` | Validate dynptr is properly initialized |
| `is_dynptr_reg_valid_uninit()` | `verifier.c:957` | Check uninitialized dynptr for output |
| `is_dynptr_type_expected()` | `verifier.c:1014` | Type matches expected type |
| `process_dynptr_func()` | `verifier.c:8826` | Process helper/kfunc dynptr arguments |

**Source:** [`kernel/bpf/verifier.c`](https://github.com/torvalds/linux/blob/master/kernel/bpf/verifier.c)

### 5.2 What the Verifier Enforces

```
  ┌──────────────────────────────────────────────────────────────┐
  │                  Verifier Safety Checks                      │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  1. INITIALIZATION                                           │
  │     ✓ dynptr must be initialized before read/write/slice     │
  │     ✓ both stack slots must be marked STACK_DYNPTR            │
  │     ✓ first_slot flag must be set on the first slot          │
  │                                                              │
  │  2. TYPE SAFETY                                              │
  │     ✓ dynptr type must match expected type                   │
  │       (e.g., ringbuf_submit requires RINGBUF type)           │
  │     ✓ no writing to read-only dynptrs                        │
  │                                                              │
  │  3. LIFETIME / REFERENCE COUNTING                            │
  │     ✓ Only RINGBUF and FILE are ref-counted                  │
  │       (dynptr_type_refcounted() in verifier.c:729)           │
  │     ✓ RINGBUF dynptr must be submitted or discarded          │
  │     ✓ FILE dynptr must be released via discard               │
  │     ✓ cannot use dynptr after release                        │
  │     ✓ LOCAL/SKB/XDP/SKB_META: no lifetime tracking needed    │
  │                                                              │
  │  4. ANTI-CORRUPTION                                          │
  │     ✓ BPF programs cannot overwrite dynptr stack slots        │
  │       directly (only via helper/kfunc calls)                 │
  │     ✓ partial dynptr passing is rejected                     │
  │     ✓ creating a new dynptr at overlapping stack slots        │
  │       destroys the old dynptr first (verifier.c:773-787)     │
  │                                                              │
  │  5. SLICE TRACKING                                           │
  │     ✓ data slices are tied to a dynptr_id                    │
  │     ✓ writing to dynptr invalidates prior slices             │
  │                                                              │
  └──────────────────────────────────────────────────────────────┘
```

### 5.3 Verifier State for Dynptr

The verifier tracks dynptr state in `struct bpf_reg_state`:

```c
/* include/linux/bpf_verifier.h */
struct bpf_reg_state {
    ...
    u32 dynptr_id;              /* unique ID, ties slices to dynptr */
    struct {
        enum bpf_dynptr_type type;
        bool first_slot;        /* true for first of 2 stack slots */
    } dynptr;
    ...
};
```

When `bpf_dynptr_slice()` returns a pointer, the verifier assigns the same
`dynptr_id` to the returned register. If a subsequent `bpf_dynptr_write()`
modifies the dynptr, the verifier **invalidates** all registers sharing that
`dynptr_id` — preventing use of stale pointers.

## 7. Real-World Example: L4 Load Balancer Packet Processing

The best real-world example is `test_cls_redirect_dynptr.c` — a Cloudflare-style
L4 load balancer (GLB) that parses GUE-encapsulated packets in a TC (traffic
control) BPF program. This is a 982-line production-quality example.

**Source:** [`tools/testing/selftests/bpf/progs/test_cls_redirect_dynptr.c`](https://github.com/torvalds/linux/blob/master/tools/testing/selftests/bpf/progs/test_cls_redirect_dynptr.c)

### 7.1 The Problem

Parse incoming packets with this encapsulation stack:

```
 Incoming Packet:
 ┌──────┬──────┬──────┬──────────┬───────────────────────┐
 │ Eth  │ IPv4 │ UDP  │ GUE+GLB  │ Inner packet          │
 │ hdr  │ hdr  │ hdr  │ headers  │ (IPv4 or IPv6)        │
 └──────┴──────┴──────┴──────────┴───────────────────────┘
 ▲                                                       ▲
 │              some of this may be in                    │
 │              paged SKB fragments!                      │
```

Without dynptr: only linear head data accessible; need manual boundary checks.

With dynptr: parse the entire packet regardless of fragmentation.

### 7.2 Step-by-Step Code Walk

#### Step 1: Initialize SKB dynptr

```c
SEC("tc")
int cls_redirect(struct __sk_buff *skb)
{
    __u8 encap_buffer[sizeof(encap_headers_t)] = {};
    struct bpf_dynptr dynptr;
    __u64 off = 0;

    /* Create dynptr wrapping the entire packet */
    bpf_dynptr_from_skb(skb, 0, &dynptr);
```

This wraps `skb` in a dynptr. Now `dynptr` represents the full packet
(including paged fragments), with `size = skb->len`.

#### Step 2: Read encapsulation headers with slice

```c
    /* Get writable pointer to encapsulation headers */
    encap = bpf_dynptr_slice_rdwr(&dynptr, 0,
                                  encap_buffer, sizeof(encap_buffer));
    if (!encap)
        return TC_ACT_OK;
```

`bpf_dynptr_slice_rdwr()` returns either:
- **Direct pointer** to SKB linear head (fast path, zero-copy)
- **Pointer to `encap_buffer`** with data copied in (if headers span fragments)

#### Step 3: Parse and validate headers

```c
    if (encap->ip.ihl != 5)
        return TC_ACT_OK;

    if (encap->ip.daddr != ENCAPSULATION_IP ||
        encap->ip.protocol != IPPROTO_UDP)
        return TC_ACT_OK;

    if (encap->udp.dest != ENCAPSULATION_PORT)
        return TC_ACT_OK;
```

Direct struct field access — no manual bounds checking needed.

#### Step 4: Parse inner packet (passing dynptr + offset)

```c
    switch (encap->gue.proto_ctype) {
    case IPPROTO_IPIP:
        verdict = process_ipv4(skb, &dynptr, &off, metrics);
        break;
    case IPPROTO_IPV6:
        verdict = process_ipv6(skb, &dynptr, &off, metrics);
        break;
    }
```

The dynptr is passed to helper functions along with an offset — each function
advances the offset as it consumes headers. This is the equivalent of
incrementing a pointer, but with bounds safety.

Here is how `pkt_parse_ipv4()` uses this pattern
([`test_cls_redirect_dynptr.c:125-139`](https://github.com/torvalds/linux/blob/master/tools/testing/selftests/bpf/progs/test_cls_redirect_dynptr.c)):

```c
static int pkt_parse_ipv4(struct bpf_dynptr *dynptr,
                          __u64 *offset, struct iphdr *iphdr)
{
    /* Read IPv4 header from dynptr at current offset */
    if (bpf_dynptr_read(iphdr, sizeof(*iphdr), dynptr, *offset, 0))
        return -1;

    /* Advance offset past the header */
    *offset += sizeof(*iphdr);

    if (iphdr->ihl < 5)
        return -1;

    /* Skip IPv4 options if present */
    *offset += (iphdr->ihl - 5) * 4;

    return 0;
}
```

Notice: **no bounds checking code at all** — `bpf_dynptr_read()` returns
an error if the read would go out of bounds, and the caller simply propagates
the failure. Compare this with the traditional approach of casting
`data + offset` and checking `> data_end` for every header.

#### Step 5: Persist modifications if data was buffered

```c
    ret = accept_locally(skb, encap);

    /* If slice returned a buffer copy, write changes back */
    if (encap == encap_buffer)
        bpf_dynptr_write(&dynptr, 0, encap_buffer,
                         sizeof(encap_buffer), 0);
    return ret;
```

This is the critical pattern: if `bpf_dynptr_slice_rdwr()` had to copy data
into the buffer (because it was in a fragment), any modifications to that
buffer must be explicitly written back.

### 7.3 Data Flow Diagram

```
  ┌────────────┐     TC hook
  │ NIC / skb  │ ──────────→ cls_redirect()
  └────────────┘                    │
                                    │ ① bpf_dynptr_from_skb()
                                    ▼
                            ┌──────────────┐
                            │  dynptr       │
                            │  data → skb   │
                            │  size = len   │
                            └──────┬───────┘
                                   │
                                   │ ② bpf_dynptr_slice_rdwr()
                                   ▼
                    ┌──────────────────────────────┐
                    │  Returns ptr to encap hdrs   │
                    │  (direct or via buffer copy) │
                    └──────────────┬───────────────┘
                                   │
                                   │ ③ Parse encap + GUE + GLB
                                   ▼
                    ┌──────────────────────────────┐
                    │  process_ipv4/ipv6()          │
                    │  - reads inner packet via     │
                    │    bpf_dynptr_read()           │
                    │  - classifies: SYN/EST/ICMP   │
                    └──────────────┬───────────────┘
                                   │
                                   │ ④ Verdict
                                   ▼
                    ┌──────────────────────────────┐
                    │  ACCEPT → accept_locally()   │
                    │  FORWARD → forward_next_hop()│

```

### 7.4 Comparison: Without Dynptr vs. With Dynptr

| Aspect | Without Dynptr | With Dynptr |
|--------|---------------|-------------|
| Bounds checking | Manual `data + N > data_end` checks (8+ checks) | Automatic, inside `slice()/read()` |
| Fragment access | Only linear head visible | Full packet accessible |
| Header parsing | Cast pointer + check boundary | `bpf_dynptr_slice()` returns struct pointer |
| Code to parse TCP | ~50 lines with bounds checks | ~15 lines with dynptr |
| Helper functions | Must pass `data`, `data_end`, `offset` separately | Pass `&dynptr` + `&offset` |
| Write-back | Not needed (direct access only) | `if (ptr == buffer) bpf_dynptr_write()` |

## 8. Another Example: Ring Buffer with Dynamic Size

This example shows how dynptr enables **dynamically-sized ring buffer
reservations** — something impossible with the original `bpf_ringbuf_reserve()`.

**Source:** [`tools/testing/selftests/bpf/progs/dynptr_success.c:37-66`](https://github.com/torvalds/linux/blob/master/tools/testing/selftests/bpf/progs/dynptr_success.c)

```c
SEC("tp/syscalls/sys_enter_nanosleep")
int test_read_write(void *ctx)
{
    char write_data[64] = "hello there, world!!";
    char read_data[64] = {};
    struct bpf_dynptr ptr;

    /* Reserve space in ring buffer via dynptr
     * (here sizeof is constant, but the API accepts any runtime value
     *  since arg2 is ARG_ANYTHING, unlike bpf_ringbuf_reserve)
     */
    bpf_ringbuf_reserve_dynptr(&ringbuf, sizeof(write_data), 0, &ptr);

    /* Write data into the reserved space via dynptr */
    err = bpf_dynptr_write(&ptr, 0, write_data, sizeof(write_data), 0);

    /* Read it back via dynptr */
    err = err ?: bpf_dynptr_read(read_data, sizeof(read_data), &ptr, 0, 0);

    /* Verify correctness */
    for (i = 0; i < sizeof(read_data); i++) {
        if (read_data[i] != write_data[i]) {
            err = 1;
            break;
        }
    }

    /* Must discard or submit — verifier enforces this! */
    bpf_ringbuf_discard_dynptr(&ptr, 0);
    return 0;
}
```

Data flow:

```
  bpf_ringbuf_reserve_dynptr()     bpf_dynptr_write()
          │                               │
          ▼                               ▼
  ┌──────────────┐                ┌──────────────┐
  │  Ring Buffer  │                │  Ring Buffer  │
  │  ┌────────┐  │  write_data   │  ┌────────┐  │
  │  │ empty  │  │ ────────────→ │  │ hello  │  │
  │  │ slot   │  │                │  │ there  │  │
  │  └────────┘  │                │  └────────┘  │
  └──────────────┘                └──────────────┘
          │                               │
          ▼                               ▼
  dynptr.data → slot              dynptr.data → slot
  dynptr.size = 64                (data now filled)
  dynptr.type = RINGBUF
```

## 11. Performance Trade-offs

Dynptr helpers are **kernel function calls** — they cannot be inlined into
JIT-compiled BPF code. This introduces overhead compared to traditional
direct packet access.

### 11.1 Direct Packet Access vs. Dynptr

**Direct packet access** (traditional, fastest path):

```c
void *data     = (void *)(long)skb->data;      // single load
void *data_end = (void *)(long)skb->data_end;   // single load
if (data + 14 > data_end)                        // single compare
    return TC_ACT_SHOT;
struct ethhdr *eth = data;                       // zero cost (cast)
eth->h_proto;                                    // direct memory load
```

The verifier converts `ctx->data`/`ctx->data_end` into JIT loads. After the
bounds check, every field access is a **single-instruction memory load** — no
function call.

**Dynptr access** (function call per API invocation):

```c
bpf_dynptr_from_skb(skb, 0, &ptr);              // CALL into kernel
bpf_dynptr_read(&eth, 14, &ptr, 0, 0);          // CALL into kernel
                                                 //  → null check
                                                 //  → bpf_dynptr_check_off_len()
                                                 //  → switch(type) dispatch
                                                 //  → __bpf_skb_load_bytes()
```

Each `bpf_dynptr_read()` or `bpf_dynptr_slice()` call has this overhead:

1. **Function call** — register save/restore, call/ret
2. **Null check** on `ptr->data`
3. **Bounds check** via `bpf_dynptr_check_off_len()`
4. **Type dispatch** via `switch(bpf_dynptr_get_type(ptr))`
5. **Data copy** (for `read()`) or pointer calculation (for `slice()`)

There is no JIT inlining for dynptr — no `convert_ctx_access` or equivalent
transformation exists in the verifier for dynptr operations.

### 11.2 Mitigating the Cost with `bpf_dynptr_slice()`

The slice pattern **amortizes** the per-access cost:

```c
// ONE function call to get a direct pointer
struct ethhdr *eth = bpf_dynptr_slice(&dynptr, 0, buf, sizeof(*eth));

// Then ALL subsequent field accesses are direct memory loads — same cost
// as traditional direct packet access, NO function calls:
eth->h_proto;    // direct load
eth->h_source;   // direct load
eth->h_dest;     // direct load
```

Once `bpf_dynptr_slice()` returns a pointer, all subsequent accesses through
that pointer are direct memory loads — identical to the traditional approach.
The overhead is **one function call per slice**, not one per field.

### 11.3 Cost Comparison

```
  ┌─────────────────────────────────────────────────────────────────┐
  │                   Cost per header access                        │
  ├──────────────────────────┬──────────────────────────────────────┤
  │  Method                  │  Cost                                │
  ├──────────────────────────┼──────────────────────────────────────┤
  │  Direct data/data_end    │  1 compare + direct loads            │
  │  (traditional)           │  Fastest. Linear head only.          │
  ├──────────────────────────┼──────────────────────────────────────┤
  │  bpf_dynptr_slice()      │  1 function call, then direct loads  │
  │                          │  Good balance. Handles fragments.    │
  ├──────────────────────────┼──────────────────────────────────────┤
  │  bpf_dynptr_read()       │  1 function call + memcpy per read   │
  │                          │  Always copies. Variable-length OK.  │
  ├──────────────────────────┼──────────────────────────────────────┤
  │  bpf_skb_load_bytes()    │  1 function call per read            │
  │  (pre-dynptr)            │  Similar cost to dynptr_read().      │
  └──────────────────────────┴──────────────────────────────────────┘
```

### 11.4 When Is the Overhead Worth It?

| Scenario | Recommendation |
|----------|---------------|
| Simple filter on linear headers | Direct `data`/`data_end` is faster |
| Complex multi-protocol parsing | Dynptr wins on ergonomics; `slice()` keeps speed close |
| Non-linear SKB data (paged fragments) | Dynptr is the clean option; `bpf_skb_load_bytes()` has similar cost |
| XDP multi-buffer fragments | Dynptr is the **only** option — no alternative API exists |
| Ring buffer with dynamic size | Dynptr is the **only** option for zero-copy + dynamic size |

The trade-off is **correctness and ergonomics vs. raw speed for the simple
case**. For high-performance packet filtering on linear packets, direct
access remains faster. Dynptr shines when you need fragment access, complex
parsing, or ring buffer flexibility.

## 12. Summary

BPF dynptr is a **unified abstraction** for safely accessing variable-length,
possibly non-contiguous memory from BPF programs. Its key design properties:

1. **Opaque 16-byte structure** — BPF programs cannot corrupt internals
2. **Type-dispatched I/O** — same API, different backends (memory, skb, xdp, file)
3. **Verifier-integrated** — stack slot tracking, lifetime enforcement, type safety
4. **Zero-copy when possible** — direct pointers for contiguous data, buffer
   fallback for fragments
5. **Extensible** — new types (SKB_META, FILE) added without changing the API

Dynptr transformed BPF packet parsing from tedious manual bounds checking into
clean, struct-oriented code — while simultaneously unlocking access to
fragmented packet data that was previously invisible to BPF programs.
