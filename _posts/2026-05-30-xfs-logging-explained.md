# Overview

XFS logs metadata, not data. Every change to an on-disk metadata
structure is first described to an in-memory log item, aggregated into
a checkpoint, written to a fixed log area on the device, and only
*later* applied back to its home block. This post explains that
pipeline end to end — from a single `creat() + write() + fsync()` down
through transactions, log items, the CIL, the on-disk log, the AIL,
and recovery — for kernel and storage engineers who already know VFS
and journaling concepts but want a working mental model of XFS's
specific machinery.

# The Running Question

A program does this:

```c
fd = creat("foo", 0644);
write(fd, data, len);
fsync(fd);
/* power fails one millisecond later */
```

After the machine comes back up, what guarantees does XFS make? In
particular:

- Does `foo` exist?
- If it does, does it contain the bytes `write()` produced?
- If it doesn't, can the filesystem still be mounted, or has the
  half-finished operation corrupted on-disk state?

The short answer is: **`foo` exists and contains the written bytes,
the filesystem mounts cleanly, and recovery does whatever home-block
work the system didn't manage to flush before the power went out.**
The rest of this post is the long answer — how the logging machinery
produces that property — and we revisit the same three-line program at
the end with every stage filled in.

# What XFS Logging Actually Guarantees

XFS does **metadata journaling with write-ahead logging**, not data
journaling. Two phrases worth pinning down before the mechanism:

- *Metadata journaling*: every modified metadata structure (inode
  core, AGF / AGI sector, btree block, free-list ring) is described
  to the log before its home block on the data device is overwritten.
  File data is **not** journaled — page-cache writeback puts it
  directly to its home blocks via the normal bio path. The log only
  ever holds the metadata that records *where* the data went.

- *Write-ahead logging (WAL)*: the log record describing a change
  must be durable on the log device *before* the change's home block
  is allowed to be written. This is the inviolable ordering rule of
  XFS — every other piece of the logging architecture exists to
  enforce, or take advantage of, this rule.

### Write-ahead logging — the rule in detail

WAL is a *discipline*, not a format. The rule above is the entirety
of it. Everything else in this section, and most of the rest of the
post, is a consequence of enforcing that rule mechanically.

**What the rule prevents.** Most metadata changes touch several home
blocks at once: a `create()` modifies an inode cluster, a dir block,
the AGI counters, possibly an inode-btree block. If the kernel
simply wrote those four blocks directly, a power loss mid-sequence
could leave the dir entry pointing at an inode the AGI counter
doesn't yet record as allocated — an inconsistency with no on-disk
evidence of what was *supposed* to happen. WAL eliminates this state
by writing a second copy of the change to a sequential log *first*;
recovery can then either replay the log onto the home blocks (if
the log record made it) or roll back to the pre-transaction state
(if it didn't). There is no third state where the home blocks
reflect a half-finished change with no log record to explain it.

**Why two writes is acceptable.** The log write is *sequential*
(one append to the log ring) and *batched* (one log record carries
many transactions via the CIL). The home-block writes are random,
but they can be **delayed and coalesced** because the log already
carries the change — ten transactions modifying the same AGF buffer
in one second produce ten log items aggregated into one log record,
and one home-block write at the AIL's convenience. WAL's cost is one
extra IO per *checkpoint*, not per *transaction*; for any realistic
workload that's a rounding error.

**What the log contains.** WAL is a discipline that admits multiple
encodings:

- *Physical*: log entire modified blocks. Simple to replay, wasteful
  of log space.
- *Logical*: log high-level operations ("create inode N"). Compact,
  but replay re-runs the operation and is hard to make deterministic.
- *Physiological* (delta): log byte-level deltas keyed to specific
  blocks. Compact like logical, deterministic-to-replay like
  physical. **This is what XFS does** — [`xfs_buf_log_item`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_buf_item.h#L39) carries
  a 1-bit-per-128-byte dirty bitmap over the buffer; the dirty bytes
  follow the format header on the wire. [`xfs_inode_log_item`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_inode_item.h#L16)
  carries the `ilf_fields` mask naming which inode regions changed.

**How the rule is enforced in the kernel.** Every dirty
[`xfs_buf`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_buf.h#L150)
has a pin counter (`b_pin_count`). When the buffer joins a CIL
context, the counter is incremented; when the matching checkpoint
reaches the AIL (i.e. the log record describing the change is
durable on the log device), the counter is decremented. While
pinned, [`xfs_buf_submit`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_buf.c#L1423) **refuses to write the buffer to its home
block**. That single `b_pin_count > 0` check is the kernel-side
enforcement of WAL — every other piece of XFS logging machinery
(CIL, AIL, the tail pin on log space, fsync's reliance on log force)
is a consequence or optimisation of this one invariant.

The crucial distinction the rest of this post leans on:

```
   "the log is durable"      ≠      "the home block is updated"

   ┌──────────────┐               ┌──────────────────────┐
   │  log device  │ ──┐           │ data device          │
   │  (xfs_log)   │   │  later    │ home blocks (AGF,    │
   │  RECORD #N   │ ──┴─ AIL ───► │  AGI, inode cluster, │
   └──────────────┘    push       │  dir block, …)       │
                                  └──────────────────────┘
   step 1:                        step 2:
   write log record;              flush dirty buffer to
   *now* the change is            its home block; *now*
   durable & recoverable.         recovery doesn't need
                                  to replay it.
```

After step 1 a power loss is recoverable: mount will scan the log,
find the unmatched record, and *replay* it into the home block. After
step 2 the log space behind that record can be reused — the change is
fully resident in its on-disk home.

`fsync(2)` and `sync(2)` only require step 1 to finish for the
involved metadata. Step 2 happens later, asynchronously, driven by
`xfsaild` (the AIL push kthread). This is why fsync is cheap on XFS:
it doesn't wait for inode cluster writes; it waits for the log to
cover the inode's log item.

The three contracts XFS logging delivers on:

1. **Crash consistency.** After a crash, the filesystem is either
   in its pre-transaction state or its post-transaction state for
   every metadata change ever logged. Never a half-modified
   structure.
2. **Ordering.** Within a single transaction, every change lands
   atomically: all log items in the transaction are emitted together
   in one log record (or set of records bracketed by
   `XLOG_START_TRANS` / `XLOG_COMMIT_TRANS`), so recovery sees the
   whole transaction or none of it.
3. **fsync durability.** When `fsync(fd)` returns, every metadata
   change the inode depends on is durable in the log (and the data
   blocks themselves have been flushed via the page cache).

# End-to-End Metadata Lifecycle

The pipeline a metadata change travels:

```
   User operation                                       fs/xfs/xfs_file.c
       │                                                fs/xfs/xfs_inode.c
       ▼
   Metadata modifications                               in-core: xfs_inode,
       │                                                xfs_buf, xfs_perag
       ▼
   Transaction                                          xfs_trans_alloc/commit
       │                                                fs/xfs/xfs_trans.c
       ▼
   Log items                                            xfs_inode_log_item,
       │                                                xfs_buf_log_item,
       │                                                xfs_efi_log_item, …
       ▼
   CIL (Committed Item List)                            xfs_cil, xfs_cil_ctx
       │                                                fs/xfs/xfs_log_cil.c
       ▼
   Physical log records                                 xlog_in_core (iclog),
       │                                                xlog_rec_header,
       │                                                xlog_op_header
       │                                                fs/xfs/xfs_log.c
       ▼
   AIL (Active Item List)                               xfsaild
       │                                                fs/xfs/xfs_trans_ail.c
       ▼
   Home metadata blocks                                 xfs_buf → submit_bio
                                                        (data device)
```

Each stage owns a different piece of state and has different
durability semantics:

| Stage              | Owner                | Durability       | Key struct                  | Key function                  |
|--------------------|----------------------|------------------|-----------------------------|-------------------------------|
| User operation     | VFS / userspace      | none             | `struct file` / `iov_iter`  | `xfs_vn_create`, `xfs_file_write_iter` |
| Metadata change    | thread, in core      | none             | `xfs_inode`, `xfs_buf`      | `xfs_dialloc`, `xfs_bmapi_write`       |
| Transaction        | thread, in core      | none             | `xfs_trans`                 | `xfs_trans_alloc`, `xfs_trans_commit`  |
| Log items          | dirty buffer/inode   | none (in core)   | `xfs_log_item` + subtypes   | `iop_format`, `iop_pin`                |
| CIL                | mount                | none (in core)   | `xfs_cil`, `xfs_cil_ctx`    | `xlog_cil_commit`, `xlog_cil_push_work`|
| Physical log       | log device           | **durable**      | `xlog_in_core`, `xlog_rec_header` | `xlog_sync`, `xlog_write`        |
| AIL                | mount, sorted by LSN | log-durable      | `xfs_ail`                   | `xfsaild`, `iop_push`                  |
| Home blocks        | data device          | fully durable    | `xfs_buf` → bio             | `xfs_buf_submit`, `xfs_iflush`         |

Reading down the table: a change is **in-core only** through the
first five stages, becomes **log-durable** the moment its CIL
checkpoint hits the iclog and is written, and is **fully home-block
durable** once `xfsaild` has flushed the relevant buffer. The log
sits between the in-core and the home-block worlds, and is what makes
crash consistency tractable: every change that's in either of those
two worlds is also recoverable from the log.

The rest of this post walks each row in order, then re-runs the
running example through every row at the end.

# User Operation → Transaction

Every metadata-modifying syscall enters XFS through a VFS hook, which
opens a transaction
([`xfs_trans_alloc`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_trans.c#L234)
or its higher-level wrappers like
[`xfs_trans_alloc_icreate`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_trans.c#L1191)),
mutates in-core metadata while attaching **log items** to the
transaction, and commits — handing the items to the CIL. Each
transaction reserves up-front the worst-case log bytes it might emit;
without that reservation the log could fill mid-transaction and
deadlock. Reservation is paid back at commit if less was used.

The captured bpftrace traces below come from
[`code/xfs/xfs-meta-observe-*.sh`]({{ site.baseurl }}/code/xfs/),
which the
[XFS Metadata Internals post]({{ site.baseurl }}/tech/2026/05/28/xfs-metadata-internals.html#operation-walkthroughs)
describes in detail. This section highlights what each operation
*logs* — the items that flow from [`xfs_trans_commit`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_trans.c#L909) into the CIL.

## `create()`

The running example's first syscall.
[`xfs_create`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_inode.c#L648)
opens one transaction that:

- allocates an inode (touches `inobt`, `finobt`, the AGI counters);
- initialises the new chunk if needed (emits an `ICREATE` intent);
- inserts a dir entry into the parent (modifies the parent inode +
  one dir-block buffer).

The single commit attaches: an [`xfs_icreate_log`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/libxfs/xfs_log_format.h#L1007) item, two
`xfs_buf_log_item` instances (AGI + dir-block), and two
`xfs_inode_log_item` instances (the new inode + the parent dir
inode). No `EFI/EFD` — `create` allocates, it doesn't free.

```
create        dp_ino=128 namelen=7            # VFS entry; parent root dir
  dialloc     (inobt/finobt/AGI dirtied)      # inode allocator
  dir_create  (parent dir-block edit)         # entry insert in parent
  cil_push    (checkpoint)                    # forced by sync
```

## `unlink()`

[`xfs_remove`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_inode.c#L1905)
splits into two paths the captured trace shows in one workload:

- **closed-file rm**: inode and its extents freed in-line within
  the single transaction. The frees go through *deferred ops* (EFI
  per extent + matching EFD when the chain runs), so the
  transaction sees both intent items.
- **open-unlinked rm**: only the inode and its parent's dir entry
  log; the *target* inode is parked on the AGI's `agi_unlinked[hash]`
  bucket. Extents stay allocated until last close drives the
  deferred cleanup.

```
remove        dp_ino=128 namelen=8                          # closed-file rm
remove        dp_ino=128 namelen=6                          # open-unlinked rm
  iu_bucket   agno=0 bucket=5 old=0xffffffff new=0x85       # park on bucket 5
  iu_bucket   agno=0 bucket=5 old=0x85 new=0xffffffff       # close drains it
  cil_push    (checkpoint)                                  # ONE push covers both
```

Both unlinks share *one* `cil_push` — the CIL aggregated the two
commits into one checkpoint.

## `rename()`

[`xfs_rename`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_inode.c#L2125)
is the most complex hot-path transaction: it may touch up to four
inodes (source dir, target dir, source inode, target inode if
overwriting) across multiple AGs. All of it lands in *one*
transaction so the on-disk log never shows an intermediate state
where the file is in neither directory or in both:

```
rename        src_dp=128(namelen=5) -> tgt_dp=128(namelen=5)
  dir_create  (target dir-block edit)        # new entry "r_dst"
  dir_remove  (source dir-block edit)        # old entry "r_src" gone
  cil_push    (checkpoint)                   # ONE push for the whole rename
```

The two dir edits, the parent inode log items, and (if blocks
were freed) the matching EFI/EFD all share one log record. That
atomicity is the property `renameat2(RENAME_EXCHANGE)` relies on.

## `truncate()`

[`xfs_setattr_size`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_iops.c#L909)
calls
[`xfs_itruncate_extents_flags`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_inode.c#L1036)
to free file extents. For a heavily-fragmented file the truncation
runs as a *sequence* of sub-transactions, each freeing a bounded
number of extents to stay within its log reservation. The
intent-chain machinery (BUI/BUD for bmap updates, EFI/EFD for the
free) makes the sequence restartable: a crash after sub-txn *k*
leaves an unmatched BUI in the log; recovery finishes the rest.

```
setattr_size  (VFS truncate entry)
  itrunc      newsize_fsb=0x0  (sub-txn extent free)
  defer       (intent chain advance)         # BUI/BUD pair drained
  defer       (intent chain advance)         # EFI/EFD pair drained
  cil_push    (checkpoint)
```

The two `defer` lines are the visible firing of the intent chain —
one for the bmap update, one for the free-space update. Each pair
goes intent → (work happens) → done; when the chain completes the
transaction commits.

## `fsync()`

[`xfs_file_fsync`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_file.c#L125)
is special: it doesn't open a new transaction *for the fsync itself*.
What it does is:

1. flush the data page-cache via `filemap_write_and_wait_range`;
2. if the inode is dirty in-core, log it in a new transaction so the
   inode's i_size and timestamps get a definite LSN;
3. call
   [`xfs_log_force_seq`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_log.c)
   to wait for the on-disk log head to cover the inode's CIL
   sequence (forces a CIL push if one isn't already in flight);
4. issue a `blkdev_issue_flush` device-cache flush.

```
fsync         pid=363 ino=164
log_force_seq seq=0x5                        # wait for CIL seq 5 on disk
cil_push      (checkpoint)                   # push happens to cover it
```

Note [`xfs_log_force_seq`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_log.c#L2970), not [`xfs_log_force`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_log.c#L2792) — modern fsync targets a
specific CIL sequence, not "force everything". The home-block
writeback of the inode cluster is *not* in this sequence; the AIL
handles it later. fsync's durability promise rests entirely on the
log force.

# Log Items

XFS does **not** log arbitrary metadata blocks. Logging a 4 KiB
buffer for every 16-byte change would balloon the log without
bound. Instead, each modification produces a **log item** — a small
in-core object describing the change — and on commit the item
*formats itself* into the log, emitting only the dirty bytes plus
a bit of structural framing.

A log item is the abstract base type
[`xfs_log_item`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_trans.h#L31):
a pointer to the owning AIL, the LSN, flags, and an **ops vector**
implemented per item subtype. The ops vector hooks the item into
the lifecycle:

```
   iop_format      called at CIL push: serialise item to on-wire bytes
   iop_pin         called when item joins a CIL ctx: prevent writeback
                   of underlying buffer until log record is durable
   iop_unpin       called when CIL ctx reaches AIL: writeback may proceed
   iop_push        called by xfsaild: write the underlying home block
   iop_committed   called when home block is durable: drop item from AIL
```

Three subtypes carry most of the traffic:

## `xfs_inode_log_item`

One per dirty in-core inode.
[`xfs_inode_log_item`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_inode_item.h#L16)
tracks an `ili_fields` bitmask of which inode regions changed
([`XFS_ILOG_CORE`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/libxfs/xfs_log_format.h#L316),
[`XFS_ILOG_DDATA`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/libxfs/xfs_log_format.h#L317),
[`XFS_ILOG_DEXT`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/libxfs/xfs_log_format.h#L318),
[`XFS_ILOG_DBROOT`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/libxfs/xfs_log_format.h#L319),
the analogous attr-fork bits,
[`XFS_ILOG_TIMESTAMP`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/libxfs/xfs_log_format.h#L334)).
On format, only the regions named by the mask are emitted as
[`xfs_inode_log_format`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/libxfs/xfs_log_format.h#L274)
followed by the dirty bytes.

A `touch -c` on an existing file produces an inode log item with
just `XFS_ILOG_CORE` set — the wire payload is 56 B (format header)
+ 176 B (v5 inode core) = 232 bytes total, against a 512 B inode
and a 4 KiB cluster. That's the structural reason XFS log volume
tracks the *amount of change*, not the size of the metadata.

## `xfs_buf_log_item`

One per dirty metadata buffer.
[`xfs_buf_log_item`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_buf_item.h#L39)
holds a 1-bit-per-128-bytes dirty bitmap (`blf_data_map`) over the
buffer. On format
([`xfs_buf_log_format`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/libxfs/xfs_log_format.h#L527)
+ the dirty regions): only the chunks whose bits are set get written.
A 16-byte change to an AGF buffer produces a buffer log item of
~88 B header + 128 B (one dirty chunk) = ~216 B on the wire, not
the full 4 KiB AGF.

The buffer's *type* (AGF / AGI / DINO / DIR_LEAF / BTREE / …) is
packed into the top 5 bits of `blf_flags` as a **blft** code, so
recovery can re-verify the buffer's CRC after replay by knowing
what kind of metadata it is.

## Intent / done items (EFI/EFD, BUI/BUD, RUI/RUD, CUI/CUD)

A *deferred operation* is one that's logically atomic with the
parent transaction but mechanically too large for a single
reservation: freeing all the extents of a heavily-fragmented file,
moving rmap entries during a refcount adjustment, etc. The
pattern:

```
  Step 1: parent transaction commits with an INTENT item:
            "I plan to free extents X..Y in AG A"
                    EFI: extent free intent

  Step 2: a deferred-op worker runs the actual work; on completion
          it commits a DONE item:
            "I freed extents X..Y; the matching intent is satisfied"
                    EFD: extent free done

  Recovery rule: if an EFI is present but the matching EFD (matched
                 by efi_id == efd_efi_id) is missing, recovery
                 finishes the free itself.
```

Four families share the same 16-byte header shape (`type + size +
nextents + id`) followed by a per-step record array:

| Family       | Use                       | Record (per step)            | Record size |
|--------------|---------------------------|------------------------------|-------------|
| **EFI/EFD**  | extent free               | [`xfs_extent_64`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/libxfs/xfs_log_format.h#L608) | 16 B |
| **BUI/BUD**  | bmap update (extent map)  | [`xfs_map_extent`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/libxfs/xfs_log_format.h#L723) | 32 B |
| **RUI/RUD**  | rmap update               | `xfs_map_extent`             | 32 B |
| **CUI/CUD**  | refcount update           | [`xfs_phys_extent`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/libxfs/xfs_log_format.h#L787)   | 16 B |

A single-extent free pair is `(16 + 16) + (16 + 16) = 64 bytes` for
the entire "I plan / I did" round trip. A 50-extent truncate's
worst case is `(16 + 50×32) × 2 + (16 + 50×16) × 2 ≈ 5 KiB` to
describe potentially hundreds of MiB of freed file data.

One special case worth pulling out: the
[`xfs_icreate_log`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/libxfs/xfs_log_format.h#L1007)
intent — ICREATE — has **no matching done item**. It's a one-way
marker recording "AG `icl_ag` block `icl_agbno` now holds an
`icl_count`-inode chunk"; on replay, recovery zero-initialises the
chunk before any references to the inodes inside it get applied.
Why not just log the chunk? Because the chunk is 64 × 512 B = 32 KiB
of mostly zeros — logging "init these blocks to gen X" is far
cheaper than logging 32 KiB of bytes.

### Mapping table

```
  Metadata change                Log item                Recovery action
  ─────────────────────          ─────────────────────   ────────────────────
  inode core / fork              xfs_inode_log_item      replay dirty regions
                                 (xfs_inode_log_format)  into cluster buffer

  metadata buffer (AGF/AGI/      xfs_buf_log_item        replay dirty chunks
   AGFL/DIR/BTREE/...)           (xfs_buf_log_format)    into home block

  extent free (deferred)         EFI + later EFD         if EFD missing,
                                                         finish the free

  bmap update (deferred)         BUI + later BUD         if BUD missing,
                                                         finish the bmap edit

  rmap update                    RUI + later RUD         if RUD missing,
                                                         finish the rmap edit

  refcount update (reflink/CoW)  CUI + later CUD         if CUD missing,
                                                         finish the refcount

  inode chunk init               ICREATE                 zero-initialise the
                                                         chunk on replay
```

# CIL — Committed Item List

If every `xfs_trans_commit` had to write a fresh log record to the
device, XFS would lose most of its scaling: a hot AGF that fields
1000 transactions per second would produce 1000 nearly-identical
log records, almost all of them re-describing the same bytes. The
**CIL** ([`xfs_cil`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_log_priv.h#L284))
is the in-memory aggregator that fixes this. It does three things:

1. **Singleton items.** The log item attached to a dirty buffer or
   inode is a singleton — a sequence of transactions that all dirty
   the same AGF buffer keeps reusing one `xfs_buf_log_item` and
   accumulating dirty byte ranges into it.
2. **Relogging.** When the next transaction modifies the same buffer
   *again* before the CIL has pushed, the same item is "relogged":
   its dirty mask grows but only one copy of the item lives in the
   CIL.
3. **Batched push.** Periodically (timer, memory threshold, or on
   demand from `xfs_log_force_seq`), the CIL pushes its current
   *context* ([`xfs_cil_ctx`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_log_priv.h#L235))
   as a single **checkpoint** — one log record describing all the
   accumulated changes.

```
  Many transactions commit
            │
            ▼
  ┌─────────────────────────────────────┐
  │           xfs_cil_ctx               │
  │  current checkpoint staging area    │
  │  ┌──────────┐ ┌──────────┐ ┌─────┐  │
  │  │ buf log  │ │ inode log│ │ EFI │  │
  │  │ item     │ │ item     │ │     │  │
  │  │ (dirty   │ │ (ilf_    │ │     │  │
  │  │  bitmap) │ │  fields) │ │     │  │
  │  └──────────┘ └──────────┘ └─────┘  │
  └──────────────────┬──────────────────┘
                     │ xlog_cil_push_work
                     ▼
  ┌─────────────────────────────────────┐
  │  one log record (= one checkpoint)  │
  │  serialised via iop_format          │
  │  written to the iclog ring          │
  └─────────────────────────────────────┘
```

The push runs in
[`xlog_cil_push_work`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_log_cil.c#L1375).
It's triggered by:

- the CIL growing past a soft threshold (default ~8 MiB per ctx);
- a `xfs_log_force_seq` from fsync demanding a specific sequence
  be durable;
- a periodic timer (~30 s) so a quiet workload still drains.

A push **does not block commits**. It snapshots the current
`xfs_cil_ctx`, swaps in a fresh one for new commits to land in, and
serialises the snapshot asynchronously. New transactions commit
into the new ctx while the old one is being written.

Once the push completes and the log record is durable, the items
move to the AIL with assigned LSNs. From that point on, recovery
can rebuild the change from the log even if the home block hasn't
been touched.

The CIL is also where the post's "delta logging keeps the log small"
and "fsync is cheap-but-correct" properties come from. Both rest on
the same fact: one push amortises many transactions, so the per-
transaction log cost is small.

# Physical Log

The on-disk log is a **fixed-size ring** of filesystem blocks. Its
location is recorded in the superblock as `sb_logstart` (start fsblock,
or 0 for an external log device) and `sb_logblocks` (length). On the
captured 2 GiB filesystem, those are `262151` and `25650` — about
100 MiB of log, living inside AG 2. See the
[Log area subsection]({{ site.baseurl }}/tech/2026/05/28/xfs-metadata-internals.html#log-area)
of the metadata post for how the kernel reaches it via
`xfs_mount.m_logdev_targp` (aliased to the data buftarg for internal
logs).

What lands in the log is a **three-layer envelope**:

```
  log record (one or more iclog flushes)
  ┌──────────────────────────────────────────────────────────────┐
  │ xlog_rec_header   { h_magicno, h_cycle, h_lsn, h_len,        │
  │                      h_crc, h_num_logops, h_fs_uuid }        │
  │ ┌──────────────────────────────────────────────────────────┐│
  │ │ xlog_op_header   { oh_tid, oh_len, oh_clientid,          ││
  │ │                    oh_flags (XLOG_START/COMMIT/CONTINUE) }││
  │ │ <log-item payload>  (xfs_buf_log_format + dirty bytes,   ││
  │ │                       or xfs_inode_log_format + regions, ││
  │ │                       or xfs_efi_log_format + extents,   ││
  │ │                       or xfs_icreate_log, …)             ││
  │ └──────────────────────────────────────────────────────────┘│
  │ ┌──────────────────────────────────────────────────────────┐│
  │ │ xlog_op_header  ...                                      ││
  │ │ <log-item payload>                                       ││
  │ └──────────────────────────────────────────────────────────┘│
  │ ...                                       h_num_logops total│
  └──────────────────────────────────────────────────────────────┘
```

Sizes from BTF on Linux v7.0 `xfs.ko`:

| Struct                  | Size | Role |
|-------------------------|------|------|
| [`xlog_rec_header`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/libxfs/xfs_log_format.h#L139) | 512 B | outer record (LSN, CRC, op count, UUID) |
| [`xlog_op_header`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/libxfs/xfs_log_format.h#L108)  | 12 B  | per-op wrapper inside the record         |
| `xfs_buf_log_format`    | 88 B (+ dirty bytes) | buffer log item header     |
| `xfs_inode_log_format`  | 56 B (+ dirty regions) | inode log item header    |
| `xfs_efi_log_format`    | 16 B (+ N × 16 B records) | EFI intent           |
| `xfs_icreate_log`       | 28 B fixed | ICREATE intent                       |

The **LSN** ([`xfs_lsn_t`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/libxfs/xfs_types.h#L27), `int64_t`) is packed as `(cycle, block)`:
the upper 32 bits count log wraps (when the head reaches the end of
the ring, cycle bumps), the lower 32 bits are the block offset
within the current cycle. The split is what lets [`XFS_LSN_CMP`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_log.h#L98) stay
monotone across wraps.

The captured `xfs-meta-dump-log.sh` output makes the layering
concrete (excerpt from `tests/xfs_logging_post_samples.sh`):

```
xfs_logprint:
    log device: 0x10300 daddr: 2097208 length: 205200

cycle: 1  version: 2  lsn: 1,2  tail_lsn: 1,2  num ops: 23
                                            # one log record, 23 ops
Oper (1): tid: 6ad14a7e  TRANS  num_items: 20         # transaction header
Oper (2): INODE  #regs:3  ino: 0x80  flags:0x3  dsize:17
                          ^^^^^^^^
                          xfs_inode_log_format (56 B), 3 regions follow
Oper (3): INODE CORE                                  # 176 B core (ILOG_CORE)
  magic 0x494e mode 040755 version 3 format 1
  ...
Oper (4): LOCAL inode data                            # the inode's data fork
          SHORTFORM DIRECTORY size 17                 #   (small dir, inline)
Oper (5): BUF: start blkno: 2 (0x2)  len: 1  flags: 0x3800
Oper (6): AGI Buffer: XAGI                            # buffer log item: AGI
  cnt: 64  root: 3  level: 1  free#: 0x3f  newino: 0x80
  bucket[0-3]: 0xffffffff 0xffffffff 0xffffffff 0xffffffff
...
```

That `INODE / INODE CORE / LOCAL inode data` triple is one log item:
`xfs_inode_log_format` (56 B) → inode core (176 B) → inline data
(varies). The whole inode "object" on the wire totals about 250 B
to describe one new inode. The same picture works for buffer log
items — `BUF: ...` is the format header, `AGI Buffer: XAGI ...` is
the dirty bytes that follow.

The on-disk log is what recovery walks. Once a checkpoint is
durable in this ring, the change is recoverable — even if the home
blocks for those AGI / dir / inode bytes never get written before
power failure.

# AIL — Active Item List

The CIL is the *in-core* side of the log items pipeline. The **AIL**
([`xfs_ail`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_trans_priv.h),
walked by
[`xfsaild`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_trans_ail.c))
is the *durable* side: once a CIL checkpoint reaches the iclog and
is written, the items in it are moved to the AIL with assigned LSNs.
The AIL is sorted by LSN, oldest first.

The crucial property the AIL exists to enforce:

```
    "the log record is durable"   ≠   "the home block is written"
```

A log item lives on the AIL until its `iop_push` callback writes
the underlying home block to disk. For a buffer log item that's a
`xfs_buf` submit_bio onto the data device; for an inode log item
it's [`xfs_iflush`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_inode.c#L2368) reformatting the in-core inode back into its
cluster buffer and writing that.

```
   commit             push                  flush                drain
  ──────────►   ──────────────►       ──────────────►       ────────────►
   xfs_trans    xlog_cil_push_work     xfs_buf_submit /      AIL removal
   _commit                             xfs_iflush
   ┌──────┐    ┌──────┐    ┌──────┐    ┌──────────┐         ┌──────┐
   │ in   │    │ in   │    │ in   │    │ home     │         │ done │
   │ trans│ →  │ CIL  │ →  │ AIL  │ →  │ block on │   →     │      │
   │      │    │      │    │      │    │ disk     │         │      │
   └──────┘    └──────┘    └──────┘    └──────────┘         └──────┘
                                       only NOW can the AIL drop the item
                                       and the log space behind it can
                                       be reused
```

The **tail LSN** is the oldest LSN still in the AIL. It *pins log
space* — every log block between the tail LSN and the head LSN is
"behind history that hasn't been retired from the home block." When
the gap approaches the log's configured size, new transactions
stall on log-grant waits until `xfsaild` flushes more items and
the tail advances.

Two facts that fall out of the AIL design:

- **fsync doesn't wait for home-block writeback.** It waits for the
  log force, which only requires the CIL push to reach the iclog.
  The AIL push for the inode cluster happens later, asynchronously.
- **Recovery's job is to make the home blocks match what the AIL
  would have eventually written.** Anything still on the AIL at
  power failure is "log-durable but not home-block-durable" — exactly
  the gap recovery fills by replaying the log into the home blocks.

The AIL push policy ([`xfsaild_push`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_trans_ail.c))
walks the AIL from oldest LSN forward, calling `iop_push` on each
item. An item that can be flushed gets its home-block IO submitted;
an item that can't (because the buffer is still pinned, or because
of locking) gets requeued for the next iteration. The push loop
runs whenever the tail needs to advance — that's normally because
log space is getting tight.

# Metadata Writeback

Why isn't metadata written to its home block immediately at
transaction commit? Because that would **break WAL** — the log
record describing the change hasn't necessarily been written yet,
and a power loss between the home-block write and the log-record
write would leave the home block updated but no log record to
authoritatively describe it.

So metadata writes are *deliberately deferred* until two things are
true: (a) the log record describing the change is durable, and
(b) the AIL push gets around to it. The mechanism is two-fold:

**The log pin.** Every
[`xfs_buf`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_buf.h#L150)
has a pin counter (`b_pin_count`). When a buffer is joined to a CIL
context, the counter is incremented; when the matching checkpoint
reaches the AIL, the counter is decremented. While pinned
(`b_pin_count > 0`) the buffer is **unflushable** — `xfs_buf_submit`
refuses. This is the kernel-side enforcement of WAL.

**The delayed-write list.** Even after the log pin drops, most
metadata buffer writes don't fire immediately. The buffer sits on
the AIL (and a delwri list inside it) until either `xfsaild` pushes
it (because its LSN is the tail and log space is needed) or a sync
forces it. This is the optimisation that lets ten transactions
modifying the same AGF buffer in one second produce *one*
home-block write, not ten — the log already carries every change,
so reading the AGF after a crash and replaying gives the same
final state as ten in-order writes.

The combination is the source of XFS's "metadata writes are bursty"
behavior — long quiet periods broken by a flurry of delayed-write
submissions when log space pressure builds.

# Recovery

On mount, if the filesystem wasn't cleanly unmounted, XFS reads the
log and replays it.
[`xlog_recover`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_log_recover.c)
walks the log in several ordered phases:

1. **Scan.** Read the log forward, build an in-memory hash of
   checkpoints and their items, and pair intent items with their
   matching `done` records as they appear.
2. **Replay buffers.** Walk the items and write each
   `xfs_buf_log_item` back to its home block, skipping any whose
   buffer LSN says the home block is already newer (the buffer
   was flushed before the crash). Buffers go *first* so that any
   inode cluster blocks (and any blocks an `icreate` references)
   exist and are zero-initialised before inodes touch them.
3. **Replay inodes and icreate.** Replay each
   `xfs_inode_log_item` into its cluster buffer, using `di_lsn` (v5)
   to skip items whose home is already current. Replay `ICREATE`
   intents to zero freshly-allocated inode chunks.
4. **Finish intents.** Drain whatever intent items are still
   unmatched at end-of-scan — for each EFI without an EFD, finish
   the free; for each BUI without a BUD, finish the bmap update;
   for each RUI/CUI without its done, finish the rmap / refcount
   update.

After phase 4 the log tail advances; the filesystem is consistent
with what the application saw at the time of the last successful
commit. Recovery does **not** roll back partial transactions — every
transaction in the log was atomic at commit time, and the log
records reflect committed state only.

Two recovery-correctness rules that are easy to miss:

- **LSN-skip prevents double-replay.** Each metadata buffer carries
  its own LSN (`bb_lsn`); each v5 inode carries `di_lsn`. Replay
  compares the log item's LSN against the on-disk value and skips
  the work if the home block is already at-or-newer. Without this,
  a clean buffer that was flushed *before* the crash would get
  re-replayed and possibly corrupted.
- **Buffer-before-inode ordering.** Inodes log items name an
  `ilf_blkno` for the cluster buffer that holds them; that buffer
  must already exist and be zero-initialised before the inode
  payload is replayed into it. The phase ordering above guarantees
  this — `ICREATE` and buffer replay both finish before inode replay
  starts.

# Parsing Real XFS Log Metadata

Several tools turn the on-disk log into human-readable form on an
**unmounted** filesystem. Assume the device is `/dev/sdb` in the
examples below.

## Transactional summary

[`xfs-meta-dump-log.sh`]({{ site.baseurl }}/code/xfs-meta-dump-log.sh)
wraps `xfs_logprint`:

```
./xfs-meta-dump-log.sh /dev/sdb
```

The first output block reports head/tail position and clean/dirty
state — useful as a quick "did this filesystem unmount cleanly":

```
xfs_logprint:
    data device: 0x10300
    log device: 0x10300 daddr: 2097208 length: 205200
    log tail: 21 head: 21 state: <CLEAN>
```

`tail == head` and `state: <CLEAN>` means everything that was ever in
the log has been flushed home. A `<DIRTY>` state with `head` ahead
of `tail` means recovery still has work to do.

## Inode log items

Inside one of the recent transactions in the captured sample:

```
Oper (2): INODE  #regs:3  ino: 0x80  flags:0x3  dsize:17
        blkno: 128  len: 32  boff: 0
Oper (3): INODE CORE
        magic 0x494e mode 040755 version 3 format 1
        nlink 2 uid 0 gid 0
        size 0x11 nblocks 0x0 nextents 0x0
        flags2 0x18 cowextsize 0x0
Oper (4): LOCAL inode data
        SHORTFORM DIRECTORY size 17
```

This is one `xfs_inode_log_item` formatting itself. `ino: 0x80` is
inode 128 (the root directory after our `creat("foo")` ran);
`#regs:3` means three dirty regions follow this header (CORE,
LOCAL inode data, plus the format struct itself). `dsize:17` says
the inline directory grew to 17 bytes — one entry's worth for the
newly-created `foo`. `blkno: 128` is the inode cluster's daddr;
recovery would write the replayed core back to it.

To correlate "this log item" with "which user file" you cross-
reference the `ino` field against `xfs_ncheck` (or
[`code/xfs-meta-name-inode.sh`]({{ site.baseurl }}/code/xfs-meta-name-inode.sh))
on the same image.

## Buffer log items

The same transaction modifies the AGI and several dir-block
buffers; each appears as a `BUF: ...` header followed by the
dirty bytes:

```
Oper (5): BUF: start blkno: 2 (0x2)  len: 1  bmap size: 1  flags: 0x3800
Oper (6): AGI Buffer: XAGI
        ver: 1  seq#: 0  len: 131072  cnt: 64  root: 3
        level: 1  free#: 0x3f  newino: 0x80
        bucket[0-3]: 0xffffffff 0xffffffff 0xffffffff 0xffffffff
```

The `blkno: 2` is the AGI sector for AG 0 (after the AGF at sector
1 and the SB at sector 0). `flags: 0x3800` encodes the buffer's
**blft** ([`XFS_BLFT_AGI_BUF`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/libxfs/xfs_log_format.h#L557)) in its upper bits, which is what lets
`xfs_logprint` print the friendly `AGI Buffer: XAGI` header rather
than a hex dump.

## Intent items

Intent items decode similarly when present — `xfs_logprint` would
show e.g.:

```
Oper (N): EFI: #regs: 1  num_extents: 3  id: 0xffff8a72c5c01a00
                                       # 3 extents staged for free
Oper (M): EFD: #regs: 1  num_extents: 3  efi_id: 0xffff8a72c5c01a00
                                       # matches the EFI above by id
```

The `id`/`efi_id` matching is what recovery uses to pair them. A
file-truncate that releases many extents at once produces a chain
of these. Note that clean-unmount workflows often drain intents
fully — the matched EFI/EFD pairs are present in the log records
behind the head, but on a freshly-unmounted filesystem they may
have been processed out of the visible decode by the time
`xfs_logprint` reads. To see intents in flight, capture the log
mid-operation via a live tracepoint instead (the
[`xfs-meta-observe-truncate.sh`]({{ site.baseurl }}/code/xfs/xfs-meta-observe-truncate.sh)
bpftrace observer above is one way — its `defer` lines are exactly
the intent-chain firing).

## Python: parse a raw log record header

A 30-line Python decoder that reads the first
[`xlog_rec_header`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/libxfs/xfs_log_format.h#L139)
from a raw log byte dump (e.g. extracted with
`xfs_logprint -C raw.bin /dev/sdb`):

```python
#!/usr/bin/env python3
import struct, sys

# struct xlog_rec_header on v5 — first 56 bytes carry the
# 16 always-present fields below; the rest of the 512 B sector
# is cycle-data + Linux-port fields + h_size + (LOGV2) extras.
REC_FMT = ">IIIIQQiIIII"   # 11 fields, big-endian
NAMES   = "h_magicno h_cycle h_version h_len h_lsn h_tail_lsn " \
          "h_crc h_prev_block h_num_logops h_cycle_data0 h_cycle_data1"

with open(sys.argv[1], "rb") as f:
    buf = f.read(struct.calcsize(REC_FMT))

vals = struct.unpack(REC_FMT, buf)
for name, val in zip(NAMES.split(), vals):
    if name in ("h_magicno", "h_crc"):
        print(f"{name:14s} 0x{val:08x}")
    elif name in ("h_lsn", "h_tail_lsn"):
        # LSN packs (cycle, block) — cycle in upper 32, block in lower 32
        cycle, block = (val >> 32) & 0xffffffff, val & 0xffffffff
        print(f"{name:14s} 0x{val:016x}  (cycle {cycle}, block {block})")
    else:
        print(f"{name:14s} {val}")
```

Running it on a captured record:

```
h_magicno      0xfeedbabe        # XLOG_HEADER_MAGIC_NUM
h_cycle        1
h_version      2
h_len          14848
h_lsn          0x0000000100000002 (cycle 1, block 2)
h_tail_lsn     0x0000000100000002
h_crc          0xabcd1234
h_num_logops   141
```

That's the entry point — 30 lines of struct.unpack tell you whether
the record is valid (`h_magicno == 0xfeedbabe`), where it lives in
the log ring (`h_lsn`), and how many operations follow
(`h_num_logops`). From there an extension that walks the 12-byte
`xlog_op_header` instances and dispatches on the 16-bit type code
at the start of each payload is straightforward.

# Source Code Navigation Guide

Where to read what, with v7.0 Bootlin links:

- [**`fs/xfs/xfs_trans.c`**](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_trans.c)
  — transaction lifecycle. `xfs_trans_alloc` (L234) opens; the
  `_icreate` / `_inode` / `_dir` wrappers (L1066+) pre-reserve for
  specific operations; `xfs_trans_commit` (L909) hands items to the
  CIL. Start here when reasoning about reservation or what an op
  attaches to its transaction.

- [**`fs/xfs/xfs_log.c`**](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_log.c)
  — log mount + iclog ring. `xfs_log_mount` (L534) reads
  `sb_logstart` / `sb_logblocks` and sets up the in-core `xlog`.
  `xfs_log_force_seq` is fsync's entry point into the log. Read
  this for "how does the log get attached at mount" and "what does
  log_force actually do."

- [**`fs/xfs/xfs_log_cil.c`**](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_log_cil.c)
  — the CIL. `xlog_cil_commit` is where each `xfs_trans_commit`
  lands; `xlog_cil_push_work` (L96) is the kthread that drains the
  CIL into iclog writes. Read this when investigating "why was my
  fsync slow" — most of fsync's latency lives in CIL push work.

- [**`fs/xfs/xfs_buf_item.c`**](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_buf_item.c)
  and
  [**`fs/xfs/xfs_inode_item.c`**](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_inode_item.c)
  — the two main log-item types. Each defines an ops vector with
  `iop_format` (serialise to wire), `iop_pin`/`iop_unpin` (WAL
  enforcement), `iop_push` (AIL flush). Read these when you need to
  understand exactly which bytes get written for a given change.

- [**`fs/xfs/xfs_trans_ail.c`**](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_trans_ail.c)
  — `xfsaild` and AIL management. `xfsaild_push` walks the AIL from
  oldest LSN forward. Read this when log space is filling and you
  want to know why the tail isn't advancing.

- [**`fs/xfs/xfs_log_recover.c`**](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_log_recover.c)
  — `xlog_recover` and the multi-phase replay. Read this when
  diagnosing a mount that's failing recovery (e.g. CRC mismatch on
  a log record) or trying to understand exactly what the replay
  policy is for a specific item type.

# Putting Everything Together

Back to the running example:

```c
fd = creat("foo", 0644);
write(fd, data, len);
fsync(fd);
/* power fails */
```

Stage by stage:

1. **`creat("foo", 0644)`** — VFS calls [`xfs_vn_create`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_iops.c#L292) →
   `xfs_create`. One transaction opens ([`xfs_trans_alloc_icreate`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_trans.c#L1191)).
   In-core: allocate a new inode in some AG ([`xfs_dialloc`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/libxfs/xfs_ialloc.c#L1889)),
   initialise its `xfs_inode_log_item` with `XFS_ILOG_CORE` set,
   modify the root directory's inline `sfdir3` data fork to add the
   entry `foo → 132`, mark the parent inode dirty too. `xfs_trans_commit`
   attaches: one `xfs_icreate_log` (if a new chunk was needed),
   one `xfs_inode_log_item` for the new inode, one for the parent,
   one `xfs_buf_log_item` for the AGI sector. **The CIL absorbs
   these into its current context.** No log record on disk yet.

2. **`write(fd, data, len)`** — VFS calls [`xfs_file_write_iter`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_file.c#L1119).
   For a buffered write the bytes go into the page cache; an
   in-core extent is reserved via [`xfs_bmapi_reserve_delalloc`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_iomap.c#L1457) with
   a `NULLSTARTBLOCK` sentinel (delayed allocation). **No
   transaction, no log item yet.** The data bytes are not journaled;
   they live in the page cache waiting for writeback.

3. **`fsync(fd)`** — VFS calls `xfs_file_fsync`. It first runs
   `filemap_write_and_wait_range`, which triggers writeback. The
   writeback path is where the delalloc reservation flips to a real
   physical extent: [`xfs_bmapi_convert_delalloc`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/libxfs/xfs_bmap.c#L4487) opens a transaction,
   allocates blocks (touching AGF/free-space btrees), updates the
   inode's bmap, commits — emitting an `xfs_inode_log_item` (with
   `XFS_ILOG_DEXT` or `XFS_ILOG_CORE` set for size/extent count)
   plus buffer log items for the AGF/free-space-btree blocks. Once
   writeback completes, fsync logs the inode again if its in-core
   state changed (i_size, timestamps), then calls
   `xfs_log_force_seq(seq)` where `seq` is the CIL sequence
   covering all of the above. The log force triggers a CIL push
   if one isn't pending; **after the push completes, the log record
   containing all of `creat`'s and `write`'s log items is durable
   on the log device**. Finally `blkdev_issue_flush` ensures the
   device write cache is flushed.

4. **Power fails.** The state on disk: the log record from step 3
   is durable on the log device; the home blocks (root-dir cluster,
   AGI sector, AGF sector, free-space-btree blocks, new inode
   cluster) **may or may not have been flushed** — `xfsaild` would
   have eventually pushed them, but had no obligation to before the
   crash. The data bytes themselves are on the data device because
   `filemap_write_and_wait_range` finished before the log force.

5. **Recovery on next mount.** `xlog_recover` scans the log, finds
   the checkpoint from step 3, replays each log item into its home
   block:
   - the buffer log items for AGI / AGF / free-space-btree get
     written into sectors 1-2 of the relevant AG;
   - the inode log items get written into the inode cluster
     (after the cluster buffer itself is initialised if a fresh
     `ICREATE` intent is present);
   - the inode's directory data fork now contains `foo → 132`
     again, the inode core has the right i_size, and the AGF/AGI
     counters reflect the new allocation.

6. **After recovery the filesystem mounts cleanly**, the file `foo`
   exists, it contains the bytes `write()` produced (because those
   were on the data device before fsync returned), and the filesystem
   is in the exact state the application saw at the time fsync
   returned — even though most of the metadata home blocks were never
   actually written before the power failure.

That's the contract the entire logging pipeline exists to deliver.
Every component named in this post — log items, the CIL, the
physical log records, the AIL, the WAL pin, the recovery phases —
is a necessary part of getting that property right under realistic
crash timing.

# Summary

**Main advantages**

- **Metadata changes are journaled atomically and recoverably.**
  Every transaction is committed all-or-nothing; recovery makes the
  filesystem consistent with what the application saw at the last
  successful commit.
- **fsync is cheap.** It forces the log up to a specific CIL
  sequence; it does not wait for inode cluster writeback. The home
  block is updated later, but the log carries the change with full
  durability.
- **Delta logging keeps the log small.** Log items emit only the
  dirty regions of their underlying structures (dirty bitmap for
  buffers, `ilf_fields` mask for inodes), so log volume tracks the
  *amount of change*, not the size of the metadata.
- **Intent/done items make multi-step ops crash-restartable.**
  EFI/EFD, BUI/BUD, RUI/RUD, CUI/CUD encode "I plan / I did" pairs
  so a partial deferred operation can be finished on the next
  mount, even when the work doesn't fit in a single transaction's
  log reservation.
- **The CIL amortises commit cost.** Many transactions modifying
  the same buffer relog into one in-core item; one push writes one
  log record describing all of it. Per-transaction log cost stays
  low even on hot metadata.

**Main problems / limitations**

- **No data journaling.** XFS does not log file data. Crash
  consistency for file *content* is the application's job — fsync
  is the only contract for "my bytes are durable on disk."
- **Log space is a global limit.** Long-running transactions or
  large rename / truncate sequences consume reservation. A log
  sized too small for the workload causes new transactions to
  stall on grant waits — the tail can't advance until `xfsaild`
  flushes enough home blocks to retire the oldest LSN.
- **Recovery cost grows with uncommitted-checkpoint window.** A
  long quiet period followed by a crash forces a large replay,
  which is why the CIL pushes on a timer (~30 s) even when no
  thread asks for it.
- **The on-wire format is rich but xfs_logprint version-dependent.**
  Decoding intent items in particular can require either a recent
  xfsprogs or live tracepoint capture — `xfs_logprint`'s
  transactional view doesn't always render every item type
  symbolically.
- **Per-AG locking on metadata buffers.** A workload that hammers
  one AG (a single huge directory; an inode-allocation-heavy
  small-file workload that pins one inobt) serialises on the AGI /
  AGF buffer locks even though the log can amortise their commits.
  The architecture answer is "spread work across AGs"; in practice
  that requires application-level discipline.
