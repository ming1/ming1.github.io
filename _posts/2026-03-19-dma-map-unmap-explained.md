## 1. Top View & Architecture

### What is DMA Mapping?

DMA (Direct Memory Access) allows hardware devices to read from and write to
system memory without CPU intervention. However, the addresses that a device
uses to access memory may differ from the CPU's physical addresses. The DMA
mapping API bridges this gap by:

1. **Translating addresses** between CPU physical space and device DMA space
2. **Managing cache coherency** to ensure CPU and device see consistent data
3. **Providing IOMMU integration** for address virtualization and protection
4. **Handling hardware limitations** via bounce buffering (SWIOTLB)

### Streaming vs Coherent DMA

The DMA API provides two models:

- **Streaming DMA** (`dma_map_*` / `dma_unmap_*`): Maps existing memory for
  short-lived transfers. Requires explicit sync operations. Used for I/O
  data buffers (e.g., disk reads, network packets).

- **Coherent DMA** (`dma_alloc_coherent` / `dma_free_coherent`): Allocates
  memory that is simultaneously accessible by CPU and device without explicit
  sync. Used for long-lived shared structures (e.g., NVMe submission/completion
  queues, DMA descriptor rings).

## 2. Basic Principle & Design

### Why Devices Need Address Translation

On most systems, three distinct address spaces exist:

1. **CPU Physical Addresses** — what the CPU uses after MMU translation
2. **Bus/DMA Addresses** — what devices use on the bus (e.g., PCI BAR space)
3. **IOVA (I/O Virtual Addresses)** — virtual addresses assigned by the IOMMU

Without an IOMMU, `dma_addr_t` typically equals the physical address (possibly
with a fixed offset). With an IOMMU, `dma_addr_t` is an IOVA that the IOMMU
hardware translates to a physical address via its own page tables.

```
 CPU View                                  Device View
 ────────                                  ───────────

 ┌──────────────┐                         ┌──────────────┐
 │  Virtual Addr │                         │  DMA Addr    │
 │  (per-process)│                         │  (dma_addr_t)│
 └──────┬───────┘                         └──────┬───────┘
        │ CPU MMU                                │
        ▼                                        │
 ┌──────────────┐                                │
 │  Physical    │                                │
 │  Address     │◄───────────────────────────────┘
 └──────┬───────┘    No IOMMU: dma_addr ≈ phys_addr
        │                                        │
        │                              ┌─────────┘
        │                              │ With IOMMU:
        │                              │ IOVA → phys via
        │                              │ IOMMU page tables
        ▼                              ▼
 ┌─────────────────────────────────────────────┐
 │              System Memory (RAM)             │
 └─────────────────────────────────────────────┘
```

### The Cache Coherency Problem

On non-coherent architectures (common in ARM), the CPU cache and device DMA
can see stale data:

```
 CPU writes data                    Device reads stale RAM
 ┌──────┐    ┌───────┐    ┌──────────┐    ┌────────┐
 │ CPU  │───►│ Cache │    │  Memory  │◄───│ Device │
 │      │    │(dirty)│    │ (stale)  │    │  DMA   │
 └──────┘    └───────┘    └──────────┘    └────────┘
                  │                            ▲
                  │     Cache flush needed      │
                  └────────────────────────────┘
```

The DMA API solves this with:

- **`DMA_TO_DEVICE`**: CPU flushes caches before device reads (device sees
  fresh data)
- **`DMA_FROM_DEVICE`**: CPU invalidates caches after device writes (CPU sees
  fresh data)
- **`DMA_BIDIRECTIONAL`**: both flush and invalidate
- **Coherent DMA**: memory allocated as non-cacheable, bypassing the problem

### Full Hardware Data Path

```
                            System Bus (e.g., PCIe)
 ┌──────┐   ┌───────┐   ┌──────────────┐   ┌─────────┐   ┌────────┐
 │      │──►│       │──►│              │──►│         │──►│        │
 │ CPU  │   │ Cache │   │   Memory     │   │  IOMMU  │   │ Device │
 │      │◄──│       │◄──│  Controller  │◄──│ (opt.)  │◄──│  DMA   │
 └──────┘   └───────┘   └──────────────┘   └─────────┘   └────────┘
                              │                   │
                              ▼                   ▼
                         ┌─────────┐        ┌──────────┐
                         │ DRAM    │        │ IO Page  │

```

### DMA Direction Enum

Defined in [`include/linux/dma-direction.h`](https://elixir.bootlin.com/linux/latest/source/include/linux/dma-direction.h):

```c
enum dma_data_direction {
    DMA_BIDIRECTIONAL = 0,   /* device may read and write */
    DMA_TO_DEVICE     = 1,   /* CPU → device (e.g., disk write) */
    DMA_FROM_DEVICE   = 2,   /* device → CPU (e.g., disk read) */
    DMA_NONE          = 3,   /* for debugging */
};
```

### The `dma_map_ops` Abstraction

The backend abstraction is [`struct dma_map_ops`](https://elixir.bootlin.com/linux/latest/source/include/linux/dma-map-ops.h#L16),
which provides a vtable of operations:

```c
struct dma_map_ops {
    void *(*alloc)(struct device *, size_t, dma_addr_t *, gfp_t, unsigned long);
    void (*free)(struct device *, size_t, void *, dma_addr_t, unsigned long);

    dma_addr_t (*map_phys)(struct device *, phys_addr_t, size_t,
                           enum dma_data_direction, unsigned long);
    void (*unmap_phys)(struct device *, dma_addr_t, size_t,
                       enum dma_data_direction, unsigned long);

    int (*map_sg)(struct device *, struct scatterlist *, int,
                  enum dma_data_direction, unsigned long);
    void (*unmap_sg)(struct device *, struct scatterlist *, int,
                     enum dma_data_direction, unsigned long);

    void (*sync_single_for_cpu)(struct device *, dma_addr_t, size_t,
                                enum dma_data_direction);
    void (*sync_single_for_device)(struct device *, dma_addr_t, size_t,
                                   enum dma_data_direction);
    /* ... plus get_sgtable, mmap, dma_supported, etc. */
};
```

Most modern systems (x86 with IOMMU, ARM with SMMU) use the built-in IOMMU
path and never set custom `dma_map_ops`. The `ops` pointer being NULL means
"use direct DMA."

### SWIOTLB (Software I/O TLB) Purpose

When a device's DMA mask is too small to address all of physical memory
(e.g., a 32-bit PCI device on a system with >4GB RAM), the kernel uses
bounce buffering:

1. Allocate a buffer in the device-addressable range (below 4GB)
2. Copy data from the original buffer to the bounce buffer
3. Give the bounce buffer's DMA address to the device
4. On completion, copy data back (if `DMA_FROM_DEVICE`)

SWIOTLB provides this transparently. It pre-allocates a pool at boot time
(`kernel/dma/swiotlb.c`).

### DMA Mask Setup

#### `dma_set_mask`

```c
int dma_set_mask(struct device *dev, u64 mask);
```

Sets the streaming DMA address mask. The device promises it can handle DMA
addresses within this mask. Returns 0 on success, `-EIO` if the mask is not
supported.

Source: [`kernel/dma/mapping.c:902`](https://elixir.bootlin.com/linux/latest/source/kernel/dma/mapping.c#L902)

```c
int dma_set_mask(struct device *dev, u64 mask)
{
    mask = (dma_addr_t)mask;
    if (!dev->dma_mask || !dma_supported(dev, mask))
        return -EIO;
    arch_dma_set_mask(dev, mask);
    *dev->dma_mask = mask;
    dma_setup_need_sync(dev);
    return 0;
}
```

#### `dma_set_coherent_mask`

```c
int dma_set_coherent_mask(struct device *dev, u64 mask);
```

Sets the coherent DMA mask (used for `dma_alloc_coherent`). Separate from the
streaming mask because some devices can access different address ranges for
coherent vs streaming.

Source: [`kernel/dma/mapping.c:921`](https://elixir.bootlin.com/linux/latest/source/kernel/dma/mapping.c#L921)

#### `dma_set_mask_and_coherent`

```c
static inline int dma_set_mask_and_coherent(struct device *dev, u64 mask)
{
    int rc = dma_set_mask(dev, mask);
    if (rc == 0)
        dma_set_coherent_mask(dev, mask);
    return rc;
}
```

Convenience function that sets both masks. The most common DMA mask call in
drivers.

Source: [`include/linux/dma-mapping.h:624`](https://elixir.bootlin.com/linux/latest/source/include/linux/dma-mapping.h#L624)

### Streaming DMA: Single Mappings

#### `dma_map_single` / `dma_unmap_single`

```c
#define dma_map_single(d, a, s, r) dma_map_single_attrs(d, a, s, r, 0)
#define dma_unmap_single(d, a, s, r) dma_unmap_single_attrs(d, a, s, r, 0)
```

Maps/unmaps a kernel virtual address for DMA. `dma_map_single_attrs` is an
inline function that converts the virtual address to a page+offset and calls
`dma_map_page_attrs`:

```c
static inline dma_addr_t dma_map_single_attrs(struct device *dev, void *ptr,
        size_t size, enum dma_data_direction dir, unsigned long attrs)
{
    if (dev_WARN_ONCE(dev, is_vmalloc_addr(ptr),
                      "rejecting DMA map of vmalloc memory\n"))
        return DMA_MAPPING_ERROR;
    debug_dma_map_single(dev, ptr, size);
    return dma_map_page_attrs(dev, virt_to_page(ptr), offset_in_page(ptr),
                              size, dir, attrs);
}
```

Source: [`include/linux/dma-mapping.h:502`](https://elixir.bootlin.com/linux/latest/source/include/linux/dma-mapping.h#L502)

**Note**: vmalloc addresses are rejected — DMA requires physically contiguous
memory (or scatter-gather for multiple physical segments).

#### `dma_map_page` / `dma_unmap_page`

```c
#define dma_map_page(d, p, o, s, r) dma_map_page_attrs(d, p, o, s, r, 0)
#define dma_unmap_page(d, a, s, r) dma_unmap_page_attrs(d, a, s, r, 0)
```

Maps a `struct page` with offset and size for DMA. This is the lower-level
interface; `dma_map_single` is built on top of it.

`dma_map_page_attrs` converts page+offset to a physical address and delegates
to `dma_map_phys`:

```c
dma_addr_t dma_map_page_attrs(struct device *dev, struct page *page,
        size_t offset, size_t size, enum dma_data_direction dir,
        unsigned long attrs)
{
    phys_addr_t phys = page_to_phys(page) + offset;
    return dma_map_phys(dev, phys, size, dir, attrs);
}
```

Source: [`kernel/dma/mapping.c:184`](https://elixir.bootlin.com/linux/latest/source/kernel/dma/mapping.c#L184)

### Coherent DMA Allocation

#### `dma_alloc_coherent` / `dma_free_coherent`

### Sync Operations

#### `dma_sync_single_for_cpu` / `dma_sync_single_for_device`

```c
static inline void dma_sync_single_for_cpu(struct device *dev, dma_addr_t addr,
        size_t size, enum dma_data_direction dir);
static inline void dma_sync_single_for_device(struct device *dev,
        dma_addr_t addr, size_t size, enum dma_data_direction dir);
```

Transfer DMA buffer ownership between CPU and device without
unmapping/remapping. Critical for streaming DMA when the buffer is reused.

The inline wrappers check `dma_dev_need_sync()` as a fast-path optimization.
On coherent devices (like x86 PCIe), the sync is typically a no-op and
`dma_skip_sync` is set to `true`:

```c
static inline void dma_sync_single_for_cpu(struct device *dev, dma_addr_t addr,
        size_t size, enum dma_data_direction dir)
{
    if (dma_dev_need_sync(dev))
        __dma_sync_single_for_cpu(dev, addr, size, dir);
}
```

### Error Handling

#### `dma_mapping_error`

```c
static inline int dma_mapping_error(struct device *dev, dma_addr_t dma_addr)
{
    debug_dma_mapping_error(dev, dma_addr);
    if (unlikely(dma_addr == DMA_MAPPING_ERROR))
        return -ENOMEM;
    return 0;
}
```

Checks if a DMA mapping returned the error sentinel. Must be called after
every `dma_map_*` call. `DMA_MAPPING_ERROR` is defined as `~(dma_addr_t)0`
(all bits set).

Source: [`include/linux/dma-mapping.h:135`](https://elixir.bootlin.com/linux/latest/source/include/linux/dma-mapping.h#L135)

## 4. Routing: How the DMA Layer Picks a Backend

Every DMA mapping operation goes through a routing decision. The key function
is `dma_go_direct()`:

```c
static bool dma_go_direct(struct device *dev, dma_addr_t mask,
                           const struct dma_map_ops *ops)
{
    if (use_dma_iommu(dev))
        return false;

    if (likely(!ops))
        return true;

#ifdef CONFIG_DMA_OPS_BYPASS
    if (dev->dma_ops_bypass)
        return min_not_zero(mask, dev->bus_dma_limit) >=
                    dma_direct_get_required_mask(dev);
#endif
    return false;
}
```

## 5. Direct DMA (No IOMMU)

When no IOMMU is active and no custom ops are set, the direct DMA path is
used. This is the simplest backend: it translates physical addresses to DMA
addresses using a fixed offset.

### `dma_direct_map_phys()`

Source: [`kernel/dma/direct.h:80`](https://elixir.bootlin.com/linux/latest/source/kernel/dma/direct.h#L80)

```c
static inline dma_addr_t dma_direct_map_phys(struct device *dev,
        phys_addr_t phys, size_t size, enum dma_data_direction dir,
        unsigned long attrs)
{
    dma_addr_t dma_addr;

    /* Force all DMA through bounce buffer */
    if (is_swiotlb_force_bounce(dev)) {
        if (attrs & DMA_ATTR_MMIO)
            return DMA_MAPPING_ERROR;
        return swiotlb_map(dev, phys, size, dir, attrs);
    }

    if (attrs & DMA_ATTR_MMIO) {
        dma_addr = phys;   /* MMIO: DMA addr = physical addr */
        if (unlikely(!dma_capable(dev, dma_addr, size, false)))
            goto err_overflow;
    } else {
        dma_addr = phys_to_dma(dev, phys);
        if (unlikely(!dma_capable(dev, dma_addr, size, true)) ||
            dma_kmalloc_needs_bounce(dev, size, dir)) {
            if (is_swiotlb_active(dev))
                return swiotlb_map(dev, phys, size, dir, attrs);
            goto err_overflow;
        }
    }

    /* Sync caches for non-coherent devices */
    if (!dev_is_dma_coherent(dev) &&
        !(attrs & (DMA_ATTR_SKIP_CPU_SYNC | DMA_ATTR_MMIO)))
        arch_sync_dma_for_device(phys, size, dir);

    return dma_addr;
}
```

### Key Steps in Direct Mapping

1. **`phys_to_dma(dev, phys)`**: Translates CPU physical address to bus
   address. On most platforms this is identity, but some have a fixed offset
   (e.g., `phys - dma_offset`).

2. **`dma_capable(dev, dma_addr, size, is_ram)`**: Checks if the resulting DMA
   address fits within the device's DMA mask and bus DMA limit. If not, the
   device cannot directly address this memory.

3. **SWIOTLB fallback**: If `dma_capable()` fails (device can't reach the
   memory), `swiotlb_map()` allocates a bounce buffer in addressable memory,
   copies data there, and returns the bounce buffer's DMA address.

4. **Cache sync**: For non-coherent devices, `arch_sync_dma_for_device()`
   flushes CPU caches so the device sees current data.

### `dma_direct_unmap_phys()`

Source: [`kernel/dma/direct.h:121`](https://elixir.bootlin.com/linux/latest/source/kernel/dma/direct.h#L121)

```c
static inline void dma_direct_unmap_phys(struct device *dev, dma_addr_t addr,
        size_t size, enum dma_data_direction dir, unsigned long attrs)
{
    if (attrs & DMA_ATTR_MMIO)
        return;   /* uncached and no swiotlb */

    phys_addr_t phys = dma_to_phys(dev, addr);
    if (!(attrs & DMA_ATTR_SKIP_CPU_SYNC))
        dma_direct_sync_single_for_cpu(dev, addr, size, dir);

    swiotlb_tbl_unmap_single(dev, phys, size, dir,
                             attrs | DMA_ATTR_SKIP_CPU_SYNC);
}
```

The unmap path:
1. Syncs caches for CPU access (invalidate on `DMA_FROM_DEVICE`)
2. Frees any SWIOTLB bounce buffer (no-op if the buffer wasn't bounced)

### SWIOTLB Bounce Buffer Flow

```
dma_direct_map_phys()
  │
  ├── phys_to_dma() → dma_addr
  │
  ├── dma_capable(dev, dma_addr)?
  │     │
  │     ├── YES: return dma_addr (fast path)
  │     │
  │     └── NO: is_swiotlb_active(dev)?
  │           │
  │           ├── YES: swiotlb_map(dev, phys, size, dir, attrs)
  │           │         │
  │           │         ├── Find free slot in SWIOTLB pool
  │           │         ├── Copy original data to bounce buffer
  │           │         ├── Return bounce buffer's DMA address
  │           │         │   (guaranteed in device-addressable range)
  │           │         └── On unmap: swiotlb_tbl_unmap_single()
  │           │                       copies back if DMA_FROM_DEVICE
  │           │
  │           └── NO: return DMA_MAPPING_ERROR
  │
  └── arch_sync_dma_for_device() (non-coherent only)
```

## 6. IOMMU DMA Mapping (Deep Dive)

When `use_dma_iommu(dev)` is true, the IOMMU DMA backend handles all mapping
operations. This path allocates IOVAs and programs the IOMMU's page tables.

### IOMMU Domain Types

Defined in [`include/linux/iommu.h:211`](https://elixir.bootlin.com/linux/latest/source/include/linux/iommu.h#L211):

| Domain Type | Flags | Description |
|-------------|-------|-------------|
| `IOMMU_DOMAIN_BLOCKED` | `0` | All DMA blocked; device isolation |
| `IOMMU_DOMAIN_IDENTITY` | `__IOMMU_DOMAIN_PT` | Identity mapping; DMA addr = phys addr |
| `IOMMU_DOMAIN_UNMANAGED` | `__IOMMU_DOMAIN_PAGING` | User-managed (e.g., VFIO for VMs) |
| `IOMMU_DOMAIN_DMA` | `PAGING \| DMA_API` | DMA API managed; strict IOTLB flush |
| `IOMMU_DOMAIN_DMA_FQ` | `PAGING \| DMA_API \| DMA_FQ` | DMA API with flush queue (lazy) |
| `IOMMU_DOMAIN_SVA` | `__IOMMU_DOMAIN_SVA` | Shared Virtual Addressing |

For DMA API usage, the kernel chooses between `IOMMU_DOMAIN_DMA` (strict) and
`IOMMU_DOMAIN_DMA_FQ` (lazy flushing) based on performance policy.

### `iommu_dma_cookie` Structure

Each IOMMU domain used for DMA has a cookie that holds the IOVA domain and
flush queue state:

Source: [`drivers/iommu/dma-iommu.c:57`](https://elixir.bootlin.com/linux/latest/source/drivers/iommu/dma-iommu.c#L57)

```c

    struct iova_domain     iovad;             /* IOVA allocator */

        struct iova_fq     *single_fq;        /* Single-queue mode */
        struct iova_fq __percpu *percpu_fq;   /* Per-CPU queue mode */
    };

    atomic64_t fq_flush_finish_cnt;  /* TLB flushes finished */
    struct timer_list fq_timer;      /* Deferred flush timer */

    struct iommu_domain *fq_domain;  /* Non-NULL if FQ active */
    struct iommu_dma_options options; /* FQ size, timeout, etc. */
};
```

### IOVA Allocation

The IOVA allocator (`drivers/iommu/iova.c`) uses a two-level scheme for
performance:

1. **Per-CPU rcache (fast path)**: Magazine-based per-CPU caching of recently
   freed IOVAs. `alloc_iova_fast()` first tries `iova_rcache_get()`.

2. **RB-tree (slow path)**: If the rcache misses, `alloc_iova()` searches a
   red-black tree of free IOVA ranges.

```
┌──────────────────────────────────────────────────────────────┐
│                     IOVA Allocator                            │
│                                                               │
│  ┌──────────────── Per-CPU rcache ────────────────────────┐  │
│  │                                                        │  │
│  │   CPU 0           CPU 1           CPU N                │  │
│  │  ┌──────────┐   ┌──────────┐   ┌──────────┐           │  │
│  │  │ loaded   │   │ loaded   │   │ loaded   │  Magazine: │  │
│  │  │ magazine │   │ magazine │   │ magazine │  127 PFN   │  │
│  │  │ [iova,.. │   │ [iova,.. │   │ [iova,.. │  entries   │  │
│  │  ├──────────┤   ├──────────┤   ├──────────┤  (1KB each)│  │
│  │  │ prev     │   │ prev     │   │ prev     │           │  │
│  │  │ magazine │   │ magazine │   │ magazine │           │  │
│  │  └──────────┘   └──────────┘   └──────────┘           │  │
│  │                         │                              │  │
│  │                         ▼ overflow                      │  │
│  │              ┌──────────────────┐                       │  │
│  │              │ Global Depot     │  Linked list of full  │  │
│  │              │ (per rcache)     │  magazines; cleaned   │  │
│  │              │ [mag]→[mag]→...  │  every 100ms          │  │
│  │              └──────────────────┘                       │  │
│  │                                                        │  │
│  │  Organized by size: rcaches[0..IOVA_RANGE_CACHE_MAX]   │  │
│  └────────────────────────┬───────────────────────────────┘  │
│                           │ rcache miss                       │
│                           ▼                                   │
│  ┌──────────────── RB-Tree ───────────────────────────────┐  │
│  │                                                        │  │
│  │          ┌───┐                                         │  │
│  │         ╱│ A │╲        Red-black tree of free          │  │
│  │        ╱ └───┘ ╲       IOVA ranges, sorted by          │  │
│  │     ┌───┐   ┌───┐     address                          │  │
│  │     │ B │   │ C │                                      │  │
│  │     └───┘   └───┘     Each node: {pfn_lo, pfn_hi}     │  │
│  │                                                        │  │
│  └────────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────────┘
```

Source: [`drivers/iommu/iova.c:376`](https://elixir.bootlin.com/linux/latest/source/drivers/iommu/iova.c#L376)

```c
unsigned long alloc_iova_fast(struct iova_domain *iovad, unsigned long size,
                              unsigned long limit_pfn, bool flush_rcache)
{
    /* Round up to power-of-two for rcache compatibility */
    if (size < (1 << (IOVA_RANGE_CACHE_MAX_SIZE - 1)))
        size = roundup_pow_of_two(size);

    /* Fast path: try per-CPU rcache */
    iova_pfn = iova_rcache_get(iovad, size, limit_pfn + 1);
    if (iova_pfn)
        return iova_pfn;

    /* Slow path: RB-tree allocation */
    new_iova = alloc_iova(iovad, size, limit_pfn, true);
    /* ... flush rcache and retry if needed ... */
}
```

### Mapping Flow: `__iommu_dma_map()`

Source: [`drivers/iommu/dma-iommu.c:835`](https://elixir.bootlin.com/linux/latest/source/drivers/iommu/dma-iommu.c#L835)

```c
static dma_addr_t __iommu_dma_map(struct device *dev, phys_addr_t phys,
        size_t size, int prot, u64 dma_mask)
{
    struct iommu_domain *domain = iommu_get_dma_domain(dev);
    struct iommu_dma_cookie *cookie = domain->iova_cookie;
    struct iova_domain *iovad = &cookie->iovad;
    size_t iova_off = iova_offset(iovad, phys);
    dma_addr_t iova;

    size = iova_align(iovad, size + iova_off);

    /* Step 1: Allocate IOVA */
    iova = iommu_dma_alloc_iova(domain, size, dma_mask, dev);
    if (!iova)
        return DMA_MAPPING_ERROR;

    /* Step 2: Program IOMMU page tables */
    if (iommu_map(domain, iova, phys - iova_off, size, prot, GFP_ATOMIC)) {
        iommu_dma_free_iova(domain, iova, size, NULL);
        return DMA_MAPPING_ERROR;
    }
    return iova + iova_off;
}
```

### Unmapping Flow: `__iommu_dma_unmap()`

Source: [`drivers/iommu/dma-iommu.c:812`](https://elixir.bootlin.com/linux/latest/source/drivers/iommu/dma-iommu.c#L812)

```c
static void __iommu_dma_unmap(struct device *dev, dma_addr_t dma_addr,
                               size_t size)
{
    struct iommu_domain *domain = iommu_get_dma_domain(dev);
    struct iommu_dma_cookie *cookie = domain->iova_cookie;
    struct iova_domain *iovad = &cookie->iovad;
    struct iommu_iotlb_gather iotlb_gather;

    dma_addr -= iova_off;
    size = iova_align(iovad, size + iova_off);
    iommu_iotlb_gather_init(&iotlb_gather);

    /* Check if lazy flushing is enabled */
    iotlb_gather.queued = READ_ONCE(cookie->fq_domain);

    /* Step 1: Remove IOMMU page table entries (fast, no TLB flush) */
    unmapped = iommu_unmap_fast(domain, dma_addr, size, &iotlb_gather);

    /* Step 2: Flush IOTLB if strict mode */
    if (!iotlb_gather.queued)
        iommu_iotlb_sync(domain, &iotlb_gather);

    /* Step 3: Free IOVA (or queue for lazy free) */
    iommu_dma_free_iova(domain, dma_addr, size, &iotlb_gather);
}
```

### SWIOTLB Behind IOMMU

An interesting case occurs when both an IOMMU and SWIOTLB are active.
`iommu_dma_map_phys()` (the IOMMU entry point) checks if SWIOTLB bounce
buffering is needed even with an IOMMU:

Source: [`drivers/iommu/dma-iommu.c:1198`](https://elixir.bootlin.com/linux/latest/source/drivers/iommu/dma-iommu.c#L1198)

```c
dma_addr_t iommu_dma_map_phys(struct device *dev, phys_addr_t phys,
        size_t size, enum dma_data_direction dir, unsigned long attrs)
{
    /* SWIOTLB needed if buffer is unaligned to IOVA granule */
    if (dev_use_swiotlb(dev, size, dir) &&
        iova_unaligned(iovad, phys, size)) {
        phys = iommu_dma_map_swiotlb(dev, phys, size, dir, attrs);
        if (phys == (phys_addr_t)DMA_MAPPING_ERROR)
            return DMA_MAPPING_ERROR;
    }

    if (!coherent && !(attrs & (DMA_ATTR_SKIP_CPU_SYNC | DMA_ATTR_MMIO)))
        arch_sync_dma_for_device(phys, size, dir);

    iova = __iommu_dma_map(dev, phys, size, prot, dma_mask);
    /* ... */
}
```

This happens when a physical buffer isn't aligned to the IOMMU page size.
The IOMMU can only map page-aligned regions, so unaligned sub-page buffers
are bounced to a properly aligned SWIOTLB buffer first, then mapped through
the IOMMU.

### The `dma_iova_*` API (Advanced IOVA Management)

A newer API enables efficient per-segment IOMMU management by separating IOVA
allocation from physical page linking:

```c
/* Allocate IOVA space (single contiguous range for entire I/O) */
bool dma_iova_try_alloc(struct device *dev, struct dma_iova_state *state,
                        phys_addr_t phys, size_t size);

/* Link a physical page range into an allocated IOVA region */
int dma_iova_link(struct device *dev, struct dma_iova_state *state,
                  phys_addr_t phys, size_t offset, size_t size,
                  enum dma_data_direction dir, unsigned long attrs);

/* Sync IOMMU TLB after linking pages */
int dma_iova_sync(struct device *dev, struct dma_iova_state *state,
                  size_t offset, size_t size);

/* Unlink pages and destroy the IOVA mapping */
void dma_iova_destroy(struct device *dev, struct dma_iova_state *state,
                      size_t mapped_len, enum dma_data_direction dir,
                      unsigned long attrs);
```

Source: [`include/linux/dma-mapping.h:350`](https://elixir.bootlin.com/linux/latest/source/include/linux/dma-mapping.h#L350)

This is used by the block layer's DMA iterator (`blk_rq_dma_map_iter_start()`)
to efficiently map multi-segment I/Os: one IOVA allocation covers all segments,
and individual pages are linked incrementally. This avoids per-segment IOVA
allocation overhead and enables better IOMMU TLB usage since the device sees
a single contiguous DMA address range.

```
Traditional per-segment approach:
  segment 0: alloc IOVA → iommu_map → dma_addr_0
  segment 1: alloc IOVA → iommu_map → dma_addr_1
  segment 2: alloc IOVA → iommu_map → dma_addr_2
  (3 IOVA allocations, 3 non-contiguous DMA addresses)

dma_iova_* approach:
  dma_iova_try_alloc(total_size) → single IOVA range
  dma_iova_link(phys_0, offset=0)
  dma_iova_link(phys_1, offset=4096)
  dma_iova_link(phys_2, offset=8192)
  dma_iova_sync()
  (1 IOVA allocation, 1 contiguous DMA address range)
```

## 7. IOMMU Lazy Flushing Optimization

IOTLB (I/O TLB) invalidation is expensive — it requires a command to the IOMMU
hardware and potentially waiting for completion. The lazy flushing optimization
batches invalidations to amortize the cost.

### Strict vs Lazy Modes

| Mode | Domain Type | Behavior | Security |
|------|-------------|----------|----------|
| **Strict** | `IOMMU_DOMAIN_DMA` | Flush IOTLB on every unmap | Higher (no stale translations) |
| **Lazy** | `IOMMU_DOMAIN_DMA_FQ` | Defer flush, batch invalidations | Lower (window for stale access) |

The mode is configurable via kernel parameter `iommu.strict=`:
- `0` = lazy (default on many systems for performance)
- `1` = strict (default when security is prioritized)

### Flush Queue Architecture

When lazy mode is active, unmapped IOVAs are queued instead of immediately
freed:

Source: [`drivers/iommu/dma-iommu.c:106`](https://elixir.bootlin.com/linux/latest/source/drivers/iommu/dma-iommu.c#L106)

```c
struct iova_fq_entry {
    unsigned long iova_pfn;        /* Start of IOVA range */
    unsigned long pages;           /* Number of pages */
    struct iommu_pages_list freelist;  /* Pages to free after flush */
    u64 counter;                   /* Flush counter at queue time */
};

struct iova_fq {
    spinlock_t lock;
    unsigned int head, tail;       /* Ring buffer indices */
    unsigned int mod_mask;         /* Ring size - 1 */
    struct iova_fq_entry entries[];  /* Flexible array */
};
```

### Flush Queue Configuration

| Parameter | Per-CPU Mode | Single Queue Mode |
|-----------|-------------|-------------------|
| Queue size | `IOVA_DEFAULT_FQ_SIZE` (256) | `IOVA_SINGLE_FQ_SIZE` (32768) |
| Timeout | `IOVA_DEFAULT_FQ_TIMEOUT` (10ms) | `IOVA_SINGLE_FQ_TIMEOUT` (1000ms) |

### `queue_iova()` Mechanism

Source: [`drivers/iommu/dma-iommu.c:197`](https://elixir.bootlin.com/linux/latest/source/drivers/iommu/dma-iommu.c#L197)

When an IOVA is freed in lazy mode, `queue_iova()` is called instead of
immediately freeing it:

1. Get the per-CPU flush queue
2. If the queue is full, flush old entries first
3. Add the IOVA to the queue with the current flush counter
4. Start the flush timer if not already running

### Counter-Based Synchronization

The cookie maintains two atomic counters:

- **`fq_flush_start_cnt`**: Incremented when an IOTLB flush is initiated
- **`fq_flush_finish_cnt`**: Incremented when a flush completes

When reclaiming entries from the flush queue, an entry is safe to free only
when its counter is less than `fq_flush_finish_cnt` — meaning the IOTLB flush
that covers it has completed.

### Timer-Based Deferred Flush

Source: [`drivers/iommu/dma-iommu.c:180`](https://elixir.bootlin.com/linux/latest/source/drivers/iommu/dma-iommu.c#L180)

```c
static void fq_flush_timeout(struct timer_list *t)
{
    struct iommu_dma_cookie *cookie = timer_container_of(cookie, t, fq_timer);
    /* Flush all per-CPU queues */
}
```

The timer fires after `fq_timeout` ms (default 10ms for per-CPU mode), flushing
any accumulated entries regardless of queue fullness.

### Lazy Flushing Flow

```
dma_unmap_*() with IOMMU_DOMAIN_DMA_FQ
  │
  ├── iommu_unmap_fast()          Remove page table entries
  │
  └── iommu_dma_free_iova()
        │
        ├── iotlb_gather.queued == true (lazy mode)
        │
        └── queue_iova(cookie, pfn, pages, &freelist)
              │
              ├── fq = this_cpu_ptr(cookie->percpu_fq)
              │
              ├── spin_lock(&fq->lock)
              │
              ├── fq_full(fq)?
              │   └── YES: fq_flush_iotlb()
              │             Flush old entries, free their IOVAs
              │
              ├── Add entry: {iova_pfn, pages, freelist, counter}
              │
              ├── Start timer if not running:
              │   mod_timer(&cookie->fq_timer,
              │     jiffies + msecs_to_jiffies(fq_timeout))
              │
              └── spin_unlock(&fq->lock)


Timer fires (10ms later):
  │
  └── fq_flush_timeout()
        │
        ├── Initiate IOTLB flush
        │   atomic64_inc(&fq_flush_start_cnt)
        │   iommu_iotlb_sync(domain)
        │   atomic64_inc(&fq_flush_finish_cnt)
        │
        └── For each per-CPU queue:
              ├── Reclaim entries where counter < fq_flush_finish_cnt
              └── free_iova_fast() for reclaimed IOVAs
```

### Performance Impact

```
Strict Mode (immediate flush):
  unmap → IOTLB flush (slow) → free IOVA → done
  Time:  ████████████████████████  per operation

Lazy Mode (flush queue):
  unmap → queue IOVA → done  (fast)
  ...
  timer → batch flush → free all queued IOVAs
  Time:  ████  per operation (amortized)

  Throughput improvement: 2-10x for high-IOPS workloads (NVMe, networking)
  Tradeoff: brief window where stale IOMMU translations exist
```

## 8. P2PDMA: Peer-to-Peer DMA Mapping

PCI Peer-to-Peer DMA (P2PDMA) allows one PCI device to DMA directly to
another PCI device's memory (typically a BAR region) without going through
system RAM. The DMA mapping layer must handle this specially because the
memory being mapped is not normal RAM — it's device MMIO space.

### What is P2PDMA?

In a typical DMA transfer, a device reads from or writes to system RAM. With
P2PDMA, a device reads from or writes to another device's BAR memory:

```
Normal DMA:
  ┌──────────┐              ┌──────────┐
  │  NVMe    │── DMA ──────►│  System  │
  │ Device   │              │  RAM     │
  └──────────┘              └──────────┘

P2PDMA (same PCI switch):
  ┌──────────┐              ┌──────────┐
  │  NVMe    │── DMA ──────►│  GPU     │
  │ Device   │   via PCIe   │  BAR mem │
  └──────────┘   switch     └──────────┘
  (data never touches system RAM or CPU caches)

P2PDMA (through host bridge):
  ┌──────────┐              ┌──────────────┐              ┌──────────┐
  │  NVMe    │── DMA ──────►│ Host Bridge  │─────────────►│  GPU     │
  │ Device   │              │ (Root Complex│              │  BAR mem │
  └──────────┘              │  routing)    │              └──────────┘
                            └──────────────┘
```

### P2PDMA Map Types

The kernel classifies P2P transfers based on the PCI topology:

Source: [`include/linux/pci-p2pdma.h:31`](https://elixir.bootlin.com/linux/latest/source/include/linux/pci-p2pdma.h#L31)

```c
enum pci_p2pdma_map_type {
    PCI_P2PDMA_MAP_NONE,             /* Not a P2P transfer */
    PCI_P2PDMA_MAP_NOT_SUPPORTED,    /* P2P not possible for this pair */
    PCI_P2PDMA_MAP_BUS_ADDR,         /* Direct via PCI switch, use bus addr */
    PCI_P2PDMA_MAP_THRU_HOST_BRIDGE, /* Goes through host bridge, use IOVA/phys */
};
```

The map type determines how the DMA layer programs the device's DMA address:

| Map Type | Routing | DMA Address Used | IOMMU Involved? | Cache Sync? |
|----------|---------|-----------------|-----------------|-------------|
| `MAP_NONE` | Normal RAM | phys_to_dma() or IOVA | Yes (if active) | Yes (non-coherent) |
| `MAP_BUS_ADDR` | PCI switch only | PCI bus address | No | No |
| `MAP_THRU_HOST_BRIDGE` | Host bridge | CPU physical or IOVA | Yes (if active) | No (MMIO) |
| `MAP_NOT_SUPPORTED` | — | Error (`-EREMOTEIO`) | — | — |

### P2PDMA Only Works Through Scatter-Gather and Block Iterator

P2PDMA pages are `ZONE_DEVICE` pages backed by device BAR memory, not regular
RAM. The single-mapping APIs (`dma_map_page`, `dma_map_single`) reject them:

```c
/* dma_map_page_attrs() rejects MMIO and zone device pages */
if (unlikely(attrs & DMA_ATTR_MMIO))
    return DMA_MAPPING_ERROR;
if (IS_ENABLED(CONFIG_DMA_API_DEBUG) &&
    WARN_ON_ONCE(is_zone_device_page(page)))
    return DMA_MAPPING_ERROR;
```

P2PDMA is only handled through:
1. **`dma_map_sg`** / **`dma_direct_map_sg`** — per-segment P2PDMA classification
2. **`blk_rq_dma_map_iter_start`** — block layer DMA iterator (preferred path)

### How P2PDMA Integrates with dma_map_sg

P2PDMA is handled per-segment inside the scatter-gather mapping functions.
Each segment's page is checked: is it a regular RAM page or a P2PDMA page?

**IOMMU path** (`iommu_dma_map_sg`): Both `MAP_THRU_HOST_BRIDGE` and `MAP_NONE`
break out of the switch and are handled identically — both get IOVA-mapped
using `iommu_map_sg()`:

Source: [`drivers/iommu/dma-iommu.c:1426`](https://elixir.bootlin.com/linux/latest/source/drivers/iommu/dma-iommu.c#L1426)

```c
switch (pci_p2pdma_state(&p2pdma_state, dev, sg_page(s))) {
case PCI_P2PDMA_MAP_THRU_HOST_BRIDGE:
    /* Mapped with regular IOVAs, same as non-P2P below */
    break;
case PCI_P2PDMA_MAP_NONE:
    break;
case PCI_P2PDMA_MAP_BUS_ADDR:
    /* Use PCI bus address directly, skip IOVA mapping */
    s->dma_address = pci_p2pdma_bus_addr_map(
        p2pdma_state.mem, sg_phys(s));
    sg_dma_mark_bus_address(s);
    continue;
default:
    ret = -EREMOTEIO;
    goto out_restore_sg;
}
/* Both MAP_THRU_HOST_BRIDGE and MAP_NONE reach here:
 * proceed with IOVA allocation and iommu_map_sg() */
```

**Direct path** (`dma_direct_map_sg`): `MAP_BUS_ADDR` uses PCI bus addresses.
`MAP_NONE` calls `dma_direct_map_phys()`. `MAP_THRU_HOST_BRIDGE` just breaks
without mapping — in practice, P2P through host bridge in the direct (non-IOMMU)
path is handled by the block layer DMA iterator instead, which uses
`dma_map_phys()` with `DMA_ATTR_MMIO`:

Source: [`kernel/dma/direct.c:454`](https://elixir.bootlin.com/linux/latest/source/kernel/dma/direct.c#L454)

```c
int dma_direct_map_sg(struct device *dev, struct scatterlist *sgl, int nents,
        enum dma_data_direction dir, unsigned long attrs)
{
    struct pci_p2pdma_map_state p2pdma_state = {};

    for_each_sg(sgl, sg, nents, i) {
        switch (pci_p2pdma_state(&p2pdma_state, dev, sg_page(sg))) {
        case PCI_P2PDMA_MAP_THRU_HOST_BRIDGE:
            /*
             * "must be mapped with CPU physical address
             *  and not PCI bus addresses."
             * Note: this break does not set sg->dma_address;
             * the block layer iterator path handles this case
             * properly with DMA_ATTR_MMIO via dma_map_phys().
             */
            break;

        case PCI_P2PDMA_MAP_NONE:
            sg->dma_address = dma_direct_map_phys(dev, sg_phys(sg),
                    sg->length, dir, attrs);
            break;

        case PCI_P2PDMA_MAP_BUS_ADDR:
            sg->dma_address = pci_p2pdma_bus_addr_map(
                p2pdma_state.mem, sg_phys(sg));
            sg_dma_mark_bus_address(sg);
            continue;

        default:
            ret = -EREMOTEIO;
            goto out_unmap;
        }
    }
}
```

### The `MAP_BUS_ADDR` Path (PCI Switch)

When two devices are behind the same PCI switch, data flows directly between
them without touching system RAM or traversing the host bridge. In this case:

1. **The DMA address is a PCI bus address**, not a CPU physical address or
   IOVA. It is computed from the provider's BAR base and an offset:

   ```c
   static inline dma_addr_t
   pci_p2pdma_bus_addr_map(struct p2pdma_provider *provider, phys_addr_t paddr)
   {
       return paddr + provider->bus_offset;
   }
   ```

2. **No IOMMU translation** — the PCI switch routes directly by bus address.
   The IOMMU is not in the data path.

3. **No cache sync** — the data is in device MMIO space, not cacheable RAM.

4. **`sg_dma_mark_bus_address(sg)`** flags the segment so that `dma_unmap_sg`
   knows to skip it (no IOMMU entry to tear down, no cache to invalidate):

   ```c
   /* In dma_direct_unmap_sg(): */
   for_each_sg(sgl, sg, nents, i) {
       if (sg_dma_is_bus_address(sg))
           sg_dma_unmark_bus_address(sg);  /* Just clear flag, no real unmap */
       else
           dma_direct_unmap_phys(dev, sg->dma_address, ...);
   }
   ```

### The `MAP_THRU_HOST_BRIDGE` Path

When P2P traffic must traverse the host bridge (root complex), the DMA address
must be something the host bridge can route. The kernel maintains an **allowlist**
of host bridges known to support P2P routing (`drivers/pci/p2pdma.c`). If the
bridge is not on the list, `MAP_NOT_SUPPORTED` is returned instead.

For allowed host bridges, the mapping depends on the backend:

- **IOMMU path** (`iommu_dma_map_sg`): Treats `MAP_THRU_HOST_BRIDGE` identically
  to `MAP_NONE` — both get regular IOVA mappings. The IOMMU creates page table
  entries pointing IOVA → BAR physical address.

- **Block layer iterator** (`blk_rq_dma_map_iter_start`): Sets `DMA_ATTR_MMIO`
  and calls `dma_map_phys()`. With `DMA_ATTR_MMIO`, `dma_direct_map_phys()`
  uses the raw physical address (`dma_addr = phys`, not `phys_to_dma()`).

`DMA_ATTR_MMIO` tells the DMA layer:
- Use raw physical address as DMA address (not `phys_to_dma()` translation)
- Skip cache sync operations (MMIO is not cacheable)
- Skip SWIOTLB bounce buffering (can't bounce MMIO)

### P2PDMA in the Block Layer DMA Iterator

The block layer's `blk_rq_dma_map_iter_start()` checks for P2PDMA at the
start of mapping:

Source: [`block/blk-mq-dma.c:170`](https://elixir.bootlin.com/linux/latest/source/block/blk-mq-dma.c#L170)

```c
/* Check first segment for P2P */
switch (pci_p2pdma_state(&iter->p2pdma, dma_dev,
             phys_to_page(vec.paddr))) {
case PCI_P2PDMA_MAP_BUS_ADDR:
    return blk_dma_map_bus(iter, &vec);   /* Use bus address directly */
case PCI_P2PDMA_MAP_THRU_HOST_BRIDGE:
    /* Fall through to normal mapping with DMA_ATTR_MMIO */
case PCI_P2PDMA_MAP_NONE:
    break;
default:
    iter->status = BLK_STS_INVAL;
    return false;
}
```

For `MAP_THRU_HOST_BRIDGE`, subsequent `dma_map_phys()` and `dma_iova_link()`
calls add `DMA_ATTR_MMIO`:

```c
static bool blk_dma_map_direct(...) {
    unsigned int attrs = 0;
    if (iter->p2pdma.map == PCI_P2PDMA_MAP_THRU_HOST_BRIDGE)
        attrs |= DMA_ATTR_MMIO;
    iter->addr = dma_map_phys(dma_dev, vec->paddr, vec->len,
            rq_dma_dir(req), attrs);
}
```

### P2PDMA Unmap

During unmap, the P2PDMA map type determines how to clean up:

- **`MAP_BUS_ADDR`**: Segments marked with `sg_dma_is_bus_address()` are simply
  cleared — no IOMMU entries to remove, no cache operations.

- **`MAP_THRU_HOST_BRIDGE`**: Unmapped normally via `dma_unmap_phys()` with
  `DMA_ATTR_MMIO` — IOMMU entries are removed but no cache sync is performed.

NVMe tracks the P2P type per-request via IOD flags and passes it to
`blk_rq_dma_unmap()`:

```c
/* In nvme_unmap_data() */
if (iod->flags & IOD_DATA_P2P)
    map = PCI_P2PDMA_MAP_BUS_ADDR;
else if (iod->flags & IOD_DATA_MMIO)
    map = PCI_P2PDMA_MAP_THRU_HOST_BRIDGE;

blk_rq_dma_unmap(req, dma_dev, &iod->dma_state, iod->total_len, map);
```

### `dma_pci_p2pdma_supported()`

Drivers can check P2PDMA support before attempting P2P transfers:

Source: [`kernel/dma/mapping.c:887`](https://elixir.bootlin.com/linux/latest/source/kernel/dma/mapping.c#L887)

```c
bool dma_pci_p2pdma_supported(struct device *dev)
{
    const struct dma_map_ops *ops = get_dma_ops(dev);
    /* P2PDMA works with direct DMA and default IOMMU, not custom ops */
    return !ops;
}
```

Only the built-in direct and IOMMU backends support P2PDMA. Devices with
custom `dma_map_ops` cannot use P2P transfers.

### Complete P2PDMA Flow Diagram

```
dma_map_sg() / blk_rq_dma_map_iter_start()
  │
  │ For each segment, check: is_pci_p2pdma_page(page)?
  │
  ├── NO (normal RAM page)
  │     └── Normal DMA mapping (direct / IOMMU / ops)
  │
  └── YES (P2PDMA page)
        │
        │ pci_p2pdma_state() determines routing:
        │
        ├── MAP_BUS_ADDR (via PCI switch, no host bridge)
        │     │
        │     ├── dma_addr = pci_p2pdma_bus_addr_map(provider, phys)
        │     │              (PCI bus address, not CPU physical)
        │     ├── sg_dma_mark_bus_address(sg)
        │     ├── No IOMMU mapping needed
        │     ├── No cache sync needed
        │     └── On unmap: just clear bus_address flag
        │
        ├── MAP_THRU_HOST_BRIDGE (goes through root complex)
        │     │  (requires host bridge on kernel allowlist)
        │     │
        │     ├── IOMMU SG path: treated same as MAP_NONE
        │     │     └── allocate IOVA, iommu_map(IOVA → bar_phys)
        │     │
        │     ├── Block iterator path: attrs |= DMA_ATTR_MMIO
        │     │     └── dma_map_phys(dev, bar_phys, size, dir, DMA_ATTR_MMIO)
        │     │           ├── Direct: dma_addr = phys (raw, no phys_to_dma)
        │     │           └── IOMMU:  allocate IOVA, iommu_map(IOVA → bar_phys)
        │     │
        │     ├── No cache sync (MMIO is uncacheable)
        │     └── On unmap: normal unmap (IOMMU entry removed if applicable)
        │
        └── MAP_NOT_SUPPORTED
              └── Return -EREMOTEIO (P2P not possible for this device pair)
```

## 9. NVMe-PCIe as Example

NVMe is an excellent real-world example of DMA API usage. It uses both coherent
and streaming DMA for different purposes.

### DMA Mask Setup

Source: [`drivers/nvme/host/pci.c:3687`](https://elixir.bootlin.com/linux/latest/source/drivers/nvme/host/pci.c#L3687)

```c
/* During nvme_probe() → nvme_pci_alloc_dev() */
if (dev->ctrl.quirks & NVME_QUIRK_DMA_ADDRESS_BITS_48)
    dma_set_mask_and_coherent(&pdev->dev, DMA_BIT_MASK(48));
else
    dma_set_mask_and_coherent(&pdev->dev, DMA_BIT_MASK(64));
dma_set_min_align_mask(&pdev->dev, NVME_CTRL_PAGE_SIZE - 1);
dma_set_max_seg_size(&pdev->dev, 0xffffffff);
```

Most NVMe devices use 64-bit DMA. Devices with the `DMA_ADDRESS_BITS_48` quirk
are limited to 48-bit addressing. The min align mask ensures IOVAs are aligned
to the NVMe controller page size (typically 4KB). The max segment size of 4GB
allows large contiguous DMA mappings.

### Queue Allocation (Coherent DMA)

NVMe Submission Queues (SQs) and Completion Queues (CQs) are allocated with
`dma_alloc_coherent` because the CPU and device continuously share them:

Source: [`drivers/nvme/host/pci.c:2103`](https://elixir.bootlin.com/linux/latest/source/drivers/nvme/host/pci.c#L2103)

```c
static int nvme_alloc_queue(struct nvme_dev *dev, int qid, int depth)
{
    struct nvme_queue *nvmeq = &dev->queues[qid];

    /* Allocate Completion Queue - shared between CPU and device */
    nvmeq->cqes = dma_alloc_coherent(dev->dev, CQ_SIZE(nvmeq),
                                     &nvmeq->cq_dma_addr, GFP_KERNEL);
    if (!nvmeq->cqes)
        goto free_nvmeq;

    /* Allocate Submission Queue */
    if (nvme_alloc_sq_cmds(dev, nvmeq, qid))
        goto free_cqdma;
    /* ... */
}
```

The DMA addresses (`cq_dma_addr`, `sq_dma_addr`) are programmed into the NVMe
controller's AQA/ASQ/ACQ registers:

```c
/* Program admin queue addresses into device registers */
writel(aqa, dev->bar + NVME_REG_AQA);
lo_hi_writeq(nvmeq->sq_dma_addr, dev->bar + NVME_REG_ASQ);
lo_hi_writeq(nvmeq->cq_dma_addr, dev->bar + NVME_REG_ACQ);
```

### Doorbell Shadow Buffer (Coherent DMA)

NVMe also uses `dma_alloc_coherent` for doorbell shadow buffers, which allow
the device to read doorbell values from memory instead of requiring MMIO writes:

Source: [`drivers/nvme/host/pci.c:471`](https://elixir.bootlin.com/linux/latest/source/drivers/nvme/host/pci.c#L471)

```c
dev->dbbuf_dbs = dma_alloc_coherent(dev->dev, mem_size,
                                    &dev->dbbuf_dbs_dma_addr, GFP_KERNEL);
dev->dbbuf_eis = dma_alloc_coherent(dev->dev, mem_size,
                                    &dev->dbbuf_eis_dma_addr, GFP_KERNEL);
```

### DMA Pool Setup

NVMe uses DMA pools for PRP/SGL descriptor lists. Pools efficiently allocate
small fixed-size DMA buffers from a pre-allocated coherent region:

Source: [`drivers/nvme/host/pci.c:598`](https://elixir.bootlin.com/linux/latest/source/drivers/nvme/host/pci.c#L598)

```c
pools->large = dma_pool_create_node("nvme descriptor page", dev->dev,
        NVME_CTRL_PAGE_SIZE, NVME_CTRL_PAGE_SIZE, 0, numa_node);

pools->small = dma_pool_create_node("nvme descriptor small", dev->dev,
        NVME_SMALL_POOL_SIZE, small_align, 0, numa_node);
```

### I/O Mapping Flow

When an I/O request arrives, NVMe maps the data buffer for DMA:

Source: [`drivers/nvme/host/pci.c:1216`](https://elixir.bootlin.com/linux/latest/source/drivers/nvme/host/pci.c#L1216)

```c
static blk_status_t nvme_map_data(struct request *req)
{
    struct nvme_iod *iod = blk_mq_rq_to_pdu(req);
    struct nvme_dev *dev = nvmeq->dev;

    /*
     * Fast path: single-segment requests skip the DMA iterator
     */
    if (blk_rq_nr_phys_segments(req) == 1) {
        ret = nvme_pci_setup_data_simple(req, use_sgl);
        if (ret != BLK_STS_AGAIN)
            return ret;
    }

    /* Multi-segment: use DMA map iterator */
    if (!blk_rq_dma_map_iter_start(req, dev->dev, &iod->dma_state, &iter))
        return iter.status;

    /* Build PRP or SGL list based on device capability */
    if (use_sgl)
        return nvme_pci_setup_data_sgl(req, &iter);
    else
        return nvme_pci_setup_data_prp(req, &iter);
}
```

#### Single-Segment Fast Path

Source: [`drivers/nvme/host/pci.c:1177`](https://elixir.bootlin.com/linux/latest/source/drivers/nvme/host/pci.c#L1177)

```c
static blk_status_t nvme_pci_setup_data_simple(struct request *req,
        enum nvme_use_sgl use_sgl)
{
    struct bio_vec bv = req_bvec(req);

    /* Direct dma_map_bvec for single bio_vec */
    dma_addr = dma_map_bvec(nvmeq->dev->dev, &bv, rq_dma_dir(req), 0);
    if (dma_mapping_error(nvmeq->dev->dev, dma_addr))
        return BLK_STS_RESOURCE;

    iod->flags |= IOD_SINGLE_SEGMENT;

    /* Fill PRP1 (or SGL) directly in the command */
    iod->cmd.common.dptr.prp1 = cpu_to_le64(dma_addr);
    /* ... */
}
```

#### Multi-Segment PRP List Building

For larger I/Os spanning multiple pages, NVMe builds PRP (Physical Region Page)
lists. Each PRP list entry is a `dma_pool_alloc`'d descriptor page containing
DMA addresses of data pages:

```c
/* From nvme_pci_setup_data_prp() */
prp_list = dma_pool_alloc(nvmeq->descriptor_pools.large, GFP_ATOMIC, &prp_dma);
/* Fill PRP list with DMA addresses from the iterator */
do {
    prp_list[i++] = cpu_to_le64(iter->dma_addr);
} while (blk_rq_dma_map_iter_next(req, dev->dev, iter));
```

### Unmapping

Source: [`drivers/nvme/host/pci.c:927`](https://elixir.bootlin.com/linux/latest/source/drivers/nvme/host/pci.c#L927)

```c
static void nvme_unmap_data(struct request *req)
{
    struct nvme_iod *iod = blk_mq_rq_to_pdu(req);
    struct device *dma_dev = nvmeq->dev->dev;

    if (iod->flags & IOD_SINGLE_SEGMENT) {
        /* Simple unmap for single-segment */
        dma_unmap_page(dma_dev, le64_to_cpu(iod->cmd.common.dptr.prp1),
                       iod->total_len, rq_dma_dir(req));
        return;
    }

    /* Unmap all DMA-mapped data segments via block layer iterator */
    if (!blk_rq_dma_unmap(req, dma_dev, &iod->dma_state,
                           iod->total_len, map)) {
        /* Free PRP or SGL chain data mappings */
        if (nvme_pci_cmd_use_sgl(&iod->cmd))
            nvme_free_sgls(req, ...);
        else
            nvme_free_prps(req, attrs);
    }

    /* Free descriptor pool allocations */
    if (iod->nr_descriptors)
        nvme_free_descriptors(req);
}
```

### NVMe I/O DMA Lifecycle

```
nvme_queue_rq()                         [Driver entry point]
  │
  ├── nvme_map_data(req)                [Map data buffer]
  │     │
  │     ├── Single segment?
  │     │   └── nvme_pci_setup_data_simple()
  │     │         └── dma_map_bvec()     ──► DMA API
  │     │               → get dma_addr
  │     │               → fill PRP1 in SQE
  │     │
  │     └── Multi segment?
  │         ├── blk_rq_dma_map_iter_start()  ──► DMA API (sg mapping)
  │         │
  │         └── nvme_pci_setup_data_prp() or nvme_pci_setup_data_sgl()
  │               ├── dma_pool_alloc()   ──► Get descriptor page
  │               ├── Fill PRP/SGL entries with dma_addr from iterator
  │               └── Write PRP1/PRP2 or SGL pointer in SQE
  │
  ├── nvme_start_request(req)           [Ring doorbell]
  │     └── SQE tail pointer → device
  │
  │ ─── Device performs DMA read/write ───
  │
  ├── nvme_irq() / nvme_poll_cq()      [Completion]
  │     └── Process CQE
  │
  └── nvme_unmap_data(req)              [Unmap data buffer]
        ├── dma_unmap_page()            ──► DMA API (single)
        │   or
        ├── blk_rq_dma_unmap()          ──► DMA API (sg)
        └── dma_pool_free()             ──► Free descriptor pages
```

## Summary

The Linux kernel DMA API provides a clean abstraction over three backends:

| Backend | When Used | Address Translation | Performance |
|---------|-----------|-------------------|-------------|
| **Direct** | No IOMMU, no custom ops | `phys_to_dma()` fixed offset | Fastest |
| **IOMMU** | `dev->dma_iommu == true` | IOVA → phys via IOMMU page tables | Slight overhead |
| **Custom** | Platform-specific `dma_map_ops` | Backend-dependent | Varies |

Key design principles:
- **Single API, multiple backends**: Drivers don't need to know which backend is active
- **Lazy optimization**: IOMMU flush queues amortize the cost of IOTLB invalidation
- **Bounce buffering**: SWIOTLB transparently handles devices with limited address ranges
- **Cache coherency management**: Automatic cache operations for non-coherent devices
- **Per-CPU optimizations**: IOVA rcache and per-CPU flush queues minimize contention

For driver developers, the key takeaways are:
1. Always set DMA masks before any DMA operation
2. Check `dma_mapping_error()` after every `dma_map_*` call
3. Use `dma_alloc_coherent` for long-lived shared structures
4. Use streaming DMA (`dma_map_*`) for I/O data buffers
5. Call `dma_sync_*` when reusing a buffer without remapping
