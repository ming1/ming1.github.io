# Understand AI Workload: CPU Side

AI workloads are commonly thought of as "GPU workloads," but the CPU plays a
critical and often underestimated role. The CPU is the **orchestrator, data
feeder, and bottleneck manager** for every AI pipeline — from loading and
preprocessing data, to launching GPU kernels, to coordinating distributed
training across hundreds of nodes.

This post examines what the CPU actually does during training and inference,
what Linux system resources matter most, and which applications form the
modern AI software stack.

## 1. Training: What the CPU Does

Training a model involves iterating over a dataset millions of times, computing
gradients, and updating model weights. While the heavy math (matrix multiplications,
convolutions) runs on the GPU, the CPU handles everything around it.

```
                     Training Pipeline: CPU vs GPU Roles
                     ────────────────────────────────────

  ┌─────────────────────────────────────────────────────────────────┐
  │                        CPU Domain                               │
  │                                                                 │
  │  Storage ──► DataLoader ──► Decode ──► Preprocess ──► Batch     │
  │   (SSD)     (workers)     (JPEG/     (resize,       (collate,   │
  │              threads)      Parquet)   normalize,     pad,        │
  │                                       tokenize,      pin_memory) │
  │                                       augment)                   │
  │                                                                 │
  └──────────────────────────┬──────────────────────────────────────┘
                             │  DMA / PCIe Transfer
                             │  (pinned memory → GPU VRAM)
                             ▼
  ┌─────────────────────────────────────────────────────────────────┐
  │                        GPU Domain                               │
  │                                                                 │
  │  Forward Pass ──► Loss Computation ──► Backward Pass ──► Update │
  │  (matmul,        (cross-entropy,      (gradient          (Adam, │
  │   attention,      MSE, etc.)           computation)       SGD)  │
  │   convolution)                                                  │
  └─────────────────────────────────────────────────────────────────┘
                             │
                             │  Gradient sync (multi-node)
                             ▼
  ┌─────────────────────────────────────────────────────────────────┐
  │                   CPU Domain (again)                            │
  │                                                                 │
  │  AllReduce control/coord ──► Checkpoint save ──► Logging/Metrics│
  │  (NCCL, gRPC, MPI)         (model state to     (TensorBoard,   │
  │                              NFS/Lustre/S3)      W&B)           │
  └─────────────────────────────────────────────────────────────────┘
```

### 1.1 Data Pipeline

The data pipeline is the most CPU-intensive part of training. A starved GPU —
sitting idle waiting for the next batch — is the most expensive waste in an AI
cluster.

**Data Loading:** Training datasets can be terabytes to petabytes. The CPU
must read data from storage (local NVMe, networked filesystem, or object
storage), decompress it, and deliver it to memory. PyTorch's `DataLoader`
spawns multiple worker processes (typically 4-16 per GPU) that use Python
`multiprocessing` to parallelize this work.

```
            DataLoader Architecture (PyTorch)
            ─────────────────────────────────

  Main Process                Worker Processes
  ────────────                ────────────────
  ┌──────────┐    fork()     ┌──────────────┐
  │ Training │──────────────►│ Worker 0     │──► read, decode,
  │ Loop     │               │              │    transform sample
  │          │    fork()     ├──────────────┤
  │ .next()  │──────────────►│ Worker 1     │──► read, decode,
  │  batch   │               │              │    transform sample
  │          │    fork()     ├──────────────┤
  │          │──────────────►│ Worker 2     │──► read, decode,
  │          │               │              │    transform sample
  │          │    fork()     ├──────────────┤
  │          │──────────────►│ Worker 3     │──► read, decode,
  │          │               │              │    transform sample
  └──────────┘               └──────────────┘
       ▲                            │
       │      shared memory         │
       └────────────────────────────┘
             (mp.Queue or
              shared memory)
```

**Decoding & Decompression:** Images are stored as JPEG/PNG/WebP; text
datasets as Parquet/JSON/CSV. The CPU must decode these into raw tensors.
For image training, JPEG decoding with `libjpeg-turbo` or NVIDIA DALI
(which can offload decoding to GPU) is a common bottleneck.

**Preprocessing & Augmentation:** Before a sample reaches the GPU, the CPU
applies transformations:

| Task | Examples | CPU Cost |
|------|----------|----------|
| Image preprocessing | Resize, crop, normalize, color jitter | Medium-High |
| Image augmentation | Random flip, rotation, mixup, cutout | Medium |
| Text tokenization | BPE encoding (tiktoken, SentencePiece) | Low-Medium |
| Audio preprocessing | Mel spectrogram, resampling | High |
| Tabular processing | Feature engineering, normalization | Low |

**Batching & Collation:** Individual samples are collated into a batch tensor,
often with padding (for variable-length sequences). The collated batch is then
copied to **pinned (page-locked) memory** for efficient DMA transfer to the GPU.

At the kernel level, pinned memory uses `mlock()` to prevent the pages from
being swapped out, and the CUDA runtime registers the memory region with the
GPU's DMA engine via IOMMU mappings. This avoids an extra copy through a
bounce buffer:

```
  Without pinned memory:              With pinned memory:
  ─────────────────────               ────────────────────

  User buffer (pageable)              User buffer (pinned/mlock'd)
       │                                   │
       ▼                                   │  DMA directly
  Staging buffer (pinned)                  │  (zero-copy)
       │                                   │
       ▼  DMA                             ▼
  GPU VRAM                            GPU VRAM

  Two copies, higher latency          One copy, lower latency
```

### 1.2 Orchestration

The CPU is the control plane for GPU computation.

**Graph Compilation & Kernel Launch:** Modern frameworks (PyTorch 2.0+
`torch.compile`, JAX's XLA, TensorFlow's XLA) compile the computation graph
into optimized GPU kernels. The compilation happens on the CPU using LLVM-based
compilers (Triton, XLA). Each GPU operation is a **kernel launch** — the CPU
queues work onto the GPU's command stream via the CUDA driver API.

In eager mode (default PyTorch), the CPU launches kernels one-by-one, which
can cause **launch overhead** if kernels are small. `torch.compile` and CUDA
Graphs batch multiple operations to reduce this overhead:

```
  Eager mode:                    torch.compile / CUDA Graph:
  ───────────                    ───────────────────────────

  CPU: launch k1                 CPU: launch graph (k1,k2,...,kN)
  CPU: wait for k1 ready              │
  CPU: launch k2                      ▼
  CPU: wait for k2 ready         GPU: k1 → k2 → ... → kN
  CPU: launch k3                      (no CPU round-trips)
  ...
  (CPU-GPU ping-pong)
```

**Learning Rate Scheduling & Optimizer State:** While the GPU computes gradients,
some optimizer logic runs on the CPU — learning rate schedules, gradient clipping
decisions, and early stopping checks. For optimizers like Adam, the optimizer
states (first and second moment estimates) are typically stored on GPU, but
hybrid CPU-GPU optimizers (like DeepSpeed ZeRO-Offload) deliberately move
optimizer states to CPU RAM to free GPU VRAM for larger models.

**Checkpointing:** Periodically, the model state is saved to persistent storage.
For a 70B parameter model, the weights alone in fp16 are ~140 GB; a full
training checkpoint including Adam optimizer states (fp32 moments) can be
~420 GB. The CPU coordinates serialization and writes to storage (often in
parallel with continued training). Asynchronous checkpointing avoids blocking
the training loop:

```
  Training step N  ──►  Snapshot state to CPU memory (fast, memcpy)
                              │
  Training step N+1 ──►      │  Background thread: write to disk/NFS
  Training step N+2 ──►      │  (overlapped with training)
                              ▼
                        Checkpoint saved
```

### 1.3 CPU-GPU Data Transfer

The interconnect between CPU and GPU is a critical path. Understanding the
hardware topology is essential.

```
              Typical Multi-GPU Server Topology
              ─────────────────────────────────

  ┌─────────────────────────────────────────────────────┐
  │                    CPU Socket 0                      │
  │  ┌──────┐  ┌──────┐                                │
  │  │Core 0│  │Core N│   DDR5 Memory (512 GB)          │
  │  └──────┘  └──────┘   Bandwidth: ~200 GB/s          │
  │       │         │                                    │
  │       └────┬────┘                                    │
  │            │  PCIe Gen5 Root Complex                  │
  │       ┌────┴────┐                                    │
  │       ▼         ▼                                    │
  │   ┌──────┐  ┌──────┐                                │
  │   │GPU 0 │  │GPU 1 │  PCIe 5.0 x16: ~64 GB/s each  │
  │   └──┬───┘  └──┬───┘                                │
  │      │ NVLink   │                                    │
  │      └────┬─────┘  NVLink: ~900 GB/s (H100, bidir)  │
  └───────────┼─────────────────────────────────────────┘
              │  UPI / Infinity Fabric
  ┌───────────┼─────────────────────────────────────────┐
  │           ▼        CPU Socket 1                      │
  │   ┌──────┐  ┌──────┐                                │
  │   │GPU 2 │  │GPU 3 │                                │
  │   └──────┘  └──────┘                                │
  │                    DDR5 Memory (512 GB)              │
  └─────────────────────────────────────────────────────┘
```

Key data transfer concepts:

| Transfer Type | Mechanism | Bandwidth | CPU Involvement |
|---------------|-----------|-----------|-----------------|
| Host→Device | PCIe DMA (pinned mem) | ~32-64 GB/s (Gen4-Gen5) | Sets up DMA descriptors |
| Device→Device (same node) | NVLink / NVSwitch | ~450-900 GB/s | Minimal (NCCL handles) |
| Device→Device (cross-node) | RDMA (GPUDirect) | ~25-100 GB/s per link | Minimal after setup |
| Device→Host | PCIe DMA | ~32-64 GB/s (Gen4-Gen5) | Receives DMA interrupt |

**IOMMU and DMA:** The Linux IOMMU subsystem (`drivers/iommu/`) translates
between device-visible I/O virtual addresses and physical memory addresses.
For AI workloads, the NVIDIA driver typically uses its own address space
management, but the IOMMU must be properly configured — misconfiguration
causes DMA failures or falls back to slow bounce buffering.

**GPUDirect RDMA:** Allows network adapters to DMA directly into GPU VRAM,
bypassing CPU memory entirely. The CPU sets up the RDMA connection and
memory registrations, but the actual data movement happens without CPU
involvement. This is essential for multi-node training.

### 1.4 Distributed Training Coordination

Large model training (LLMs, vision foundation models) runs on hundreds to
thousands of GPUs across many nodes. The CPU is the coordination backbone.

```
         Distributed Training: Data Parallel (AllReduce)
         ────────────────────────────────────────────────

  Node 0                          Node 1
  ┌──────────────────┐           ┌──────────────────┐
  │ CPU: coordinator │           │ CPU: coordinator │
  │ ┌──────┐┌──────┐│           │ ┌──────┐┌──────┐│
  │ │GPU 0 ││GPU 1 ││           │ │GPU 2 ││GPU 3 ││
  │ │grad_0││grad_1││           │ │grad_2││grad_3││
  │ └──┬───┘└──┬───┘│           │ └──┬───┘└──┬───┘│
  │    │NVLink  │    │           │    │NVLink  │    │
  │    └───┬────┘    │           │    └───┬────┘    │
  │        │         │           │        │         │
  └────────┼─────────┘           └────────┼─────────┘
           │    InfiniBand / RoCE          │
           │◄──────────────────────────────┤
           │      AllReduce (Ring)         │
           │──────────────────────────────►│
           │                               │
      avg(grad_0..3)                  avg(grad_0..3)
      on all GPUs                     on all GPUs
```

**NCCL (NVIDIA Collective Communications Library):** Handles GPU-to-GPU
communication. While the actual data transfer happens between GPUs (via
NVLink or RDMA), the CPU runs the NCCL control thread that:
- Establishes connections (TCP/IB) during initialization
- Computes communication topology (ring, tree, or hybrid algorithms)
- Handles error detection and recovery

**Parallelism Strategies and CPU Load:**

| Strategy | Description | CPU Overhead |
|----------|-------------|-------------|
| Data Parallel (DDP) | Same model on each GPU, split data, AllReduce gradients | Low — just gradient sync |
| Pipeline Parallel | Model split across GPUs by layer, micro-batches pipelined | Medium — scheduling micro-batches |
| Tensor Parallel | Individual layers split across GPUs | Low — mostly GPU-GPU |
| Expert Parallel (MoE) | Mixture-of-experts routing | Medium — token dispatch routing |
| ZeRO (Stage 1-3) | Partition optimizer/gradient/param states across GPUs | High — CPU manages scattering/gathering |
| FSDP | Fully Sharded Data Parallel (ZeRO-3 in PyTorch) | High — all-gather before forward, reduce-scatter after backward |

**gRPC and Parameter Servers:** Some distributed training architectures use
gRPC-based parameter servers where the CPU aggregates gradients from multiple
workers. This is increasingly rare for LLM training (AllReduce is preferred)
but still used for recommendation models with large sparse embedding tables
that don't fit in GPU VRAM.

## 2. Inference: What the CPU Does

Inference has a fundamentally different CPU profile compared to training:
lower throughput demands per request, but strict latency requirements and
much more complex request scheduling.

```
                    Inference Pipeline: CPU vs GPU Roles
                    ────────────────────────────────────

  ┌─────────────────────────────────────────────────────────────┐
  │                       CPU Domain                            │
  │                                                             │
  │  HTTP Request ──► Tokenize ──► Batch ──► Schedule           │
  │  (FastAPI,       (tiktoken,   (dynamic   (continuous        │
  │   gRPC)           SPM)         batching)  batching)         │
  │                                                             │
  └─────────────────────────┬───────────────────────────────────┘
                            │
                            ▼
  ┌─────────────────────────────────────────────────────────────┐
  │                       GPU Domain                            │
  │                                                             │
  │  Prefill ──────────────► Decode (autoregressive)            │
  │  (process full prompt)   (one token at a time)              │
  │  (compute-bound)         (memory-bandwidth-bound)           │
  │                                                             │
  └─────────────────────────┬───────────────────────────────────┘
                            │
  ┌─────────────────────────┼───────────────────────────────────┐
  │                       CPU Domain (again)                    │
  │                                                             │
  │  Detokenize ──► Sampling ──► Stream Response ──► Metrics    │
  │  (token→text)   (top-p,     (SSE/WebSocket)    (latency,   │
  │                  top-k,                          throughput) │
  │                  temperature)                                │
  └─────────────────────────────────────────────────────────────┘
```

**Prefill and Decode — the Two Phases of Transformer Inference:**

A transformer-based LLM generates text in two distinct phases. Understanding
them is key to understanding the CPU and memory profile of inference.

First, some notation used throughout this section:

```
  Transformer notation:
  ─────────────────────

  X       — Input matrix. Each row is one token's embedding vector.
             Shape: [N × d_model] where N = number of tokens.
             Before the first layer, X comes from the embedding table
             (mapping token IDs → vectors). For subsequent layers,
             X is the output of the previous layer.

  d_model — Hidden dimension of the model. This is the width of the
             embedding vector for each token. For LLaMA-70B, d_model = 8192.
             Every token is represented as a vector of 8192 numbers.

  W_q, W_k, W_v — Learned weight matrices that project X into Query,
             Key, and Value spaces. Shape: [d_model × d_model] each.
             These are the main parameters of the model — the "knowledge"
             learned during training. Each layer has its own set.

  W_o     — Output projection weight matrix after attention.
  W_up, W_down — FFN (feed-forward network) weight matrices.

  d_k     — Dimension per attention head = d_model / n_heads.
             For LLaMA-70B: 8192 / 64 = 128.

  Example sizes (LLaMA-70B):
    d_model = 8192
    n_layers = 80
    n_heads = 64 (query), n_kv_heads = 8 (GQA)
    Each W_q: [8192 × 8192] = 134 million parameters per layer
    Total across all layers and all weight matrices: ~70 billion parameters
```

**Prefill (prompt processing):** The entire input prompt is fed through the
transformer in one pass. At each transformer layer, the self-attention
mechanism computes Query, Key, and Value matrices from **all** input tokens
simultaneously:

```
  Prefill: processing prompt "The capital of France is"
  ─────────────────────────────────────────────────────

  Input: [The, capital, of, France, is]  (5 tokens, processed in parallel)

  Each token is first converted to a vector via the embedding table:
    "The"     → [0.12, -0.34, ..., 0.56]  (8192 numbers)
    "capital" → [0.78, 0.01, ..., -0.23]   (8192 numbers)
    ...
  Together they form X: a [5 × 8192] matrix.

  For each of the 80 layers:
  ┌──────────────────────────────────────────────────────────┐
  │ 1. Linear projections (matrix × matrix):                 │
  │    Q = X · W_q    [5 × 8192] · [8192 × 8192]            │
  │    K = X · W_k    [5 × 8192] · [8192 × 1024] (GQA: 8kv) │
  │    V = X · W_v    [5 × 8192] · [8192 × 1024] (GQA: 8kv) │
  │                                                          │
  │ 2. Attention (all tokens attend to all prior tokens):    │
  │    scores = Q · K^T / √d_k       [5 × 5] attention map  │
  │    attn = softmax(scores) · V     [5 × d_head]           │
  │                                                          │
  │ 3. Output projection + FFN:                              │
  │    out = attn · W_o               matrix × matrix        │
  │    ffn = GELU(out · W_up) · W_down  (two large matmuls)  │
  │                                                          │
  │ 4. Save K, V to KV cache for this layer                  │
  └──────────────────────────────────────────────────────────┘

  Compute profile: matrix × matrix (all N tokens at once)
  → High arithmetic intensity → GPU compute-bound
  → KV cache is populated as a side effect
```

Because all tokens are processed in parallel, the weight matrices are
multiplied against a **matrix** (N tokens × hidden_dim), giving high
arithmetic intensity — the GPU's compute units stay busy.

**Decode (token generation):** After prefill, the model generates output
tokens one at a time. Each decode step takes the **single** most recent
token, runs it through all layers, and produces the next token:

```
  Decode: generating token after "The capital of France is"
  ────────────────────────────────────────────────────────

  Input: [Paris]  (1 new token)

  For each of the 80 layers:
  ┌──────────────────────────────────────────────────────────┐
  │ 1. Linear projections (vector × matrix):                 │
  │    q = x · W_q    [1 × 8192] · [8192 × 8192]            │
  │    k = x · W_k    [1 × 8192] · [8192 × 1024] → cache    │
  │    v = x · W_v    [1 × 8192] · [8192 × 1024] → cache    │
  │                                                          │
  │ 2. Attention (new token attends to ALL cached tokens):   │
  │    scores = q · K_cache^T    [1 × seq_len] (read cache)  │
  │    attn = softmax(scores) · V_cache  [1 × d_head]        │
  │                                                          │
  │ 3. Output projection + FFN:                              │
  │    out = attn · W_o            vector × matrix            │
  │    ffn = GELU(out · W_up) · W_down                       │
  │                                                          │
  │ 4. Append new k, v to KV cache                           │
  └──────────────────────────────────────────────────────────┘

  Compute profile: vector × matrix (only 1 token at a time)
  → Low arithmetic intensity → Memory bandwidth-bound
  → KV cache grows by one position each step
  → Must read ALL weights (~140 GB for 70B) for just 1 token
```

The difference is dramatic. Prefill processes N tokens and reuses weights
across all of them. Decode processes 1 token but still reads every weight
in the model. This makes decode ~N times less compute-efficient per byte
of memory read:

| | Prefill | Decode |
|--|---------|--------|
| **Tokens processed** | N (full prompt) | 1 (per step) |
| **Matmul shape** | [N × d] · [d × d] | [1 × d] · [d × d] |
| **FLOPs per weight byte** | ~N | ~1 |
| **Bottleneck** | GPU compute (TFLOPS) | Memory bandwidth (GB/s) |
| **KV cache** | Created (written) | Read + appended |
| **Latency metric** | Time-to-first-token (TTFT) | Time-per-output-token (TPOT) |

This two-phase nature shapes everything in inference serving: the scheduler
must handle prefill and decode differently, memory allocation for KV cache
must be dynamic, and batching strategies must account for the fact that
prefill requests are compute-heavy while decode requests are bandwidth-heavy.

### 2.1 Request Handling & Batching

Unlike training (where batch size is fixed), inference must handle requests
arriving at unpredictable rates with varying sequence lengths.

**Dynamic Batching:** Collects requests over a short time window (e.g., 10ms)
and groups them into a batch. Simple but wastes GPU time — short requests
wait for long ones to finish.

**Continuous Batching (Iteration-Level Batching):** Used by vLLM, TensorRT-LLM,
and TGI. The CPU scheduler inserts new requests into the running batch at
each decode iteration. When a request finishes (hits EOS token), its slot is
immediately freed for a new request:

```
  Time ──────────────────────────────────────────────────►

  Slot 0: [Req A: prefill ── decode ── decode ── EOS]  [Req D: prefill ──...
  Slot 1: [Req B: prefill ── decode ── decode ── decode ── decode ── EOS]
  Slot 2: [Req C: prefill ── decode ── EOS]  [Req E: prefill ── decode ──...

  ───────────────────────────────────────────────────────────────────────────
  CPU scheduler decisions at each step:
    • Which pending requests to admit?  (memory budget check)
    • Which running requests to preempt? (if memory pressure)
    • Which requests are done?          (EOS / max_length check)
```

The CPU scheduler must make these decisions at every decode step (~10-50ms),
so scheduler overhead directly impacts latency.

### 2.2 Pre/Post Processing

**Tokenization:** The CPU converts input text into token IDs using a trained
tokenizer. Modern tokenizers (tiktoken for GPT, SentencePiece for LLaMA) use
BPE (Byte Pair Encoding). Tokenization is fast (~microseconds per request)
but must happen on the CPU since it's inherently sequential string processing.

**Sampling:** After the GPU produces logits (probabilities for the next token),
the CPU often handles sampling strategies:
- Temperature scaling
- Top-k filtering (keep only k highest-probability tokens)
- Top-p (nucleus) sampling (keep tokens summing to probability p)
- Repetition penalty
- Beam search (maintaining multiple candidate sequences)

Some inference engines do sampling on the GPU for better throughput, but many
still use CPU sampling because of the complex control flow involved (e.g.,
structured output / grammar-guided decoding).

**Detokenization & Streaming:** As tokens are produced one-by-one, the CPU
decodes them back to text and streams the response over HTTP (Server-Sent
Events or WebSocket). UTF-8 multibyte characters require the CPU to buffer
tokens until a complete character is formed.

### 2.3 KV Cache Management

The KV (Key-Value) cache is the largest memory consumer during inference. For
each token generated, the attention keys and values from all previous tokens
must be stored to avoid recomputation.

```
  KV Cache Memory per Request:
  ────────────────────────────

  KV_size = 2 × n_layers × n_kv_heads × head_dim × seq_len × dtype_size

  Example (LLaMA-70B, 4K context, fp16):
  LLaMA-70B uses GQA (Grouped Query Attention): 64 query heads, but only
  8 KV heads — keys and values are shared across groups of 8 query heads.

  = 2 × 80 × 8 × 128 × 4096 × 2 bytes
  = ~1.25 GB per request

  With 32 concurrent requests at 4K context:
  = ~40 GB of KV cache alone

  Note: without GQA (standard MHA with 64 KV heads), this would be
  ~10 GB per request — GQA reduces KV cache by 8x.
```

The CPU manages KV cache allocation and lifecycle. **PagedAttention** (pioneered
by vLLM) manages KV cache like OS virtual memory — allocating fixed-size blocks
and mapping them to requests via a page table maintained on the CPU:

```
  PagedAttention: KV Cache as Virtual Memory
  ───────────────────────────────────────────

  CPU-side Page Table               GPU VRAM (Physical KV Blocks)
  ┌─────────────────┐              ┌──────────────────────────┐
  │ Req A:          │              │ Block 0: [Req A, tok 0-15]│
  │   page 0 → blk 0│             │ Block 1: [Req B, tok 0-15]│
  │   page 1 → blk 3│             │ Block 2: [Req A, tok 32-47]│
  │   page 2 → blk 5│             │ Block 3: [Req A, tok 16-31]│
  │ Req B:          │              │ Block 4: [free]            │
  │   page 0 → blk 1│             │ Block 5: [Req A, tok 48-63]│
  │   page 1 → blk 6│             │ Block 6: [Req B, tok 16-31│
  └─────────────────┘              └──────────────────────────┘

  • No internal fragmentation (fixed block size)
  • Copy-on-write for shared prefixes (prompt caching)
  • CPU scheduler checks free blocks before admitting new requests
```

### 2.4 CPU-Only Inference

Not all inference requires a GPU. For smaller models or cost-sensitive
deployments, CPU-only inference is viable and increasingly optimized.

**llama.cpp / GGML:** Runs quantized models (Q4, Q5, Q8) entirely on the CPU
using hand-tuned SIMD kernels (AVX2, AVX-512, ARM NEON). A 7B model quantized
to Q4_K_M fits in ~4 GB of RAM and runs at ~20-40 tokens/second on a modern
CPU.

**ONNX Runtime:** Supports CPU inference with kernel optimizations including
Intel MKL-DNN (oneDNN) and ARM Compute Library. Used for non-LLM models
(vision, speech, recommendation) where latency requirements are less strict.

**Key CPU features for CPU-only inference:**

| Feature | Use | Impact |
|---------|-----|--------|
| AVX-512 | 512-bit SIMD vector operations | 2-4x speedup for quantized matmul |
| AVX-512 VNNI | Int8/Int16 dot product | Essential for INT8 quantized models |
| AMX (Intel) | Matrix multiplication tiles | Up to 8x for bf16/int8 matmul |
| SVE/SVE2 (ARM) | Scalable vector operations | Competitive with x86 for inference |
| Large L3 cache | Keeps model weights near cores | Reduces memory bandwidth pressure |
| High memory bandwidth | Feeds matrix operations | Often the bottleneck for large models |

## 3. Training vs Inference: CPU Role Comparison

| Aspect | Training | Inference |
|--------|----------|-----------|
| **Data pipeline** | Heavy — continuous streaming, decode, augment | Light — single request preprocessing |
| **Batch management** | Fixed batch size, simple | Dynamic/continuous batching, complex scheduling |
| **GPU kernel launch** | Many kernels per step, compilation overhead | Repetitive decode kernel, CUDA Graphs effective |
| **Memory management** | Gradient + optimizer state + activations | KV cache lifecycle (PagedAttention) |
| **Network coordination** | AllReduce, gradient sync across nodes | Load balancing across replicas |
| **Storage I/O** | Sustained read (dataset) + periodic write (checkpoint) | Model loading at startup, minimal after |
| **Latency sensitivity** | Throughput-oriented (minutes per epoch) | Latency-critical (ms per token) |
| **CPU core usage** | Many cores for DataLoader workers | Fewer cores, but low-latency scheduling |

## 4. Linux System Requirements

### 4.1 Memory

Memory is the most critical CPU-side resource for AI workloads. Both capacity
and bandwidth matter.

**Capacity Requirements:**

| Workload | Minimum | Recommended | Why |
|----------|---------|-------------|-----|
| Training (7B model) | 64 GB | 128 GB | DataLoader workers + CPU tensors + OS |
| Training (70B model) | 256 GB | 512 GB-1 TB | ZeRO offload needs CPU RAM for optimizer states |
| Inference (7B, GPU) | 32 GB | 64 GB | Model loaded to GPU, CPU needs headroom |
| Inference (70B, GPU) | 128 GB | 256 GB | KV cache overflow, request buffers |
| Inference (7B, CPU-only) | 16 GB | 32 GB | Quantized model + KV cache in RAM |

**NUMA Topology:** Multi-socket servers have Non-Uniform Memory Access.
Accessing memory attached to a remote socket costs 1.5-2x latency. For AI
workloads, NUMA-aware placement is critical:

```
  NUMA Effects on AI Workload
  ───────────────────────────

  Socket 0 (NUMA Node 0)              Socket 1 (NUMA Node 1)
  ┌────────────────────┐              ┌────────────────────┐
  │ Cores 0-31         │              │ Cores 32-63        │
  │ DDR5: 256 GB       │    UPI       │ DDR5: 256 GB       │
  │ GPU 0, GPU 1       │◄───────────►│ GPU 2, GPU 3       │
  │ NIC 0              │   ~40 GB/s   │ NIC 1              │
  └────────────────────┘              └────────────────────┘

  GOOD: DataLoader workers for GPU 0 run on cores 0-31
        (local memory access, ~100 ns latency)

  BAD:  DataLoader workers for GPU 0 run on cores 32-63
        (remote memory access, ~150-200 ns latency)

  Use: numactl --cpunodebind=0 --membind=0 python train.py
  Or:  taskset + set_mempolicy() in code
```

**Huge Pages:** AI workloads allocate large contiguous buffers (gigabytes for
tensors). Standard 4 KB pages cause massive TLB (Translation Lookaside Buffer)
pressure — a 100 GB allocation requires ~26 million page table entries.

Huge pages (2 MB or 1 GB) reduce TLB misses dramatically:

```bash
# Enable transparent huge pages (THP)
echo always > /sys/kernel/mm/transparent_hugepage/enabled

# Or use explicit huge pages for more predictable performance
echo 16384 > /proc/sys/vm/nr_hugepages   # Reserve 32 GB of 2MB pages

# Verify
cat /proc/meminfo | grep -i huge
```

At the kernel level, `madvise(MADV_HUGEPAGE)` hints to the kernel that a
memory region would benefit from transparent huge pages. The CUDA driver
typically uses this for pinned memory allocations.

**Memory Bandwidth:** Training data pipelines and CPU-only inference are often
**memory-bandwidth-bound**, not compute-bound. DDR5-4800 provides ~38.4 GB/s
per channel (4800 MT/s × 64 bits ÷ 8); a dual-socket server with 8 channels
per socket gets ~614 GB/s aggregate (or ~922 GB/s with 12-channel EPYC
platforms).

Why does memory bandwidth matter so much for inference? During autoregressive
decoding, **each generated token requires a full forward pass through all model
layers** — reading essentially all weight data from memory. For a 70B fp16
model, that means reading ~140 GB of weights to produce a single token. The
computation itself is a matrix-vector multiply (tiny input × huge weight
matrix), so the arithmetic intensity is very low — the processor spends most
of its time waiting for data, not computing:

```
  Prefill vs Decode: Why decode is memory-bound
  ──────────────────────────────────────────────

  Prefill (process full prompt):
    Input:   [N tokens] × weight matrix  →  matrix × matrix
    Compute: HIGH (many FLOPs per byte read)
    Bottleneck: GPU compute (FLOPS)

  Decode (generate 1 token):
    Input:   [1 token]  × weight matrix  →  vector × matrix
    Compute: LOW (few FLOPs per byte read)
    Bottleneck: Memory bandwidth (GB/s)

  Both read ~140 GB of weights, but prefill amortizes the cost
  across thousands of tokens. Decode cannot.
```

At ~614 GB/s aggregate bandwidth, a CPU-only system can do roughly
~4 decode steps/second for a single 70B fp16 model (614 ÷ 140 ≈ 4.4),
ignoring KV cache reads and other overhead. Batching helps — N concurrent
requests share the same weight reads — but memory bandwidth remains the
fundamental ceiling for decode throughput.

### 4.2 CPU

**Core Count vs Clock Speed:**

| Workload | Preference | Reason |
|----------|-----------|--------|
| Training data pipeline | More cores (32-64+) | DataLoader parallelism, many workers per GPU |
| Inference scheduling | Fewer, faster cores | Low-latency scheduling decisions |
| CPU-only inference | Balanced (16-32 cores) | Matrix ops parallelize well but need clock speed |
| Distributed coordination | Moderate cores | NCCL control, gRPC servers |

A typical 8-GPU training node runs 8×8 = 64 DataLoader workers. Each worker
does I/O, decoding, and preprocessing — so 64+ physical cores (128+ threads
with SMT) is common in training servers. AMD EPYC (96-128 cores per socket)
and Intel Xeon Sapphire Rapids / Emerald Rapids are popular choices.

**Instruction Set Extensions:**

| Extension | Use in AI | Platform |
|-----------|-----------|----------|
| AVX2 | 256-bit SIMD, baseline for most AI code | Intel/AMD (2013+) |
| AVX-512 | 512-bit SIMD, significant speedup for quantized inference | Intel Xeon, AMD Zen4+ |
| AVX-512 VNNI | INT8/INT16 vector dot products | Intel (Ice Lake+), AMD (Zen4+) |
| AMX | Tile-based matrix multiply (bf16, int8) | Intel Sapphire Rapids+ |
| AES-NI | Encrypted data pipeline, TLS for distributed | Intel/AMD |
| ARM SVE/SVE2 | Scalable vector processing | ARM Neoverse V1/V2, Grace |
| ARM SME | Scalable matrix extensions | ARM Neoverse V3+ |

**CPU Scheduling:** For latency-sensitive inference, the Linux CFS (Completely
Fair Scheduler) may introduce unwanted scheduling jitter. Options:

```bash
# Pin inference process to specific cores (avoid migration)
taskset -c 0-15 ./inference_server

# Set real-time priority for the scheduler thread
chrt -f 50 ./scheduler_thread

# Disable CPU frequency scaling for consistent performance
echo performance | tee /sys/devices/system/cpu/cpu*/cpufreq/scaling_governor

# Isolate cores from the general scheduler (boot parameter)
# isolcpus=16-31  (kernel boot param)
```

### 4.3 Storage I/O

**Training Dataset Access:**

| Storage Type | Bandwidth | Latency | Use Case |
|-------------|-----------|---------|----------|
| Local NVMe SSD | 3-14 GB/s per drive | ~10-100 μs | Single-node training, fast random access |
| NVMe-oF (over fabric) | 10-25 GB/s | ~50-200 μs | Shared storage with near-local performance |
| Lustre / GPFS | 10-100+ GB/s aggregate | ~100-500 μs | HPC clusters, parallel filesystem |
| NFS v4.2 / pNFS | 1-50 GB/s (depending on layout) | ~200 μs-1 ms | Shared datasets, easier management |
| S3 / Object Storage | 10-100+ Gbps (many connections) | ~5-50 ms | Cloud training, infinite capacity |

**Dataset Streaming Patterns:**

Sequential reads dominate training workloads (iterating over shuffled datasets).
The Linux kernel's readahead mechanism (`/sys/block/nvme0n1/queue/read_ahead_kb`)
can be tuned to prefetch more aggressively:

```bash
# Increase readahead for training data drives (default is 128 KB)
echo 4096 > /sys/block/nvme0n1/queue/read_ahead_kb

# Or per-process with fadvise
posix_fadvise(fd, 0, 0, POSIX_FADV_SEQUENTIAL);
```

**Checkpoint I/O:** Checkpoints are large (tens to hundreds of GB), written
periodically. The key concern is write throughput and ensuring checkpointing
doesn't stall training:

```
  Checkpoint I/O Strategy
  ───────────────────────

  Option 1: Synchronous (simple but blocks training)
  ──────────────────────────────────────────────────
  Training step 1000 → [BLOCKED: write 140 GB checkpoint] → step 1001

  Option 2: Async via background thread
  ──────────────────────────────────────
  Training step 1000 → snapshot to RAM → step 1001 → step 1002 → ...
                              │
                              └──► background: write 140 GB to disk

  Option 3: Distributed checkpointing (FSDP/DeepSpeed)
  ────────────────────────────────────────────────────
  Each rank saves its own shard in parallel
  Node 0: write shard_0 (17.5 GB)  ──┐
  Node 1: write shard_1 (17.5 GB)  ──┤  parallel writes
  ...                                 │  total: 140 GB across 8 nodes
  Node 7: write shard_7 (17.5 GB)  ──┘  = ~17.5 GB each
```

**io_uring for Async I/O:** Modern AI data loaders can leverage Linux's
`io_uring` (since kernel 5.1) for high-performance async I/O. Unlike the
older `aio` interface, `io_uring` uses shared ring buffers between user and
kernel space, avoiding syscall overhead per I/O operation:

```
  Traditional read():           io_uring:
  ─────────────────            ──────────

  for each file:               Setup: io_uring_setup()
    syscall(read)                │
    context switch               ▼
    copy to user buffer        Submit N reads to SQ ring (no syscall)
    return to user               │
    (one syscall per read)       ▼
                               Kernel processes in batch
                                 │
                                 ▼
                               Reap completions from CQ ring
                               (amortized: fewer context switches)
```

### 4.4 Network I/O

Distributed training is the most network-intensive workload in modern computing.

**Bandwidth Requirements:**

| Scenario | Required Bandwidth | Technology |
|----------|-------------------|------------|
| Single-node multi-GPU | N/A (NVLink within node) | NVLink/NVSwitch |
| Multi-node DDP (8B model) | 25-100 Gbps per node | 100GbE, RoCE |
| Multi-node DDP (70B+ model) | 200-400 Gbps per node | 400G InfiniBand, 400GbE |
| LLM training (1T+ params) | 400-3200 Gbps per node | 8×400G InfiniBand (NDR) |
| Inference (single model) | 10-25 Gbps | Standard Ethernet |
| Inference (pipeline parallel) | 100+ Gbps | RoCE / InfiniBand |

**RDMA (Remote Direct Memory Access):** Bypasses the kernel network stack
entirely. The NIC reads/writes application memory directly via DMA, eliminating
memcpy and context switch overhead. Linux supports two RDMA fabrics for AI:

```
  TCP/IP Stack (traditional):          RDMA (InfiniBand / RoCE):
  ───────────────────────────          ──────────────────────────

  Application                          Application
      │                                    │
      ▼ syscall                            ▼ verbs API (user-space)
  Socket layer                         ┌──────────────────┐
      │                                │ RDMA NIC (HCA)   │
      ▼                                │ • DMA from app   │
  TCP/IP stack                         │   memory directly │
  (segmentation,                       │ • No kernel       │
   retransmit,                         │   involvement     │
   checksums)                          │   in data path    │
      │                                └────────┬─────────┘
      ▼                                         │
  NIC driver                                    │ wire
      │                                         │
      ▼ DMA                                     ▼
  NIC hardware                          Remote NIC → Remote memory

  Latency: ~10-50 μs                   Latency: ~1-2 μs
  CPU overhead: HIGH                    CPU overhead: MINIMAL
```

**NCCL Backend Selection:** NCCL automatically selects the best communication
backend based on available hardware:

| Backend | Technology | When Used |
|---------|-----------|-----------|
| `nccl` (default) | NVLink + InfiniBand/RoCE | Multi-GPU, multi-node with RDMA |
| `gloo` | TCP sockets | CPU tensors, fallback when no RDMA |
| `mpi` | MPI (OpenMPI, MPICH) | HPC environments with MPI infrastructure |

**TCP Tuning for Non-RDMA Environments:** When RDMA isn't available (cloud
instances, commodity hardware), TCP tuning helps:

```bash
# Increase socket buffer sizes for large transfers
sysctl -w net.core.rmem_max=16777216
sysctl -w net.core.wmem_max=16777216
sysctl -w net.ipv4.tcp_rmem="4096 87380 16777216"
sysctl -w net.ipv4.tcp_wmem="4096 65536 16777216"

# Enable TCP window scaling and timestamps
sysctl -w net.ipv4.tcp_window_scaling=1
sysctl -w net.ipv4.tcp_timestamps=1

# Use BBR congestion control (better for datacenter)
sysctl -w net.ipv4.tcp_congestion_control=bbr
```

### 5.1 Deep Learning Frameworks

These are the foundation — every AI workload runs through one of these.

**PyTorch** is the dominant framework (~80% of research, growing in production).
It owns the CPU-side orchestration:

- **CPU thread pool:** PyTorch uses `ATen` thread pool (backed by OpenMP or
  the internal `pthreadpool`) for CPU tensor operations. Default thread count
  is `OMP_NUM_THREADS` or `torch.set_num_threads()`.
- **DataLoader workers:** Spawns separate Python processes (not threads, due
  to the GIL) for data loading. Each worker uses `fork()` or `spawn()`,
  consuming CPU cores and memory.
- **torch.compile:** Compiles computation graphs using TorchDynamo (Python
  bytecode analysis) and TorchInductor (code generation). Compilation is
  CPU-intensive (can take minutes for large models) but amortized over
  training.
- **Distributed:** `torch.distributed` provides CPU-managed communication
  primitives (init_process_group, AllReduce, broadcast) that coordinate
  with NCCL for GPU tensors or Gloo for CPU tensors.

**JAX** (Google) takes a functional approach with XLA compilation:

- **XLA (Accelerated Linear Algebra):** Compiles entire computation graphs to
  optimized HLO (High Level Operations), then to target-specific code. The
  compilation is CPU-intensive but produces highly optimized kernels. JAX's
  `jit` compilation can spend significant CPU time on tracing and lowering.
- **PJIT / sharding:** JAX's automatic parallelism uses CPU-side sharding
  annotations to partition tensors across devices, with the CPU managing the
  SPMD (Single Program Multiple Data) execution.

**TensorFlow** remains relevant in production deployments:

- **tf.data:** Built-in data pipeline with automatic prefetching, parallel
  mapping, and caching. Uses C++ threading (not Python multiprocessing),
  which avoids GIL issues and fork() overhead.
- **SavedModel serving:** TensorFlow Serving uses a C++ runtime, minimizing
  CPU overhead for inference compared to Python-based servers.

### 5.2 Data Loading & Processing

**NVIDIA DALI (Data Loading Library):**

- Offloads image decoding and augmentation to the GPU, freeing CPU cores
- Provides a pipeline API that overlaps data loading (CPU), preprocessing
  (GPU), and training (GPU) in a three-stage pipeline
- Particularly effective for image training where JPEG decode is a bottleneck
- CPU role: reading from storage, pipeline coordination

**Hugging Face Datasets:**

- Memory-mapped Arrow files avoid loading entire datasets into RAM
- Uses Apache Arrow columnar format — the CPU performs zero-copy reads via
  `mmap()`, with the kernel's page cache handling prefetching
- Streaming mode for datasets too large to fit on local storage

**Apache Arrow / Polars:**

- Columnar in-memory format used by many data processing tools
- Polars (Rust-based) provides multi-threaded query execution that saturates
  all available CPU cores for feature engineering and data preprocessing
- Zero-copy integration with PyTorch via the Arrow C data interface

### 5.3 Serving & Inference Runtimes

**vLLM:**

- The most popular open-source LLM inference engine
- CPU-side innovations: PagedAttention (KV cache as virtual memory), continuous
  batching scheduler, prefix caching
- Written in Python + C++/CUDA. The CPU runs the scheduler loop (Python) and
  manages memory allocation. Each scheduling step is CPU-bound and must
  complete within one decode iteration (~10-50ms)
- Supports distributed inference (tensor parallelism) via CPU-coordinated
  Ray or `torch.distributed`

**TensorRT-LLM (NVIDIA):**

- NVIDIA's optimized LLM inference runtime
- The CPU compiles the model into an optimized TensorRT engine (which can take
  10-60 minutes of CPU time) but inference is then extremely GPU-efficient
- In-flight batching (continuous batching) with C++ scheduler
- CPU handles: batch scheduling, KV cache management, beam search

**Text Generation Inference (TGI, Hugging Face):**

- Rust-based HTTP server wrapping PyTorch/CUDA inference
- Rust provides predictable low-latency request handling without GIL or GC
  pauses — an important CPU-side advantage over pure Python servers
- Supports flash attention, continuous batching, quantization

**llama.cpp:**

- C/C++ implementation for CPU and GPU inference
- Extensively optimized CPU kernels: AVX2, AVX-512, ARM NEON, Metal
- Uses `mmap()` to load model files — the kernel manages page-in on demand,
  allowing fast startup and memory sharing between processes
- Quantization formats (GGUF) designed for CPU cache efficiency: Q4_K, Q5_K
  group quantization keeps dequantization data cache-local

**ONNX Runtime:**

- Microsoft's cross-platform inference engine
- CPU execution providers: Intel oneDNN (MKL-DNN), ARM Compute Library
- Graph optimizations (constant folding, operator fusion) happen on CPU at
  model load time
- Used for non-LLM models: vision, speech, recommendation

### 5.4 Distributed Training Tools

**DeepSpeed (Microsoft):**

- ZeRO (Zero Redundancy Optimizer) stages partition optimizer state,
  gradients, and parameters across GPUs. ZeRO-Offload and ZeRO-Infinity
  extend this to CPU RAM and NVMe, using the CPU to:
  - Store and update optimizer states (Adam compute on CPU)
  - Page model parameters between NVMe → CPU → GPU on demand
  - Manage a priority queue of parameter prefetching
- CPU impact: ZeRO-Offload can use 100% of multiple CPU cores for optimizer
  updates. The `deepspeed.ops.adam.DeepSpeedCPUAdam` is a C++ extension
  optimized with AVX-512.

**Megatron-LM (NVIDIA):**

- Combines data, tensor, and pipeline parallelism for training very large
  models (100B+ parameters)
- CPU manages the complex parallelism schedule: micro-batch interleaving,
  pipeline stage assignment, communication topology
- Used for training the largest foundation models (GPT-3 scale and beyond)

**Horovod (Uber, now LF AI):**

- Ring-AllReduce implementation that wraps PyTorch/TensorFlow
- CPU handles MPI process management, gradient compression, and bandwidth
  negotiation
- Simpler than DeepSpeed/Megatron but less flexible for very large models

**Ray (Anyscale):**

- Distributed computing framework used as the backbone for many AI systems
- Ray Core: CPU-managed task and actor scheduling across a cluster
- Ray Train: distributed training orchestration (wraps PyTorch DDP, DeepSpeed)
- Ray Serve: model serving with dynamic batching and multi-model composition
- CPU-intensive: Ray's scheduler, object store (Plasma, backed by shared
  memory), and GCS (Global Control Store) all run on CPU

### 5.5 Container & Orchestration

**Docker + NVIDIA Container Toolkit:**

- The NVIDIA Container Toolkit (`nvidia-ctk`) provides GPU access inside
  containers via the NVIDIA Container Runtime
- CPU overhead: container runtime, cgroup management, namespace isolation.
  For AI workloads, the main concern is that cgroup memory limits must account
  for both the training process and DataLoader worker RSS
- `--shm-size` must be increased (default 64 MB is far too small for PyTorch
  DataLoader shared memory, typical setting: 8-64 GB)

```bash
# Typical Docker run for AI training
docker run --gpus all \
  --shm-size=32g \
  --ulimit memlock=-1 \
  --ulimit stack=67108864 \
  --ipc=host \
  -v /data:/data \
  nvcr.io/nvidia/pytorch:24.05-py3 \
  python train.py
```

**Kubernetes + GPU Scheduling:**

- NVIDIA GPU Operator and Device Plugin expose GPUs as schedulable resources
- CPU-side concerns: the Kubernetes scheduler must be NUMA-aware (via
  Topology Manager) to colocate GPU and CPU resources on the same NUMA node
- Resource requests should include CPU cores for DataLoader workers (not just
  GPU count)
- MIG (Multi-Instance GPU) partitioning is configured by the CPU-side GPU
  Operator

**Slurm:**

- Dominant job scheduler in HPC and AI supercomputing clusters (MLCommons,
  most national labs)
- CPU manages: job queuing, resource allocation (GPU + CPU + memory),
  node selection, job accounting
- `srun` and `sbatch` launch distributed training with `SLURM_PROCID` as
  rank, `SLURM_NNODES` as world size

```bash
# Typical Slurm job for multi-node training
#!/bin/bash
#SBATCH --nodes=4
#SBATCH --ntasks-per-node=8
#SBATCH --gpus-per-node=8
#SBATCH --cpus-per-task=12
#SBATCH --mem=512G

srun torchrun \
  --nnodes=$SLURM_NNODES \
  --nproc_per_node=8 \
  --rdzv_backend=c10d \
  --rdzv_endpoint=$MASTER_ADDR:$MASTER_PORT \
  train.py
```

### 5.6 Monitoring & Profiling

Understanding where the CPU is spending time is essential for optimizing AI
workloads.

**nvidia-smi / DCGM:**

- `nvidia-smi` reports GPU utilization, memory, temperature — but also
  reveals CPU-side bottlenecks indirectly: if GPU utilization is low (<80%)
  during training, the CPU pipeline is likely the bottleneck
- NVIDIA DCGM (Data Center GPU Manager) provides continuous monitoring with
  Prometheus/Grafana integration

**PyTorch Profiler:**

- Traces both CPU and GPU operations, showing the overlap (or lack thereof)
  between data loading, computation, and communication
- Outputs Chrome Trace format for visualization in `chrome://tracing` or
  TensorBoard

```python
with torch.profiler.profile(
    activities=[
        torch.profiler.ProfilerActivity.CPU,
        torch.profiler.ProfilerActivity.CUDA,
    ],
    schedule=torch.profiler.schedule(wait=1, warmup=1, active=3),
    on_trace_ready=torch.profiler.tensorboard_trace_handler('./logs'),
) as prof:
    for step, batch in enumerate(dataloader):
        train_step(batch)
        prof.step()
```

**Linux perf:**

- `perf stat` for CPU counters (IPC, cache misses, branch mispredictions)
- `perf record` + `perf report` for CPU profiling of DataLoader workers,
  tokenization, and other CPU-bound code
- Useful for diagnosing NUMA effects, cache thrashing, and lock contention

```bash
# Profile a training run to find CPU hotspots
perf record -g -p $(pgrep -f train.py) -- sleep 30
perf report --sort=dso,symbol

# Check NUMA statistics
numastat -p $(pgrep -f train.py)
```

**NVIDIA Nsight Systems:**

- System-wide profiler showing CPU, GPU, network, and storage activity on a
  unified timeline
- Best tool for finding CPU-GPU overlap issues (e.g., CPU preprocessing not
  overlapping with GPU compute)
- Shows CUDA API calls, kernel launches, memory copies, and NCCL
  communications with CPU-side timestamps
