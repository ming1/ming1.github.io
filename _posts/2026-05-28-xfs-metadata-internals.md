# Overview

XFS is a large, mature, allocation-group-partitioned, fully journaled
filesystem. From a kernel-engineer point of view it is best understood
as **three storage tiers tied together by transactions**: structures
live in memory, get logged into a journal, and only later get rewritten
back to their home blocks. This post is a top-down walk through the
metadata that matters in practice — superblock, allocation groups,
inodes, extent btrees, the transaction/log machinery, the metadata
buffer cache — and then a set of operation walkthroughs (`create`,
`unlink`, `rename`, `truncate`, `fsync`, append) that tie the layers
together. Every parsing recipe is a standalone script you can run
against a live image.

# The Three Storage Tiers

Every piece of XFS metadata exists in three places at different times:

```
   in-core           journal-durable          home-block-durable
   (RAM only)        (CIL or on-disk log)     (sector rewritten in place)
   xfs_buf           xfs_cil_ctx              physical block on disk
   xfs_inode         xlog_in_core             AGF / AGI / inode / btree
        │                  │                          │
        └─── transaction ──┴────── checkpoint ────────┘
```

State flows strictly left to right, never the other way:

1. A thread starts a transaction (`xfs_trans_alloc`).
2. It mutates in-core copies of buffers and inodes, attaching **log
   items** to the transaction that describe the deltas.
3. `xfs_trans_commit` hands those log items to the **Committed Item
   List (CIL)** — an in-memory aggregator.
4. A CIL push writes a **checkpoint** out to the on-disk log; only now
   is the change journal-durable.
5. The same log items are also stitched onto the **Active Item List
   (AIL)**, sorted by LSN.
6. `xfsaild` walks the AIL and asks each item to **flush**, which
   eventually rewrites the *home block* (the AGF sector, the inode
   cluster, the btree leaf, …).
7. Only after the home block hits the platter is the log space
   reclaimable.

Recovery is the inverse: replay every log item whose target home
block hasn't yet caught up. The reason the rest of this post talks
constantly about "the AIL", "the CIL", and "home blocks" is that *all*
metadata in XFS sits somewhere on this pipeline, and every operation
moves it along.

Two design rules follow from this picture and explain a lot of XFS code:

- **Write-ahead-logging is absolute.** No metadata block is ever
  rewritten in place before the corresponding log records are durable.
- **Metadata is "logical" in the log, but "physical" at home.** Many
  log items describe *deltas* (e.g. "this 16-byte region of this buffer
  changed"); the home block is the result of all deltas applied. This
  is why the log can be tiny relative to filesystem size.

# High-Level Architecture

## Allocation groups

XFS partitions the volume into **allocation groups (AGs)**, each a
contiguous, equally-sized chunk of blocks (`sb_agblocks` blocks each,
`sb_agcount` of them). The AG is the **scalability unit**: it has its
own free-space btrees, its own inode btrees, its own inode space, and
its own header locks. Operations inside one AG never contend on locks
held by operations inside another AG, which is most of why XFS scales
to wide CPUs.

```
 byte 0                                            byte (sb_agblocks * 4096) - 1
 ├──────────────── allocation group 0 ─────────────────┤
 │ SB │ AGF │ AGI │ AGFL │  inode chunks + data extents + btree blocks ...
 │ 0  │ 1   │ 2   │ 3    │ (rest of the AG)
 │           512-B sectors

 ├──────────────── allocation group 1 ─────────────────┤
 │ SB │ AGF │ AGI │ AGFL │  ...
 ...

 ├──────────────── allocation group N-1 ────────────────┤
```

Sector 0 holds a **superblock copy**; AG 0's copy is authoritative.
Sector 1 holds the **AGF** (per-AG free-space root + counters). Sector
2 holds the **AGI** (per-AG inode root + counters). Sector 3 holds
the **AGFL**, a small circular list of free blocks reserved for
splitting the free-space btrees without recursing into themselves.

The structures live in
[`fs/xfs/libxfs/xfs_format.h`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/libxfs/xfs_format.h):
`struct xfs_dsb`, `struct xfs_agf`, `struct xfs_agi`,
`struct xfs_agfl`.

## Superblock

The superblock is small but central: block size, AG count and stride,
inode size, log location, and the three feature bitmaps
(`sb_features2`, `sb_features_compat`, `sb_features_ro_compat`,
`sb_features_incompat`). The mount path refuses unknown `incompat`
bits; that bitmap is why a kernel built without rmap support cannot
even read-only-mount an rmap filesystem.

In core, the superblock is `struct xfs_sb` cached on
`xfs_mount->m_sb`. Counters that move on every operation
(`fdblocks`, `ifree`, …) are not updated atomically on the on-disk
copy; instead, deltas are accumulated per-CPU and folded back into the
superblock when a transaction commits. That folding produces a
*superblock log item* whose home-block flush is the only path that
ever rewrites the superblock sector.

Dump it with [`xfs-meta-parse-sb.sh`]({{ site.baseurl }}/code/xfs-meta-parse-sb.sh):

```
./xfs-meta-parse-sb.sh /dev/loop0
```

The script prints the magic, version bits, block size, AG geometry and
inode size — exactly the fields you need before doing anything else
with an image.

## AGF, AGI, AGFL

Three short, hot, per-AG headers. They behave very differently, so it
helps to look at them together:

```
 AGF (free-space root)         AGI (inode root)         AGFL (free list)
 ─────────────────────         ─────────────────         ────────────────
 agf_roots[BNO]                agi_root                  ring of free
 agf_roots[CNT]                agi_free_root              block numbers
 agf_levels[..]                agi_count
 agf_freeblks                  agi_freecount              consumed when
 agf_longest                   agi_unlinked[64]           a free-space
 agf_flcount                                              btree splits
```

- **AGF** points at the bnobt and cntbt roots (free space by start
  block, free space by extent length), records how many free blocks the
  AG has, and how long the longest contiguous free extent is. The
  allocator reads `agf_longest` to decide if it can satisfy the request
  in this AG without scanning. Kernel entry point:
  [`xfs_read_agf`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/libxfs/xfs_alloc.c#L3365).
- **AGI** points at the inode btree (`inobt`) and the free-inode btree
  (`finobt`), and owns the **`agi_unlinked[64]`** array — 64
  hash-bucket heads of a per-AG list of inodes that have nlink == 0
  but are still open. Their blocks are *not yet* freed; the buckets
  exist so a crash before the last close can't orphan them: the next
  mount walks each bucket and finishes the eviction. That is also
  why `unlink()` of an open file in XFS only logs a small AGI edit
  rather than freeing anything: the on-disk state is "still
  reachable from the unlinked list", and the eventual free runs from
  the last close (or from recovery).
- **AGFL** is a ring of single free blocks reserved to bootstrap btree
  splits. The free-space btree itself needs free blocks to grow, so
  pulling those out of the btree would be circular. The AGFL pre-stages
  them. The ring is sized so the deepest possible split (root → leaf,
  on both bnobt and cntbt) can always proceed without recursion;
  `xfs_alloc_fix_freelist` refills it as part of any allocation that
  drains it.

In core, each AG is wrapped by `struct xfs_perag`. The buffers backing
AGF/AGI/AGFL are normal metadata buffers (`xfs_buf`); mutations attach
a **buffer log item** to the current transaction.

[`xfs-meta-parse-ag.sh`]({{ site.baseurl }}/code/xfs-meta-parse-ag.sh)
dumps all three for one AG:

```
./xfs-meta-parse-ag.sh /dev/loop0 0
```

## Lifecycle and crash consistency

- **In core:** AGF/AGI/AGFL buffers are read on demand into the
  metadata buffer cache and pinned in the AG perag.
- **Logged:** any mutation to an AG header goes through a transaction
  and produces a buffer log item.
- **Home block:** the AIL eventually flushes the buffer; the new bytes
  go back to sectors 1/2/3 of the AG.
- **Recovery:** if a crash happens after the log record but before the
  home-block flush, `xlog_recover` replays the buffer log item and
  rewrites the home block from the journal. AG headers are never the
  source of "lost a metadata update" bugs in production — only the
  journal can be.

# Inodes

## Purpose and layout

An inode in XFS is a fixed-size record, typically 512 bytes today on
v5 filesystems (`sb_inodesize`). It lives inside an **inode chunk** —
a logically contiguous group of 64 inodes (`XFS_INODES_PER_CHUNK`)
allocated inside an AG. The inode number encodes its (AG, AG block,
offset-within-chunk) tuple. With the `sparse_inodes` feature (on by
default in recent `mkfs.xfs`) a chunk's on-disk blocks may be a
strict subset of the full 64-inode footprint — the inobt record then
carries a `holemask` marking which 4K inode-cluster blocks are
actually allocated, and inodes inside an unallocated cluster simply
don't exist. The structure of the chunk is otherwise unchanged.

```
 ┌──────────────────────────────────────────────────────────────┐
 │ inode chunk (64 inodes contiguous on disk)                   │
 │ ┌──────┐ ┌──────┐ ┌──────┐ ... ┌──────┐                      │
 │ │ ino0 │ │ ino1 │ │ ino2 │ ... │ ino63│                      │
 │ └──────┘ └──────┘ └──────┘     └──────┘                      │
 └──────────────────────────────────────────────────────────────┘
```

Each inode has two parts: the **core** (`struct xfs_dinode`) and the
**literal area** that holds the data fork and (optionally) the attr
fork. The split between forks is parameterised by `di_forkoff`, an
8-byte-granular offset *inside* the literal area; this is how XFS fits
both the data extent map and small extended attributes inside one
512-byte inode.

Core fields that matter for debugging:

- `di_magic`, `di_version`: integrity / format gate.
- `di_format`, `di_aformat`: encoding of the data and attr forks.
  Possible values: `dev` (0, only on special files — char/block
  devices, sockets, FIFOs), `local` (1, payload inline), `extents`
  (2, array of `xfs_bmbt_rec`), `btree` (3, bmbt root in the literal
  area). Regular files only ever use `local`/`extents`/`btree`.
  Local-format dirs and symlinks have no extents at all.
- `di_nextents` / `di_anextents`: counts that match the array size in
  *extents* format.
- `di_lsn` (v5 / CRC inodes): last log sequence number that touched
  this inode. Recovery compares against the log item's LSN to skip
  replays whose home block is already newer. On legacy v4 inodes the
  equivalent role is played by `di_flushiter`, which is bumped each
  `xfs_iflush`; v5 made it a proper LSN and `di_flushiter` is no
  longer maintained.

The full structure is in
[`xfs_format.h`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/libxfs/xfs_format.h);
the in-core form is
[`struct xfs_inode`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_inode.h#L25).

## In-memory: `xfs_inode`

`xfs_inode` is the in-core counterpart. It embeds the generic VFS
`struct inode`, the metadata cluster buffer pointer, and per-fork
`struct xfs_ifork` objects that hold the *expanded* fork (the in-core
extent list is decompressed from the packed on-disk form for fast
lookup). It also owns the inode log item
(`struct xfs_inode_log_item`) which is what threads attach to a
transaction when they dirty inode fields.

The cache is **per-AG**: each `xfs_perag` keeps a radix tree of
`xfs_inode` keyed by inode number.
[`xfs_iget`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_icache.c)
is the central lookup; it either returns the existing in-core inode,
reactivates a reclaimable one, or reads the cluster off disk and
constructs a fresh `xfs_inode`. Reclaim is asynchronous and tagged in
the same radix tree, which means a recently-evicted inode can be
recovered without IO if VFS asks for it again before the reclaim
worker runs.

## Dirty tracking and lifecycle

The inode lifecycle, in transaction terms:

```
 clean (in-core matches inode cluster on disk)
   │
   │  xfs_trans_ijoin + xfs_trans_log_inode
   ▼
 logged-in-trans   (xfs_inode_log_item attached, mask records dirty fields)
   │
   │  xfs_trans_commit  -> CIL insertion
   ▼
 in-CIL            (not on disk yet; LSN unassigned)
   │
   │  CIL push     -> writes a checkpoint to the log device
   ▼
 in-AIL            (LSN assigned; journal-durable)
   │
   │  xfs_inode_item_push -> xfs_iflush -> cluster-buffer write
   ▼
 clean again      (home block updated; AIL item removable)
```

The mask in the inode log item
([`xfs_inode_item.c`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_inode_item.c))
is critical: it says *which* fields changed (core, data fork, attr
fork, broot). Only those regions are formatted into the log record.
That is why XFS log volume is roughly proportional to the *amount of
change*, not the inode size.

## When is an inode update where?

- **Memory only** — between `xfs_trans_log_inode()` and the CIL push.
  A crash here loses the change.
- **Journal-durable** — after the CIL push completes and the checkpoint
  is on the log device. A crash here is recoverable; replay will
  reapply the inode delta.
- **Home block** — after `xfs_iflush` writes the inode cluster.
  Only now can the log space backing this change be released.

[`xfs-meta-parse-inode.sh`]({{ site.baseurl }}/code/xfs-meta-parse-inode.sh)
dumps an inode and its forks; for environments without `xfs_db`,
[`xfs-meta-parse-inode-raw.py`]({{ site.baseurl }}/code/xfs-meta-parse-inode-raw.py)
decodes the first 96 bytes (the inode core) from a raw byte blob
extracted by `dd`. Both tag the most important fields back to
`xfs_dinode`.

# Extents and Btrees

## Why btrees everywhere

XFS uses B+ trees for every variable-cardinality lookup. Per-AG:
free space by start block (bnobt), free space by length (cntbt),
allocated inode chunks (inobt), inode chunks with at least one free
slot (finobt), reverse maps (rmapbt), and reference counts
(refcountbt) — six on a modern v5 filesystem. Per-inode: the file
extent map (bmbt). The shape is shared via
[`struct xfs_btree_cur`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/libxfs/xfs_btree.h) —
a per-cursor object that hides the per-tree differences (key compare,
record format, leaf size) behind ops vectors. Code that walks the
tree (`xfs_btree_lookup`, `xfs_btree_insert`, `xfs_btree_delete`) is
shared across all five trees.

## Per-AG btrees

- **bnobt** — free space by *starting block* — for "allocate near
  this hint" requests.
- **cntbt** — free space by *extent length* — for "give me 32
  contiguous blocks" requests.
  Both index the same set of free extents; two orderings give O(log n)
  lookups against both query shapes.
- **inobt** — allocated inode chunks. One record per 64-inode chunk,
  with a 64-bit "is-free" mask.
- **finobt** — only chunks that have at least one free inode. Speeds
  up inode allocation when most chunks are full.

The roots and heights live in the AGF (for free-space) and AGI (for
inodes). Records are defined in
[`xfs_format.h`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/libxfs/xfs_format.h):
`xfs_alloc_rec`, `xfs_inobt_rec`, `xfs_bmbt_rec`.

## Per-inode btree: the extent map

File extents are kept in a btree **rooted inside the inode** when
`di_format == XFS_DINODE_FMT_BTREE`. The root block (`xfs_bmdr_block_t`)
lives in the inode literal area; the rest of the tree is in normal
filesystem blocks. For small files, `di_format == EXTENTS` keeps the
records inline; for tiny files (or short symlinks, or small dirs),
`di_format == LOCAL` skips extents altogether.

Each `xfs_bmbt_rec` is a packed 128-bit value:

```
 ┌───────────────────────────── 128 bits ─────────────────────────────┐
 │ flag (1) │ logical offset (54) │ physical block (52) │ blockcount(21)│
 └────────────────────────────────────────────────────────────────────┘
```

The flag distinguishes **unwritten extents** — allocated but not yet
written, exposed as zeroes — which is how XFS implements
`fallocate(FALLOC_FL_ZERO_RANGE)` without doing IO.

Walks happen through `xfs_iext_lookup_extent` for the in-core list,
or `xfs_bmapi_read` / `xfs_bmapi_write` for the
allocate-or-map path.

## Walking from userspace

[`xfs-meta-walk-bnobt.sh`]({{ site.baseurl }}/code/xfs-meta-walk-bnobt.sh)
reads the bnobt, cntbt, and inobt roots out of the AGF/AGI and prints
the root block. From there `xfs_db`'s `addr` command takes you to
children. The same cursor structure shows up at every level — which
is why the kernel has only one btree walker.

## Synchronization

A btree mutation always happens inside a transaction. The buffers that
back tree blocks get joined (`xfs_trans_bjoin`), logged with their
range of modified bytes (`xfs_trans_log_buf`), and on commit attach a
buffer log item to the CIL. Where a mutation has to free blocks that
the transaction itself can't finish in-line — e.g. a bmap update on
the data fork that releases an extent, or a merge that releases an
interior btree block — the free is deferred and logged as an
EFI/EFD pair (and any matching BUI/BUD or RUI/RUD), so a crash
between the intent and the done leaves enough state in the log for
the next mount to finish the work. Allocations themselves don't
produce intents; they edit the free-space btrees directly inside the
transaction.

# Transactions and the Log

This is XFS's nervous system. Get this section, and everything else
follows.

## The transaction

`struct xfs_trans` is an in-memory object that:

- carries a **log space reservation** (worst-case bytes the transaction
  may need to log, paid up-front to keep the log from filling under
  load),
- accumulates **log items** that describe the work,
- and, on commit, atomically hands the items to the CIL.

Entry points:
[`xfs_trans_alloc`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_trans.c#L234),
plus higher-level wrappers like
[`xfs_trans_alloc_inode`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_trans.c#L1066),
[`xfs_trans_alloc_icreate`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_trans.c#L1191),
[`xfs_trans_alloc_dir`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_trans.c#L1353).
Commit is
[`xfs_trans_commit`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_trans.c#L909).

## Log items

A **log item** is one entry on the CIL/AIL pipeline. It is the
abstract base type for everything the log knows how to recover.
[`struct xfs_log_item`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_trans.h#L31)
holds a pointer to the owning AIL, the LSN, flags, and an ops vector
(`iop_format`, `iop_pin`, `iop_unpin`, `iop_push`, `iop_committed`,
etc.) implemented per item type:

- **`xfs_buf_log_item`** — a metadata buffer was modified.
  `iop_format` emits the dirty byte ranges, not the whole buffer.
- **`xfs_inode_log_item`** — an inode was modified.
- **EFI / EFD** — Extent Free Intent / Done. Pair: EFI = "I intend
  to free this extent"; EFD = "I have freed it". If the system crashes
  between the two, recovery sees the unmatched EFI and finishes the
  free.
- **RUI / RUD, CUI / CUD, BUI / BUD** — rmap, refcount, and bmap
  intent/done pairs, with the same protocol.
- **ICREATE** — inode-chunk initialization.
- **Unlinked-list items** — link/unlink ops on the AGI buckets.

The intent/done pattern is what lets XFS keep multi-step operations
(allocate one extent + free another + update rmap + update refcount)
atomic in a single transaction even though each sub-step touches a
different btree.

## The CIL

The **Committed Item List** is an in-memory aggregator between
transaction commit and on-disk log record. It absorbs **relogging**:
the log item attached to a dirty buffer or inode is a singleton, so a
sequence of transactions that all dirty the same AGF buffer's
`freeblks` field within one checkpoint window keeps reusing one
buffer log item and accumulating dirty byte ranges into it. The CIL
push then formats that one item once, no matter how many
transactions touched the buffer. (Concurrent transactions can't race
here — the buffer lock serializes them; the win is across *sequential*
transactions inside the checkpoint window.) The data structure is
[`struct xfs_cil`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_log_priv.h#L284),
holding a current context (`xfs_cil_ctx`) plus per-CPU staging lists.

The push happens in
[`xlog_cil_push_work`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_log_cil.c#L96)
when:

- the CIL grows past a soft threshold,
- a synchronous transaction (e.g. fsync's log force) demands it,
- or a periodic timer fires.

A push produces a **checkpoint**: a self-contained set of log records
on disk. Recovery replays at checkpoint granularity, never partial
ones.

## The AIL

The **Active Item List** is the *durable* side of the same items.
Once a checkpoint is written, the CIL hands its items to the AIL with
a freshly-assigned LSN. The AIL is sorted by LSN, oldest first.
[`xfsaild`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_trans_ail.c)
walks it and asks the oldest items to `iop_push`, which translates
into:

- *for buffers*: write the buffer to its home block.
- *for inodes*: format the in-core inode back into its cluster buffer
  and write that.
- *for intent items*: drive the deferred op forward until the matching
  `done` item also drains, which removes both.

An item leaves the AIL once its home block is durable. The oldest
LSN still in the AIL is the **tail LSN**; it pins log space because
recovery may still need to replay through that point.

## State diagram

```
 (in-trans)             (committed)            (checkpointed)         (clean)
 +-----------+   commit  +------------+  push   +-----------+  flush  +------+
 | log item  | --------> |  in CIL    | ------> | in AIL    | ------> | gone |
 +-----------+           +------------+         +-----------+         +------+
       ▲                       │                       │
       │                       │ aggregation           │ writes home
       │                       │ + dedup               │ block (xfs_buf
       │                       │                       │ or xfs_iflush)
       │                       ▼                       │
       │                  on-disk log record           │
       │                  ("checkpoint")               │
       └──── crash ────────────┴─────── replay ────────┘
                          (xlog_recover)
```

## Log force and fsync

`xfs_log_force(mp, flags)` flushes everything currently in the CIL
out to the log device; `xfs_log_force_seq(mp, seq, flags, log_flushed)`
does the same but only up to a specific CIL sequence (used to wait for
"my transaction" instead of "all transactions"). `fsync(2)` on XFS
lives in
[`xfs_file_fsync`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_file.c#L125)
and does, roughly:

1. Flush in-flight data IO (page cache writeback to the data device).
2. If the inode is dirty in core, log it (so the i_size update gets a
   stable LSN).
3. Push the CIL and force the log up through the inode's
   `xfs_inode_log_item->li_lsn`.
4. Issue a barrier / flush to the underlying device.

Crucially, fsync does **not** wait for the home-block writeback of the
inode cluster or the AGF or anything else. It only needs the *log* to
be durable; recovery will materialise the home blocks if a crash
intervenes.

## Recovery

On mount of an uncleanly-unmounted filesystem,
[`xlog_recover`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_log_recover.c)
walks the log in several ordered phases:

- **Scan**: read the log forward and build an in-memory hash of
  checkpoints and the items inside them, pairing intents with their
  matching `done` records as they're seen.
- **Replay buffers**: walk the items and write buffer log items back
  to their home blocks, skipping any whose buffer LSN says the home
  block is already newer. Buffers go first so that any inode cluster
  blocks (and any blocks an `icreate` references) exist and are
  zero-initialised before inodes touch them.
- **Replay inodes and icreate**: replay inode log items into their
  cluster buffers, using `di_lsn` (v5) or `di_flushiter` (v4) to
  skip items whose home is already current; replay `icreate`
  intents to zero freshly allocated inode chunks.
- **Finish intents**: drain whatever intents remain unmatched — for
  each EFI without an EFD, finish the free; for each BUI without a
  BUD, finish the bmap update; for each RUI/CUI without its done,
  finish the rmap/refcount update.

After replay, the log tail advances; the filesystem is consistent.

## Inspecting the log

[`xfs-meta-dump-log.sh`]({{ site.baseurl }}/code/xfs-meta-dump-log.sh)
wraps `xfs_logprint`:

```
./xfs-meta-dump-log.sh /dev/loop0
```

The output is transaction-grouped: each TID lists the items that were
committed together. Reading a few of these is the fastest way to build
intuition about what a real operation looks like on the wire.

## What is *not* journaled

Two large things:

- **File data**. XFS journals metadata only. Data lives in the page
  cache and is written through normal writeback. Crash consistency for
  data is the user's job (call `fsync`).
- **Allocation-group counters as raw values**. The superblock has
  counters but XFS logs *deltas*, applied on the in-core sb during
  commit and folded into the on-disk sb during AIL push of the sb log
  item.

# Buffer Cache and Metadata IO

## Why a separate cache

XFS reads metadata through
[`struct xfs_buf`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_buf.h),
not through the page cache. The page cache is keyed by file inode +
offset; metadata is keyed by *device + block + length*, can span
sector sizes different from `PAGE_SIZE`, and needs explicit pin/unpin
semantics to interact correctly with the log. Mixing the two would
also create ordering hazards between data and metadata writes.

The buffer cache lives in per-AG hash buckets (so reads from different
AGs don't contend), holds metadata blocks as a private kmem buffer, and
exposes the standard read/get/put API. Mutations attach an
**`xfs_buf_log_item`** (the buffer's counterpart to the inode log
item) to the active transaction.

## Lifecycle

```
 (idle on LRU)
    │  xfs_buf_get / xfs_buf_read
    ▼
 (referenced; locked while modifying)
    │  xfs_trans_bjoin + xfs_trans_log_buf  (dirty byte range tracked)
    ▼
 (joined to txn; XBF_DONE)
    │  xfs_trans_commit; xfs_buf_log_item → CIL
    ▼
 (in CIL; pinned by log)
    │  CIL push → xfs_buf_log_item → AIL
    ▼
 (in AIL; pinned by log AND on delayed-write list)
    │  xfsaild iop_push → submit_bio
    ▼
 (write in flight; pinned by IO)
    │  bio completion → xfs_buf_iodone → AIL removal
    ▼
 (clean; back on LRU)
```

A buffer carries three independent pieces of state that gate writes:

- **Log pin** (`b_pin_count`) — incremented when the buffer is joined
  to a CIL context, decremented when its log item is unpinned after
  the checkpoint reaches the AIL. While `b_pin_count > 0` the buffer
  cannot be written back; flushing a buffer whose log record isn't
  durable yet would break WAL.
- **Lock** (`b_sema`) — held by the thread currently mutating or
  formatting the buffer; not a pin, but the only thing that
  serialises in-place modifications.
- **In-flight IO** (`b_io_remaining`) — a refcount on outstanding bios
  carrying the buffer back to its home block. Completion drains this
  before `xfs_buf_iodone` runs.

## Locking and ordering

Metadata buffers are locked in **AG number order** and, within an AG,
in a canonical order (AGF before AGI before AGFL). This is the
deadlock-avoidance rule for cross-AG operations like rename across
directory parents. Inode locks do install lockdep subclasses
(`XFS_ILOCK_*`, `XFS_IOLOCK_*`) that encode parent/child and
inumber-ascending ordering, so violations of *those* trip lockdep
directly. Buffer ordering, by contrast, is convention enforced by
review; reading the relevant `xfs_buf_lock` calls in
`xfs_inode.c::xfs_rename` is the usual onboarding exercise.

## Delayed write

Most metadata buffer writes do **not** happen at transaction commit.
They sit on a delayed-write list, dirty, until either:

- the AIL pushes them (because their LSN is the tail and log space
  is needed), or
- a sync (`syncfs`, `umount`, an explicit `xfs_buf_delwri_submit`)
  forces them.

This is a deliberate optimization: a buffer modified by ten
transactions in the same second only needs *one* home-block write at
the end. The journal already guarantees durability, so we're free to
coalesce.

# Operation Walkthroughs

Each walkthrough lists *what metadata is touched*, *what log items
appear*, and *when the change reaches the home blocks*. References to
the kernel are functions in the local tree at v7.0.

## `create()` of a regular file

Entry: VFS calls
[`xfs_create`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_inode.c#L648),
typically through `xfs_vn_create`.

Touched:

- Parent directory inode (link count, dir entries).
- Target AG: inobt (allocate one inode from a chunk; allocate a chunk
  if none has free slots), finobt (update), AGI (counters,
  `agi_freecount`).
- Parent dir's data fork (insert a new entry — may grow into local /
  block / leaf / node formats).
- Superblock counters (`sb_ifree`).

Trans flow:

```
 xfs_trans_alloc_icreate
   │
   ├── xfs_dialloc                     find / allocate inode
   │     ├── inobt edits               -> buffer log items
   │     ├── finobt edits              -> buffer log items
   │     └── AGI counter update        -> buffer log item
   │
   ├── xfs_icreate_log_item            ICREATE intent (chunk init)
   │
   ├── xfs_init_new_inode              fill in core; mark dirty
   │     └── xfs_trans_log_inode       -> inode log item
   │
   ├── xfs_dir_createname              insert entry into parent
   │     ├── parent inode change       -> inode log item (parent)
   │     └── dir-block buffer edits    -> buffer log items
   │
   └── xfs_trans_commit                -> CIL
```

Dirty transitions:

- **Memory only:** every change up to `xfs_trans_commit`.
- **Journal-durable:** after the next CIL push.
- **Home blocks updated:** asynchronously, when xfsaild pushes the
  AGI, the inode cluster, and the dir block.

If a crash happens after commit but before flush, replay re-applies
the inobt edits, the AGI counters, the parent inode link/size, the dir
block, and the newly-created inode core. The ICREATE intent ensures a
freshly-allocated chunk is fully zero-initialised before any extent
references it.

## Append-write with extent allocation

A write past i_size that needs a new extent:

1. VFS calls into iomap; XFS's iomap-begin maps the write through
   `xfs_bmapi_write` which allocates `n` blocks.
2. Allocation contacts the AGF/cntbt, walks the bnobt by hint, may
   pull blocks from the AGFL, and updates `agf_freeblks` /
   `agf_longest`.
3. The inode's bmap btree gains a record (or its `EXTENTS`-format
   list grows by one).
4. Data is written into the page cache and tagged dirty. Writeback
   submits the actual bio later.
5. On `fsync`, `xfs_file_fsync` (above) issues a data flush, then a
   log force up through the LSN that committed the bmap update, then a
   device flush.

Log items produced: buffer log items for the AGF, AGFL, free-space
btree blocks, possibly cntbt; an inode log item for the inode (size,
extent count, fork format); EFI/EFD if any free was needed alongside
the alloc (e.g. an unwritten-to-written conversion that splits an
extent record).

The data itself is *never* journaled — only the metadata that locates
it.

## `unlink()`

[`xfs_remove`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_inode.c#L1905)

Two cases:

- **Last reference, file closed**: the inode and its blocks are freed
  in one transaction. Free-space btrees and AGI counters get updated;
  EFI/EFD pairs are produced for each extent freed; the inode chunk
  bitmask in inobt flips a bit; if the chunk is now fully free, the
  chunk's blocks themselves are freed.
- **Last name removed, but file still open**: the inode goes on the
  AGI's `agi_unlinked[hash]` bucket. The inode log item records the
  new state. The actual free is deferred until the last close,
  triggered through the same unlinked-list path on close or on the
  next mount.

The unlinked list is exactly why an unmount-recovery sequence
correctly cleans up half-deleted open files: at recovery the kernel
scans `agi_unlinked` for each AG and drains everything still on it.

On the wire, an unlink of an open file therefore commits with a small
set of log items: an inode log item for the parent (dir entry removed,
nlink decremented, timestamps updated), an inode log item for the
target (nlink → 0, `iflags` updated to mark it on the unlinked list),
and a buffer log item for the AGI sector whose `agi_unlinked` bucket
now points at the target's inode number. No EFIs yet — the extents
are not freed until the last close drives the deferred cleanup.

## `rename()`

[`xfs_rename`](https://elixir.bootlin.com/linux/v7.0/source/fs/xfs/xfs_inode.c#L2125)
is the most complex update on the hot path. It may touch up to four
inodes (source dir, target dir, source inode, target inode if
overwriting), each potentially in a different AG, plus the dir-block
buffers in both directories and the AGI/inobt of any inode being
removed. All of this is one transaction — atomic on commit.

Lock ordering by AG number is the rule that prevents AB/BA deadlocks
across two directories on different AGs. The transaction reservation
is sized for the worst case (cross-AG rename overwriting an
inode-chunk-freeing target).

Log items: inode log items for each touched inode; buffer log items
for dir blocks and AG headers; EFI/EFD for any blocks freed.

A subtle correctness point: because the *entire* rename is one
transaction, the source dir's "entry removed" and the target dir's
"entry added" land in the same checkpoint. There is no observable
intermediate state — even at the on-disk-log level — where the file
exists in neither directory or in both. This is the property
`renameat2(RENAME_EXCHANGE)` relies on, and it is also why renames
across AGs are reservation-expensive: the worst-case set of log items
spans the maximum of both AGs' touched buffers.

## `truncate()`

`xfs_setattr_size` calls `xfs_itruncate_extents_flags`. For a file
with many extents, the truncation runs as a sequence of transactions,
each freeing a bounded number of extents to keep within the log
reservation. The intent-chain machinery (BUI/BUD for bmap updates,
EFI/EFD for the free) makes this restartable: a crash after extent
range *k* has been freed but before *k+1* leaves a BUI in the log;
replay finishes the rest.

Dirty transitions: each sub-transaction independently moves through
in-CIL → in-AIL; the inode's `di_size` only reaches the home block
after the last sub-transaction's inode log item is flushed.

## `fsync()`

The interesting case. With a freshly written file:

1. Writeback flushes data bios.
2. `xfs_file_fsync` checks the inode's dirty mask; if dirty in core,
   logs the inode (so the i_size and timestamps are captured in a log
   item with a definite LSN).
3. Push the CIL.
4. `xfs_log_force_seq` waits until the on-disk log has covered the
   CIL sequence that committed the inode's log item.
5. Issue a device flush.

Run [`xfs-meta-observe-fsync.sh`]({{ site.baseurl }}/code/xfs-meta-observe-fsync.sh)
in one terminal while doing `xfs_io -f -c "pwrite 0 4k" -c "fsync"
/mnt/foo` in another, and you'll see the trio: `xfs_file_fsync` →
`xfs_log_force` → `xlog_cil_push_work`. The inode-cluster home-block
write is *not* in that sequence; AIL flushes it later. The fsync
guarantee holds because the log is durable and recovery reproduces the
home block from it.

# Summary

**Main advantages**

- **One pipeline for all metadata.** Every change passes through
  transaction → CIL → AIL → home block, so recovery, ordering, and
  durability have a single mental model regardless of which structure
  changed.
- **AGs give cheap parallelism.** Each AG has its own free-space and
  inode btrees and its own header locks; concurrent allocations from
  different AGs never contend.
- **Delta logging keeps the log small.** Buffer and inode log items
  record dirty byte ranges, not whole structures; the CIL deduplicates
  repeated edits inside a checkpoint window.
- **Intent items make multi-step ops crash-restartable.** EFI/EFD,
  BUI/BUD, RUI/RUD, CUI/CUD encode "I plan to / I did" pairs so a
  partial step can be finished on the next mount.
- **fsync is cheap-but-correct.** It only forces the log; home-block
  writes happen later, but durability is intact because replay can
  rebuild any home block from the journal.

**Main problems / limitations**

- **Hot AGs hurt.** Allocation policy spreads work across AGs by
  inode placement, but workloads that hammer one AG (a single huge
  directory, an inode-allocation-heavy small-file workload that pins
  one inobt) still serialise on per-AG locks.
- **Log space is a global limit.** Long-running transactions or large
  rename/truncate operations consume reservation; if the log is sized
  too small for the workload, transactions stall on log-grant waits
  before they ever run.
- **Metadata replay can be slow.** A long uncommitted-checkpoint
  window means a crash forces a large replay; this is the main reason
  XFS pushes the CIL periodically even when no thread asks for it.
- **No data journaling.** Crash consistency for file data is the
  application's responsibility — XFS will recover its metadata to a
  consistent state, but `fsync` is the only contract for "my bytes are
  durable".
- **The on-disk format is rich.** Six btrees per AG on v5 (bnobt,
  cntbt, inobt, finobt, rmapbt, refcountbt), packed bmbt records,
  intent/done item types, and the CRC-everywhere format mean
  debugging requires fluency with
  `xfs_db`, `xfs_logprint`, and the structures in `xfs_format.h`.
  Scripts like the ones linked from this post are the practical entry
  point to that fluency.
