# Overview

Writeback is the kernel layer that turns *dirty page cache* into *device
I/O*. Filesystems mark pages dirty when an application writes; writeback
decides **when** those dirty pages reach the disk, **how fast** they get
there, and **which writer task** gets throttled when memory fills up with
dirty data. This post walks the subsystem outside-in: boundary, entry
points, core objects, the dirty-throttling math, cgroup writeback, and
the contracts it exposes to neighbouring subsystems.

> Based on Linux mainline (commit `4cbfe4502e3d`, May 2026). Primary
> sources:

Edges of the box, with direction:

- **userspace → writeback** (syscalls / sysctls / sysfs) — entry only.
- **VFS ↔ writeback** — VFS owns `inode->i_state` dirty bits; writeback
  reads them and links inodes onto `wb` lists.
- **MM → writeback** — the buffered-write paths in `mm/filemap.c`,
  `fs/iomap/buffered-io.c`, and `fs/buffer.c` call
  `balance_dirty_pages_ratelimited()` after dirtying pages; reclaim
  (`mm/vmscan.c`) calls `wakeup_flusher_threads()`.
- **cgroup → writeback** — memcg/blkcg lifetime hooks drive
  `cgwb_create` / `cgwb_release_workfn`.
- **writeback → block** — bios reach the block layer through filesystem
  `->writepages`; block completion calls `end_page_writeback()`, which
  feeds `wb_update_bandwidth()` back up.
- **writeback → filesystem** — `wb_workfn` and
  `filemap_fdatawrite_range` (used by `->fsync` implementations)
  dispatch into `ops->writepages`; the filesystem owns how dirty pages
  turn into bios.

**Inside the box** (the writeback subsystem proper): three C files
plus their headers, listed below. **Outside the box**: every other
subsystem writeback touches — none of those files belong to writeback
even though writeback cannot function without them.

The writeback subsystem is split across two source trees:

| File | Role |
|------|------|
| [`fs/fs-writeback.c`](https://elixir.bootlin.com/linux/latest/source/fs/fs-writeback.c) | The **scheduler**: per-bdi work queues, inode dirty-list management, `wb_workfn` cycle, `sync(2)`/`fsync(2)` paths. |
| [`mm/page-writeback.c`](https://elixir.bootlin.com/linux/latest/source/mm/page-writeback.c) | The **economist**: dirty-page accounting, dirty thresholds, `balance_dirty_pages` PI controller, per-wb bandwidth estimation. |
| [`mm/backing-dev.c`](https://elixir.bootlin.com/linux/latest/source/mm/backing-dev.c) | BDI lifecycle: `bdi_alloc`, `bdi_register`, cgroup-writeback (`cgwb_*`) creation/release. |
| [`include/linux/writeback.h`](https://elixir.bootlin.com/linux/latest/source/include/linux/writeback.h) | `struct writeback_control`, `wb_domain`, exported writeback API. |
| [`include/linux/backing-dev-defs.h`](https://elixir.bootlin.com/linux/latest/source/include/linux/backing-dev-defs.h) | `struct bdi_writeback`, `struct backing_dev_info`, `struct wb_writeback_work`, `enum wb_reason`. |

What is **not** part of writeback, even though it touches dirty pages
on every operation:

- **Filesystem `->writepages` callbacks** (in ext4/xfs/btrfs/...). Writeback
  *calls* them; their implementation is filesystem code. The writeback
  layer hands them a populated `struct writeback_control` and a folio
  batch; what bytes go where on disk is the filesystem's problem.
- **Block layer.** Writeback emits bios via the filesystem's `writepages`;
  request merging, scheduling, and completion live in `block/`. The only
  shared object is `struct backing_dev_info`, which the block layer
  attaches to each request queue.
- **Page cache itself.** Dirty *tracking* (the `PAGECACHE_TAG_DIRTY` xarray
  tag, folio flags) is in `mm/filemap.c`. Writeback consumes that
  state — it does not own it.
- **Reclaim.** Memory reclaim only *triggers* writeback through
  `wakeup_flusher_threads()`; the LRU walk and the shrinker are in
  `mm/vmscan.c`.

Maintainership tracks the split: writeback scheduling is touched by the
block-layer maintainer (Jens Axboe), the throttling and accounting half
by MM (Andrew Morton et al.), and the inode-side hooks by VFS.

# Use Cases and Task Contexts

Writeback exists to serve five distinct workloads. Each runs in its own
task context, and each pins a different code path.

## Periodic background writeback

The classic "flusher": every `dirty_writeback_centisecs` (default 5 s,
[`mm/page-writeback.c:103`](https://elixir.bootlin.com/linux/latest/source/mm/page-writeback.c#L103))
the per-bdi delayed work `bdi_writeback.dwork` fires and runs
`wb_workfn()` on the `bdi_wq` workqueue. The work drains the
`b_dirty` / `b_io` lists, calling each filesystem's `writepages`, until
either the configured chunk count is met or the lists are empty.

This is the path that ages out dirty data even when there is no memory
pressure. The age threshold is `dirty_expire_interval` (default 30 s):
inodes older than that get queued by `move_expired_inodes()`.

## Dirty-ratio throttling on writers

When a task dirties pages faster than the disks can drain them, the
*dirtier itself* is throttled — not via a separate thread. Every call to
`folio_mark_dirty()` is paired with `balance_dirty_pages_ratelimited()`
([`mm/page-writeback.c`](https://elixir.bootlin.com/linux/latest/source/mm/page-writeback.c)).
That function has a per-CPU fast path (no locks, ~32-page budget) and a
slow path that calls `balance_dirty_pages()`, which sleeps the task on
`io_schedule_timeout()` for a duration computed by the PI controller in
§5.

The point is to keep dirty memory below `vm_dirty_ratio` (default 20 %
of dirtyable RAM) without bursting — a steady producer-rate pushback
rather than hard caps.

## `sync(2)` / `fsync(2)` / `syncfs(2)`

User-driven durability. `fsync(2)` calls the filesystem's `->fsync`,
which typically routes through `filemap_fdatawrite_range()` and waits.
`sync(2)` calls `ksys_sync()`, which iterates all superblocks and runs
`sync_inodes_sb()`
([`fs/fs-writeback.c:2932`](https://elixir.bootlin.com/linux/latest/source/fs/fs-writeback.c#L2932)).

The defining flag is `sync_mode = WB_SYNC_ALL` in
`writeback_control` — every dirty page must be both *submitted* and
*waited on*, not merely queued. Background writeback uses `WB_SYNC_NONE`
and fires-and-forgets.

## Memory pressure

When reclaim cannot find clean pages to evict, it calls
`wakeup_flusher_threads()`
([`fs/fs-writeback.c:2474`](https://elixir.bootlin.com/linux/latest/source/fs/fs-writeback.c#L2474))
with `reason = WB_REASON_VMSCAN`. This walks every BDI and queues a
non-blocking writeback work item per wb. It is intentionally async — the
reclaim path cannot afford to wait on a disk.

## cgroup writeback

The same machinery runs *per memory cgroup × per blkcg*: each cgroup
gets its own `bdi_writeback`, its own dirty thresholds, and its own
throttling state. This is what stops a busy container from monopolising
a shared disk's dirty budget. We give it its own section in §6.

The task contexts in play:

| Context | What runs there |
|---------|-----------------|
| `bdi_wq` workqueue | `wb_workfn`, cgroup-wb switch work |
| Dirtier task (syscall) | `balance_dirty_pages` (may sleep on `io_schedule_timeout`) |
| `kswapd` / direct reclaim | `wakeup_flusher_threads` (async kick only) |
| `kworker` on `mm_percpu_wq` | `wb_update_bandwidth` deferred work |

# Interfaces

Writeback's surface is small relative to its size — most of the file is
internal helpers. The exported edges fall into four groups, plus a
uAPI surface.

## Exported symbols — kick writeback

Queue async writeback work; return immediately.

- `writeback_inodes_sb(sb, reason)` /
  `writeback_inodes_sb_nr(sb, nr, reason)` — flush a superblock's dirty
  inodes. Used by `sync` and quota.
- `try_to_writeback_inodes_sb(sb, reason)` — same, but returns false if
  work is already in flight. Used by paths that mustn't pile up.
- `wb_start_writeback(wb, reason)` — internal, for the periodic timer
  and reclaim wakeup.
- `wakeup_flusher_threads(reason)` /
  `wakeup_flusher_threads_bdi(bdi, reason)` — bulk wakeup; the reclaim
  entry point.

## Exported symbols — wait for writeback

These block until the I/O finishes; use them only when you actually
need durability.

- `sync_inodes_sb(sb)` — `WB_SYNC_ALL` on all of `sb`. The `sync(2)`
  path.
- `write_inode_now(inode, sync)` — single-inode synchronous write.
  Used by `dput()`, `iput()`, and quota.
- `sync_inode_metadata(inode, wait)` — metadata-only sync.
- `filemap_fdatawrite_range(mapping, start, end)` /
  `filemap_fdatawait_range()` — page-range sync. The `fsync` building
  blocks (called from filesystem code, not directly).

## Exported symbols — dirty accounting

Called by anything that dirties or cleans page-cache state.

- `filemap_dirty_folio(mapping, folio)` — default `->dirty_folio` for
  most filesystems. Increments per-cgroup and global dirty counters.
- `folio_mark_dirty(folio)` / `folio_mark_dirty_lock(folio)` — the
  public dirtying API.
- `tag_pages_for_writeback(mapping, start, end)` — promote dirty tags
  to `PAGECACHE_TAG_TOWRITE` so writeback can iterate consistently
  without racing concurrent dirtiers.
- `balance_dirty_pages_ratelimited(mapping)` /
  `balance_dirty_pages_ratelimited_flags(mapping, flags)` — the
  throttle hook. Called *before* the dirty mutation, on every page.
- `__mark_inode_dirty(inode, flags)` — set inode dirty bits and link
  it onto the wb's `b_dirty` list.

## Exported symbols — BDI lifecycle

- `bdi_alloc(node)` — allocate a BDI.
- `bdi_register(bdi, fmt, ...)` — register with sysfs and the global
  list. Without this, no writeback runs for the device.
- `bdi_unregister(bdi)` — tear down sysfs, stop work, release cgwbs.
- `bdi_put(bdi)` — drop a reference; final put frees.
- `inode_to_bdi(inode)` — fetch an inode's BDI via its superblock.
- `bdi_set_max_ratio(bdi, max_ratio)` — admin tunable for proportional
  dirty share.

## uAPI surface

Syscalls: `sync(2)`, `syncfs(2)`, `fsync(2)`, `fdatasync(2)`,
`sync_file_range(2)`.

Sysctls under `/proc/sys/vm/`:

| Knob | Default | Meaning |
|------|---------|---------|
| `dirty_ratio` | 20 % | Hard throttle level (% of dirtyable RAM). |
| `dirty_background_ratio` | 10 % | Background writeback start level. |
| `dirty_bytes` / `dirty_background_bytes` | 0 (unused) | Absolute overrides for the above. |
| `dirty_writeback_centisecs` | 500 | Periodic flusher interval. |
| `dirty_expire_centisecs` | 3000 | Inode age before kupdate writeback. |
| `dirtytime_expire_seconds` | 43200 | Lazy timestamp flush interval. |

Sysfs: `/sys/class/bdi/<dev>/` exposes per-BDI ratios, `stable_pages_required`,
and writeback stats.

## Real call stacks

The three entry points worth seeing through:

**`writeback_inodes_sb(sb, reason)`** — the `sync(2)` and friends path:

```
writeback_inodes_sb(sb, reason)
  └─ writeback_inodes_sb_nr(sb, get_nr_dirty_pages(), reason)
       └─ __writeback_inodes_sb_nr(sb, nr, reason, false)
            └─ bdi_split_work_to_wbs(sb->s_bdi, &work, ...)
                 └─ wb_queue_work(wb, work)          # per cgroup wb
                      └─ mod_delayed_work(bdi_wq, &wb->dwork, 0)
```

Notice it never writes anything itself: the call returns once work is
*queued*. The actual I/O happens on `bdi_wq` later, in `wb_workfn`.

**`balance_dirty_pages_ratelimited(mapping)`** — the dirtier-side
throttle:

```
balance_dirty_pages_ratelimited()
  └─ balance_dirty_pages_ratelimited_flags()
       ├─ this_cpu_add()                 # fast path: bump per-CPU counter
       └─ balance_dirty_pages()          # slow path on overflow
            ├─ wb_position_ratio()       # compute throttle strength
            ├─ wb_update_bandwidth()     # refresh bandwidth estimate
            └─ io_schedule_timeout(pause)
```

The fast path is the common case: a few-instruction `this_cpu_add`,
no locks. The PI controller only kicks in once a CPU's local budget is
exhausted.

**`wb_workfn(work)`** — the workqueue side:

```
wb_workfn()
  └─ wb_do_writeback()
       └─ wb_writeback()                  # loop until nr_pages done
            └─ writeback_sb_inodes()      # walk wb->b_io
                 └─ __writeback_single_inode()
                      └─ do_writepages()
                           └─ ops->writepages()  # filesystem callback
```

`wb_do_writeback` runs in a loop, draining the `work_list` until empty,
then optionally reschedules `dwork` for the next periodic tick.

# Core Data Structures

Five objects carry the subsystem. Read them top-down: what they
represent, where they live, who pins them, when they go away.

## `struct bdi_writeback` (the "wb")

Defined in
[`include/linux/backing-dev-defs.h`](https://elixir.bootlin.com/linux/latest/source/include/linux/backing-dev-defs.h).

**Represents** an independently throttleable writeback context.
Non-cgroup case: one per BDI (`bdi->wb`). cgroup case: also one per
(memcg, blkcg) pair, kept in `bdi->cgwb_tree`.

**Key fields:**

- `b_dirty`, `b_io`, `b_more_io`, `b_dirty_time` — inode lists. Inodes
  flow `dirty → io → (back to dirty or removed)`; `b_more_io` collects
  inodes that ran out of budget mid-flush so they get re-tried this
  round. `b_dirty_time` is the lazy-timestamp queue.
- `work_list` — pending `wb_writeback_work` items.
- `dwork` — the delayed_work that drives the 5-second flusher tick.
- `state` — bitflags: `WB_registered`, `WB_writeback_running`,
  `WB_has_dirty_io`, `WB_start_all`.
- `write_bandwidth`, `avg_write_bandwidth` — pages/sec, updated every
  200 ms by `wb_update_bandwidth()`. Feeds the PI controller.
- `dirty_ratelimit`, `balanced_dirty_ratelimit` — PI-controller output:
  per-task pages-per-second allowance.
- `memcg_css`, `blkcg_css` — owning cgroups (cgwb only).

**Lifetime.** The root wb (`bdi->wb`) lives as long as the BDI. cgroup
wbs are allocated by `cgwb_create()` on first dirty in that cgroup, ref-
counted by `percpu_ref`, and released asynchronously via
`cgwb_release_workfn()` once the memcg or blkcg goes away. The async
teardown is the source of multiple historic use-after-frees (see
commits `6689f01d6740`, `8f5857be99f1`, `49717c7bd6b8`) and is why
inode-to-wb pointer chasing is always done under `rcu_read_lock()`.

## `struct backing_dev_info` (the BDI)

Defined in
[`include/linux/backing-dev-defs.h`](https://elixir.bootlin.com/linux/latest/source/include/linux/backing-dev-defs.h).
Represents a backing device — a request queue, an NFS mount, a FUSE
connection, or anything else with its own write bandwidth.

**Key fields:**

- `wb` — the embedded root writeback. Always present.
- `wb_list` — list of all wbs (root + cgwbs) for this device. Iterated
  by `bdi_split_work_to_wbs()` to fan a single request out across cgroups.
- `cgwb_tree` — radix tree of cgwbs keyed by `(memcg, blkcg)` pair.
- `tot_write_bandwidth` — sum across wbs; the denominator in
  proportional dirty-share math.
- `min_ratio`, `max_ratio`, `max_prop_frac` — admin caps on this BDI's
  share of the global dirty budget.
- `capabilities` — `BDI_CAP_*` flags. `BDI_CAP_STRICTLIMIT` makes the
  BDI use *its own* dirty threshold rather than the global one, which
  is what NFS and FUSE want.

**Lifetime.** Allocated by `bdi_alloc()` (which the block layer calls
for every request queue, and filesystems call directly for fabric
mounts). Registered with `bdi_register()`, which gives it a sysfs
node. Unregister tears down cgwbs first (asynchronously), then the
root wb, then frees on the last `bdi_put()`.

## `struct wb_writeback_work`

Defined at
[`fs/fs-writeback.c:38`](https://elixir.bootlin.com/linux/latest/source/fs/fs-writeback.c#L38).
A single unit of "please flush X pages from Y with mode Z".

**Key fields:**

- `nr_pages` — how many pages to flush. `LONG_MAX` for "everything".
  Decremented as pages are submitted.
- `sb` — restrict to one superblock, or NULL.
- `sync_mode` — `WB_SYNC_NONE` (queue & go) or `WB_SYNC_ALL` (queue &
  wait).
- `for_kupdate`, `for_background`, `for_sync` — picks which inode list
  to walk and which expiry rule applies.
- `tagged_writepages` — flip dirty tags to TOWRITE first, so concurrent
  dirtiers don't cause livelock.
- `reason` — `WB_REASON_*` enum for tracepoints.
- `done` — optional `wb_completion *` for synchronous callers to wait
  on.
- `auto_free` — set if the work was kmalloc'd; cleared if on-stack.

**Lifetime.** On-stack in `__writeback_inodes_sb_nr()` for synchronous
callers (the caller blocks on `done`). For async kicks, kmalloc'd with
`auto_free=1` and freed by the workqueue after processing.

## `struct writeback_control` (`wbc`)

Defined in
[`include/linux/writeback.h`](https://elixir.bootlin.com/linux/latest/source/include/linux/writeback.h).
The argument bundle filesystems receive in their `->writepages`
callback.

**Key fields:**

- `nr_to_write` — page budget. The filesystem must decrement this; when
  it hits zero, writeback stops walking.
- `range_start`, `range_end` — byte range for `sync_file_range(2)`.
- `range_cyclic` — start from `mapping->writeback_index` rather than 0.
  Required for fairness on large files.
- `sync_mode` — copied from the parent `wb_writeback_work`.
- `tagged_writepages` — together with `tag_pages_for_writeback()`,
  guarantees forward progress.
- `wb` — the `bdi_writeback` this I/O is being attributed to (cgroup
  case only).
- `fbatch` — folio batch the filesystem fills as it iterates.

`wbc` is stack-allocated for the duration of one `writepages` call;
filesystems read `sync_mode` to decide whether to wait for journal
flushes, and read `nr_to_write` to decide when to break out of the loop.

## Inode dirty bits

In [`include/linux/fs.h`](https://elixir.bootlin.com/linux/latest/source/include/linux/fs.h),
six bits in `inode->i_state`:

| Bit | Meaning |
|-----|---------|
| `I_DIRTY_SYNC` | Metadata that `fsync` must flush. |
| `I_DIRTY_DATASYNC` | Subset: metadata `fdatasync` must also flush (e.g. i_size). |
| `I_DIRTY_PAGES` | Data pages are dirty. |
| `I_DIRTY_TIME` | Only timestamps changed; written out lazily (12 h). |
| `I_SYNC` | Inode is *currently* under writeback. |
| `I_DIRTY_INODE` | Alias = `I_DIRTY_SYNC | I_DIRTY_DATASYNC`. |

The exact bit positions are in
[`include/linux/fs.h:724-752`](https://elixir.bootlin.com/linux/latest/source/include/linux/fs.h#L724):
`I_DIRTY_SYNC = 1U<<4`, `I_DIRTY_DATASYNC = 1U<<5`, `I_DIRTY_PAGES = 1U<<6`,
`I_DIRTY_TIME = 1U<<12`, and `I_SYNC = 1U << __I_SYNC` where `__I_SYNC = 1`
(the bit position, used by `wait_on_bit`).

**Invariants:**

- `I_DIRTY_TIME` is mutually exclusive with `I_DIRTY_INODE`: any
  metadata change clears `I_DIRTY_TIME` and sets `I_DIRTY_SYNC`.
- `I_SYNC` is held for the duration of writeback on the inode; readers
  must `inode_wait_for_writeback()` to coordinate.
- All bit manipulation runs under `inode->i_lock`. List membership
  (`inode->i_io_list`) is under `wb->list_lock`. Acquiring both at
  once: `i_lock` first.

# Dirty Throttling Math

The throttling code in
[`mm/page-writeback.c`](https://elixir.bootlin.com/linux/latest/source/mm/page-writeback.c)
exists to answer one question, asked millions of times a second: *how
long should this dirtier sleep right now?* The answer comes from a
feedback loop that targets a steady-state dirty count, not a hard cap.

## Thresholds

Two thresholds bound the dirty population:

```
background_thresh = dirtyable_memory * dirty_background_ratio / 100
dirty_thresh      = dirtyable_memory * dirty_ratio / 100
```

(Or absolute values via `dirty_background_bytes` / `dirty_bytes` if set.)
"Dirtyable memory" is RAM minus reserved minus unevictable; computed in
`global_dirtyable_memory()`.

Per-wb, the share is proportional to recent write bandwidth:

```
wb_thresh = dirty_thresh * wb->avg_write_bandwidth / bdi->tot_write_bandwidth
```

A fast disk gets a bigger slice of the global budget — the system
naturally redirects pressure toward the device that can absorb it.

## `balance_dirty_pages` loop

Once a dirtier exceeds its ratelimit budget, it enters
`balance_dirty_pages()`. Each iteration:

1. Read global and per-wb dirty counts into
   `struct dirty_throttle_control`.
2. If under `background_thresh`, return — no throttle.
3. If between `background_thresh` and `dirty_thresh`, kick background
   writeback if it isn't already running.
4. Call `wb_position_ratio()` to compute a *position ratio* (the
   "proportional" term in PI), based on how close to the threshold the
   per-wb count is.
5. Call `wb_update_bandwidth()` if the 200 ms window has rolled over.
6. Compute pause length from `wb->dirty_ratelimit` (pages/sec) and the
   number of pages the task has dirtied since its last pause.
7. `io_schedule_timeout(pause)`. Loop.

The pause is clamped to `wb_max_pause()` — `MAX_PAUSE` is `max(HZ/5, 1)`
ticks at
[`mm/page-writeback.c:49`](https://elixir.bootlin.com/linux/latest/source/mm/page-writeback.c#L49),
i.e. ~200 ms at typical `HZ=1000`. If more pause is needed the loop
runs again, but the task remains preemptible.

## The PI controller

`wb_update_bandwidth()` runs every 200 ms (`BANDWIDTH_INTERVAL`) and
updates `wb->dirty_ratelimit`. Two terms feed it:

- **Proportional (`pos_ratio`):** how far the per-wb dirty count is from
  its setpoint. Above the setpoint → ratelimit decreases; below →
  increases.
- **Integral (`balanced_dirty_ratelimit`):** smoothed estimate of what
  dirty rate would keep the system in equilibrium, given the observed
  write bandwidth.

The final `dirty_ratelimit` interpolates between the two so transient
spikes don't whipsaw producers. The detailed coefficients are derived
in the comments above `wb_position_ratio()`; the key intuition is that
**the loop targets a dirty count, not a throughput** — it lets the disk
run at its natural rate and only sleeps producers as needed to keep
the dirty pool stable.

## Why a fast path matters

Every `folio_mark_dirty()` calls
`balance_dirty_pages_ratelimited()`. A naive implementation that
recomputed thresholds on every page would be a CPU disaster — modern
filesystems can dirty millions of pages per second. The fast path is:

```c
preempt_disable();
this_cpu_add(bdp_ratelimits, nr_pages);
if (this_cpu_read(bdp_ratelimits) < ratelimit_pages)
    return;        /* common case: no locks, no math */
balance_dirty_pages(...);  /* drained the budget */
```

`ratelimit_pages` is 32 by default. A dirtier hits the slow path once
every 32 pages — and even then usually finds no throttling needed.

# cgroup Writeback

`CONFIG_CGROUP_WRITEBACK` makes everything above run *per cgroup*. A
process inside a memcg dirties pages charged to that memcg, throttled
against that memcg's share of the dirty budget, with I/O attributed to
that blkcg.

## Per-cgroup `bdi_writeback`

On the first dirty in a `(memcg, blkcg)` pair, `cgwb_create()`
([`mm/backing-dev.c`](https://elixir.bootlin.com/linux/latest/source/mm/backing-dev.c))
allocates a fresh `bdi_writeback` and inserts it into `bdi->cgwb_tree`
keyed by that pair. Subsequent dirtiers in the same cgroup find it via
`wb_get_create()`.

Each cgwb has its own `b_dirty` / `b_io` lists, its own `dwork`, and
its own `dirty_ratelimit`. This is the unit at which the PI controller
runs: a noisy memcg gets *its* `dirty_ratelimit` clamped without
penalising other cgroups sharing the same BDI.

## `bdi_split_work_to_wbs`

A `sync(2)` on a superblock needs to flush dirty pages across *all*
cgwbs sharing that BDI. `bdi_split_work_to_wbs()` iterates `bdi->wb_list`
and queues one `wb_writeback_work` to each — fan-out is explicit, not
implicit, so individual cgwbs can refuse work (e.g. if their state says
no dirty I/O).

## Inode-to-cgwb attribution

An inode written by tasks from multiple cgroups needs an owner — you
can only attribute I/O to one. The kernel uses *foreign-inode
detection*:

- `wbc_attach_fdatawrite_inode(wbc, inode)` records the dirtying memcg
  on each `writepages` call.
- `wbc_account_cgroup_owner(wbc, page, bytes)` accumulates per-cgroup
  byte counts inside the wbc.
- `wbc_detach_inode(wbc)` compares those counts against the inode's
  current owner. If a different cgroup dominates by a margin, it
  schedules `inode_switch_wbs()` to migrate the inode to that cgroup's
  wb.

The switch is asynchronous (a work_struct on `bdi_wq`) and runs under
RCU because the old wb may still be visible to in-flight writers.

## Why this is the buggy half

The async cgwb release path has produced multiple UAFs over the years
(commits `6689f01d6740`, `8f5857be99f1`, `49717c7bd6b8`). The pattern:
an inode (or a writer) holds an indirect reference to a cgwb whose
percpu_ref already started zero-counting. Every fix has tightened the
RCU contract — *never* dereference a cgwb pointer outside
`rcu_read_lock()` without an explicit `wb_get()`. New code that touches
cgwb pointers should follow that pattern by default.

# Relations With Other Subsystems

The writeback subsystem sits between several others and is usually the
*coordinator*, not the *actor*.

| Subsystem | Contract |
|-----------|----------|
| **Block layer** | Provides `request_queue`, which embeds a `backing_dev_info`. Writeback emits bios via filesystem `writepages` → block layer; block I/O completion calls `end_page_writeback()`, which feeds `wb_update_bandwidth()`. blk-wbt (block writeback throttle) reads writeback hints to avoid oversubscribing the device. |
| **Memory management** | Page cache provides the `PAGECACHE_TAG_DIRTY` tag; writeback consumes it via `tag_pages_for_writeback`. Reclaim (`mm/vmscan.c`) triggers writeback via `wakeup_flusher_threads()` when LRU scanning finds dirty pages. `global_dirtyable_memory()` reads MM zone state to compute thresholds. |
| **VFS** | Inode lifecycle drives writeback list membership: `__mark_inode_dirty` queues to `wb->b_dirty`, `evict_inodes()` waits on `I_SYNC`. Superblock `s_inodes` list anchors the per-sb walk in `sync_inodes_sb`. |
| **cgroup** | memcg and blkcg provide the keys that index `bdi->cgwb_tree`; lifetime hooks (`mem_cgroup_css_offline`, `blkcg_destroy_blkgs`) drive cgwb release. The contract is "writeback always reads memcg/blkcg pointers under RCU and via `wb_get()` only". |
| **Filesystem `->writepages`** | Writeback hands the filesystem a `writeback_control` with a budget and a sync mode. Filesystem must (a) respect `nr_to_write`, (b) honour `WB_SYNC_ALL` by waiting on submitted I/O, (c) call `tag_pages_for_writeback` when iterating cyclic ranges. |

# Summary

**Main advantages**

- **Separation of policy from mechanism.** `fs/fs-writeback.c` decides
  *when* and *what* to write; `mm/page-writeback.c` decides *how fast*.
  Each half stays small enough to reason about.
- **Producer-side throttling.** Dirtiers slow themselves down on
  `folio_mark_dirty`, with a per-CPU fast path that costs an
  `add+compare` for the common case. No background "dirty police"
  thread is needed.
- **Proportional fairness across devices.** `wb_thresh` is a function of
  `avg_write_bandwidth`, so fast disks naturally absorb more of the
  dirty budget than slow ones — without admin tuning.
- **Per-cgroup isolation.** `CONFIG_CGROUP_WRITEBACK` runs the entire
  pipeline per `(memcg, blkcg)` pair, so noisy containers can't
  starve their neighbours of the dirty budget.
- **Bounded latency.** `wb_max_pause` caps individual throttle pauses
  at 200 ms; the loop reruns rather than sleeping arbitrarily long,
  keeping latency predictable for interactive workloads.

**Main problems / limitations**

- **PI controller tuning is fragile.** `wb_position_ratio` constants
  were derived empirically against ~2010-era spinning disks. NVMe and
  fast NVMe-oF targets can saturate the system before the controller
  catches up, manifesting as periodic spikes in dirty count.
- **cgroup writeback teardown has been a UAF magnet.** The async cgwb
  release through RCU + percpu_ref has been re-fixed across many
  releases (`6689f01d6740`, `8f5857be99f1`, `49717c7bd6b8`). The pattern
  recurs whenever a new caller forgets the "always under RCU, always
  via `wb_get()`" rule.
- **`balance_dirty_pages` runs in the dirtier's context.** That is
  great for backpressure but bad for tasks holding locks while
  dirtying — they sleep in `io_schedule_timeout()` with the lock held,
  which has caused real priority-inversion and deadlock bugs in
  filesystems that dirty inside transactions.
- **Foreign-inode migration is best-effort.** When two cgroups
  alternate writes to the same inode, the wb keeps switching, and
  accounting accuracy degrades. The kernel deliberately under-corrects
  to avoid switch storms, at the cost of attribution drift.
- **No explicit priority class.** Background writeback, sync(2)
  writeback, and reclaim-triggered writeback all flow through the same
  `bdi_wq` queue. Distinguishing "I'm trying to free memory right now"
  from "the periodic timer just fired" is the caller's responsibility
  via `wb_reason`, but the scheduler treats them identically.
