## 1. What is the NVMe Target?

NVMe is the wire protocol that talks to local PCIe SSDs. **NVMe over
Fabrics (NVMe-oF)** stretches that same protocol across a network so a
remote host can use a server's storage as if it were a local NVMe drive.

The Linux kernel ships **two halves** of NVMe-oF:

| Half | Where | Role |
|------|-------|------|
| **Host** (initiator) | [`drivers/nvme/host/`](https://elixir.bootlin.com/linux/latest/source/drivers/nvme/host) | Consumer — turns a remote subsystem into `/dev/nvmeXnY` |
| **Target** (this post) | [`drivers/nvme/target/`](https://elixir.bootlin.com/linux/latest/source/drivers/nvme/target) | Producer — exports local block devices/files as NVMe namespaces |

Think of it as the NVMe equivalent of the kernel's iSCSI target (LIO), but
speaking native NVMe semantics: queue pairs, namespaces, command sets —
not SCSI LUNs.

### Why it exists

- **Disaggregation.** Pool SSDs in a JBOF (Just a Bunch Of Flash) and serve
  them over the network. Compute nodes get diskless flexibility.
- **Lower overhead than iSCSI.** NVMe's submission/completion queue model
  maps cleanly to RDMA and TCP; no SCSI translation layer.
- **Multiple fabrics, one protocol.** TCP for commodity Ethernet, RDMA for
  RoCE/IB, FC for legacy SAN, loop for in-kernel testing, PCI-EPF for
  exposing the *host CPU itself* as an NVMe controller to another machine
  over PCIe.

```
   Host (initiator)                  Target (this post)
   ┌──────────────┐  NVMe-oF PDUs   ┌─────────────────────┐
   │ /dev/nvme0n1 │ ◄────────────► │  /dev/sdb, /file.img │
   │ blk-mq queue │   TCP/RDMA/FC   │  (real backing store)│
   └──────────────┘                  └─────────────────────┘
        Linux nvme-fabrics              Linux nvmet + transport
        (drivers/nvme/host)             (drivers/nvme/target)
```

## 2. Top View: The Object Graph

Everything in `nvmet` revolves around six core objects, defined in
[`nvmet.h`](https://elixir.bootlin.com/linux/latest/source/drivers/nvme/target/nvmet.h):

```
                     ┌──────────────────────┐
                     │  struct nvmet_port   │   listening endpoint
                     │  (TCP:4420, IB GID…) │   one per IP/transport tuple
                     └──────────┬───────────┘
                                │  tr_ops (transport vtable)
                                │  links subsystems via configfs symlinks
                                ▼
                     ┌──────────────────────┐
                     │ struct nvmet_subsys  │   the "device" (subsysnqn)
                     │  e.g. nqn.…:storage  │
                     └────┬─────────┬───────┘
                          │         │
              namespaces  │         │  ctrls (active sessions)
                          ▼         ▼
              ┌────────────────┐  ┌──────────────────────┐
              │ struct nvmet_ns│  │ struct nvmet_ctrl    │
              │  NSID, bdev or │  │  per-host controller │
              │  file, size,…  │  │  cntlid, kato, AERs  │
              └────────────────┘  └──────────┬───────────┘
                                             │
                                  sqs[]/cqs[] (queue arrays)
                                             ▼
                                ┌──────────────────────────┐
                                │  struct nvmet_sq / _cq   │
                                │  qid 0 = admin, >0 = I/O │
                                └────────────┬─────────────┘
                                             │  every in-flight command
                                             ▼
                                ┌──────────────────────────┐
                                │     struct nvmet_req     │
                                │  cmd, cqe, sg, ns,       │
                                │  execute(), inline_bvec  │
                                └──────────────────────────┘
```

A quick guide to who owns whom:

- **`nvmet_port`** — a network listener. It does *not* own subsystems; it
  references them via configfs symlinks (one port can expose many
  subsystems; one subsystem can be reachable through many ports).
- **`nvmet_subsys`** — the logical "device" identified by its NQN. Holds an
  `xarray` of namespaces and a list of currently connected controllers.
- **`nvmet_ns`** — one namespace, backed by either a `struct block_device`
  *or* a `struct file`. The choice is per-namespace.
- **`nvmet_ctrl`** — a per-host controller; created when a host sends the
  fabrics `Connect` command. Holds the SQ/CQ arrays and async-event slots.
- **`nvmet_sq` / `nvmet_cq`** — the queue pair. `qid==0` is the admin
  queue; `qid>0` are I/O queues. A `percpu_ref` protects in-flight commands
  from racing with queue teardown.
- **`nvmet_req`** — one command in flight. This is the unit that travels
  from the transport through parsing to the backend executor.

### The `nvmet_req` is the workhorse

```c
struct nvmet_req {
    struct nvme_command   *cmd;        /* inbound SQE */
    struct nvme_completion *cqe;       /* outbound CQE */
    struct nvmet_sq       *sq;
    struct nvmet_ns       *ns;
    struct scatterlist    *sg;
    void (*execute)(struct nvmet_req *req);   /* set by the parser */
    const struct nvmet_fabrics_ops *ops;      /* transport */
    union {
        struct { struct bio inline_bio; }   b;   /* bdev backend */
        struct { struct kiocb iocb; … }     f;   /* file backend */
        struct { struct request *rq; }      p;   /* passthru */
    };
    struct bio_vec inline_bvec[NVMET_MAX_INLINE_BIOVEC];
    …
};
```

Two things to notice:

1. **`execute` is a function pointer.** Parsers in `admin-cmd.c`,
   `io-cmd-bdev.c`, `io-cmd-file.c`, `zns.c`, `pr.c` set it based on opcode
   and backend. The transport doesn't care what it points to — it just
   calls it.
2. **`inline_bio` and `inline_bvec[]` are embedded.** Small reads/writes
   avoid heap allocation entirely: the bio is built inside the request
   object using inline `bio_vec`s.

See: [`nvmet.h` ~ line 380](https://elixir.bootlin.com/linux/latest/source/drivers/nvme/target/nvmet.h#L380).

## 3. The Command Lifecycle (Top to Bottom)

Here is the journey of a single NVMe `Write` from wire to backing storage:

```
  ┌───────────────────────────────────────────────────────────────┐
  │  Host sends NVMe SQE (Submission Queue Entry) over fabric     │
  └───────────────────────────────┬───────────────────────────────┘
                                  ▼
  (1) Transport receives the PDU (e.g. nvmet_tcp recv path)
      Allocates a struct nvmet_req, copies cmd, sets up sg list
                                  │
                                  ▼
  (2) Transport calls nvmet_req_init(req, sq, ops)         core.c
        - dispatches on queue type:
            sq->ctrl == NULL  → nvmet_parse_connect_cmd()
            sq->qid == 0      → nvmet_parse_admin_cmd()
            sq->qid >  0      → nvmet_parse_io_cmd()
        - parser sets req->execute and validates fields
        - percpu_ref_tryget_live(&sq->ref) pins the queue
                                  │
                                  ▼
  (3) Transport pulls in-bound data (if WRITE),
      then invokes req->execute(req).
                                  │
                                  ▼
  (4) Backend executor (e.g. nvmet_bdev_execute_rw)        io-cmd-bdev.c
        - builds a bio over req->sg (inline if it fits)
        - bio->bi_end_io = nvmet_bio_done
        - submit_bio(bio)        ←── returns immediately, async
                                  │
                  block layer / file VFS completion
                                  │
                                  ▼
  (5) nvmet_bio_done → nvmet_req_complete(req, status)
        - fills req->cqe
        - calls ops->queue_response(req)
                                  │
                                  ▼
  (6) Transport serializes the CQE on the wire,
      releases the percpu_ref.
```

The two pivot points worth memorizing:

- **`nvmet_req_init()`** in
  [`core.c`](https://elixir.bootlin.com/linux/latest/source/drivers/nvme/target/core.c)
  — this is where "raw bytes" become "a command with a known
  destination."
- **`nvmet_req_complete()`** in the same file — the symmetric exit gate.
  Every backend, every transport, every error path eventually goes through
  it.

### Admin vs. I/O divergence

- Admin queue (qid 0) commands — `Identify`, `Get/Set Features`, `Get Log
  Page`, `Async Event Request`, `Keep Alive`, etc. — are handled in
  [`admin-cmd.c`](https://elixir.bootlin.com/linux/latest/source/drivers/nvme/target/admin-cmd.c).
  Their executors are usually synchronous: build a buffer, copy it into
  the SGL, complete.
- I/O queue commands — `Read`, `Write`, `Flush`, `Dataset Management`,
  `Write Zeroes`, plus Zoned variants — go through
  [`io-cmd-bdev.c`](https://elixir.bootlin.com/linux/latest/source/drivers/nvme/target/io-cmd-bdev.c),
  [`io-cmd-file.c`](https://elixir.bootlin.com/linux/latest/source/drivers/nvme/target/io-cmd-file.c),
  or
  [`zns.c`](https://elixir.bootlin.com/linux/latest/source/drivers/nvme/target/zns.c).
  They are *async* — the executor submits and returns; completion comes
  later from the block layer or VFS.

### Dispatcher snapshot

Boiled down from `nvmet_parse_io_cmd()`:

```c
switch (req->ns->csi) {
case NVME_CSI_NVM:
    if (req->ns->file)
        return nvmet_file_parse_io_cmd(req);   /* sets req->execute */
    return nvmet_bdev_parse_io_cmd(req);
case NVME_CSI_ZNS:
    return nvmet_bdev_zns_parse_io_cmd(req);
}
```

And inside `nvmet_bdev_parse_io_cmd()`:

```c
switch (req->cmd->common.opcode) {
case nvme_cmd_read:
case nvme_cmd_write:    req->execute = nvmet_bdev_execute_rw;          break;
case nvme_cmd_flush:    req->execute = nvmet_bdev_execute_flush;       break;
case nvme_cmd_dsm:      req->execute = nvmet_bdev_execute_dsm;         break;
case nvme_cmd_write_zeroes:
                        req->execute = nvmet_bdev_execute_write_zeroes;break;
default:                return nvmet_report_invalid_opcode(req);
}
```

## 4. Two Backends: Block Device and File

NVMe-oF targets need to put bytes *somewhere*. `nvmet` supports three
backend styles, picked per namespace:

| Backend | Set by | Path |
|---------|--------|------|
| **Block device** | `device_path` = `/dev/sdb`, etc. | `bdev_file_open_by_path()` + `submit_bio()` |
| **File**         | `device_path` = `/data/img.raw`  | `filp_open()` + `vfs_iter_*` via `kiocb`  |
| **Passthru**     | `passthru/device_path` = real `/dev/nvmeX` | forward the NVMe command itself to a local NVMe ctrl |

### 4.1 Block device backend
[`io-cmd-bdev.c`](https://elixir.bootlin.com/linux/latest/source/drivers/nvme/target/io-cmd-bdev.c)

`nvmet_bdev_execute_rw()` is the canonical example of the zero-copy path:

```c
if (nvmet_use_inline_bvec(req))
    bio = &req->b.inline_bio;          /* embedded, no kmalloc */
else
    bio = bio_alloc(req->ns->bdev, …);

for_each_sg(req->sg, sg, req->sg_cnt, i)
    bio_add_page(bio, sg_page(sg), sg->length, sg->offset);

bio->bi_end_io = nvmet_bio_done;
bio->bi_private = req;
submit_bio(bio);                       /* returns immediately */
```

Notable details:

- **T10-PI / Data Integrity Field**: if the underlying bdev advertises
  integrity, `nvmet_bdev_ns_enable_integrity()` enables end-to-end CRC
  checks across the fabric.
- **Zoned block devices (ZNS)**: detected via `bdev_is_zoned()`; the
  ZNS-specific opcodes (`Zone Append`, `Zone Management Send/Recv`) are
  routed into
  [`zns.c`](https://elixir.bootlin.com/linux/latest/source/drivers/nvme/target/zns.c).

### 4.2 File backend
[`io-cmd-file.c`](https://elixir.bootlin.com/linux/latest/source/drivers/nvme/target/io-cmd-file.c)

```c
iov_iter_bvec(&iter, rw, req->f.bvec, nr_segs, count);
iocb->ki_filp     = req->ns->file;
iocb->ki_complete = nvmet_file_io_done;
file->f_op->write_iter(iocb, &iter);   /* AIO via kiocb */
```

By default the file is opened with `O_DIRECT`; setting the `buffered_io=1`
configfs attribute drops `O_DIRECT` and lets the page cache do its thing
(useful for files on top of tiered/networked filesystems).

### 4.3 Picking a backend

`nvmet_ns_enable()` (in `core.c`) tries each backend in order until one
opens the path successfully — typically bdev first, then file. The choice
is sticky for the namespace's lifetime.

## 5. The Transport Abstraction

All transports register a vtable:

```c
struct nvmet_fabrics_ops {
    struct module *owner;
    unsigned int   type;          /* NVMF_TRTYPE_TCP/RDMA/FC/LOOP/PCI */
    unsigned int   flags;         /* KEYED_SGLS, METADATA_SUPPORTED */
    void (*queue_response)(struct nvmet_req *req);
    int  (*add_port)   (struct nvmet_port *port);
    void (*remove_port)(struct nvmet_port *port);
    void (*delete_ctrl)(struct nvmet_ctrl *ctrl);
    void (*disc_traddr)(struct nvmet_req *, struct nvmet_port *, char *);
    …
};
```

See [`nvmet.h` ~ line 411](https://elixir.bootlin.com/linux/latest/source/drivers/nvme/target/nvmet.h#L411).

Each transport drops a single global table pointer into
`nvmet_transports[type]`. Ports load the right transport module on demand
via `request_module("nvmet-transport-%d", trtype)`.

```
   ┌──────────────────────────────────────────────────────────┐
   │                    core.c (transport-agnostic)           │
   │   nvmet_req_init → parse → execute → req_complete        │
   └────────▲────────────────────────────────────────▲────────┘
            │ queue request                          │ queue_response
            │                                        │
   ┌────────┴────────┐   ┌──────────────┐   ┌────────┴─────────┐
   │  tcp.c          │   │  rdma.c      │   │  loop.c          │
   │  socket + PDU   │   │  IB QP + RW  │   │  blk-mq pipe     │
   └─────────────────┘   └──────────────┘   └──────────────────┘
   ┌─────────────────┐   ┌──────────────────────┐
   │  fc.c           │   │  pci-epf.c           │
   │  FC-NVMe LS/IO  │   │  PCIe endpoint HW    │
   └─────────────────┘   └──────────────────────┘
```

### TCP transport ([`tcp.c`](https://elixir.bootlin.com/linux/latest/source/drivers/nvme/target/tcp.c))

The most widely deployed fabric. Each connection is a `struct
nvmet_tcp_queue` around a kernel socket. Inbound bytes get parsed into
NVMe-TCP PDUs (`CapsuleCmd`, `H2CData`, …); each command capsule becomes
an `nvmet_tcp_cmd` which holds an `nvmet_req`. `queue_response()` serializes
the CQE into a `RspPDU` and writes it to the socket. TLS handshake hooks
sit at the front of the queue state machine.

### Loop transport ([`loop.c`](https://elixir.bootlin.com/linux/latest/source/drivers/nvme/target/loop.c))

A tiny in-kernel pipe: the host's blk-mq request and the target's
`nvmet_req` share memory. No PDUs, no copies. Used heavily for **testing
the target stack itself** and for validating new opcodes without standing
up real fabric hardware.

### RDMA transport ([`rdma.c`](https://elixir.bootlin.com/linux/latest/source/drivers/nvme/target/rdma.c))

Each queue is an RDMA QP. The host's command capsule carries a *keyed
SGL* — basically "here is a remote key and address, please fetch the data
yourself." The target then issues RDMA `Read` to pull WRITE data, or RDMA
`Write` to push READ data, before signalling the CQE with a SEND.

### PCI-EPF transport ([`pci-epf.c`](https://elixir.bootlin.com/linux/latest/source/drivers/nvme/target/pci-epf.c))

This one is wild: the local CPU acts as a PCIe **endpoint function** —
i.e., it pretends to be an NVMe SSD plugged into *another* machine's PCIe
slot. The peer machine sees an NVMe controller and uses its normal
`drivers/nvme/host/pci.c` driver. This is how SmartNIC / DPU vendors
present compute-side storage as a PCIe NVMe device.

## 6. configfs: How Userspace Wires it Up

`nvmet` has no ioctls. Configuration lives entirely under
`/sys/kernel/config/nvmet/`, served by
[`configfs.c`](https://elixir.bootlin.com/linux/latest/source/drivers/nvme/target/configfs.c).
The userspace tool is `nvmetcli`, but you can use raw `mkdir`/`echo` too.

```
/sys/kernel/config/nvmet/
├── subsystems/
│   └── nqn.2026-05.org.example:storage/
│       ├── attr_allow_any_host
│       ├── attr_serial, attr_model, attr_firmware
│       ├── namespaces/
│       │   └── 1/
│       │       ├── device_path     ← echo /dev/sdb here
│       │       ├── buffered_io
│       │       ├── enable          ← echo 1 to activate
│       │       └── ana_grpid
│       └── allowed_hosts/          ← symlinks to hosts/<nqn>
├── ports/
│   └── 1/
│       ├── addr_trtype             ← tcp | rdma | fc | loop
│       ├── addr_traddr             ← IP/WWPN/PCI BDF
│       ├── addr_trsvcid            ← TCP port (e.g. 4420)
│       ├── addr_adrfam             ← ipv4 | ipv6 | ib | fc
│       ├── subsystems/             ← symlinks to subsystems/<nqn>
│       └── ana_groups/
└── hosts/
    └── nqn.2026-05.org.example:host1/
        ├── dhchap_key
        └── dhchap_ctrl_key
```

Minimal recipe:

```bash
modprobe nvmet nvmet-tcp
mount -t configfs none /sys/kernel/config

NQN=nqn.2026-05.org.example:storage
mkdir /sys/kernel/config/nvmet/subsystems/$NQN
echo 1 > /sys/kernel/config/nvmet/subsystems/$NQN/attr_allow_any_host

mkdir /sys/kernel/config/nvmet/subsystems/$NQN/namespaces/1
echo /dev/sdb > /sys/kernel/config/nvmet/subsystems/$NQN/namespaces/1/device_path
echo 1       > /sys/kernel/config/nvmet/subsystems/$NQN/namespaces/1/enable

mkdir /sys/kernel/config/nvmet/ports/1
echo tcp           > /sys/kernel/config/nvmet/ports/1/addr_trtype
echo ipv4          > /sys/kernel/config/nvmet/ports/1/addr_adrfam
echo 192.168.1.10  > /sys/kernel/config/nvmet/ports/1/addr_traddr
echo 4420          > /sys/kernel/config/nvmet/ports/1/addr_trsvcid

ln -s /sys/kernel/config/nvmet/subsystems/$NQN \
      /sys/kernel/config/nvmet/ports/1/subsystems/$NQN
```

The symlink in the last step is what actually arms the port: it triggers
`add_port` on the TCP transport, opens the listening socket, and makes
the subsystem discoverable on that endpoint.

## 7. Discovery and ANA

When a host doesn't know which subsystems exist yet, it connects to a
**Discovery Controller** at the fixed NQN
`nqn.2014-08.org.nvmexpress.discovery`. The target satisfies the Discovery
Log Page request from
[`discovery.c`](https://elixir.bootlin.com/linux/latest/source/drivers/nvme/target/discovery.c)
by walking every port × subsystem pairing the host is allowed to see.

When config changes (new namespace, port enabled, host added), `nvmet`
increments a global generation counter and sends an **Async Event
Notification (AEN)** to every connected discovery controller. The host
then re-reads the log page.

**ANA (Asymmetric Namespace Access)** lets a multi-port target tell the
host that namespace N is *Optimized* via port A but only *Non-optimized*
via port B — so the host's `nvme-multipath` layer prefers the right path.
Each `nvmet_ns` belongs to one of `NVMET_MAX_ANAGRPS` groups; the per-port
state lives in `nvmet_port->ana_state[grpid]`.

## 8. Authentication

Optional, gated by `CONFIG_NVME_TARGET_AUTH`. The mechanism is
**DH-HMAC-CHAP** — Diffie-Hellman key exchange followed by HMAC-based
challenge/response. Source files:
[`auth.c`](https://elixir.bootlin.com/linux/latest/source/drivers/nvme/target/auth.c),
[`fabrics-cmd-auth.c`](https://elixir.bootlin.com/linux/latest/source/drivers/nvme/target/fabrics-cmd-auth.c).

Flow (per queue):

```
   Host → Connect                                    target allocates ctrl
        ←  Connect rsp (auth required)               sq->authenticated = false
   Host → AuthSend  (NegotiateChallenge)
        ←  AuthRecv (Challenge + DH params + salts)
   Host → AuthSend  (Response = HMAC(secret, ...))   target verifies
        ←  AuthRecv (Success / target's response)
                                                     sq->authenticated = true
   Host → I/O commands now accepted
```

`nvmet_parse_io_cmd()` refuses I/O until `sq->authenticated` flips true.
Session keys can also be used to set up **TLS 1.3 on TCP** (PSK
established by the auth exchange), giving in-line encryption for free
without certificates.

| [`trace.c`/`debugfs.c`](https://elixir.bootlin.com/linux/latest/source/drivers/nvme/target) | — | Tracepoints + debugfs |

Total: ~27k lines of C — small for a feature this consequential, and a
reflection of how cleanly the transport/backend split was drawn.

## 10. Going Userspace: a Zero-Copy NVMe Target with io_uring + AF_XDP

Everything above lives in the kernel. But there is a parallel universe —
**userspace NVMe targets** — where the same protocol is served by a
process that polls hardware directly. SPDK is the well-known example; it
ships its own NIC driver (DPDK), its own TCP stack, and talks to NVMe
SSDs through `vfio`. The benefits are real (sub-10µs added latency, full
core isolation), but the cost is high: you re-implement large chunks of
the kernel's network and storage subsystems.

A more pragmatic modern design **borrows from the kernel where it helps
and bypasses it where it hurts**, using two relatively new interfaces:

- **`io_uring`** for the storage side — submit NVMe I/O to a real
  `/dev/nvmeXnY` (or even a raw `/dev/ng*` passthrough char device) with
  registered buffers and a polling submission queue.
- **`AF_XDP`** (with XDP redirection) for the network side — let the NIC
  DMA frames straight into a userspace memory region (UMEM) that the
  kernel never copies.

When you make those two interfaces share **the same memory**, you get an
end-to-end zero-copy data path with no kernel networking stack on the
hot path.

### 10.1 Why userspace at all?

| Pain point in kernel `nvmet`                | Userspace win                     |
|---------------------------------------------|-----------------------------------|
| Context switches per command                | Polled I/O, single core, no switch |
| TCP stack overhead on small PDUs            | Custom slim stack or AF_XDP bypass |
| Hard to tune per-tenant QoS / scheduling    | Fully under app control            |
| Iteration / debug cycle (kernel rebuilds)   | Just restart a process             |
| Hard to integrate exotic backends           | Any user library is fair game      |

The price is fragility: you own NIC reset, page pinning lifetimes,
hugepage allocation, and NUMA placement.

### 10.2 The data path at 30,000 feet

```
                ┌────────────────────────────────────────────────────┐
                │            One physical hugepage region            │
                │  (UMEM — registered as both AF_XDP UMEM and        │
                │   io_uring fixed buffers via IORING_REGISTER_BUFFERS) │
                └──────▲──────────────────────────────────▲──────────┘
                       │ NIC DMA (RX/TX)                  │ NVMe DMA (R/W)
                       │                                  │
        ┌──────────────┴───────────┐         ┌────────────┴────────────┐
        │  NIC (XDP-capable)       │         │  NVMe SSD               │
        └──────────────▲───────────┘         └────────────▲────────────┘
                       │                                  │
                ┌──────┴──────┐                    ┌──────┴───────┐
                │ XDP program │                    │ io_uring SQE │
                │ redirect →  │                    │ WRITE_FIXED  │
                │  AF_XDP sk  │                    │ buf_index=N  │
                └──────▲──────┘                    └──────▲───────┘
                       │ rx ring desc                    │ submit
                       │                                  │
                       └──────────► userspace worker ◄────┘
                                    (one busy-poll loop)
```

The userspace worker thread:

1. Pulls a descriptor off the AF_XDP RX ring (NIC has already DMA'd the
   frame into UMEM).
2. Parses the NVMe-TCP PDU **in-place** — the header is a tiny prefix in
   the same frame.
3. For a `WRITE` capsule: hands the data range (UMEM offset + length) to
   `io_uring` as `IORING_OP_WRITE_FIXED`.
4. For a `READ`: submits `IORING_OP_READ_FIXED` into a free UMEM frame,
   then enqueues an AF_XDP TX descriptor for the eventual response PDU.
5. Reaps completions from `io_uring`'s CQ ring, builds the NVMe-TCP
   `RspPDU` header (24 bytes), and pushes the frame onto AF_XDP TX.

No `read()`, no `recv()`, no `memcpy()` on the data path. The CPU only
touches PDU headers.

### 10.3 The storage half: io_uring done right

Three `io_uring` features matter for this design:

**Registered buffers (`IORING_REGISTER_BUFFERS`)** — pre-pin the entire
UMEM region once at startup. Subsequent `WRITE_FIXED`/`READ_FIXED` ops
reference it by index, skipping per-I/O `get_user_pages()`.

```c
struct iovec iov = { .iov_base = umem_area, .iov_len = umem_size };
io_uring_register_buffers(&ring, &iov, 1);
…
io_uring_prep_write_fixed(sqe, fd_nvme,
                          umem_area + frame_off + data_off,
                          data_len,
                          lba_offset,
                          /* buf_index */ 0);
```

**SQPOLL (`IORING_SETUP_SQPOLL`)** — a kernel thread polls the SQ ring,
so your worker never makes a syscall to kick I/O. Pair with
`IORING_SETUP_IOPOLL` (or `NVMe poll queues`) and the storage side runs
syscall-free under load.

**NVMe passthrough (`IORING_OP_URING_CMD`)** — for the truly bold,
`io_uring` can submit raw NVMe commands to `/dev/ng0n1` via
`nvme_uring_cmd`. This lets the userspace target forward host commands
*as-is* to a local NVMe SSD — effectively kernel `nvmet/passthru.c` but
without the kernel hop. Useful when the local drive's command set
already matches what the remote host requests (vendor specific opcodes,
ZNS, KV).

### 10.4 The network half: AF_XDP with zero-copy mode

```c
struct xsk_umem_config ucfg = {
    .fill_size      = 4096,
    .comp_size      = 4096,
    .frame_size     = 4096,     /* match hugepage alignment */
    .flags          = XDP_UMEM_UNALIGNED_CHUNK_FLAG,
};
xsk_umem__create(&umem, umem_area, umem_size, &fq, &cq, &ucfg);

struct xsk_socket_config scfg = {
    .rx_size  = NUM_FRAMES,
    .tx_size  = NUM_FRAMES,
    .libxdp_flags = 0,
    .xdp_flags    = XDP_FLAGS_DRV_MODE,   /* native, not generic */
    .bind_flags   = XDP_ZEROCOPY,
};
xsk_socket__create(&xsk, ifname, queue_id, umem, &rx, &tx, &scfg);
```

An attached XDP program decides which packets to redirect:

```c
SEC("xdp")
int nvme_tcp_filter(struct xdp_md *ctx)
{
    /* tcp.dst_port == 4420 → redirect to AF_XDP queue */
    if (is_nvme_tcp_pdu(ctx))
        return bpf_redirect_map(&xsks_map, ctx->rx_queue_index, 0);
    return XDP_PASS;     /* let kernel handle everything else */
}
```

With `XDP_ZEROCOPY`, the NIC's RX descriptor ring is populated with UMEM
chunk addresses; no `skb` is ever allocated for redirected frames.

### 10.5 The hard part: TCP framing

A `WRITE` capsule may carry a 16 KB data block. That block will be split
by TCP into multiple segments, each landing in a separate UMEM chunk —
likely *not contiguous*. The naïve "build a bio from one chunk" plan
breaks immediately.

Three options, in increasing order of complexity:

1. **Scatter-gather submit.** Walk the chunks belonging to one PDU and
   submit an `IORING_OP_WRITEV_FIXED` with an `iovec` listing each.
   Kernel builds a multi-bvec bio. Cheap, but you must reassemble PDUs
   from segments in userspace.
2. **Header split offload.** Tell the NIC to split NVMe-TCP headers from
   data into separate buffer pools (modern NICs can do this with HDS or
   "split header" flow-director rules). Headers land in a small ring,
   data lands page-aligned. Now each PDU's data sits in one or two
   chunks.
3. **Custom TCP stack.** Reassemble byte streams in userspace, place
   reconstructed PDUs into a second UMEM-resident buffer pool. This is
   what SPDK does. Maximum control, maximum code.

Option (1) is the sweet spot for an "io_uring + AF_XDP" prototype: you
get true zero-copy with a few hundred lines of TCP state machine.

### 10.6 Completion path

Sending a response is symmetric:

```c
/* 1. Submit storage read */
io_uring_prep_read_fixed(sqe, fd_nvme, umem + tx_frame,
                         len, lba, /*buf_index=*/0);
sqe->user_data = (u64)cmd_ctx;
io_uring_submit(&ring);

/* 2. On CQE arrival */
struct io_uring_cqe *cqe;
io_uring_wait_cqe(&ring, &cqe);
struct cmd_ctx *c = (void *)cqe->user_data;
build_nvme_tcp_rsp_header(umem + c->tx_frame);   /* only header touched */

/* 3. Hand the frame to AF_XDP TX */
struct xdp_desc *d = xsk_ring_prod__tx_desc(&tx, idx);
d->addr = c->tx_frame;
d->len  = sizeof(nvme_tcp_rsp_pdu) + c->data_len;
xsk_ring_prod__submit(&tx, 1);
sendto(xsk_fd, NULL, 0, MSG_DONTWAIT, NULL, 0);   /* kick TX */
```

The same physical pages that the SSD DMA'd into are now DMA'd back out
by the NIC. The CPU only wrote 24 bytes of header.

### 10.7 Where this design lands

| Concern                | SPDK (DPDK+TCP)        | io_uring + AF_XDP      | Kernel `nvmet-tcp`         |
|------------------------|------------------------|------------------------|----------------------------|
| Zero-copy data         | yes                    | yes                    | partial (sendpage / MSG_SPLICE) |
| Zero-copy headers      | yes                    | no (24 B touched)      | no                         |
| Lines of code          | very large             | medium                 | small                      |
| Reuses kernel drivers  | no (own NIC + NVMe)    | **yes** (XDP NIC drv, blk-mq) | yes                |
| Idle-CPU cost          | high (busy poll)       | tunable                | near zero                  |
| Latency floor          | lowest                 | ~1–2 µs above SPDK     | a few µs above us          |
| Multi-tenant fairness  | DIY                    | DIY + cgroups          | cgroups for free           |

The `io_uring + AF_XDP` design is the **interesting middle ground**:
near-SPDK performance, but you keep the kernel's NIC and block drivers,
and most of your code is application logic — not driver re-implementation.

### 10.8 Practical hints if you build one

- **Hugepages, always.** 2 MB pages give you 512 UMEM chunks per page;
  fewer TLB misses on the NIC's IOMMU walk.
- **One queue per core.** Bind an AF_XDP socket and an `io_uring` to the
  same CPU, IRQ-pin the NIC queue, and run a single busy-poll loop. No
  locks anywhere on the fast path.
- **Header split if the NIC supports it.** Saves the only memcpy left.
- **`io_uring` SQPOLL idle timeout.** Set `sq_thread_idle` low; the SQ
  poller and your worker both end up sleeping the moment traffic stops.
- **AF_XDP `XDP_USE_NEED_WAKEUP`.** Lets the worker actually sleep
  instead of spinning when both rings are empty.
- **NVMe poll queues (`/sys/class/nvme/.../poll_queues`).** Combined
  with `IORING_SETUP_IOPOLL`, you remove storage-side interrupts too.
- **Memory registration is per-ring.** If you have one ring per core,
  register the *full* UMEM in each — registration is cheap and lets any
  worker complete I/O on any buffer.

### 10.9 What's still missing

A few features kernel `nvmet` gives you for free and a userspace target
must reimplement:

- **DH-HMAC-CHAP + TLS 1.3** — userspace `kTLS`/OpenSSL on top of the
  raw byte stream, *or* skip authentication in trusted environments.
- **ANA state and discovery AENs** — periodic broadcasts, generation
  counter, all live in your daemon.
- **Persistent reservations** — surprisingly subtle; cluster managers
  rely on this for failover. `nvmet/pr.c` is 1100 lines for a reason.
- **Multi-queue host fairness** — `nvmet` benefits from blk-mq's
  scheduler; in userspace you build your own.

These aren't reasons not to do it — they're reasons to scope carefully.
A research prototype that does WRITE/READ/FLUSH at line rate with no
auth and one namespace is a few thousand lines of code. A production
target rivals `nvmet` in size.

## 11. Benchmarking Methodology

An NVMe target — kernel or userspace — is only as credible as the
numbers you publish for it. The measurement setup is at least as easy
to get wrong as the implementation, and the failure modes are sneakier.

### 11.1 What to measure, in priority order

1. **Tail latency at fixed sub-saturation load** — p99, p99.9, p99.99 at
   ~70 % of peak. This is the engineering number.
2. **CPU efficiency** — kIOPS per polled core, or cycles per I/O.
3. **Peak IOPS / bandwidth.** The marketing number.

Always report percentiles, not averages. NVMe target work lives in the
tail; averages hide every interesting bug. Fio's default percentile list
is too coarse — request your own:

```
--percentile_list=50:90:99:99.9:99.99
```

### 11.2 Topology — pick three points on the curve

```
A) nvme-loop                  no NIC, no TCP   → measures bare nvmet core
B) nvmet-tcp, two-host        real NIC + TCP   → the real thing
C) bare /dev/nvmeXnY locally  no nvmet         → the ceiling
```

Always run all three. (A) tells you target overhead, (C) tells you the
backend ceiling, and (B) is the one you ship. **Skip nvmet-tcp over
`lo`** — it has none of the interesting NIC behaviors and the numbers
look better than reality.

### 11.3 Eliminate the noise floor first

The single biggest source of variance on a modern server is power
management, not your code. Before *any* measurement run:

```bash
# CPU frequency: lock to performance, optionally pin turbo
cpupower frequency-set -g performance
echo 1 > /sys/devices/system/cpu/intel_pstate/no_turbo   # for reproducibility

# C-states: shallowest only
cpupower idle-set -D 0

# IRQs: kill irqbalance, pin NIC + NVMe IRQs to dedicated cores
#   /proc/irq/<N>/<driver-name>/ is just a marker subdir; the writable
#   file lives one level up at /proc/irq/<N>/smp_affinity_list.
systemctl stop irqbalance
for irq in $(awk -F: '/eth0|nvme/ {print $1}' /proc/interrupts); do
    echo 2-9 > /proc/irq/$irq/smp_affinity_list
done

# Kernel cmdline (set in /etc/default/grub, then reboot):
#   isolcpus=2-15 nohz_full=2-15 rcu_nocbs=2-15 intel_idle.max_cstate=1
```

For sub-microsecond claims, also:

- Disable SMT *or* pin job + sibling identically — both work; flipping
  between them ruins comparability
- Pre-allocate hugepages, not on-demand
- Put target, `fio`, and NIC IRQs **all on the NIC's NUMA node** — or
  all off it, but pick one
- `echo never > /sys/kernel/mm/transparent_hugepage/defrag` to stop
  compaction stalls

### 11.4 Tooling

| Tool | Use for |
|------|---------|
| `fio --ioengine=io_uring` | Standard end-to-end host driver |
| `t/io_uring` (in fio source tree) | Lowest-overhead host driver; isolates target cost |
| SPDK `perf` | Polled host driver; eliminates host kernel from the path |
| `bpftrace` on `nvmet:*` tracepoints | Per-command latency histograms inside the target |
| `perf record -g` + flamegraph | Where target CPU actually goes |
| `ethtool -S` | NIC drops, retransmits, OOO — any non-zero is a clue |
| `nstat -az \| grep -i tcp` | Kernel TCP counters during a TCP run |
| `mpstat -P ALL 1` | Per-core asymmetry → usually bad pinning |

Use `t/io_uring` (not plain `fio --ioengine=io_uring`) when you want to
attribute observed latency to the **target** rather than the host
driver. It strips fio of everything except the submit/complete loop.

### 11.5 Backend choice and SSD preconditioning

To measure target overhead in isolation, use `null_blk` as backend:

```bash
modprobe null_blk nr_devices=1 queue_mode=2 irqmode=2 \
                  completion_nsec=0 hw_queue_depth=2048
```

`completion_nsec=0` makes the device complete instantly — what you see
is *purely* target + fabric + host. Then switch to a real SSD for the
end-to-end run.

For SSD-backed runs, **precondition** before measuring:

```bash
nvme format /dev/nvme0n1 --ses=1                       # secure erase
fio --name=precond --filename=/dev/nvme0n1 --rw=write \
    --bs=128k --iodepth=32 --numjobs=1 --loops=2       # fill device twice
```

A fresh NVMe SSD does ~1M IOPS sequential write; preconditioned, ~300k.
Both numbers are honest — pick one, label it loudly.

### 11.6 Workload matrix

Minimum credible matrix:

```
   block size:    4K, 16K, 128K
   pattern:       randread, randwrite, 70/30 randrw, seqread, seqwrite
   queue depth:   1, 4, 32, 128
   numjobs:       1, 4, 16          (= host connections)
```

The two **must-publish** points:

1. `--bs=4k --rw=randread --iodepth=1 --numjobs=1` — minimum end-to-end
   latency. This is the number you cannot fake; every wakeup, syscall,
   and context switch shows up here.
2. `--bs=4k --rw=randread --iodepth=128 --numjobs=N` ramped until target
   CPU saturates — peak IOPS per polled core.

Together they pin both ends of the curve. Everything else interpolates.

### 11.7 A representative fio invocation

```bash
fio --name=nvmeof --filename=/dev/nvme1n1 \
    --ioengine=io_uring --direct=1 --thread \
    --rw=randread --bs=4k --iodepth=128 --numjobs=8 \
    --cpus_allowed=2-9 --cpus_allowed_policy=split \
    --ramp_time=10 --runtime=120 --time_based \
    --group_reporting --percentile_list=50:90:99:99.9:99.99 \
    --output-format=json+ --output=run1.json
```

Knobs that matter:

- `--direct=1` is non-negotiable. Without it you measure the host page
  cache and your numbers are 10× too good.
- `--ramp_time=10` lets TCP windows, fio buffer pools, and NIC ring
  caches settle before measurement starts.
- `--runtime=120` minimum. Under a minute, you don't see SSD garbage
  collection, TCP window collapse, or thermal events.
- `--time_based` so changing `runtime` doesn't silently change the work
  done per cell.
- `--output-format=json+` archives histograms, not just summary stats.
  Never throw away raw data.

### 11.8 Watching the target during a run

On the **target** machine, while `fio` is running:

```bash
# 1. Per-core utilization — asymmetry usually means bad pinning
mpstat -P ALL 1

# 2. In-target per-command latency histogram (kernel nvmet)
#    Tracepoints defined in drivers/nvme/target/trace.h.
#    Key is (qid, cid) — cid alone is not unique across queues.
bpftrace -e '
  tracepoint:nvmet:nvmet_req_init {
      @s[args->qid, args->cid] = nsecs;
  }
  tracepoint:nvmet:nvmet_req_complete /@s[args->qid, args->cid]/ {
      $d = nsecs - @s[args->qid, args->cid];
      delete(@s[args->qid, args->cid]);
      @us = hist($d / 1000);
  }'

# 3. NIC counters — any drop / discard / retransmit invalidates the run
ethtool -S eth0 | grep -Ei 'drop|discard|retrans|oo|err'

# 4. Kernel TCP counters (TCP transport)
nstat -az | grep -iE 'retrans|oofo|listenoverflow'

# 5. CPU flame graph (run once when the workload is in steady state)
perf record -F 999 -a -g -- sleep 10
perf script | stackcollapse-perf.pl | flamegraph.pl > target.svg
```

For an `io_uring + AF_XDP` userspace target (the §10 design), substitute:

- `ethtool -S eth0 | grep -i xdp` for XDP drop reasons
- `bpftrace` on `tracepoint:io_uring:io_uring_submit_req` /
  `io_uring_complete` for in-process latency
- `perf stat -e instructions,cycles,cache-misses -p $pid` for per-process
  efficiency

### 11.9 Common pitfalls

- **No `--direct=1`** → measuring page cache, not the device.
- **Fresh SSD without preconditioning** → measuring the empty-state cliff.
- **Sub-minute runs** → no time for GC, TCP CC, or thermals.
- **A single run** → run three, report median + spread (or std-dev).
- **Implicit cross-NUMA** → default IRQ/scheduler placement straddles
  nodes; you'll see ±15 % wobble that isn't your code.
- **Wrong unit comparison** → "kIOPS / core" needs both sides at the
  same frequency, same SMT mode, same CPU SKU. Otherwise it's noise.
- **No baselines** → without bare-backend (C) and `nvme-loop` (A)
  numbers, your transport number has no scale. If your fabric beats
  `nvme-loop`, your measurement is wrong.
- **`--ramp_time=0`** → first seconds are TCP slow-start and buffer
  allocation. They are not your steady state.
- **Hidden interrupt coalescing** → `ethtool -c eth0` shows `rx-usecs`;
  zero it for latency runs, leave default for throughput runs, and
  *state which you used*.
- **Forgetting to pin fio** → an unpinned host moves between cores under
  load and adds 5–20 µs of variance.

### 11.10 What a defensible report contains

If you publish numbers, the writeup should include:

1. **System table** — CPU SKU, RAM, NUMA layout, NIC + firmware, SSD +
   firmware, kernel version, target commit hash.
2. **Topology diagram** — which cores run target, which run `fio`, where
   NIC IRQs land.
3. **Tunables list** — every non-default `sysctl`, `ethtool`, IRQ
   affinity, governor, kernel cmdline.
4. **Workload matrix** with raw JSON archived (commit it to the repo).
5. **Results table** with percentiles *and* run-to-run variance, not
   just one bold number per cell.
6. **Baselines** — bare backend (C) and `nvme-loop` (A) for every cell
   where you cite (B).
7. **Profile evidence** — at least one flamegraph showing where target
   CPU actually went.

If a report is missing items 1–3 and 6, treat the numbers as decorative.

## 12. Mental Model in One Paragraph

`nvmet` is a thin, opcode-driven dispatcher with two pluggable edges. On
the **outer edge**, a transport vtable (`tcp`, `rdma`, `fc`, `loop`,
`pci-epf`) feeds parsed NVMe commands into a common core; on the **inner
edge**, a backend chosen per namespace (`bdev`, `file`, or pass-through)
performs the actual data movement. Every command in flight is a
`nvmet_req` whose `execute()` pointer was set by the parser, and whose
completion always funnels back through `nvmet_req_complete()` →
`ops->queue_response()`. The whole thing is configured at runtime through
configfs, with no daemon, no ioctls, and no kernel patches needed to add
or remove an exported namespace.

That's it — once you internalize the `port → subsys → ns / ctrl → sq →
req → execute` chain and the transport/backend split, the rest of
`drivers/nvme/target/` reads as variations on that one theme.
