# Overview

This post walks through writing a loadable kernel module in Rust on
mainline Linux, anchored on the upstream teaching driver
`samples/rust/rust_misc_device.rs` — a misc character device that
exposes `open`/`read`/`write`/`ioctl` to userspace. The goal is not
"here is the API" but *why the abstractions are shaped the way they
are*: how a Rust `struct` becomes a `module_init`, how a safe
`impl MiscDevice` becomes a C `struct file_operations`, who owns each
object, when it is freed, and which lock protects what. By the end you
should be able to read any in-tree Rust driver and predict its
lifetime and locking from its types.

The reader is assumed to know C kernel programming (modules, char
devices, `copy_to_user`, refcounts, sleeping locks) but to be new to
Rust-for-Linux. Rust concepts are introduced as they map onto kernel
concepts you already know.

# Motivation: what problem Rust-for-Linux solves

The kernel's defect history is dominated by a small set of memory- and
concurrency-safety bugs: use-after-free, double-free, out-of-bounds
access, uninitialized reads, data races, and forgetting to drop a
refcount or unlock on an error path. C gives the compiler no way to
reject these at build time; they surface as runtime oopses, often
discovered by fuzzers or in production.

Rust-for-Linux exists to move that class of bug from runtime to compile
time. The mechanism is the **type system**, not runtime checks:

- **Ownership and borrowing** — every value has exactly one owner; the
  compiler tracks when it is dropped and forbids use after that point.
  This statically eliminates use-after-free and double-free for objects
  the Rust side owns.
- **`Send`/`Sync` marker traits** — the compiler refuses to share a
  value across threads unless its type proves that is sound. This is
  data-race prevention encoded in types.
- **RAII guards** — a lock returns a guard object; the data is only
  reachable *through* the guard; dropping the guard unlocks. You cannot
  access the protected data without holding the lock, and you cannot
  forget to unlock on an error path, because the unlock is the guard's
  destructor.

The cost: kernel objects do not naturally obey Rust's rules. They are
allocated by C, pinned in memory, refcounted by C, and freed by C. The
bulk of the `kernel` crate is a layer of *abstractions* that wrap these
C objects so that safe Rust driver code can use them while all the
genuinely-unsafe FFI is confined to audited `unsafe` blocks inside the
crate. The rest of this post is about how that wrapping works.

# Big-picture architecture

A Rust kernel module is three layers stacked between your driver logic
and the C kernel:

```
         ┌──────────────────────────────────────────────┐
         │ Your driver  (samples/rust/rust_misc_device.rs)│
         │   struct RustMiscDevice                        │
         │   impl MiscDevice for RustMiscDevice           │   100% safe Rust
         │   module! { type: RustMiscDeviceModule, ... }  │
         └───────────────┬────────────────────────────────┘
                         │ uses safe traits & types
         ┌───────────────▼────────────────────────────────┐
         │ kernel crate  (rust/kernel/*.rs)                │
         │   MiscDeviceRegistration / MiscDevice trait     │   safe API on top,
         │   Mutex / ARef / Arc / UserSlice / Module       │   `unsafe` FFI inside
         │   #[vtable], module! macro (rust/macros/)       │
         └───────────────┬────────────────────────────────┘
                         │ bindgen-generated `extern "C"`
         ┌───────────────▼────────────────────────────────┐
         │ bindings crate  (rust/bindings/)                │
         │   raw `bindings::miscdevice`, `misc_register`,  │   1:1 with C headers
         │   `file_operations`, `file`, `inode`, ...       │
         └───────────────┬────────────────────────────────┘
   ══════════════════════╪══════════════════════════════════ C/Rust boundary
                         ▼
         ┌─────────────────────────────────────────────────┐
         │ C kernel: drivers/char/misc.c, fs/, kernel/module │
         │   misc_register(), VFS, module loader             │
         └─────────────────────────────────────────────────┘
```

Two boundaries matter:

1. **bindings ↔ C.** `rust/bindings/` is generated by `bindgen` from
   the C headers. It contains raw, `unsafe`, `repr(C)` mirrors of C
   structs and `extern "C"` declarations of C functions. Nothing in
   here is safe; it is the literal C ABI expressed in Rust syntax.

2. **kernel crate ↔ bindings.** `rust/kernel/` wraps the raw bindings in
   safe types. The trick used throughout is
   [`Opaque<T>`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/types.rs#L323) — a
   `#[repr(transparent)]` wrapper around `UnsafeCell<MaybeUninit<T>>`
   that holds a C struct whose internals Rust never touches directly,
   only via C functions. Every C struct that the Rust side must hold but
   not introspect (`miscdevice`, `mutex`, `file`) lives inside an
   `Opaque`.

Your driver only ever touches layer 1. It never writes `unsafe`. That
is the entire point: the `unsafe` is *concentrated and reviewed once* in
the crate, then reused by every driver.

# Core kernel concepts involved

Before the Rust abstractions make sense, four C-side concepts must be
crisp, because the Rust types exist precisely to encode their rules.

## Misc devices

A misc device is the lightweight way to expose a single char device
node without allocating a major number. Owned by the **char device /
driver-base** subsystem; implemented in
[`drivers/char/misc.c`](https://elixir.bootlin.com/linux/v7.0/source/drivers/char/misc.c).
You fill a `struct miscdevice`
([`include/linux/miscdevice.h:84`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/miscdevice.h#L84))
with a name, a minor number (usually `MISC_DYNAMIC_MINOR`, value `255`,
meaning "allocate one for me"), and a `struct file_operations`, then
call
[`misc_register()`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/miscdevice.h#L96).
This creates `/dev/<name>`. `misc_deregister()` tears it down. The
single most important lifetime rule: **the `struct miscdevice` you pass
to `misc_register()` must stay alive and at a fixed address until
`misc_deregister()` returns**, because the kernel stores the pointer,
not a copy.

## `struct file_operations`

[`struct file_operations`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/fs.h#L1926)
is the vtable of a char device: a struct of function pointers
(`open`, `release`, `read_iter`, `write_iter`, `unlocked_ioctl`,
`mmap`, ...). When userspace calls `open("/dev/foo")`, the VFS routes
through to `fops->open`. The contract is all in the heads of kernel
developers: which lock is held, what `private_data` points to, what the
return value means. Rust's job is to turn this informal contract into
types.

## `private_data`

[`struct file`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/fs.h#L1259)
has a `void *private_data` field. By convention a driver
stores its per-open instance pointer there in `open`, retrieves it in
every other callback, and frees it in `release`. This is hand-rolled
ownership: the C compiler does nothing to ensure you free it exactly
once. This convention is exactly what Rust's ownership model formalizes.

## Sleeping locks and refcounts

[`struct mutex`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/mutex_types.h#L41)
is a sleeping lock — you may block while holding it, but
not take it in atomic context.
[`struct kref`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/kref.h#L19)
/ [`refcount_t`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/refcount_types.h#L15)
is the
kernel's reference count: `*_get()` increments, `*_put()` decrements and
frees at zero. Both have informal rules ("unlock on every path", "every
get needs a put") that account for a large fraction of kernel bugs.
Rust encodes the mutex rule in `Mutex<T>`/`Guard` and the refcount rule
in `ARef<T>`/`Arc<T>`.

# Rust abstractions

Now the layer-2 types. For each: *why it exists*, *which C object it
wraps*, *lifetime*, *locking/refcount rules*, and *how it interacts with
Rust ownership*.

## `module!` and the `Module` / `InPlaceModule` traits

`module!`
([`rust/macros/module.rs`](https://elixir.bootlin.com/linux/v7.0/source/rust/macros/module.rs))
is a procedural macro that generates everything `MODULE_*` and
`module_init`/`module_exit` do in C: the `.modinfo` strings (license,
author, description), the module struct, and the C-ABI entry points.

Your driver implements one of two traits from
[`rust/kernel/lib.rs`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/lib.rs):

- [`Module`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/lib.rs#L178)
  — `fn init(module: &'static ThisModule) -> Result<Self>`. The simple
  case: build your module struct by value and return it.
- [`InPlaceModule`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/lib.rs#L189)
  — `fn init(...) -> impl PinInit<Self, Error>`. Used when the module
  struct must be **pinned** — initialized directly in its final memory
  location and never moved. The misc-device sample uses this because it
  embeds a `MiscDeviceRegistration`, which contains a `struct
  miscdevice` whose address the kernel records.

There is a blanket impl
([`lib.rs:196`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/lib.rs#L196)):
every `Module` is automatically an `InPlaceModule`. So `InPlaceModule`
is the lower-level primitive; `Module` is the convenient case.

Why `Sized + Sync + Send` bounds on `Module`? A module's state is a
single static instance reachable from any CPU running any of its
functions — so it must be `Sync` (shareable by reference across
threads). It is initialized on one CPU and dropped on another, so it
must be `Send`.

[`ThisModule`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/lib.rs#L220)
is the Rust equivalent of C's
[`THIS_MODULE`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/init.h#L174):
a wrapper around
`*mut bindings::module`. It is `unsafe impl Sync` because the raw
pointer is shared but only ever read.

## `MiscDeviceRegistration<T>` — registration as an owned object

[`MiscDeviceRegistration<T>`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/miscdevice.rs#L56)
is the keystone abstraction. It encodes the misc-device lifetime rule as
a Rust type:

```rust
#[repr(transparent)]
#[pin_data(PinnedDrop)]
pub struct MiscDeviceRegistration<T> {
    #[pin]
    inner: Opaque<bindings::miscdevice>,
    _t: PhantomData<T>,
}
```

Line by line:

- `#[repr(transparent)]` — the Rust struct has the exact memory layout
  of its one real field, the `Opaque<miscdevice>`. No wrapper overhead,
  and a `*mut MiscDeviceRegistration<T>` is interchangeable with a
  `*mut miscdevice` at the ABI level.
- `Opaque<bindings::miscdevice>` — holds the raw C `struct miscdevice`.
  Rust never reads its fields; it only hands its address to
  `misc_register`/`misc_deregister`. `Opaque` is `UnsafeCell<
  MaybeUninit<T>>` plus `PhantomPinned`, giving interior mutability (C
  mutates it behind Rust's back) and a "never move me" marker.
- `#[pin]` on `inner` + `#[pin_data]` on the struct — declares that this
  field is address-sensitive. The pin-init machinery guarantees the
  struct is constructed in place and never relocated, satisfying "the
  `miscdevice` must stay at a fixed address."
- `PhantomData<T>` — zero-size; carries the `T: MiscDevice` type
  parameter so the right vtable is selected, without storing anything.

The crucial design move is in
[`register()`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/miscdevice.rs#L71)
and the matching
[`PinnedDrop`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/miscdevice.rs#L104):

```rust
#[pinned_drop]
impl<T> PinnedDrop for MiscDeviceRegistration<T> {
    fn drop(self: Pin<&mut Self>) {
        unsafe { bindings::misc_deregister(self.inner.get()) };
    }
}
```

`register` calls `misc_register`; `drop` calls `misc_deregister`. So the
*existence* of a `MiscDeviceRegistration` value is exactly the window in
which `/dev/<name>` exists. Because the module struct owns this value as
a field, dropping the module (on `rmmod`) drops the registration, which
deregisters the device. You cannot leak the device or deregister it
twice — those are now type errors, not review burdens. This is the
RAII-equals-lifetime pattern that recurs through the whole crate.

`unsafe impl Send`/`Sync` on the registration
([`miscdevice.rs:64-67`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/miscdevice.rs#L64))
encode two real kernel facts: `misc_deregister` may run on a different
thread than `misc_register` (so `Send`), and the `&self` methods are
written to be callable concurrently (so `Sync`). These `unsafe impl`s
are the human asserting an invariant the compiler cannot derive — the
exact spot a reviewer must check.

## The `MiscDevice` trait and `#[vtable]`

[`MiscDevice`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/miscdevice.rs#L114)
is the safe replacement for `struct file_operations`:

```rust
#[vtable]
pub trait MiscDevice: Sized {
    type Ptr: ForeignOwnable + Send + Sync;
    fn open(file: &File, misc: &MiscDeviceRegistration<Self>) -> Result<Self::Ptr>;
    fn release(device: Self::Ptr, file: &File) { drop(device); }
    fn ioctl(device: ..., file: &File, cmd: u32, arg: usize) -> Result<isize> { ... }
    fn read_iter(...) -> Result<usize> { ... }
    fn write_iter(...) -> Result<usize> { ... }
    // mmap, compat_ioctl, show_fdinfo ...
}
```

Three things to understand:

- **`type Ptr: ForeignOwnable`** is the typed replacement for
  `void *private_data`.
  [`ForeignOwnable`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/types.rs#L27)
  is the trait for "a pointer
  that can be converted to/from a raw `*mut c_void` and back without
  losing ownership." In the sample, `Ptr = Pin<KBox<Self>>` — a pinned,
  heap-allocated, owned box. `open` returns one; the crate's glue stores
  its raw form in `private_data`; `release` reconstructs and drops it.
  The whole `private_data` ownership convention is now expressed in the
  type signature.

- **`#[vtable]`** (a proc-macro) generates, for each method, an
  associated `const HAS_<METHOD>: bool` that is `true` only if the trait
  impl overrides that method. This lets the crate populate the C
  `file_operations` with `Some(fn)` only for methods you actually
  implemented, and `None` otherwise — matching C semantics where a null
  fop means "not supported." The default method bodies call
  `build_error!(VTABLE_DEFAULT_ERROR)`, which is a *compile-time* error
  if ever actually invoked, proving the `None` paths are unreachable.

- **`open` takes `&File` and `&MiscDeviceRegistration<Self>`, not raw
  pointers.** The unsafe pointer-to-reference conversion happens once,
  in the crate's glue, with a documented `# Safety` justification.

## The vtable glue: where safe meets unsafe

[`MiscdeviceVTable<T>`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/miscdevice.rs#L196)
is the bridge. It is a zero-size type carrying `T`, with one
`unsafe extern "C" fn` per fop and a `const VTABLE: file_operations`
that wires them up
([`miscdevice.rs:390`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/miscdevice.rs#L390)):

```rust
const VTABLE: bindings::file_operations = bindings::file_operations {
    open:    Some(Self::open),
    release: Some(Self::release),
    unlocked_ioctl: if T::HAS_IOCTL { Some(Self::ioctl) } else { None },
    read_iter:  if T::HAS_READ_ITER  { Some(Self::read_iter)  } else { None },
    write_iter: if T::HAS_WRITE_ITER { Some(Self::write_iter) } else { None },
    ..pin_init::zeroed()
};
```

Look at the C-ABI `open` shim
([`miscdevice.rs:203`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/miscdevice.rs#L203)):

```rust
unsafe extern "C" fn open(inode: *mut bindings::inode,
                          raw_file: *mut bindings::file) -> c_int {
    let ret = unsafe { bindings::generic_file_open(inode, raw_file) };
    if ret != 0 { return ret; }
    let misc_ptr = unsafe { (*raw_file).private_data };
    let misc = unsafe { &*misc_ptr.cast::<MiscDeviceRegistration<T>>() };
    let file = unsafe { File::from_raw_file(raw_file) };
    let ptr = match T::open(file, misc) {
        Ok(ptr) => ptr,
        Err(err) => return err.to_errno(),
    };
    unsafe { (*raw_file).private_data = ptr.into_foreign() };
    0
}
```

This is the *only* place the `void *` dance happens for `open`. Note the
two-phase use of `private_data`: when `misc_open()` in C calls this, the
field still points at the `miscdevice` (so we read the registration out
of it); then we *overwrite* it with `ptr.into_foreign()` — the raw form
of your `Pin<KBox<Self>>`. From this point every other fop reads
`private_data` as your instance. `release`
([`miscdevice.rs:244`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/miscdevice.rs#L244))
calls `from_foreign` to turn it back into an owned `Pin<KBox<Self>>` and
drops it. `into_foreign`/`from_foreign` are the typed encoding of
"transfer ownership across the C boundary and back."

`ioctl`
([`miscdevice.rs:328`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/miscdevice.rs#L328))
uses `borrow` (not `from_foreign`): an ioctl borrows your instance for
the duration of the call but does not consume it, mirroring that the
file stays open afterward.

## `Mutex<T>` — data and its lock fused

C separates the lock from the data it protects; the connection lives
only in comments. Rust fuses them. `Mutex<T>`
([`rust/kernel/sync/lock/mutex.rs:87`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/sync/lock/mutex.rs#L87))
is a type alias for the generic
[`Lock<T, MutexBackend>`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/sync/lock.rs#L106):

```rust
pub struct Lock<T: ?Sized, B: Backend> {
    state: Opaque<B::State>,   // the C `struct mutex`
    data:  UnsafeCell<T>,      // the protected data, owned inline
    ...
}
```

The protected data lives *inside* the lock. The only way to reach it is
[`lock()`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/sync/lock.rs#L174),
which acquires the C mutex and returns a
[`Guard<'_, T, B>`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/sync/lock.rs#L201).
The guard derefs to `&mut T`; dropping the guard
([`lock.rs:301`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/sync/lock.rs#L301))
releases the mutex. Consequences, all compiler-enforced:

- You cannot touch the data without holding the lock — there is no path
  to `&mut T` except through `lock()`.
- You cannot forget to unlock, including on early `return`/`?` error
  paths — `Drop` runs at end of scope regardless. This kills the classic
  "unlock missing on the error branch" bug.
- The guard's lifetime `'_` is tied to the `&Lock`, so a `&mut T`
  obtained from it cannot outlive the locked section.

`MutexBackend`
([`mutex.rs:98`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/sync/lock/mutex.rs#L98))
implements `Backend` by calling the C
[`mutex_lock`](https://elixir.bootlin.com/linux/v7.0/source/kernel/locking/mutex.c#L285)/`mutex_unlock`.
Note
the implication of "sleeping lock": because `lock()` may sleep, you must
not call it in atomic context — a rule Rust does *not* yet check, and a
live pitfall (see Debugging).

`new_mutex!(value)` is the constructor macro; it must be used inside a
pin-init because the C `struct mutex` is address-sensitive (it has a
self-referential wait list), hence the `#[pin]` on the field in the
sample's `Inner`.

## `ARef<T>` and `Arc<T>` — two flavors of reference counting

These are easy to confuse; the distinction is *where the refcount
lives*.

- [`ARef<T>`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/sync/aref.rs#L68)
  ("Always-Refcounted Ref") wraps a pointer to an object that *embeds
  its own kernel refcount* — i.e. `T: AlwaysRefCounted`
  ([`aref.rs:40`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/sync/aref.rs#L40)),
  whose `inc_ref`/`dec_ref` call the C `get`/`put` (e.g.
  [`get_device`](https://elixir.bootlin.com/linux/v7.0/source/drivers/base/core.c#L3784)/[`put_device`](https://elixir.bootlin.com/linux/v7.0/source/drivers/base/core.c#L3794)).
  `ARef<Device>` is exactly a held
  reference on a
  [`struct device`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/device.h#L565):
  cloning calls `get_device`, dropping
  calls `put_device`. This is the typed version of "balance every get
  with a put." It `Deref`s to `&T`
  ([`aref.rs:149`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/sync/aref.rs#L149)).
  The sample holds `dev: ARef<Device>` so the device stays pinned for as
  long as the open instance lives.

- [`Arc<T>`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/sync/arc.rs#L132)
  is the kernel crate's own refcounted heap allocation: it boxes your
  `T` together with a `Refcount` in an
  [`ArcInner<T>`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/sync/arc.rs#L147).
  Use it to share *your own* data structure between multiple owners
  (multiple opens, a workqueue item, a timer). `ARef` is for
  C-allocated objects with a built-in refcount; `Arc` is for
  Rust-allocated data you want shared. The misc sample uses `ARef`
  (for the device) and a single `Pin<KBox>` (for the per-open instance,
  which is not shared), so it does not need `Arc` — but a driver that
  hands its state to a workqueue would.

## `UserSlice` — typed `copy_to/from_user`

[`UserSlice`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/uaccess.rs#L147)
wraps a userspace pointer + length. C lets you `memcpy` from a `void *`
that happens to be a user address — the source of countless missing
`copy_from_user` / direct-deref CVEs. `UserSlice` makes the user/kernel
boundary a type:

- [`UserSlice::new(ptr, len)`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/uaccess.rs#L165)
  then `.reader()` / `.writer()` yields a
  [`UserSliceReader`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/uaccess.rs#L212)
  or `UserSliceWriter`. You cannot dereference the user pointer
  directly; you must go through `reader.read::<T>()`
  ([`uaccess.rs:337`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/uaccess.rs#L337))
  or `writer.write::<T>(&value)`
  ([`uaccess.rs:531`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/uaccess.rs#L531)),
  which call the checked `copy_from_user`/[`copy_to_user`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/uaccess.h#L228)
  under the hood.
- `read::<T>` requires `T:`&nbsp;[`FromBytes`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/transmute.rs#L38)
  and `write::<T>` requires
  `T:`&nbsp;[`AsBytes`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/transmute.rs#L202) — marker traits asserting the type has no padding/invalid
  bit patterns, so copying raw bytes is sound.

## Imports and ioctl numbers

```rust
const RUST_MISC_DEV_HELLO:     u32 = _IO('|' as u32, 0x80);
const RUST_MISC_DEV_GET_VALUE: u32 = _IOR::<i32>('|' as u32, 0x81);
const RUST_MISC_DEV_SET_VALUE: u32 = _IOW::<i32>('|' as u32, 0x82);
```

[`_IO`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/ioctl.rs#L27)/[`_IOR`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/ioctl.rs#L33)/[`_IOW`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/ioctl.rs#L39)
are the Rust equivalents of the C macros of the same
name: they pack direction, type, size, and command number into the
32-bit ioctl code. `_IOR::<i32>` encodes that this ioctl reads an `i32`
out to userspace; the generic parameter supplies the size, so it cannot
drift from the actual transferred type.

## Module declaration and struct

```rust
module! {
    type: RustMiscDeviceModule,
    name: "rust_misc_device",
    authors: ["Lee Jones"],
    description: "Rust misc device sample",
    license: "GPL",
}

#[pin_data]
struct RustMiscDeviceModule {
    #[pin]
    _miscdev: MiscDeviceRegistration<RustMiscDevice>,
}
```

`module!` names `RustMiscDeviceModule` as the module type and generates
the C entry points (next section). The module struct holds a single
field: the registration. `#[pin_data]`/`#[pin]` because the registration
embeds the address-sensitive `miscdevice`. The leading underscore on
`_miscdev` says "I never read this field; I hold it purely for its
lifetime" — its existence *is* the registered device, and its drop
deregisters.

## `init`: registering the device

```rust
impl kernel::InPlaceModule for RustMiscDeviceModule {
    fn init(_module: &'static ThisModule) -> impl PinInit<Self, Error> {
        pr_info!("Initialising Rust Misc Device Sample\n");
        let options = MiscDeviceOptions { name: c"rust-misc-device" };
        try_pin_init!(Self {
            _miscdev <- MiscDeviceRegistration::register(options),
        })
    }
}
```

`init` returns a `PinInit` *recipe*, not a value — the actual
construction happens in place at the module's final address (the macro
machinery writes into `__MOD`, below). `try_pin_init!` is the
fallible, pinned struct initializer; the `<-` operator means "initialize
this pinned field in place using this `PinInit`," as opposed to `:` for
a plain value. `register(options)` is itself a `PinInit` that runs
`misc_register` during construction. If it fails, `try_pin_init!`
propagates the error and nothing is left half-built. `c"rust-misc-device"`
is a C string literal (NUL-terminated) — the name of the `/dev` node.

## Per-open state

```rust
struct Inner {
    value:  i32,
    buffer: KVVec<u8>,
}

#[pin_data(PinnedDrop)]
struct RustMiscDevice {
    #[pin]
    inner: Mutex<Inner>,
    dev:   ARef<Device>,
}
```

`RustMiscDevice` is the per-open instance (one per `open()`). Its mutable
state — an integer and a byte buffer — lives in `Inner`, wrapped in a
`Mutex` so concurrent ioctls/reads/writes on the same fd are serialized.
`dev: ARef<Device>` holds a counted reference to the backing `struct
device`, keeping it alive for logging (`dev_info!`) throughout the open.
`#[pin]` on `inner` because the mutex is address-sensitive.

## `open`: building the instance

```rust
fn open(_file: &File, misc: &MiscDeviceRegistration<Self>) -> Result<Pin<KBox<Self>>> {
    let dev = ARef::from(misc.device());
    dev_info!(dev, "Opening Rust Misc Device Sample\n");
    KBox::try_pin_init(
        try_pin_init! {
            RustMiscDevice {
                inner <- new_mutex!(Inner { value: 0_i32, buffer: KVVec::new() }),
                dev: dev,
            }
        },
        GFP_KERNEL,
    )
}
```

`misc.device()`
([`miscdevice.rs:94`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/miscdevice.rs#L94))
returns `&Device`; `ARef::from` bumps its refcount (`get_device`) and
gives us an owned `ARef<Device>`. `KBox::try_pin_init` allocates a
pinned heap box with `GFP_KERNEL` and constructs the instance in place;
`new_mutex!` initializes the embedded mutex. The returned
`Pin<KBox<Self>>` is `Self::Ptr` — the crate stores its raw form in
`file->private_data`. The pin matters: the mutex inside must never move
after construction.

## `ioctl` dispatch

```rust
fn ioctl(me: Pin<&RustMiscDevice>, _file: &File, cmd: u32, arg: usize) -> Result<isize> {
    let arg  = UserPtr::from_addr(arg);
    let size = _IOC_SIZE(cmd);
    match cmd {
        RUST_MISC_DEV_GET_VALUE => me.get_value(UserSlice::new(arg, size).writer())?,
        RUST_MISC_DEV_SET_VALUE => me.set_value(UserSlice::new(arg, size).reader())?,
        RUST_MISC_DEV_HELLO     => me.hello()?,
        _ => { dev_err!(me.dev, "-> IOCTL not recognised: {}\n", cmd); return Err(ENOTTY); }
    };
    Ok(0)
}
```

`me: Pin<&RustMiscDevice>` is your borrowed instance (the crate's glue
called `borrow` on the `private_data`). The unknown-cmd arm returns
`Err(ENOTTY)` — the conventional "inappropriate ioctl" errno; the crate
turns the `Err` into the negative errno the C ABI expects. `GET_VALUE`
builds a *writer* (kernel → user), `SET_VALUE` a *reader* (user →
kernel) — direction encoded in the type, matching `_IOR`/`_IOW`.

## `set_value` / `get_value`: locking + user copy together

```rust
fn set_value(&self, mut reader: UserSliceReader) -> Result<isize> {
    let new_value = reader.read::<i32>()?;       // copy_from_user, checked
    let mut guard = self.inner.lock();           // acquire mutex
    guard.value = new_value;                     // mutate under lock
    Ok(0)
}                                                // guard drops -> unlock

fn get_value(&self, mut writer: UserSliceWriter) -> Result<isize> {
    let guard = self.inner.lock();
    let value = guard.value;
    drop(guard);                                 // release lock early
    writer.write::<i32>(&value)?;                // copy_to_user after unlock
    Ok(0)
}
```

Two subtle, deliberate things here:

- In `set_value`, `reader.read::<i32>()` happens *before* `self.inner.lock()`.
  The user copy can fault and sleep; doing it outside the lock minimizes
  hold time. Then the write to `guard.value` is the only work under the
  lock.
- In `get_value`, `drop(guard)` is *explicit* and deliberate: it
  releases the mutex before `writer.write` does the (potentially
  sleeping, faulting) `copy_to_user`. Copying a snapshot out from under
  the lock is the right pattern — never hold a lock across a user copy
  if you can avoid it. Rust makes the unlock point visible as `drop`
  rather than an invisible end-of-scope.

## Teardown

```rust
#[pinned_drop]
impl PinnedDrop for RustMiscDevice {
    fn drop(self: Pin<&mut Self>) {
        dev_info!(self.dev, "Exiting the Rust Misc Device Sample\n");
    }
}
```

When the file is closed, the crate's `release` shim reconstructs the
owned `Pin<KBox<Self>>` from `private_data` and drops it; this runs
`PinnedDrop::drop`, then frees the box, then drops `dev` (which calls
`put_device`). You wrote no `kfree`, no `put_device`, no `private_data =
NULL`; ownership did it. The `read_iter`/`write_iter` methods follow the
same pattern with `IovIterDest`/`IovIterSource` for the buffer field.

# Internal kernel execution path

Two flows tie the Rust back to the C kernel. These are real call chains;
the Rust entry points are generated, so they are stable.

## Module load

```
userspace: insmod rust_misc_device.ko
  -> syscall finit_module()
     -> load_module()                      kernel/module/main.c
        -> do_init_module(mod)
           -> mod->init()                  == generated init_module()
              ::kernel::ffi::c_int init_module()      rust/macros/module.rs:554
                -> __init()                            module.rs:622
                   -> <LocalModule as InPlaceModule>::init(&THIS_MODULE)
                      -> RustMiscDeviceModule::init()   your code
                         -> MiscDeviceRegistration::register()
                            -> bindings::misc_register()   drivers/char/misc.c
                   -> initer.__pinned_init(__MOD.as_mut_ptr())  // construct in place
```

The generated
[`init_module`](https://elixir.bootlin.com/linux/v7.0/source/rust/macros/module.rs#L554)
is the C-ABI symbol the loader calls. It delegates to
[`__init`](https://elixir.bootlin.com/linux/v7.0/source/rust/macros/module.rs#L622),
which obtains the `PinInit` recipe from your `init` and runs
`__pinned_init` to construct the module *directly into the static
`__MOD`* (a `MaybeUninit<LocalModule>`). On error it returns the negated
errno and the module fails to load. (For built-in modules the symbol is
placed in an initcall section instead; same `__init` underneath —
[`module.rs:600`](https://elixir.bootlin.com/linux/v7.0/source/rust/macros/module.rs#L600).)

Unload mirrors this:
[`cleanup_module`](https://elixir.bootlin.com/linux/v7.0/source/rust/macros/module.rs#L570)
→
[`__exit`](https://elixir.bootlin.com/linux/v7.0/source/rust/macros/module.rs#L640)
→ `__MOD.assume_init_drop()` — which drops your module struct, dropping
its `MiscDeviceRegistration` field, which calls `misc_deregister`.

## `open()` and `ioctl()` from userspace

```
userspace: fd = open("/dev/rust-misc-device", O_RDWR)
  -> VFS chrdev open
     -> misc_open()                         drivers/char/misc.c
        (sets file->private_data = &miscdevice; finds fops)
        -> fops->open == MiscdeviceVTable::<T>::open    miscdevice.rs:203
           -> generic_file_open()
           -> read MiscDeviceRegistration out of private_data
           -> T::open(file, misc)            your open(): build Pin<KBox<Self>>
           -> file->private_data = ptr.into_foreign()   // overwrite with instance

userspace: ioctl(fd, RUST_MISC_DEV_SET_VALUE, &v)
  -> VFS -> fops->unlocked_ioctl == MiscdeviceVTable::<T>::ioctl  miscdevice.rs:328
     -> borrow instance from private_data
     -> T::ioctl(me, file, cmd, arg)         your dispatch
        -> set_value(): UserSliceReader::read (copy_from_user) + Mutex::lock
```

The single most important detail:
[`misc_open()`](https://elixir.bootlin.com/linux/v7.0/source/drivers/char/misc.c#L118)
initially stores the
`miscdevice` pointer in `private_data`; the Rust `open` shim reads the
registration from it, then **overwrites** `private_data` with the
foreign form of your instance. Every later fop reads the instance. This
two-phase use is documented inline in the shim and is the linchpin of
the whole typed-`private_data` scheme.

# Advanced topics

## Contrast: platform driver registration

The misc device is a *static* registration — `init` registers, `drop`
deregisters, no hardware binding. A bus driver instead registers a
*driver* and is called back on `probe`/`remove` per matching device.
Compare
[`samples/rust/rust_driver_platform.rs`](https://elixir.bootlin.com/linux/v7.0/source/samples/rust/rust_driver_platform.rs):

```rust
impl platform::Driver for SampleDriver {
    type IdInfo = Info;
    const OF_ID_TABLE:   Option<of::IdTable<Self::IdInfo>>   = Some(&OF_TABLE);
    const ACPI_ID_TABLE: Option<acpi::IdTable<Self::IdInfo>> = Some(&ACPI_TABLE);
    fn probe(pdev: &platform::Device<Core>, info: Option<&Self::IdInfo>)
        -> impl PinInit<Self, Error> { ... Ok(Self { pdev: pdev.into() }) }
}

kernel::module_platform_driver! {
    type: SampleDriver, name: "rust_driver_platform", ...
}
```

Differences worth internalizing:

- **Registration model.**
  [`module_platform_driver!`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/platform.rs#L153)
  registers a
  [`struct platform_driver`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/platform_device.h#L234)
  whose `probe`/`remove` are bound by the bus.
  The module's lifetime registers the *driver*; instances come and go as
  devices are matched, not at module init.
- **Match tables.** `of_device_table!` / `acpi_device_table!` generate
  the `MODULE_DEVICE_TABLE` data that lets the kernel autoload the module
  for matching device-tree/ACPI nodes. The misc device has no such
  matching — it just appears at load.
- **`probe` returns `PinInit<Self>`** like `init`, and the driver struct
  is dropped on `remove` (see its `impl Drop`), so probe/remove balance
  the same RAII way init/exit do.
- **`pdev.into()`** produces an `ARef<platform::Device>` — the same
  always-refcounted pattern, here keeping the platform device pinned for
  the driver instance's lifetime.

## Why pinning is pervasive

Address-sensitive C objects —
[`struct mutex`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/mutex_types.h#L41)
(self-referential wait list),
[`struct miscdevice`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/miscdevice.h#L84)
(kernel stores the pointer),
[`struct list_head`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/types.h#L204)
(nodes point at each other) — must never move after they are
"live." Rust's default is that values *can* move (a move is a
byte-copy). `Pin<P>` is the type-level promise "this will not move," and
`#[pin_data]` + `<-` in `try_pin_init!` are how you construct such a
struct in place rather than building it on the stack and moving it. Any
field wrapping such a C object carries `#[pin]`. If you forget, the
compiler rejects the `<-` initialization — pinning bugs become build
errors.

## Error handling and the `?` operator

Kernel C returns negative errnos; Rust returns `Result<T, Error>`.
[`Error`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/error.rs#L101)
converts to/from errnos
([`to_errno`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/error.rs#L161)),
and the crate's C shims do
that conversion at the boundary (see every `Err(err) => err.to_errno()`
above). Inside your driver you use `?` to propagate — and because of
RAII, an early `?`-return still drops every guard and owned object on
the way out. There is no `goto out_unlock;` ladder, and no way to skip
it.

## Build prerequisites


is the first thing to run when `CONFIG_RUST` won't turn on; it reports

[`Documentation/rust/quick-start.rst`](https://elixir.bootlin.com/linux/v7.0/source/Documentation/rust/quick-start.rst).

## Reading panics and `BUG`

A Rust `panic!` in the kernel routes through the crate's
[`#[panic_handler]`](https://elixir.bootlin.com/linux/v7.0/source/rust/kernel/lib.rs#L245),
which `pr_emerg!`s the message and calls
[`BUG()`](https://elixir.bootlin.com/linux/v7.0/source/include/asm-generic/bug.h#L73).
So a Rust panic looks
like a C `BUG` in dmesg, with the panic message on the line above.
Common triggers: `unwrap()` on an `Err`/`None`, slice index out of
bounds, integer overflow in debug builds.

## Sleeping-in-atomic — a pitfall Rust does *not* catch

`Mutex::lock` and `UserSlice` copies may sleep. The compiler does **not**
track whether you are in atomic context (holding a spinlock, in an IRQ).
If you call `lock()` on a sleeping mutex from atomic context, you get the
same `scheduling while atomic` splat as in C. The type system covers
memory and data-race safety, not context safety. Build with
`CONFIG_DEBUG_ATOMIC_SLEEP=y` to catch it at runtime, exactly as for C.

## Tracing the boundary with bpftrace

Because the Rust shims are thin `extern "C"` functions and the C side is
unchanged, ordinary kernel tracing works. To watch misc-device opens
live:

```sh
bpftrace -e 'kprobe:misc_open { printf("%s (pid %d) opened a misc dev\n", comm, pid); }'
```

Real capture (Linux 7.0.0, x86-64, in a `virtme-ng` VM), with the
module loaded via `insmod rust_misc_device.ko` and the workload being
the C test program embedded at the top of the sample file, built as
`rust_misc_test`:

```
Attaching 1 probe...
rust_misc_test (pid 329) opened a misc dev
```

The whole walkthrough is observable end-to-end in `dmesg` from the same
run — note the lifecycle methods firing in order (`open`, `ioctl`
dispatch, the user copies, the unknown-ioctl `ENOTTY`, and
`PinnedDrop::drop` on close):

```
rust_misc_device: Initialising Rust Misc Device Sample   # init -> register
misc rust-misc-device: Opening Rust Misc Device Sample    # T::open
misc rust-misc-device: -> Hello from the Rust Misc Device # HELLO ioctl
misc rust-misc-device: -> Copying data to userspace (value: 0)   # GET_VALUE
misc rust-misc-device: -> Copying data from userspace (value: 1) # SET_VALUE
misc rust-misc-device: -> IOCTL not recognised: 31744     # FAIL -> ENOTTY
misc rust-misc-device: Exiting the Rust Misc Device Sample # drop on close
```

`misc_register`/`misc_deregister`
are equally probeable to confirm load/unload balance. Rust symbols are
mangled but visible in `/proc/kallsyms`; prefer probing the stable C
boundary functions over mangled Rust names.

## `rmmod` hangs or device won't disappear

If `/dev/<name>` lingers after `rmmod`, the `MiscDeviceRegistration`
was leaked — almost always because the module struct that owns it was
not actually dropped (e.g. it was stored somewhere with a static
lifetime, or an `Arc` cycle keeps it alive). Since `drop` is what calls
`misc_deregister`, a missing deregister is a missing drop. Trace
`cleanup_module`/`misc_deregister` to confirm teardown ran.

# Summary

**Main advantages**

- **Memory- and data-race-safety by construction.** Use-after-free,
  double-free, missing-unlock, and unbalanced refcounts become compile
  errors or are eliminated by RAII drops, because the *why* — ownership
  and the lock-data fusion — is encoded in types rather than comments.
- **Lifetime equals object existence.** `MiscDeviceRegistration`,
  `Mutex` guards, `ARef`, and `Pin<KBox>` make "registered", "locked",
  "referenced", and "allocated" coincide with the lifetime of a value,
  so teardown is automatic and ordered correctly without `goto` ladders.
- **`unsafe` is concentrated.** All genuinely unsafe FFI lives in the
  `kernel` crate's reviewed shims; driver authors write zero `unsafe`,
  yet still call into the unchanged C kernel with normal tracing/debug
  tooling working as before.

**Main problems / limitations**

- **Context safety is not modeled.** The type system does not know
  atomic vs. process context, so sleeping-in-atomic (e.g. `Mutex::lock`
  under a spinlock or in IRQ) is still a runtime bug — rely on
  `CONFIG_DEBUG_ATOMIC_SLEEP` as in C.
- **Pinning and pin-init have a real learning curve.** `#[pin_data]`,
  `<-` vs `:`, `try_pin_init!`, and `Pin<KBox>` are unfamiliar and
  necessary precisely because so many C objects are address-sensitive;
  getting them wrong yields dense compiler errors.
- **Abstraction coverage is incomplete and moving.** Only a subset of
  subsystems have safe wrappers; many drivers still need new
  abstractions written (and reviewed) before they can be expressed
  safely, and APIs like `miscdevice.rs` are recent (2024) and still
  evolving across releases.
