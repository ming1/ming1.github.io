# Overview

`io_uring/net.c` is the bridge between the io_uring core and the kernel's
BSD-socket layer. It turns a submitted SQE into a `struct io_kiocb`, drives
that request through the socket send/recv machinery (`sock_sendmsg()` /
`sock_recvmsg()` and friends), and reports the result as a CQE — without the
caller ever entering `sendmsg(2)`/`recvmsg(2)`. This post follows one request
end to end: how a net handler in `net.c` is *reached* from the io_uring core,
how it *calls into* the networking stack, and how the `io_kiocb` is born,
parked, retried, and freed. The focus is **send and recv and all of their
variants** — plain, `msg`, vectorized, bundled, multishot, and zero-copy.

> Sources: Linux **v7.0** (`git tag v7.0`). Primary files:
> [`io_uring/net.c`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c),

# The two boundaries net.c sits between

`net.c` is glue. It has an **upper boundary** to the io_uring core (which
calls it through a per-opcode `prep`/`issue` vtable) and a **lower boundary**
to the socket layer (which it calls through the ordinary
[`struct socket`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/net.h#L116)
send/recv functions). Everything in this post is about what happens between
those two lines.

```
            io_uring SQE ring  (userspace)
   ─────────────────────────────────────────────  syscall: io_uring_enter(2)
            io_uring core   (io_uring.c)
              io_submit_sqe → io_init_req → def->prep   ← prep half
              io_queue_sqe  → io_issue_sqe → def->issue  ← issue half
   ─────────────────────────────────────────────  per-opcode vtable (opdef.c)
            net.c handlers
              io_send / io_sendmsg / io_send_zc / ...
              io_recv / io_recvmsg / io_recvzc
   ─────────────────────────────────────────────  struct socket ops
            socket layer    (net/socket.c)
              sock_sendmsg / sock_recvmsg / __sys_*msg_sock
                          │
                          ▼
            protocol (TCP/UDP) → device
```

The vtable is [`io_issue_defs[]`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/opdef.c#L54)
in `opdef.c`. Each opcode names a `.prep` and an `.issue` function; the
send/recv family maps like this:

| Opcode | What it does | `.prep` | `.issue` |
|---|---|---|---|
| [`IORING_OP_SEND`](https://elixir.bootlin.com/linux/v7.0/A/ident/IORING_OP_SEND) | Send from one linear buffer (also drives provided-buffer / bundle sends). | [`io_sendmsg_prep`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L418) | [`io_send`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L642) |
| [`IORING_OP_SENDMSG`](https://elixir.bootlin.com/linux/v7.0/A/ident/IORING_OP_SENDMSG) | Send a full `msghdr`: iovec array, ancillary control data, destination address. | [`io_sendmsg_prep`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L418) | [`io_sendmsg`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L545) |
| [`IORING_OP_SEND_ZC`](https://elixir.bootlin.com/linux/v7.0/A/ident/IORING_OP_SEND_ZC) | Zero-copy send of a linear buffer: pins user pages, posts a second notification CQE when the NIC is done. | [`io_send_zc_prep`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L1330) | [`io_send_zc`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L1461) |
| [`IORING_OP_SENDMSG_ZC`](https://elixir.bootlin.com/linux/v7.0/A/ident/IORING_OP_SENDMSG_ZC) | Zero-copy `sendmsg`: full `msghdr` with the same pinned-pages + notification model. | [`io_send_zc_prep`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L1330) | [`io_sendmsg_zc`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L1528) |
| [`IORING_OP_RECV`](https://elixir.bootlin.com/linux/v7.0/A/ident/IORING_OP_RECV) | Receive into one linear/provided buffer; supports multishot and bundles. | [`io_recvmsg_prep`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L785) | [`io_recv`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L1174) |
| [`IORING_OP_RECVMSG`](https://elixir.bootlin.com/linux/v7.0/A/ident/IORING_OP_RECVMSG) | Receive a full `msghdr`: payload plus peer address and control messages. | [`io_recvmsg_prep`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L785) | [`io_recvmsg`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L1015) |
| [`IORING_OP_RECV_ZC`](https://elixir.bootlin.com/linux/v7.0/A/ident/IORING_OP_RECV_ZC) | Zero-copy receive: payloads land directly in pre-registered device memory (zcrx), no userspace buffer. | [`io_recvzc_prep`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L1250) | [`io_recvzc`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L1278) |

Note the deliberate sharing: `SEND`/`SENDMSG` share one `prep`, both ZC sends
share one `prep`, and `RECV`/`RECVMSG` share one `prep`. The `prep` functions
branch on `req->opcode` to specialize. This is why the data structure
([`io_sr_msg`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L65))
is a union that serves all of them.

# What net.c does NOT do

- It is **not** the socket layer. `net.c` never touches TCP state machines or
  [`sk_buff`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/skbuff.h#L885)
  queues directly; it hands a
  [`struct msghdr`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/socket.h#L71)
  to `sock_sendmsg()` /
  `sock_recvmsg()` and lets `net/` do protocol work.
- It is **not** the io_uring core. Reference counting, task-work, CQE posting,
  and the SQE ring live in `io_uring.c`; `net.c` only fills in `req->cqe` and
  returns a status code.
- `IORING_OP_RECV_ZC` is a thin shim. The real device-memory receive logic
  lives in [`io_uring/zcrx.c`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/zcrx.c);
  `io_recvzc()` just calls [`io_zcrx_recv()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/zcrx.c#L1510).
- Accept/connect/bind/listen/shutdown/socket also live in `net.c` but are out
  of scope here; this post is send/recv only.

# Core data structures

These are the nouns. The handlers are verbs that operate on them.

## struct io_kiocb — the request

[`struct io_kiocb`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/io_uring_types.h#L688)
is *the* io_uring request object — one per in-flight operation, used by every
opcode, not just networking. The first member is a union whose first field is
always [`struct file`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/fs.h#L1259)` *file`,
so any opcode can read `req->file` regardless of
which sub-command view it uses. Fields that matter for net:

- `opcode`, `flags` (`REQ_F_*`) — drive dispatch and per-request state.
- `cqe` — the embedded completion (`res` + `flags`); the handler fills this.
- `buf_index` — fixed-buffer index or selected provided-buffer ID.
- `async_data` — opcode-private heap state; for net it points at a
  [`struct io_async_msghdr`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.h#L8).
- `refs` — atomic refcount governing lifetime (see below).

The opcode-private command view is overlaid on the `cmd` union member. For
networking that view is
[`struct io_sr_msg`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L65),
obtained with [`io_kiocb_to_cmd`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/io_uring_types.h#L678)`(req, struct io_sr_msg)`. It carries the user
buffer/`msghdr` pointer, `len`, `msg_flags`, the io_uring-level `flags`
(`IORING_RECVSEND_*`), `done_io` (bytes already transferred across retries),
multishot bookkeeping, and `notif` (the zero-copy notification request).

**Lifetime.** A request is allocated from a per-context cache in the core
([`io_alloc_req`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/io_uring.h#L529)),
initialized by `io_init_req()`, and reference-counted via
`req->refs` (initialized to 1; extra refs taken for async poll, linked
timeouts, multishot). It is freed back to the cache when the last ref drops
after the CQE is posted. The whole §"The io_kiocb lifetime" walks this in
order.

## struct io_async_msghdr — the persistent message state

[`io_async_msghdr`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.h#L8)
is the heap companion that lets a net request **survive blocking and resume**.
It embeds a `struct msghdr msg`, a fast single-entry `fast_iov`, an
[`iou_vec`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/io_uring_types.h#L131)` vec`
for multi-segment imports, and the
[`sockaddr_storage`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/socket.h#L63)` addr`. The
`struct_group(clear, ...)` lets the core memset the volatile part cheaply on
reuse.

Why it must persist: a non-blocking `sock_sendmsg()` can do a *partial*
transfer and return `-EAGAIN`. The request is parked and re-issued later; the
`msg.msg_iter` inside `io_async_msghdr` already records how far we got, so the
retry resumes at the right offset instead of resending from the start. This is
the structure the io_kiocb's `async_data` points at.

**Lifetime.** Allocated lazily by
[`io_msg_alloc_async()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L185)
during `prep`, drawn from the per-context `netmsg_cache` slab cache so the
common case is allocation-free. On completion it is recycled back to that
cache by
[`io_netmsg_recycle()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L166);
any imported `iovec` is freed first. Caching the `iovec` across reuse is why
`REQ_F_NEED_CLEANUP` is set when a cached vec is found.

## struct io_recvzc — the zero-copy receive view

[`io_recvzc`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L111)
is the much smaller command view for `IORING_OP_RECV_ZC`: just `file`,
`flags`, `len`, and a pointer to the
[`struct io_zcrx_ifq`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/zcrx.h#L41)
(the registered RX interface queue that owns the device
memory area). It deliberately does *not* use `io_async_msghdr` — there is no
userspace buffer and no `msghdr`, because data lands in pre-registered device
memory and is reported by the zcrx layer.

# The io_kiocb lifetime

This is the spine of the post: one request from SQE to free. Send and recv
share the same skeleton; the per-opcode work happens only at the `issue` step.

## 1. Birth and prep

The core reads an SQE and calls
[`io_submit_sqe()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/io_uring.c#L1868),
which calls
[`io_init_req()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/io_uring.c#L1715).
`io_init_req()` allocates nothing for the request itself (it came from the
cache), sets `opcode`/`flags`/`file`, then ends with `def->prep(req, sqe)`
— [the prep dispatch](https://elixir.bootlin.com/linux/v7.0/source/io_uring/io_uring.c#L1828).

For send that lands in
[`io_sendmsg_prep()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L418);
for recv,
[`io_recvmsg_prep()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L785).
Prep validates flags against `SENDMSG_FLAGS`/`RECVMSG_FLAGS`, reads the buffer
pointer and length from the SQE, allocates the `io_async_msghdr`, and — for
the `msg` variants — copies the
[`user_msghdr`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/socket.h#L98)
header in from userspace once
([`io_sendmsg_setup`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L392)
/ [`io_recvmsg_prep_setup`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L754)).
Doing the import in `prep`
means the volatile SQE is fully consumed before the request can be parked;
nothing later re-reads userspace SQE memory.

`prep` runs in the submitter's syscall context with `uring_lock` held. It must
not block on the socket — that is the `issue` step's job.

## 2. First issue (the fast path)

After init, `io_submit_sqe()` calls
[`io_queue_sqe()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/io_uring.c#L1627),
which calls
[`io_issue_sqe()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/io_uring.c#L1399)
with `IO_URING_F_NONBLOCK | IO_URING_F_COMPLETE_DEFER`. `io_issue_sqe()`
assigns the file, then invokes
[`__io_issue_sqe()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/io_uring.c#L1366)
which finally calls `def->issue(req, issue_flags)` — the net handler.

Because the first issue carries `IO_URING_F_NONBLOCK`, the handler asks the
socket for a non-blocking transfer. The three outcomes:

- **Completed (or hard error):** the handler fills `req->cqe` and returns
  `IOU_COMPLETE`. `io_issue_sqe()` posts the completion (deferred via
  [`io_req_complete_defer()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/io_uring.h#L487)
  to the submit-flush batch) and the request heads toward teardown.
- **Would block (`-EAGAIN`):** the handler returns `-EAGAIN`; the request is
  punted (step 3).
- **More to do (multishot/bundle/retry):** the handler returns `IOU_RETRY` or
  re-loops internally; see the variant sections.

## 3. Parking: poll vs io-wq

On `-EAGAIN`, `io_queue_sqe()` calls
[`io_queue_async()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/io_uring.c#L1602).
For a socket the usual outcome is **armed poll**
([`io_arm_poll_handler()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/poll.c#L701)):
io_uring registers a poll
wait on the socket and returns; the submitter syscall is done while the
request waits. When the socket becomes readable/writable, task-work resumes
the request via
[`io_req_task_submit()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/io_uring.c#L1033),
which re-issues the same `io_kiocb` — now with its `io_async_msghdr` carrying
any partial-transfer offset. If poll can't be armed, the request is handed to
an **io-wq** worker thread that can call the socket in blocking mode
(`IO_URING_F_UNLOCKED` set, no `IO_URING_F_NONBLOCK`).

This park/resume cycle can repeat. `sr->done_io` accumulates bytes across
re-issues so the final CQE reports the total, not just the last chunk.

## 4. Completion and free

When the handler returns `IOU_COMPLETE`, the core posts the CQE
([`io_req_complete_post`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/io_uring.c#L900)
or, batched,
[`__io_submit_flush_completions()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/io_uring.c#L1120)),
drops the final reference, runs the opcode `cleanup` callback
([`io_sendmsg_recvmsg_cleanup`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L342)
for the plain/msg paths,
[`io_send_zc_cleanup`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L1313)
for ZC), recycles the `io_async_msghdr` to the
`netmsg_cache`, and returns the `io_kiocb` to its cache. If the request failed
mid-flight, the `.fail` callback
[`io_sendrecv_fail()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L1600)
salvages `done_io` so a short transfer is still reported.

# Send path and its variants

All send handlers follow the same five beats: resolve the socket
(`sock_from_file`), honour `POLL_FIRST`, set `MSG_DONTWAIT` under
`IO_URING_F_NONBLOCK`, call the socket, then post-process the return for
retry/partial/error.

## io_send — plain IORING_OP_SEND

[`io_send()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L642)
is the simplest. After the preamble it optionally selects a provided buffer
(`io_send_select_buffer`), computes `min_ret` (the whole length when
`MSG_WAITALL` or bundle), and calls
[`sock_sendmsg()`](https://elixir.bootlin.com/linux/v7.0/source/net/socket.c#L753).
On a short send with retry possible it advances `sr->buf`/`sr->len`, adds to
`sr->done_io`, and returns `IOU_RETRY`. The `io_async_msghdr.msg.msg_iter`
holds the iterator across that retry.

## io_sendmsg — IORING_OP_SENDMSG

[`io_sendmsg()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L545)
handles a full `struct msghdr` (multiple iovecs, ancillary control data,
destination address). The header was already imported in
`io_sendmsg_setup()`, so `io_sendmsg()` just restores `msg_control_user` and
calls
[`__sys_sendmsg_sock()`](https://elixir.bootlin.com/linux/v7.0/source/net/socket.c#L2655)
— the same internal entry the `sendmsg(2)` syscall uses, minus the userspace
copy. Control messages disable partial retry (it can't resume cleanly), so on
a retried short send it zeroes `msg_controllen` before re-arming.

## Vectorized send — IORING_SEND_VECTORIZED

[`IORING_SEND_VECTORIZED`](https://elixir.bootlin.com/linux/v7.0/A/ident/IORING_SEND_VECTORIZED)
(bit 5 of `sqe->ioprio`) gives plain `IORING_OP_SEND` scatter-gather *without*
the `sendmsg` overhead. With the flag set, `sqe->addr` no longer points at a
single contiguous buffer — it points at an array of `struct iovec`, and
`sqe->len` is the segment count. The uAPI doc sums it up: *"SEND[_ZC] will
take a pointer to a io_vec to allow vectorized send operations."* It was added
by commit `6f02527729bd` ("io_uring/net: Allow to do vectorized send").

The whole effect lives in
[`io_send_setup()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L349):
with the flag, the source is imported by
[`io_net_import_vec()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L212)
(the same `__import_iovec()` machinery `sendmsg` uses); without it, by
`import_ubuf()` (one linear buffer). The resulting iterator lands in
`io_async_msghdr.msg.msg_iter` exactly as in the linear case, so a partial
`MSG_WAITALL` send resumes mid-iovec across poll re-arms.

The value over `IORING_OP_SENDMSG` is that there is no `user_msghdr` copy and
no `msg_name`/control plumbing — just data segments. For `IORING_OP_SENDMSG`
the flag is accepted but redundant (that opcode is always vectored). Zero-copy
`IORING_OP_SEND_ZC` honours it too — it shares `io_send_setup()` — but only
with a generic iovec: `IORING_SEND_VECTORIZED | IORING_RECVSEND_FIXED_BUF` is
rejected with `-EINVAL`. There is no recv counterpart; vectored receive is the
`RECVMSG` path only.

## Bundles — IORING_RECVSEND_BUNDLE

A bundle lets one send drain *multiple* provided buffers in a single syscall.
`io_send()` loops at the `retry_bundle` label:
[`io_send_select_buffer()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L594)
pulls a batch of buffers into an `ITER_IOVEC`, the data goes out in one
`sock_sendmsg()`, and
[`io_send_finish()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L507)
posts a CQE with
`IORING_CQE_F_MORE` and loops again if more buffers remain and the socket
still accepts data.
[`io_bundle_nbufs()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L467)
works out how many provided buffers a
short transfer consumed so the right number are committed. Bundles are
send-only for `IORING_OP_SEND` (rejected for `SENDMSG`).

## Zero-copy send — IORING_OP_SEND_ZC / SENDMSG_ZC

[`io_send_zc()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L1461)
and
[`io_sendmsg_zc()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L1528)
avoid the data copy into kernel `sk_buff`s: the user pages are pinned and
referenced directly by the network stack, and a *second* completion (the
notification, `req->notif`) is posted once the NIC is done with the pages.
Two consequences visible in the code:

- The socket must advertise `SOCK_SUPPORT_ZC`, else `-EOPNOTSUPP`.
- `kmsg->msg.msg_ubuf` is set to the notif's
  [`ubuf_info`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/skbuff.h#L546),
  and the CQE carries
  `IORING_CQE_F_MORE` to tell userspace a notification CQE will follow.

These paths also accept `IORING_RECVSEND_FIXED_BUF`: the source is a
pre-registered buffer imported by
[`io_send_zc_import()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L1448)
via [`io_import_reg_buf()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/rsrc.c#L1124)
(single) or
[`io_import_reg_vec()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/rsrc.c#L1471)
(vectorized). Registered buffers are imported lazily at *issue* time, not
prep, because resolving the buffer node needs the issue-time locking context.

# Recv path and its variants

Recv mirrors send but adds two features send lacks: **multishot** (one SQE
yields many CQEs as data arrives) and provided-buffer selection on every
arrival.

## io_recv — plain IORING_OP_RECV

[`io_recv()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L1174)
optionally selects a provided buffer
([`io_recv_buf_select()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L1102)),
then calls
[`sock_recvmsg()`](https://elixir.bootlin.com/linux/v7.0/source/net/socket.c#L1096).
Like send it handles `-EAGAIN` (retry via poll), partial transfer with
`MSG_WAITALL` (advance buffer, accumulate `done_io`), and `MSG_TRUNC`/`CTRUNC`
as failure under `MSG_WAITALL`.
[`io_recv_finish()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L848)
decides whether to loop (multishot) or complete.

## io_recvmsg — IORING_OP_RECVMSG

[`io_recvmsg()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L1015)
fills a full `msghdr` (peer address + control messages) via
[`__sys_recvmsg_sock()`](https://elixir.bootlin.com/linux/v7.0/source/net/socket.c#L2863).
It sets `msg_get_inq` so the socket reports bytes still queued (`msg_inq`),
which feeds the `IORING_CQE_F_SOCK_NONEMPTY` hint and the multishot retry
decision.

## Multishot — IORING_RECV_MULTISHOT

Multishot requires provided buffers (`REQ_F_BUFFER_SELECT`) and forbids
`MSG_WAITALL` — see the checks in
[`io_recvmsg_prep()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L785).
At issue, the handler loops at `retry_multishot`: select a buffer, receive,
post a CQE with `IORING_CQE_F_MORE`, and loop again as long as the socket
keeps delivering and a buffer is available. For `RECVMSG`,
[`io_recvmsg_multishot()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L961)
packs the name + control + payload into one provided buffer per CQE. The
optional `IORING_RECV_MSHOT_LIM` caps total bytes (`mshot_total_len`) so a
busy socket can't monopolize one request forever. When buffers run out or the
socket drains, the final CQE omits `F_MORE` and the request completes.

## Recv bundles

`IORING_RECVSEND_BUNDLE` on `IORING_OP_RECV` fills several provided buffers
from one `sock_recvmsg()` and reports them together; `io_recv_finish()` +
`io_bundle_nbufs()` commit the consumed count. As with send, bundles are
rejected for the `msg` opcode.

## Zero-copy recv — IORING_OP_RECV_ZC

[`io_recvzc()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/net.c#L1278)
is the odd one out: no buffer, no `msghdr`. It forwards to
[`io_zcrx_recv()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/zcrx.c#L1510),
which lands packet payloads directly in a pre-registered device-memory area
(the `io_zcrx_ifq`) and reports filled regions through the zcrx refill/
completion rings. `io_recvzc()` returns `IOU_RETRY` to stay armed for the next
batch, or `IOU_COMPLETE`/`IOU_REQUEUE` on terminal conditions. Use it for
high-throughput RX where even the copy to a provided buffer is too expensive.

# How net.c calls the networking stack

The lower boundary is small and stable. Every send/recv handler ends in one of
four socket entry points, all in
[`net/socket.c`](https://elixir.bootlin.com/linux/v7.0/source/net/socket.c):

| net.c handler | socket call | notes |
|---|---|---|
| `io_send` | [`sock_sendmsg()`](https://elixir.bootlin.com/linux/v7.0/source/net/socket.c#L753) | linear/iovec/bundle source |
| `io_sendmsg` / `io_sendmsg_zc` | [`__sys_sendmsg_sock()`](https://elixir.bootlin.com/linux/v7.0/source/net/socket.c#L2655) | full msghdr incl. cmsg |
| `io_send_zc` | [`sock_sendmsg()`](https://elixir.bootlin.com/linux/v7.0/source/net/socket.c#L753) + `msg_ubuf` | pinned pages, notif |
| `io_recv` | [`sock_recvmsg()`](https://elixir.bootlin.com/linux/v7.0/source/net/socket.c#L1096) | linear/provided buffer |
| `io_recvmsg` | [`__sys_recvmsg_sock()`](https://elixir.bootlin.com/linux/v7.0/source/net/socket.c#L2863) | full msghdr incl. cmsg |
| `io_recvzc` | [`io_zcrx_recv()`](https://elixir.bootlin.com/linux/v7.0/source/io_uring/zcrx.c#L1510) | device memory, not socket.c |

The contract is the ordinary `struct socket` one: io_uring builds a
`struct msghdr` (with an
[`iov_iter`](https://elixir.bootlin.com/linux/v7.0/source/include/linux/uio.h#L43)
over user or registered memory), sets
`MSG_DONTWAIT` for the non-blocking first attempt, and lets the protocol do
the rest. The only io_uring-specific twist is `msg_ubuf` for zero-copy send
and the `msg_get_inq`/`msg_inq` plumbing that feeds the multishot decision.
[`sock_from_file()`](https://elixir.bootlin.com/linux/v7.0/source/net/socket.c#L530)
recovers the `struct socket` from `req->file` on every issue.

# Putting it together: a recv multishot trace

1. Userspace submits one `IORING_OP_RECV` SQE with `IORING_RECV_MULTISHOT` and
   a provided-buffer group.
2. Core: `io_submit_sqe` → `io_init_req` → `io_recvmsg_prep` allocates the
   `io_async_msghdr`, sets `REQ_F_APOLL_MULTISHOT`.
3. Core: `io_queue_sqe` → `io_issue_sqe` → `io_recv` (non-blocking). Socket is
   empty → `-EAGAIN` → `io_queue_async` arms poll. Syscall returns.
4. Packet arrives. Task-work re-issues `io_recv`. It selects a buffer,
   `sock_recvmsg()` copies the payload, posts a CQE with `F_MORE`, loops.
5. Socket drains → `-EAGAIN` → re-arm poll (the request stays alive, refcount
   held). Steps 4–5 repeat per arrival.
6. Buffer group empties (or `MSHOT_LIM` reached): final CQE without `F_MORE`,
   `io_recv` returns completion, core posts CQE, recycles the
   `io_async_msghdr`, frees the `io_kiocb`.

The single `io_kiocb` and its `io_async_msghdr` are reused across every loop
and every poll re-arm — that buffer reuse is what makes multishot cheap.

# Summary

**Main advantages**

- **One small glue layer, two clean boundaries.** `net.c` only translates
  between the io_uring `prep`/`issue` vtable and the standard `struct socket`
  send/recv calls — protocol work stays in `net/`, lifetime/refcount stays in
  `io_uring.c`.
- **The `io_async_msghdr` makes blocking transparent.** Persisting `msg_iter`
  and `done_io` across poll re-arms and io-wq punts lets a partial transfer
  resume correctly while the submitter syscall has long returned.
- **Variants reuse one skeleton.** Plain/msg/bundle/multishot/zero-copy all
  share the same socket-call-plus-retry shape; only buffer sourcing and
  completion semantics differ, which keeps the code auditable.
- **Allocation-light hot path.** Requests and `io_async_msghdr`s come from
  per-context caches; multishot/bundle reuse a single request for many CQEs.

**Main problems / limitations**

- **Feature-flag combinatorics.** `IORING_RECVSEND_*` flags interact (bundle
  vs msg, multishot vs `MSG_WAITALL`, fixed-buf vs vectorized), and the rules
  live in scattered `prep` checks rather than one table — easy to get wrong
  when extending.
- **Partial-transfer correctness is subtle.** `done_io` accounting, control-
  message retry suppression, and `MSG_TRUNC` handling are per-variant special
  cases; a missed case silently misreports bytes.
- **Zero-copy is not free.** SEND_ZC needs `SOCK_SUPPORT_ZC`, page pinning,
  and a second notification CQE; RECV_ZC needs a registered device-memory
  interface (`io_zcrx_ifq`). Both add setup and completion complexity that
  only pays off at high throughput.
- **`net.c` is a catch-all.** Send/recv share the file with accept/connect/
  bind/listen/socket/shutdown; the single translation unit is large and the
  shared `io_sr_msg` union is doing a lot of double duty.
