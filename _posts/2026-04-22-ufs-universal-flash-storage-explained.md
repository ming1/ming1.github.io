## 1. What Is UFS?

**Universal Flash Storage (UFS)** is a high-performance flash storage interface
standard defined by **JEDEC** (Joint Electron Device Engineering Council). It is
the dominant storage technology in modern smartphones and is expanding into
automotive, IoT, and embedded systems.

**One sentence**: UFS is a serial, full-duplex, command-queued flash storage
interface that replaced eMMC's parallel half-duplex bus, delivering 10-20x
higher throughput and 20-200x higher random IOPS.

```
   eMMC (legacy)                  UFS (modern)
   ─────────────                  ────────────
   8-bit parallel bus             Serial differential lanes
   Half-duplex                    Full-duplex (TX + RX)
   One command at a time          Up to 32 queued commands
   ~400 MB/s max                  ~4,200 MB/s (UFS 4.0)
   ~7K random read IOPS           ~140K random read IOPS
```

### Key Industry Players

- **Standard body**: JEDEC (JESD220 family)
- **Physical/transport layers**: MIPI Alliance (M-PHY + UniPro)
- **Major vendors**: Samsung, SK Hynix, Micron, KIOXIA, Western Digital
- **SoC integrators**: Qualcomm, MediaTek, Samsung LSI, HiSilicon

## 2. Motivation and Technology Evolution

### 2.1 Why eMMC Was Not Enough

eMMC 5.1 (the final eMMC revision, released 2015) hit fundamental limits:

| Limitation | Detail |
|------------|--------|
| **Half-duplex** | 8-bit parallel bus can only read OR write, never both |
| **No command queuing** | Commands processed strictly one-at-a-time |
| **Speed ceiling** | HS400 mode maxes out at ~400 MB/s read, ~125 MB/s write |
| **Frozen standard** | No new eMMC spec since 2015 |
| **Capacity limit** | Typically tops out at 256 GB |

Modern workloads that exposed these limits:
- 4K/8K video recording at 100+ MB/s sustained write
- App launch times on 8+ GB RAM devices with aggressive multitasking
- On-device AI inference loading multi-GB models
- 1 TB storage tiers in flagship phones

## 3. Hardware and Software Interfaces

### 3.1 Layered Architecture Overview

UFS uses a layered architecture modeled after the OSI reference model. This is
the most important diagram to understand — everything else builds on it:

```
┌───────────────────────────────────────────────────────┐
│                   Application Layer                    │
│  ┌──────────┐  ┌─────────────┐  ┌──────────────────┐ │
│  │   UCS    │  │    Task     │  │     Device       │ │
│  │ (SCSI    │  │   Manager   │  │    Manager       │ │
│  │  Cmds)   │  │ (queue ctl) │  │ (config/query)   │ │
│  └────┬─────┘  └──────┬──────┘  └────────┬─────────┘ │
│       │               │                  │            │
│       ▼               ▼                  ▼            │
│  ┌────────────────────────────────────────────────┐   │
│  │    UTP (UFS Transport Protocol) Layer          │   │
│  │    Encapsulates requests into UPIU frames      │   │
│  └────────────────────┬───────────────────────────┘   │
│                       │                               │
│  ┌────────────────────▼───────────────────────────┐   │
│  │    UIC (UFS Interconnect) Layer                │   │
│  │  ┌─────────────────────────────────────────┐   │   │
│  │  │  UniPro (PA, DL, N, T layers + DME)     │   │   │
│  │  └─────────────────┬───────────────────────┘   │   │
│  │  ┌─────────────────▼───────────────────────┐   │   │
│  │  │  MIPI M-PHY (physical signaling)        │   │   │
│  │  └─────────────────────────────────────────┘   │   │
│  └────────────────────────────────────────────────┘   │
└───────────────────────────────────────────────────────┘
         │                              ▲
         ▼  (TX differential pair)      │  (RX differential pair)
    ┌─────────────────────────────────────────┐
    │          UFS Device (NAND flash)        │
    └─────────────────────────────────────────┘
```

Let's examine each layer from bottom to top.

### 3.2 MIPI M-PHY — Physical Layer

M-PHY is the physical layer standard from the MIPI Alliance. It provides
the electrical signaling for UFS.

**Key characteristics:**
- **Differential NRZ signaling** (through HS-G5); PAM4 in future HS-G6
- **8b/10b line encoding** (20% overhead)
- **Dedicated unidirectional lanes**: each TX lane and RX lane is independent

**Gear speeds:**

| Gear | Data Rate/Lane | Effective BW/Lane | M-PHY Version |
|------|---------------|-------------------|---------------|
| HS-G1 | 1.248 Gbps | ~150 MB/s | v2.1+ |
| HS-G2 | 2.496 Gbps | ~300 MB/s | v2.1+ |
| HS-G3 | 5.83 Gbps | ~600 MB/s | v3.x |
| HS-G4 | 11.6 Gbps | ~1,450 MB/s | v4.1 |
| HS-G5 | 23.32 Gbps | ~2,900 MB/s | v5.0 |
| HS-G6 | 46.694 Gbps | ~5,800 MB/s | v6.0 (future) |

**Lane configuration**: UFS uses 1 or 2 lanes per direction. A typical UFS 4.0
setup has 2 TX + 2 RX lanes, giving 4 active differential pairs.

**Power states** (low to high power):

```
HIBERN8 ──→ SLEEP ──→ STALL ──→ HS-BURST (active transfer)
(lowest)                         (highest)
```

**Low-speed mode**: M-PHY also supports PWM (Pulse Width Modulation) signaling
for low-speed, self-clocked communication. Link startup always begins in PWM-G1
before negotiating up to high-speed gears.

### 3.3 UniPro — Transport Layer

MIPI UniPro (Unified Protocol) sits between M-PHY and UTP. It implements
reliable packet transport over the M-PHY lanes.

**Sublayers:**

| Layer | Name | Function |
|-------|------|----------|
| PA | PHY Adapter | Interfaces with M-PHY; gear/lane negotiation |
| DL | Data Link | Reliable delivery with credit-based flow control |
| N | Network | Point-to-point addressing |
| T | Transport | Connection management, segmentation (up to 1144 bytes/segment in UniPro 2.0) |
| DME | Device Management Entity | Configuration and management of the entire UniPro stack |

UniPro versions correspond to UFS versions: v1.6 for UFS 2.x, v1.8 for UFS
3.x, v2.0 for UFS 4.x.

### 3.4 SCSI Command Set

A surprising design choice: **UFS uses SCSI commands**. Rather than inventing a
new command protocol, UFS adopted a subset of SCSI Primary Commands (SPC-4) and
SCSI Block Commands (SBC-3). This means UFS devices appear as standard SCSI
devices to the operating system.

**Commonly used SCSI commands in UFS:**

| Command | Opcode | Purpose |
|---------|--------|---------|
| READ (10/16) | 0x28/0x88 | Read data from LBA |
| WRITE (10/16) | 0x2A/0x8A | Write data to LBA |
| SYNCHRONIZE CACHE | 0x35 | Flush write cache to NAND |
| START STOP UNIT | 0x1B | Power state transitions |
| REQUEST SENSE | 0x03 | Error information |
| INQUIRY | 0x12 | Device identification |
| MODE SENSE/SELECT | 0x1A/0x15 | Device parameters |
| REPORT LUNS | 0xA0 | List logical units |
| UNMAP | 0x42 | Trim/discard (like ATA TRIM) |

**Three Service Access Points (SAPs):**

```
 ┌──────────┐    ┌──────────────┐    ┌──────────────┐
 │   UCS    │    │ Task Manager │    │Device Manager│
 │(SCSI I/O)│    │ (abort/reset)│    │(query/config)│
 └────┬─────┘    └──────┬───────┘    └──────┬───────┘
      │                 │                   │
      ▼                 ▼                   ▼
 UTP_CMD_SAP       UTP_TM_SAP          UDM_SAP
      │                 │                   │
      └────────┬────────┴───────────────────┘
               ▼
    UTP Layer (builds UPIUs)
```

### 3.5 UPIU — The Message Format

**UFS Protocol Information Unit (UPIU)** is the fundamental message format
exchanged between host and device. Every operation — reads, writes, queries,
task management — is wrapped in a UPIU.

**UPIU structure:**

```
┌──────────────────────────────────────┐
│  Basic Header (12 bytes, fixed)      │
│  - Transaction type (1 byte)         │
│  - Flags, LUN, Task Tag             │
│  - Command set type                  │
│  - Data segment length               │
├──────────────────────────────────────┤
│  Transaction-Specific Fields         │
│  (variable, depends on UPIU type)    │
├──────────────────────────────────────┤
│  Extended Header Segments (optional) │
├──────────────────────────────────────┤
│  Data Segment (0 or more bytes)      │
│  (SCSI CDB, sense data, or payload) │
└──────────────────────────────────────┘
Min size: 32 bytes    Max size: 65,600 bytes
```

**UPIU types and their roles:**

| UPIU Type | Direction | Purpose |
|-----------|-----------|---------|
| COMMAND | Host → Device | Carries a SCSI command (CDB) |
| DATA OUT | Host → Device | Write data payload |
| READY TO TRANSFER | Device → Host | "I'm ready, send me write data" |
| DATA IN | Device → Host | Read data payload |
| RESPONSE | Device → Host | Command completion status |
| QUERY REQUEST | Host → Device | Device management (read/write attributes, flags, descriptors) |
| QUERY RESPONSE | Device → Host | Response to query |
| TASK MGMT REQUEST | Host → Device | Abort task, LUN reset, etc. |
| TASK MGMT RESPONSE | Device → Host | Task management result |
| NOP OUT / NOP IN | Both | Keepalive / link verification |
| REJECT | Device → Host | Malformed or unsupported request |

**Typical read command flow:**

```
Host                                    Device
  │                                       │
  │──── COMMAND UPIU (READ 10) ──────────▶│
  │     [Task Tag=5, LBA=0x1000, Len=8]  │
  │                                       │  (device fetches data from NAND)
  │◀──── DATA IN UPIU ───────────────────│
  │     [Task Tag=5, 4096 bytes payload] │
  │◀──── RESPONSE UPIU ─────────────────│
  │     [Task Tag=5, Status=GOOD]        │
  │                                       │
```

**Typical write command flow:**

```
Host                                    Device
  │                                       │
  │──── COMMAND UPIU (WRITE 10) ─────────▶│
  │     [Task Tag=7, LBA=0x2000, Len=8]  │
  │                                       │
  │◀──── READY TO TRANSFER UPIU ─────────│
  │     [Task Tag=7, offset=0, count=8]  │
  │                                       │
  │──── DATA OUT UPIU ──────────────────▶│
  │     [Task Tag=7, 4096 bytes payload] │
  │                                       │  (device programs NAND)
  │◀──── RESPONSE UPIU ─────────────────│
  │     [Task Tag=7, Status=GOOD]        │
  │                                       │
```

Each command is tagged with a unique **Task Tag** (0-31), enabling up to 32
outstanding commands.

### 3.6 UFS Host Controller Interface (UFSHCI)

UFSHCI defines the MMIO register interface that host software uses to
communicate with the UFS host controller hardware.

**Key register groups (MMIO-mapped):**

```
Offset   Register                   Purpose
──────   ────────                   ───────
0x0000   CAP                        Host Controller Capabilities
0x0008   VER                        Version
0x0018   AHIT                       Auto-Hibernate Idle Timer
0x0020   IS                         Interrupt Status
0x0024   IE                         Interrupt Enable
0x0030   HCS                        Host Controller Status
0x0034   HCE                        Host Controller Enable
0x0050   UTRLBA/UTRLBAU             Transfer Request List Base Address (lo/hi)
0x0058   UTRLDBR                    Transfer Request Doorbell Register
0x005C   UTRLCLR                    Transfer Request Clear
0x0060   UTRLRSR                    Transfer Request Run-Stop
0x0070   UTMRLBA/UTMRLBAU           Task Mgmt Request List Base Address
0x0078   UTMRLDBR                   Task Mgmt Doorbell
0x0080   UTMRLRSR                   Task Mgmt Run-Stop
0x0090   UICCMD                     UIC Command Register
0x0094   UCMDARG1-3                 UIC Command Arguments
```

**Data structures in host memory:**

```
                    UTRL (UTP Transfer Request List)
                    ┌─────────────────────────────────┐
 UTRLBA ──────────▶ │  UTRD[0]                        │
                    │  UTRD[1]                        │
                    │  ...                            │
                    │  UTRD[31]                       │
                    └─────────────────────────────────┘

Each UTRD (Transfer Request Descriptor) points to:

    ┌──────────────────────────────────────────┐
    │  UCD (UFS Command Descriptor)            │
    │  ┌────────────────────────────────────┐  │
    │  │  Command UPIU (request)            │  │
    │  ├────────────────────────────────────┤  │
    │  │  Response UPIU (buffer for reply)  │  │
    │  ├────────────────────────────────────┤  │
    │  │  PRDT (Physical Region Desc Table) │  │
    │  │  (scatter-gather list for DMA)     │  │
    │  └────────────────────────────────────┘  │
    └──────────────────────────────────────────┘
```

**Command submission flow (Single Doorbell mode):**

```
1. SCSI midlayer calls ufshcd_queuecommand()
2. Driver selects a free slot (0-31) in the UTRL
3. Driver builds a Command UPIU, populates the UTRD and PRDT
4. Driver writes the slot bit to UTRLDBR (doorbell register)
   ──── hardware takes over ────
5. Controller DMAs the UTRD, sends UPIU to device via UniPro/M-PHY
6. Device processes command, returns Response UPIU
7. Controller DMAs response into the Response UPIU buffer
8. Controller clears the doorbell bit and raises an interrupt
   ──── back to software ────
9. Driver reads the Response UPIU and reports status to SCSI midlayer
```

### 3.7 Multi-Circular Queue (MCQ) — UFS 4.0+

UFSHCI 4.0 introduced MCQ to overcome the single-doorbell bottleneck:

```
 Traditional SDB Mode              MCQ Mode (UFS 4.0+)
 ──────────────────                ─────────────────────
 Single 32-slot UTRL              Multiple Submission + Completion Queues
 Single doorbell register         Per-queue head/tail pointers
 All CPUs contend on              Each CPU can own its own queue
   one doorbell

                     ┌────────────────────────┐
  CPU 0 ───────────▶ │  SQ[0]  →  CQ[0]      │
  CPU 1 ───────────▶ │  SQ[1]  →  CQ[1]      │
  CPU 2 ───────────▶ │  SQ[2]  →  CQ[2]      │
  CPU 3 ───────────▶ │  SQ[3]  →  CQ[3]      │
                     └────────────────────────┘
```

MCQ advantages:
- Up to 32 submission + 32 completion queues
- Queue depth is implementation-defined (not limited to 32)
- Circular queue design with head/tail pointers (modeled after NVMe)
- Eliminates doorbell contention on multi-core SoCs
- Better interrupt steering via per-queue MSI/MSI-X vectors

### 3.8 Logical Unit Model

A UFS device supports multiple **Logical Units (LUs)**:

```
┌──────────────────────────────────────┐
│            UFS Device                │
│                                      │
│  ┌──────┐ ┌──────┐     ┌──────┐    │
│  │ LU 0 │ │ LU 1 │ ... │LU 31│    │  ← Up to 32 configurable LUs
│  │(data)│ │(data)│     │(data)│    │    (each has independent LBA space)
│  └──────┘ └──────┘     └──────┘    │
│                                      │
│  Well-Known LUs:                     │
│  ┌──────────┐ ┌──────┐ ┌────────┐  │
│  │ Boot LU  │ │ RPMB │ │Device  │  │
│  │(A and B) │ │  LU  │ │  LU    │  │
│  └──────────┘ └──────┘ └────────┘  │
└──────────────────────────────────────┘
```

- **Boot LUs (A/B)**: Firmware/bootloader storage for A/B boot schemes
- **RPMB LU**: Replay Protected Memory Block for secure storage
- **Device LU**: Device-level management (power, health)
- Each data LU has its own independent LBA address space starting from 0

Because UFS sits on the SCSI stack, UFS devices appear as standard SCSI block

### 4.3 Key Code Path: ufshcd_queuecommand()

The main SCSI entry point is `ufshcd_queuecommand()` in
`drivers/ufs/core/ufshcd.c`:

```c
/* drivers/ufs/core/ufshcd.c */
static enum scsi_qc_status ufshcd_queuecommand(struct Scsi_Host *host,
                                                struct scsi_cmnd *cmd)
{
    /* 1. Get a free transfer request slot (task tag) */
    /* 2. Build the Command UPIU from the SCSI CDB */
    /* 3. Set up PRDT for scatter-gather DMA */
    /* 4. Write doorbell register to submit to hardware */
    /* 5. Return — completion happens in interrupt handler */
}
```

The completion path runs in the interrupt handler (`ufshcd_intr()`), which
reads the Response UPIU and calls `scsi_done()` to complete the SCSI command.

## 5. Performance Features Deep Dive

### 5.1 Full-Duplex Communication

UFS uses physically separate TX and RX lanes, enabling **simultaneous
bidirectional data transfer**:

```
          Host                          Device
  ┌───────────────┐            ┌───────────────┐
  │    TX Lane 0 ─┼───────────▶┤ RX Lane 0     │
  │    TX Lane 1 ─┼───────────▶┤ RX Lane 1     │
  │               │            │               │
  │    RX Lane 0 ◀┼────────────┼─ TX Lane 0    │
  │    RX Lane 1 ◀┼────────────┼─ TX Lane 1    │
  └───────────────┘            └───────────────┘

  Host can WRITE on TX lanes while simultaneously
  READing on RX lanes. eMMC cannot do this.
```

In practice, this doubles effective bandwidth for mixed read/write workloads
compared to eMMC's half-duplex bus. However, actual simultaneous read+write
at the application level depends on the host controller implementation —
some controllers schedule commands with limited overlap.

### 5.2 Command Queuing

UFS supports SCSI Tagged Command Queuing with up to **32 outstanding commands**:

```
          eMMC                              UFS
  ┌─────────────────┐          ┌──────────────────────┐
  │ cmd1 → done     │          │ cmd1 ──┐             │
  │ cmd2 → done     │          │ cmd2 ──┤             │
  │ cmd3 → done     │          │ cmd3 ──┼── all in    │
  │  (sequential)   │          │ cmd4 ──┤  flight     │
  │                 │          │ cmd5 ──┘             │
  │ Total: T1+T2+T3 │          │ Total: max(T1..T5)  │
  └─────────────────┘          └──────────────────────┘
```

Benefits:
- Device can **reorder commands** for optimal NAND page/block access patterns
- **Overlapping operations**: read one block while programming another
- High Priority and Data Tag features for latency-sensitive commands
- This is the primary reason UFS achieves 20-200x higher random IOPS than eMMC

### 5.3 Write Booster

Write Booster uses an **SLC (Single-Level Cell) cache** to accelerate writes:

```
Write Phase:
  Host data ──▶ SLC Cache (fast, 1 bit/cell)
                     │
Flush Phase (idle):  │
                     ▼
               TLC/QLC Main Storage (dense, 3-4 bits/cell)
```

- SLC writes are ~2-4x faster than TLC/QLC
- Data is migrated to main storage during idle periods
- UFS 4.1 adds **Write Booster Buffer Resizing** — the host can dynamically
  adjust cache size based on workload
- **Limitation**: Once the SLC cache fills (typically a few GB), write speed
  drops to native TLC/QLC speeds. This affects sustained write workloads like
  large file copies.

### 5.4 HPB (Host Performance Booster)

HPB caches the device's **L2P (Logical-to-Physical) address mapping** in host
DRAM to accelerate random reads:

```
Without HPB:                       With HPB:
┌──────┐     ┌──────────────┐     ┌──────┐     ┌──────────────┐
│ Host │────▶│  UFS Device  │     │ Host │────▶│  UFS Device  │
│      │     │              │     │      │     │              │
│      │     │ 1. Look up   │     │ HPB  │     │ Skip L2P     │
│      │     │    L2P table │     │cache │     │ lookup!      │
│      │     │    in SRAM   │     │(DRAM)│     │              │
│      │     │ 2. If miss,  │     │  │   │     │ Go directly  │
│      │     │    read from │     │  ▼   │     │ to NAND page │
│      │     │    NAND      │     │ Host │     │              │
│      │     │ 3. Read data │     │sends │     │              │
│      │     │    page      │     │phys  │     │              │
│      │     │              │     │addr  │     │              │
└──────┘     └──────────────┘     └──────┘     └──────────────┘
       ~100-200 μs                        ~50-70 μs
```

- Eliminates the L2P table lookup latency on the device
- Up to **46-67% improvement** in random read performance
- Particularly effective for app launch (lots of small random reads)
- Uses HPB READ BUFFER and HPB WRITE BUFFER SCSI commands

### 5.5 RPMB (Replay Protected Memory Block)

RPMB is a dedicated, authenticated secure partition:

```
┌───────────────────────────────────────┐
│            RPMB Partition             │
│                                       │
│  Write:  data + HMAC(key, data+counter)
│          ──────────────────────▶      │
│          Counter increments           │
│          (prevents replay)            │
│                                       │
│  Read:   data + HMAC for auth         │
│          ◀──────────────────────      │
└───────────────────────────────────────┘
```

Use cases:
- Storing cryptographic keys
- DRM license management
- Secure boot parameters
- Anti-rollback counters
- UFS 4.0 introduced advanced RPMB with increased bandwidth

### 5.6 Power Management

UFS defines a multi-level power management scheme:

```
Power Level  │ Device State │ Link State │ Power    │ Wake-up
─────────────┼──────────────┼────────────┼──────────┼─────────
     0       │ Active       │ Active     │ Highest  │ Immediate
     1       │ Active       │ Hibernate  │          │ ~μs
     2       │ Sleep        │ Active     │          │ ~ms
     3       │ Sleep        │ Hibernate  │          │ ~ms
     4       │ PowerDown    │ Hibernate  │          │ ~10s ms
     5       │ PowerDown    │ Off        │          │ Power cycle
     6       │ DeepSleep    │ Off        │ Lowest   │ HW reset
```

**Auto-Hibernate**: The UFSHCI AHIT register allows configuring an idle timer.
After the specified idle period, the controller automatically transitions the
link to HIBERN8, saving power without software intervention.

**Reference clock gating**: In HIBERN8 and low-speed modes, the reference clock
can be gated for additional power savings. UFS is approximately **2x more
power-efficient at idle** compared to eMMC.

### 5.7 Performance Throttling Notification (UFS 3.1+)

The device can notify the host when temperature approaches critical thresholds.
This is especially important for:
- **Automotive** applications (operating up to 115 degrees C)
- **Sustained workloads** (4K video recording, large file transfers)
- The host can proactively throttle I/O or trigger cooling measures

## 6. Advantages and Disadvantages

### 6.1 UFS vs eMMC — Feature Comparison

| Feature | eMMC 5.1 | UFS 4.0 | Improvement |
|---------|----------|---------|-------------|
| Interface | 8-bit parallel | Serial LVDS | Scalable bandwidth |
| Duplex | Half-duplex | Full-duplex | 2x effective BW for mixed I/O |
| Command queuing | None (1 at a time) | 32 commands + MCQ | Massive random IOPS gain |
| Seq. read | ~400 MB/s | ~4,200 MB/s | **10.5x** |
| Seq. write | ~125 MB/s | ~4,000 MB/s | **32x** |
| Random read IOPS | ~7,000 | ~140,000 | **20x** |
| Random write IOPS | ~500 | ~100,000 | **200x** |
| Idle power | Baseline | ~0.5x baseline | **2x more efficient** |
| Max capacity | ~256 GB | 1 TB+ | 4x+ |
| Standard status | Frozen (2015) | Active (UFS 5.0 coming) | Continued evolution |

### 6.2 UFS vs NVMe — In-Depth Comparison

UFS and NVMe are both modern flash storage interfaces, but they were designed
for different worlds. Understanding their architectural differences explains
why each dominates its respective market.

#### 6.2.1 Architecture & Physical Layer

```
UFS Stack                              NVMe Stack
────────────────                       ────────────────
SCSI Command Set (SPC/SBC)             NVMe Command Set (native)
        │                                      │
UTP (UPIU framing)                     NVMe over PCIe (TLP framing)
        │                                      │
UniPro (PA/DL/N/T layers)             PCIe Transaction/Data Link/Physical
        │                                      │
MIPI M-PHY                             PCIe PHY (PIPE)
  2 TX + 2 RX lanes                      x1/x2/x4 PCIe lanes
  HS-G5: 23.3 Gbps/lane                 Gen 4: 16 GT/s/lane
  8b/10b encoding (20% OH)              128b/130b encoding (1.5% OH)
```

| Aspect | UFS 4.0 | NVMe (PCIe Gen 4 x4) |
|--------|---------|----------------------|
| **Physical layer** | MIPI M-PHY (2 lanes) | PCIe PHY (1-4 lanes) |
| **Raw link BW** | 46.4 Gbps (2x HS-G5) | 64 GT/s (x4 Gen 4) |
| **Encoding overhead** | 20% (8b/10b) | 1.5% (128b/130b) |
| **Effective max BW** | ~5.8 GB/s (both dirs) | ~7.88 GB/s (each dir) |
| **Pin count** | ~13 pins (2-lane) | ~50+ pins (x4) |
| **Topology** | Point-to-point | Point-to-point (or fabric in servers) |

**Key difference**: NVMe's PCIe PHY is far more bandwidth-efficient — only
1.5% encoding overhead vs. UFS's 20%. But PCIe requires significantly more
pins and silicon area for the PHY IP, which matters in mobile SoC design.

#### 6.2.2 Protocol & Command Model

This is the deepest architectural divergence between UFS and NVMe:

```
UFS Command Path:
  App → Block Layer → SCSI Midlayer → UFSHCD → SCSI CDB → UPIU → UniPro → M-PHY
                      ▲
                      │ Decades of SCSI code:
                      │ error recovery, device
                      │ discovery, LUN scanning,
                      │ sense data parsing...
                      │ (good for features,
                      │  bad for latency)

NVMe Command Path:
  App → Block Layer → NVMe Driver → NVMe Command → PCIe TLP → PCIe PHY
                      ▲
                      │ Purpose-built for flash:
                      │ no legacy protocol layers,
                      │ submission/completion queues
                      │ in host memory,
                      │ doorbell registers only
                      │ (lean, low latency)
```

| Aspect | UFS | NVMe |
|--------|-----|------|
| **Command set** | SCSI (SPC-4 + SBC-3) | Native NVMe commands |
| **Command size** | UPIU: 32-65,600 bytes | NVMe SQE: fixed 64 bytes |
| **Protocol layers** | 5+ layers (UCS→UTP→UniPro→M-PHY) | 3 layers (NVMe→PCIe TL→PHY) |
| **Command overhead** | Higher (UPIU framing + UniPro headers) | Lower (direct PCIe TLP) |
| **Software complexity** | SCSI midlayer + UFSHCD (~15K lines) | NVMe driver (~8K lines) |

**Why this matters**: Every I/O in UFS passes through the Linux SCSI midlayer
(`scsi_mod`), which adds function call depth, memory allocations for
`scsi_cmnd` structures, and SCSI sense data parsing on every error path. NVMe
bypasses all of this — commands go straight from `blk-mq` into the NVMe driver
with a fixed 64-byte command structure. This gives NVMe a measurable latency
advantage, especially at queue depth 1 (single outstanding I/O).

#### 6.2.3 Queue Depth & Submission Model

```
UFS (SDB Mode):                    NVMe:
┌─────────────────────┐           ┌──────────────────────────┐
│  1 Transfer Request │           │  Up to 65,535 Submission │
│  List (UTRL)        │           │  Queues (SQs)            │
│  32 slots max       │           │                          │
│  1 doorbell register│           │  Each SQ: up to 65,536   │
│  All CPUs contend   │           │  entries                 │
└─────────────────────┘           │                          │
                                  │  Per-queue doorbells     │
UFS (MCQ, UFS 4.0+):             │  Per-CPU queue ownership │
┌─────────────────────┐           │  MSI-X per queue         │
│  Up to 32 SQ + 32 CQ│           └──────────────────────────┘
│  Implementation-     │
│  defined depth       │
│  Per-CPU ownership   │
│  Modeled after NVMe! │
└─────────────────────┘
```

| Aspect | UFS (SDB) | UFS (MCQ) | NVMe |
|--------|-----------|-----------|------|
| **Max queues** | 1 | 32 SQ + 32 CQ | 65,535 SQ + 65,535 CQ |
| **Max queue depth** | 32 | Impl. defined | 65,536 |
| **Max outstanding cmds** | 32 | Hundreds+ | ~4 billion |
| **Multi-core scaling** | Poor (contention) | Good | Excellent |
| **Interrupt steering** | Single IRQ | Per-queue MSI | Per-queue MSI-X |

UFS 4.0's MCQ closes the gap significantly — it was explicitly designed to
borrow NVMe's multi-queue model. But NVMe still has 3 orders of magnitude
more queue capacity, which matters in server/datacenter workloads (not mobile).

#### 6.2.4 Performance Comparison

| Metric | UFS 4.0 | NVMe (mobile) | NVMe (PC, Gen 4 x4) |
|--------|---------|---------------|---------------------|
| **Seq. read** | ~4,200 MB/s | ~5,000 MB/s | ~7,000 MB/s |
| **Seq. write** | ~4,000 MB/s | ~5,000 MB/s | ~5,000 MB/s |
| **Random read IOPS** | ~140K | ~160K | ~1,000K |
| **Random write IOPS** | ~100K | ~130K | ~600K |
| **QD1 read latency** | ~80-120 μs | ~50-80 μs | ~10-15 μs |
| **QD1 write latency** | ~30-50 μs | ~20-30 μs | ~10-20 μs |

Notes: Mobile NVMe figures based on Apple's custom PCIe-based controller in
iPhones. PC NVMe figures from Samsung 990 PRO / WD SN850X class drives.
QD1 = queue depth 1 (single outstanding command), the most latency-sensitive
metric.

**Latency analysis**: NVMe's leaner protocol stack gives it a ~30-50%
advantage in QD1 latency. At higher queue depths, command queuing and
parallelism dominate, and the protocol overhead matters less — the gap
narrows to ~15-20%.

#### 6.2.5 Power & Thermal

| Aspect | UFS 4.0 | NVMe (mobile) | NVMe (PC M.2) |
|--------|---------|---------------|---------------|
| **Active read power** | ~300-500 mW | ~500-800 mW | ~5-8 W |
| **Active write power** | ~400-700 mW | ~600-1000 mW | ~5-8 W |
| **Idle power** | ~1-5 mW (HIBERN8) | ~5-20 mW (L1.2) | ~30-50 mW |
| **Deep sleep** | ~0.5 mW | ~2-5 mW | ~2-5 mW (L1.2) |
| **Thermal throttle** | Device notifies host | Controller-managed | Controller-managed |
| **Operating temp** | -40 to 85°C (consumer) | -40 to 85°C | 0 to 70°C |
|                     | -40 to 115°C (auto) | | |

**Why UFS wins on power**: UFS's M-PHY was designed from the ground up for
mobile power profiles. M-PHY's HIBERN8 state has near-zero leakage, and the
transition back to active takes microseconds. PCIe's ASPM (Active State Power
Management) L1/L1.2 states achieve similar goals but with higher baseline
power due to the more complex PHY circuitry.

#### 6.2.6 Form Factor & Integration

```
UFS BGA Package:                   NVMe M.2 2230 (smallest common):
┌──────────┐                       ┌────────────────────────────┐
│          │                       │                            │
│  11.5mm  │  ~1mm height          │         22mm               │  ~2.4mm
│    x     │                       │          x                 │  height
│  13mm    │                       │         30mm               │
│          │                       │                            │
└──────────┘                       └────────────────────────────┘
 ~150 mm²                           ~660 mm²
 BGA soldered to PCB                M.2 connector (removable)
                                    or BGA (embedded NVMe)
```

| Aspect | UFS | NVMe |
|--------|-----|------|
| **Smallest form factor** | 11.5x13 mm BGA | 22x30 mm M.2 2230 |
| **Board area** | ~150 mm² | ~660 mm² (4.4x larger) |
| **Height** | ~1 mm | ~2.4 mm |
| **Mounting** | BGA solder | M.2 connector or BGA |
| **SoC integration** | M-PHY + UniPro IP in SoC | PCIe controller in SoC |
| **IP licensing cost** | Lower (MIPI IP) | Higher (PCIe IP) |
| **Pin count to SoC** | ~13 (2-lane) | ~50+ (x4 PCIe) |

UFS's compact BGA package and low pin count are critical for smartphone PCB
design where every mm² and every pin count.

#### 6.2.7 Software Ecosystem

```
Linux Kernel:

UFS Path:                          NVMe Path:
/dev/sda ← sd_mod                  /dev/nvme0n1 ← nvme driver
    │                                  │
SCSI Midlayer (scsi_mod)           blk-mq (direct)
    │                                  │
UFSHCD (ufshcd-core)               NVMe driver (nvme-core)
    │                                  │
Platform (ufs-qcom, etc.)         PCI driver (nvme-pci)
```

| Aspect | UFS | NVMe |
|--------|-----|------|
| **Device nodes** | `/dev/sda` (SCSI disk) | `/dev/nvme0n1` (NVMe namespace) |
| **Admin tools** | `sg_utils`, `lsscsi`, `smartctl` | `nvme-cli`, `smartctl` |
| **Userspace access** | BSG (`/dev/bsg/ufs-bsg0`) | NVMe passthrough ioctls |
| **Hot plug** | Not applicable (soldered) | Supported (M.2, U.2) |
| **Boot support** | UFS Boot LUs (A/B) | UEFI NVMe boot driver |
| **Error recovery** | SCSI EH (abort → LUN reset → host reset) | NVMe reset (controller reset) |

#### 6.2.8 Where Each Wins

```
UFS wins when:                     NVMe wins when:
─────────────                      ──────────────
✓ Mobile/embedded form factor      ✓ Raw throughput matters most
✓ Power budget < 1W active         ✓ Low latency is critical (QD1)
✓ BOM cost is constrained          ✓ Deep queue depths needed
✓ Pin count / PCB area limited     ✓ Hot-plug / removable required
✓ Wide temp range (automotive)     ✓ Server / datacenter workloads
✓ Existing MIPI ecosystem in SoC   ✓ PC / laptop / desktop storage

Overlap (either works):
○ Automotive infotainment
○ High-end IoT gateways
○ Edge AI devices
```

**Bottom line**: UFS and NVMe are converging in capability (MCQ borrows from
NVMe's queue model, UFS 5.0 narrows the bandwidth gap). The choice is driven
by **system-level constraints** — power, form factor, pin count, cost — rather
than raw storage performance. In a smartphone, UFS wins on every axis except
peak throughput and QD1 latency. In a PC or server, NVMe wins on every axis
except power and cost.

### 6.3 Advantages of UFS

1. **Dramatically faster than eMMC**: 10-200x improvement across all metrics
2. **Full-duplex**: Simultaneous read and write on independent lanes
3. **Command queuing**: Up to 32 commands in flight, enabling device-side
   optimization
4. **Low power**: Multi-level power management (HIBERN8, Deep Sleep) with
   auto-hibernate support
5. **Mature ecosystem**: Broad SoC support from all major mobile chipmakers
6. **Compact form factor**: BGA package fits in smartphones and IoT devices
7. **Secure storage**: RPMB, inline encryption, secure erase
8. **Active development**: New versions every 2 years with 2x bandwidth gains
9. **Standard SCSI interface**: Leverages decades of SCSI tooling and OS support

### 6.4 Disadvantages of UFS

1. **Higher cost than eMMC**: Added manufacturing expense keeps eMMC in budget
   devices
2. **Integration complexity**: Requires M-PHY + UniPro IP blocks (more complex
   than eMMC's parallel bus)
3. **Write Booster exhaustion**: SLC cache provides fast burst writes, but
   sustained writes fall to native TLC/QLC speeds (can be 3-5x slower)
4. **SCSI stack overhead**: The SCSI software stack is heavier than NVMe's
   purpose-built stack, adding latency and code complexity
5. **Non-upgradeable**: Soldered to the motherboard — no user replacement
6. **Thermal throttling**: High-gear operation (HS-G5, 2 lanes) generates
   significant heat during sustained I/O
7. **Write endurance**: Like all NAND flash, limited write-erase cycles
   (mitigated by wear leveling, but still a concern for write-heavy workloads)

### 6.5 Use Cases

| Market | UFS Version | Why UFS |
|--------|------------|---------|
| **Flagship smartphones** | UFS 4.0/4.1 | Maximum performance for apps, camera, AI |
| **Mid-range smartphones** | UFS 3.1/2.2 | Good performance at lower cost |
| **Automotive (ADAS)** | UFS 3.1/4.1 | 4K cameras at 100+ MB/s, fast boot, wide temp range |
| **Automotive (infotainment)** | UFS 3.1 | Fast boot, map databases, OTA updates |
| **IoT edge devices** | UFS 2.2/3.1 | Low-power Deep Sleep, small form factor |
| **Digital cameras** | UFS 3.1/4.0 | Burst photography, 8K video recording |
| **Industrial** | UFS 2.2/3.1 | Edge AI, security cameras, data logging |

Budget phones ($100-200) still use eMMC 5.1 — UFS adds $5-15 to the BOM.

## 7. Summary

UFS represents a generational leap from eMMC, bringing NVMe-class performance
to mobile and embedded systems while maintaining a compact, power-efficient
design.

```
Evolution timeline:

2011        2013        2016        2018        2020        2022        2024        ~2027
 │           │           │           │           │           │           │           │
UFS 1.0 → UFS 2.0 → UFS 2.1 → UFS 3.0 → UFS 3.1 → UFS 4.0 → UFS 4.1 → UFS 5.0
300MB/s   1.2GB/s    1.2GB/s    2.9GB/s    2.9GB/s    5.8GB/s    5.8GB/s   ~10.8GB/s
           +lanes     +HPB       +2x BW     +WB        +MCQ       +Zoned     +PAM4
           +FD        +crypto               +DeepSleep +RPMB v2   +QLC opt   +HS-G6
```

The key insight: UFS's performance gains come not just from faster signaling
(M-PHY gear upgrades), but from architectural innovations at every layer —
full-duplex communication, command queuing, MCQ, Write Booster, and HPB.
Together, these deliver 10-200x improvement over eMMC while keeping power
consumption lower.
