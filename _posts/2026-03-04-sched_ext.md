## Background: Linux CPU Scheduling

The Linux kernel uses a modular scheduling architecture based on **scheduling
classes**. Each class implements a set of operations for managing tasks, and
the classes are arranged in a strict priority hierarchy:

| Priority | Class | Purpose |
|----------|-------|---------|
| Highest | `stop_sched_class` | Per-CPU stopper tasks (migration, etc.) |
| | `dl_sched_class` | SCHED_DEADLINE — earliest-deadline-first |
| | `rt_sched_class` | SCHED_FIFO / SCHED_RR — real-time |
| | **`ext_sched_class`** | **SCHED_EXT — BPF-defined scheduling** |
| | `fair_sched_class` | SCHED_NORMAL / SCHED_BATCH / SCHED_IDLE — CFS |
| Lowest | `idle_sched_class` | Per-CPU idle tasks |

When a CPU needs the next task to run, it walks this chain from highest to
lowest priority until it finds a runnable task. The **ext** class sits between
real-time and fair, meaning BPF-scheduled tasks preempt CFS tasks but yield
to RT and deadline tasks.

Each scheduling class implements a common interface: `enqueue_task`,
`dequeue_task`, `pick_next_task`, `put_prev_task`, and others. The scheduler
core calls into whichever class owns a given task. This is the extension
point that sched_ext exploits — it implements all these class operations
but delegates the actual policy decisions to BPF programs.

## Motivation: Why sched_ext?

### The Problem

Traditional kernel scheduler development has a painfully slow iteration
cycle:

1. Modify C code in the kernel
2. Recompile (minutes to hours)
3. Reboot into the new kernel
4. Test, find a bug, repeat

A scheduling policy bug can hang or crash the entire system, requiring a
hard reboot. Experimentation is risky, deployment is all-or-nothing, and
the barrier to entry is extremely high.

Meanwhile, different workloads have fundamentally different scheduling needs:

- **Latency-sensitive** (gaming, audio): minimize tail latency
- **Throughput-oriented** (batch compute, CI): maximize CPU utilization
- **Cooperative pairs** (io_uring + userspace): minimize handoff latency
- **Container-dense** (cloud): fair sharing across many cgroups

CFS is a good general-purpose scheduler, but no single policy can be
optimal for all these use cases.

### The sched_ext Solution

sched_ext addresses these problems by allowing scheduling policies to be
implemented as BPF programs that can be **loaded and unloaded at runtime**:

- **Rapid iteration**: change policy, recompile a small BPF program, reload
  — no reboot needed
- **Safety**: if the BPF scheduler crashes, stalls, or misbehaves, the
  kernel automatically falls back to CFS
- **Accessibility**: scheduling researchers and application developers can
  experiment without deep kernel expertise
- **Specialization**: deploy workload-specific schedulers on production
  systems, swap them live
- **Debuggability**: structured error reporting, debug dumps, and
  tracepoints built in

The key design insight is that **system integrity must be maintained no
matter what the BPF scheduler does**. The kernel retains authority to:

- Revert to CFS at any time (SysRq-S, error detection, watchdog timeout)
- Reject invalid CPU selections
- Enforce task starvation timeouts
- Guarantee forward progress via the bypass mechanism

## Design Overview

### Architecture

```
 ┌─────────────────────────────────────────────┐
 │              Userspace daemon                │
 │  (loads BPF, reads stats, handles signals)   │
 └──────────┬──────────────────────────────────┘
            │  BPF skeleton (load/attach)
 ───────────┼──────────────────────────────────
            │  Kernel
 ┌──────────▼──────────────────────────────────┐
 │         BPF Scheduler Programs               │
 │  (struct_ops implementing sched_ext_ops)      │
 │                                               │
 │  ops.select_cpu()  ops.enqueue()              │
 │  ops.dispatch()    ops.running()   ...        │
 └──────────┬──────────────────────────────────┘
            │ callbacks
 ┌──────────▼──────────────────────────────────┐
 │         sched_ext core (ext.c)               │
 │                                               │
 │  ┌──────┐  ┌──────┐      ┌──────────────┐   │
 │  │Local │  │Local │ ...  │   Global     │   │
 │  │DSQ 0 │  │DSQ 1 │      │    DSQ       │   │
 │  └──────┘  └──────┘      └──────────────┘   │
 │       ┌──────────────────────────┐           │
 │       │   Custom DSQs (BPF)     │           │
 │       └──────────────────────────┘           │
 └──────────┬──────────────────────────────────┘
            │
 ┌──────────▼──────────────────────────────────┐
 │     Scheduler Core (blk-mq style per-CPU)    │
 │     pick_next_task → context switch          │
 └─────────────────────────────────────────────┘
```

### Key Concepts

1. **struct_ops BPF programs**: The BPF scheduler is a collection of BPF
   programs attached to `struct sched_ext_ops` via BPF struct_ops. Each
   callback handles a specific scheduling event.

2. **Dispatch Queues (DSQs)**: The intermediary between BPF scheduling
   decisions and the kernel's task execution. Tasks are "inserted" into
   DSQs, then "moved" to a CPU's local DSQ for execution.

3. **Two-phase dispatch**: Tasks are first queued by `ops.enqueue()`, then
   dispatched to CPUs by `ops.dispatch()`. This separation allows flexible
   scheduling algorithms while keeping the core simple.

4. **Bypass mode**: A fallback FIFO scheduler that activates when the BPF
   scheduler fails, ensuring forward progress under all circumstances.

## Dispatch Queues (DSQs)

DSQs are the central abstraction in sched_ext. They bridge the BPF
scheduler's decisions and the kernel's per-CPU execution model.

### Built-in DSQs

| DSQ | ID | Description |
|-----|----|-------------|
| `SCX_DSQ_LOCAL` | `0x8000000000000002` | Per-CPU local DSQ. A CPU always runs the first task from its own local DSQ. FIFO only. |
| `SCX_DSQ_GLOBAL` | `0x8000000000000001` | Global DSQ, internally split per-NUMA-node for scalability. FIFO only. Consumed automatically when local DSQ is empty. |
| `SCX_DSQ_LOCAL_ON \| cpu` | `0xC000000000000000 \| cpu` | Target a specific CPU's local DSQ. Allows cross-CPU dispatch. |
| `SCX_DSQ_BYPASS` | `0x8000000000000003` | Per-CPU bypass DSQ used during bypass mode. Not directly used by BPF schedulers. |

### Custom DSQs

BPF schedulers can create arbitrary DSQs via `scx_bpf_create_dsq()`:

- IDs must be less than 2^63 (bit 63 clear)
- Support both **FIFO** ordering (`scx_bpf_dsq_insert()`) and **vtime
  priority queue** ordering (`scx_bpf_dsq_insert_vtime()`)
- Stored in a kernel hash table (`sch->dsq_hash`)
- Must be destroyed with `scx_bpf_destroy_dsq()` when no longer needed

### DSQ Ordering

- **FIFO**: Tasks execute in insertion order. Use `scx_bpf_dsq_insert()`
  with `SCX_ENQ_HEAD` to insert at the front.
- **Priority (vtime)**: Tasks ordered by `p->scx.dsq_vtime` in a red-black
  tree. Lower vtime = higher priority. Use `scx_bpf_dsq_insert_vtime()`.
- Built-in DSQs (`SCX_DSQ_LOCAL`, `SCX_DSQ_GLOBAL`) support FIFO only.
  Custom DSQs support both modes.

### DSQ Task Flow

```
  BPF scheduler decision
         │
         ▼
  scx_bpf_dsq_insert(p, dsq_id, ...)    ← "insert" into any DSQ
         │
         ▼
  ┌─────────────────┐
  │   Custom DSQ    │  or  Global DSQ
  └────────┬────────┘
           │
           ▼
  scx_bpf_dsq_move_to_local(dsq_id)     ← "move" to local DSQ
           │
           ▼
  ┌─────────────────┐
  │ CPU Local DSQ   │
  └────────┬────────┘
           │
           ▼
     Task executes
```

## The Scheduling Cycle

When a task wakes up or needs to be scheduled, sched_ext follows this
sequence:

### Step 1: CPU Selection — `ops.select_cpu()`

The first callback invoked when a task wakes up. It serves two purposes:

- **CPU hint**: suggest which CPU should run the task (not binding — the
  core may override)
- **Idle wakeup**: the selected CPU is woken from idle if necessary

**Fast path optimization**: if `ops.select_cpu()` calls
`scx_bpf_dsq_insert()` to insert the task into `SCX_DSQ_LOCAL`, the task
is placed directly on the returned CPU's local DSQ, skipping
`ops.enqueue()` entirely. This is the common fast path for idle CPU wakeups.

### Step 2: Enqueue — `ops.enqueue()`

Called when the task wasn't directly dispatched in `ops.select_cpu()`. The
BPF scheduler must decide what to do with the task:

- Insert into `SCX_DSQ_GLOBAL` for simple global scheduling
- Insert into a custom DSQ for later dispatch
- Insert into `SCX_DSQ_LOCAL` or `SCX_DSQ_LOCAL_ON | cpu` for immediate
  placement
- Queue internally in BPF data structures (the task becomes invisible to
  the kernel until dispatched)

If `ops.enqueue()` doesn't dispatch the task, it will stall, eventually
triggering the watchdog.

### Step 3: Dispatch — `ops.dispatch()`

Called when a CPU's local DSQ is empty and the global DSQ has no tasks for
it. The BPF scheduler populates the local DSQ using:

- `scx_bpf_dsq_insert()` — insert a task (up to `ops.dispatch_max_batch`,
  default 32)
- `scx_bpf_dsq_move_to_local()` — move a task from a custom DSQ to the
  local DSQ

### Step 4: Task Selection

After `ops.dispatch()` returns, the kernel:

1. If local DSQ has tasks → run the first one
2. Else, try moving from global DSQ → if found, run it
3. Else, if `ops.dispatch()` dispatched any tasks → retry from step 3
4. Else, if previous task is still runnable (and `SCX_OPS_ENQ_LAST` not
   set) → keep running it
5. Else → go idle

## Task Lifecycle

The complete lifecycle of a task under sched_ext:

```
  Task created (fork/exec)
         │
         ▼
  ops.init_task()          ← Allocate per-task resources
         │
         ▼
  ops.enable()             ← Task enters SCHED_EXT
         │
         ▼
  ┌──────────────── scheduling loop ────────────────┐
  │                                                  │
  │   ops.select_cpu()     ← Pick target CPU         │
  │         │                                        │
  │         ▼                                        │
  │   ops.runnable()       ← Task is ready to run    │
  │         │                                        │
  │    ┌────┴──── while runnable ──────────┐         │
  │    │                                   │         │
  │    │  ops.enqueue()   ← Queue task     │         │
  │    │  ops.dispatch()  ← Populate DSQ   │         │
  │    │  ops.running()   ← Start execute  │         │
  │    │       │                           │         │
  │    │  ┌────┴── while slice > 0 ──┐     │         │
  │    │  │  ops.tick()   ← 1/HZ     │     │         │
  │    │  └──────────────────────────┘     │         │
  │    │       │                           │         │
  │    │  ops.stopping()  ← Stop execute   │         │
  │    │                                   │         │
  │    └───────────────────────────────────┘         │
  │         │                                        │
  │   ops.quiescent()      ← Task sleeps             │
  │                                                  │
  └──────────────────────────────────────────────────┘
         │
         ▼
  ops.disable()            ← Task leaves SCHED_EXT
         │
         ▼
  ops.exit_task()          ← Free per-task resources
```

## The `sched_ext_ops` Interface

All callbacks are optional. The only mandatory field is `ops.name`.

### Task Selection and Dispatch

#### `ops.select_cpu(p, prev_cpu, wake_flags) → s32`

First callback on task wakeup. Returns a CPU number as a scheduling hint.
Can directly dispatch via `scx_bpf_dsq_insert()` to skip `ops.enqueue()`.
The returned CPU is woken from idle if necessary. This is the primary
fast-path optimization point.

**Parameters:**
- `p`: the waking task
- `prev_cpu`: CPU the task last ran on
- `wake_flags`: bitmask (includes `SCX_WAKE_*` flags)

#### `ops.enqueue(p, enq_flags) → void`

Called to enqueue a task that wasn't directly dispatched in
`ops.select_cpu()`. Must eventually insert the task into a DSQ or the task
will stall (triggering the watchdog).

**Parameters:**
- `p`: task to enqueue
- `enq_flags`: bitmask of `SCX_ENQ_*` flags (see [Enqueue Flags](#enqueue-flags))

#### `ops.dequeue(p, deq_flags) → void`

Removes a task from the BPF scheduler. Called when a task that was queued
on the BPF side (not in a kernel DSQ) needs to be dequeued. Optional —
the core handles spurious dispatches gracefully.

#### `ops.dispatch(cpu, prev) → void`

Called when a CPU's local DSQ is empty and needs work. Can dispatch tasks
using `scx_bpf_dsq_insert()` (up to `ops.dispatch_max_batch` at a time) or
move tasks from custom DSQs using `scx_bpf_dsq_move_to_local()`.

**Parameters:**
- `cpu`: the CPU needing work
- `prev`: the previous task (non-NULL if its slice expired and it's still
  runnable but not yet re-enqueued)

### Execution State

#### `ops.running(p) → void`

Task is about to start executing on its assigned CPU.

#### `ops.stopping(p, runnable) → void`

Task is being taken off the CPU. `runnable` indicates whether the task
remains runnable (true) or is going to sleep (false). Common place to
update vtime accounting.

#### `ops.tick(p) → void`

Called every `1/HZ` seconds while `p` is running. Setting
`p->scx.slice = 0` triggers immediate dispatch (preemption).

#### `ops.runnable(p, enq_flags) → void`

Task has become runnable (wakeup, fork, migration). Useful for maintaining
runnable task counts per CPU or domain.

#### `ops.quiescent(p, deq_flags) → void`

Task is no longer runnable (going to sleep). Paired with `ops.runnable()`.

#### `ops.yield(from, to) → bool`

A task is yielding the CPU. `to` may be NULL (yield to any) or a specific
task (directed yield). Returns true if the yield can be fulfilled.

### Task Properties

#### `ops.set_weight(p, weight) → void`

Task's weight has changed. Weight range is `[1, 10000]`.

#### `ops.set_cpumask(p, cpumask) → void`

Task's CPU affinity has changed. The scheduler may need to migrate or
re-evaluate placement.

### CPU Management

#### `ops.update_idle(cpu, idle) → void`

CPU idle state changed. **Important**: implementing this callback disables
the built-in idle CPU tracking unless `SCX_OPS_KEEP_BUILTIN_IDLE` is set
in `ops.flags`.

#### `ops.cpu_acquire(cpu, args) → void`

A CPU is becoming available to the BPF scheduler (returned from a higher-
priority scheduling class like RT).

#### `ops.cpu_release(cpu, args) → void`

A CPU is being taken from the BPF scheduler by a higher-priority class.
`args->reason` indicates why: `SCX_CPU_PREEMPT_RT`, `SCX_CPU_PREEMPT_DL`,
`SCX_CPU_PREEMPT_STOP`, or `SCX_CPU_PREEMPT_UNKNOWN`.

#### `ops.cpu_online(cpu) → void` / `ops.cpu_offline(cpu) → void`

CPU hotplug events. If not implemented and a hotplug event occurs, the
scheduler exits with `SCX_ECODE_ACT_RESTART | SCX_ECODE_RSN_HOTPLUG`.

### Task Lifecycle

#### `ops.init_task(p, args) → s32`

Initialize per-task state for BPF scheduling. Called once per task on fork
or when the scheduler is loaded. This is a **sleepable** context — it can
allocate memory, take mutexes, etc. Return 0 on success, negative errno
on failure.

`args->fork` indicates whether this is a new task (true) or an existing
task being initialized on scheduler load (false).

#### `ops.exit_task(p, args) → void`

Free per-task state. `args->cancelled` is true if `ops.init_task()`
succeeded but the task never actually ran under the scheduler.

#### `ops.enable(p) → void`

BPF scheduling is now active for this task. Good place to initialize
vtime. Always paired with a later `ops.disable()`.

#### `ops.disable(p) → void`

BPF scheduling is being disabled for this task (policy change or scheduler
unload).

### Core Scheduling (SMT)

#### `ops.core_sched_before(a, b) → bool`

For core-scheduling on SMT systems. Returns true if task `a` should be
preferred over task `b` for simultaneous execution ordering.

### Debugging

#### `ops.dump(ctx) → void`

Generate global scheduler debug output. Called on error or SysRq-D.
Use `scx_bpf_dump()` to produce output.

#### `ops.dump_cpu(ctx, cpu, idle) → void`

Generate per-CPU debug output.

#### `ops.dump_task(ctx, p) → void`

Generate per-task debug output.

### Cgroup Operations (requires `CONFIG_EXT_GROUP_SCHED`)

#### `ops.cgroup_init(cgrp, args) → s32`

Initialize a cgroup for BPF scheduling. `args` contains weight and
bandwidth parameters. Return 0 on success, negative errno on failure.

#### `ops.cgroup_exit(cgrp) → void`

Clean up cgroup state.

#### `ops.cgroup_prep_move(p, from, to) → s32`

Prepare a task for migration between cgroups. May block. Return 0 to
allow, negative errno to abort the move.

#### `ops.cgroup_move(p, from, to) → void`

Commit a cgroup migration. The task is dequeued during this operation.

#### `ops.cgroup_cancel_move(p, from, to) → void`

Cancel a prepared cgroup migration (called if `cgroup_prep_move()`
succeeded but the move subsequently failed).

#### `ops.cgroup_set_weight(cgrp, weight) → void`

Cgroup's `cpu.weight` changed. Weight range `[1, 10000]`.

#### `ops.cgroup_set_bandwidth(cgrp, period_us, quota_us, burst_us) → void`

Cgroup's `cpu.max` bandwidth parameters changed.

#### `ops.cgroup_set_idle(cgrp, idle) → void`

Cgroup's idle state changed.

### Initialization and Cleanup

#### `ops.init() → s32`

Initialize the BPF scheduler. This is **sleepable** — create custom DSQs,
allocate resources, set up data structures here. Return 0 on success,
negative errno on failure.

#### `ops.exit(info) → void`

Clean up the BPF scheduler. Called on unload, error, or `ops.init()`
failure. `info` is a `struct scx_exit_info` containing:
- `kind`: exit reason (`SCX_EXIT_*`)
- `exit_code`: encoded exit code
- `reason`, `msg`: human-readable strings
- `dump`: debug dump output

### Configuration Fields

| Field | Default | Description |
|-------|---------|-------------|
| `dispatch_max_batch` | 32 | Max tasks `ops.dispatch()` can dispatch per invocation |
| `flags` | 0 | `SCX_OPS_*` flags (see [Ops Flags](#ops-flags)) |
| `timeout_ms` | 30000 | Watchdog timeout in milliseconds (max 30s) |
| `exit_dump_len` | 32768 | Debug dump buffer length |
| `hotplug_seq` | 0 | If non-zero, loading fails if it doesn't match the kernel's hotplug sequence counter |
| `name` | (required) | Scheduler name, max 128 chars |

## Flags and Constants

### Ops Flags (`SCX_OPS_*`)

Set in `ops.flags` to configure scheduler behavior:

| Flag | Description |
|------|-------------|
| `SCX_OPS_KEEP_BUILTIN_IDLE` | Keep built-in idle CPU tracking even when `ops.update_idle()` is implemented |
| `SCX_OPS_ENQ_LAST` | Pass `SCX_ENQ_LAST` to `ops.enqueue()` when a task is the only one available on a CPU |
| `SCX_OPS_ENQ_EXITING` | Don't auto-dispatch exiting tasks to local DSQ; route through `ops.enqueue()` |
| `SCX_OPS_SWITCH_PARTIAL` | Only schedule `SCHED_EXT` tasks; `SCHED_NORMAL`/`SCHED_BATCH`/`SCHED_IDLE` stay on CFS |
| `SCX_OPS_ENQ_MIGRATION_DISABLED` | Route migration-disabled tasks through `ops.enqueue()` instead of auto-dispatching locally |
| `SCX_OPS_ALLOW_QUEUED_WAKEUP` | Enable `ttwu_queue` optimization; `ops.enqueue()` may be called without prior `ops.select_cpu()` |
| `SCX_OPS_BUILTIN_IDLE_PER_NODE` | Use per-NUMA-node idle cpumasks instead of a single global mask |

### Enqueue Flags (`SCX_ENQ_*`)

Passed to `ops.enqueue()` and `scx_bpf_dsq_insert()`:

| Flag | Value | Description |
|------|-------|-------------|
| `SCX_ENQ_WAKEUP` | from core | Task is waking up |
| `SCX_ENQ_HEAD` | from core | Insert at head of queue |
| `SCX_ENQ_CPU_SELECTED` | from core | CPU was selected by `ops.select_cpu()` |
| `SCX_ENQ_PREEMPT` | `1 << 32` | Preempt current task on target CPU, clear its slice |
| `SCX_ENQ_REENQ` | `1 << 40` | Task re-enqueued via `scx_bpf_reenqueue_local()` |
| `SCX_ENQ_LAST` | `1 << 41` | Only task on CPU (requires `SCX_OPS_ENQ_LAST`) |

### Slice Constants

| Constant | Value | Description |
|----------|-------|-------------|
| `SCX_SLICE_DFL` | 20ms | Default time slice |
| `SCX_SLICE_BYPASS` | 5ms | Slice used during bypass mode |
| `SCX_SLICE_INF` | `U64_MAX` | Infinite slice; implies tickless (nohz) |

### Other Constants

| Constant | Value | Description |
|----------|-------|-------------|
| `SCX_DSP_DFL_MAX_BATCH` | 32 | Default `dispatch_max_batch` |

## Safety and Error Handling

### The Watchdog

sched_ext includes a watchdog that detects runnable task starvation. If
any task remains runnable for longer than `ops.timeout_ms` (default 30s)
without being scheduled, the watchdog fires. Each task's
`scx.runnable_at` timestamp is checked periodically.

When the watchdog fires:
1. Debug information is dumped (via `ops.dump`, `ops.dump_cpu`,
   `ops.dump_task`)
2. The BPF scheduler is aborted
3. All tasks revert to CFS scheduling

### Bypass Mode

Bypass mode is a fallback FIFO scheduler that guarantees forward progress:

**When activated:**
- `ops.select_cpu()`, `ops.enqueue()`, `ops.dispatch()` are all ignored
- Tasks are placed in per-CPU `bypass_dsq` queues with 5ms slices
- A load balancer (`bypass_lb_cpu()`) periodically redistributes tasks
  across CPUs to prevent starvation

**Activation triggers:**
- Scheduler error or `scx_bpf_error()` call
- Watchdog timeout
- During scheduler load/unload transitions
- SysRq-S key sequence

Bypass is reference-counted (`scx_bypass_depth`), supporting nested
activation (e.g., error during shutdown).

### SysRq Integration

- **SysRq-S**: Abort the BPF scheduler and revert to CFS
- **SysRq-D**: Trigger a debug dump without terminating the scheduler

### Error Reporting

When the scheduler exits (normally or due to error), `ops.exit()` receives
a `struct scx_exit_info` containing:
- `kind`: categorized exit reason (see `SCX_EXIT_*`)
- `exit_code`: encoded exit code with system and user fields
- `reason`, `msg`: human-readable descriptions
- `dump`: debug dump output

The exit code format uses bitfields:
- Bits `[31..0]`: user-defined exit codes
- Bits `[47..32]`: system reasons (e.g., `SCX_ECODE_RSN_HOTPLUG`)
- Bits `[63..48]`: system actions (e.g., `SCX_ECODE_ACT_RESTART`)

The `SCX_ECODE_ACT_RESTART` action tells the userspace daemon to reload
the scheduler automatically (useful for handling CPU hotplug events).

### Sysfs Interface

Runtime status is available at:

```
/sys/kernel/sched_ext/state         # "enabled" or "disabled"
/sys/kernel/sched_ext/root/ops      # name of loaded scheduler
/sys/kernel/sched_ext/enable_seq    # monotonic load counter
```

## Cgroup Integration

When `CONFIG_EXT_GROUP_SCHED` is enabled, sched_ext can participate in
the kernel's cgroup CPU controller. This allows BPF schedulers to
implement hierarchical CPU bandwidth and weight policies.

### How It Works

1. On scheduler load, `ops.cgroup_init()` is called for every existing
   cgroup, passing weight and bandwidth parameters.
2. When tasks move between cgroups, the `prep_move → move` or
   `prep_move → cancel_move` sequence executes.
3. When cgroup properties change (`cpu.weight`, `cpu.max`), the
   corresponding callbacks fire.
4. On scheduler unload, `ops.cgroup_exit()` is called for each cgroup.

### Cgroup Weight

The `cpu.weight` value maps to the range `[1, 10000]` passed to
`ops.cgroup_set_weight()`. Schedulers typically use this to compute
compound weights through the cgroup hierarchy (see `scx_flatcg` for an
example of flattening hierarchical weights).

---

## CPU Hotplug

sched_ext tracks CPU hotplug events through a global sequence counter
(`scx_hotplug_seq`). When a CPU comes online or goes offline:

1. `scx_hotplug_seq` is incremented
2. `ops.cpu_online()` or `ops.cpu_offline()` is called

If these callbacks are **not implemented**, the scheduler is automatically
exited with `SCX_ECODE_ACT_RESTART | SCX_ECODE_RSN_HOTPLUG`, signaling the
userspace daemon to reload. The `ops.hotplug_seq` field can be set during
load to detect races: if it doesn't match `scx_hotplug_seq`, loading fails.

## Writing a Scheduler: Practical Guide

### Project Structure

A sched_ext scheduler consists of two files:

```
my_scheduler.bpf.c    ← BPF programs (runs in kernel)
my_scheduler.c        ← Userspace daemon (loads BPF, monitors)
```

### BPF Side Template

```c
#include <scx/common.bpf.h>

char _license[] SEC("license") = "GPL";

/* Configuration (set from userspace before loading) */
const volatile u64 slice_ns = SCX_SLICE_DFL;

/* Exit info for communicating exit reason to userspace */
UEI_DEFINE(uei);

/* Custom DSQ ID */
#define SHARED_DSQ 0

/*
 * CPU selection: try to find an idle CPU and dispatch directly.
 * This is the fast path — skips ops.enqueue() entirely.
 */
s32 BPF_STRUCT_OPS(my_select_cpu, struct task_struct *p,
                   s32 prev_cpu, u64 wake_flags)
{
    bool is_idle = false;
    s32 cpu;

    cpu = scx_bpf_select_cpu_dfl(p, prev_cpu, wake_flags, &is_idle);
    if (is_idle)
        scx_bpf_dsq_insert(p, SCX_DSQ_LOCAL, slice_ns, 0);

    return cpu;
}

/*
 * Enqueue: called when select_cpu() didn't directly dispatch.
 * Insert into our shared DSQ for later dispatch.
 */
void BPF_STRUCT_OPS(my_enqueue, struct task_struct *p, u64 enq_flags)
{
    scx_bpf_dsq_insert(p, SHARED_DSQ, slice_ns, enq_flags);
}

/*
 * Dispatch: CPU is idle, move a task from our DSQ to the local DSQ.
 */
void BPF_STRUCT_OPS(my_dispatch, s32 cpu, struct task_struct *prev)
{
    scx_bpf_dsq_move_to_local(SHARED_DSQ);
}

/*
 * Init: create our custom DSQ. This is sleepable.
 */
s32 BPF_STRUCT_OPS_SLEEPABLE(my_init)
{
    return scx_bpf_create_dsq(SHARED_DSQ, -1);
}

/*
 * Exit: record exit info for the userspace daemon.
 */
void BPF_STRUCT_OPS(my_exit, struct scx_exit_info *ei)
{
    UEI_RECORD(uei, ei);
}

/* Register the scheduler with struct_ops */
SCX_OPS_DEFINE(my_ops,
    .select_cpu         = (void *)my_select_cpu,
    .enqueue            = (void *)my_enqueue,
    .dispatch           = (void *)my_dispatch,
    .init               = (void *)my_init,
    .exit               = (void *)my_exit,
    .name               = "my_scheduler");
```

### Userspace Side Template

```c
#include <scx/common.h>
#include "my_scheduler.bpf.skel.h"

static volatile int exit_req;

static void sigint_handler(int sig) { exit_req = 1; }

int main(int argc, char **argv)
{
    struct my_scheduler *skel;
    struct bpf_link *link;
    __u64 ecode;

    signal(SIGINT, sigint_handler);
    signal(SIGTERM, sigint_handler);

restart:
    /* Open and verify BPF programs */
    skel = SCX_OPS_OPEN(my_ops, my_scheduler);

    /* Configure (write to BPF rodata before loading) */
    skel->rodata->slice_ns = 5000000;  /* 5ms */

    /* Load BPF programs into kernel */
    SCX_OPS_LOAD(skel, my_ops, my_scheduler, uei);

    /* Attach — scheduler is now active */
    link = SCX_OPS_ATTACH(skel, my_ops, my_scheduler);

    /* Monitor loop */
    while (!exit_req && !UEI_EXITED(skel, uei)) {
        /* Read stats, print status, etc. */
        sleep(1);
    }

    /* Detach and cleanup */
    bpf_link__destroy(link);
    ecode = UEI_REPORT(skel, uei);
    my_scheduler__destroy(skel);

    /* Auto-restart on hotplug */
    if (UEI_ECODE_RESTART(ecode))
        goto restart;

    return 0;
}
```

### Common Patterns

#### Weighted vtime Scheduling (Fairness)

```c
static u64 vtime_now;

void BPF_STRUCT_OPS(my_running, struct task_struct *p)
{
    /* Advance global vtime to at least the running task's vtime */
    if (time_before(vtime_now, p->scx.dsq_vtime))
        vtime_now = p->scx.dsq_vtime;
}

void BPF_STRUCT_OPS(my_stopping, struct task_struct *p, bool runnable)
{
    /* Charge consumed time, inversely weighted by task priority */
    p->scx.dsq_vtime +=
        (slice_ns - p->scx.slice) * 100 / p->scx.weight;
}

void BPF_STRUCT_OPS(my_enable, struct task_struct *p)
{
    /* Initialize new tasks to the current vtime */
    p->scx.dsq_vtime = vtime_now;
}

void BPF_STRUCT_OPS(my_enqueue, struct task_struct *p, u64 enq_flags)
{
    /* Insert with vtime ordering — lower vtime = higher priority */
    scx_bpf_dsq_insert_vtime(p, SHARED_DSQ, slice_ns,
                              p->scx.dsq_vtime, enq_flags);
}
```

#### Per-Task BPF Storage

```c
struct task_ctx {
    u64 total_runtime;
};

struct {
    __uint(type, BPF_MAP_TYPE_TASK_STORAGE);
    __uint(map_flags, BPF_F_NO_PREALLOC);
    __type(key, int);
    __type(value, struct task_ctx);
} task_ctx_stor SEC(".maps");

s32 BPF_STRUCT_OPS(my_init_task, struct task_struct *p,
                   struct scx_init_task_args *args)
{
    struct task_ctx *tctx;

    tctx = bpf_task_storage_get(&task_ctx_stor, p, 0,
                                BPF_LOCAL_STORAGE_GET_F_CREATE);
    if (!tctx)
        return -ENOMEM;
    tctx->total_runtime = 0;
    return 0;
}
```

#### Preemption via Tick

```c
void BPF_STRUCT_OPS(my_tick, struct task_struct *p)
{
    /* Check if a higher-priority task is waiting */
    if (should_preempt(p))
        p->scx.slice = 0;  /* triggers immediate dispatch */
}
```

## Example Schedulers

The `tools/sched_ext/` directory contains reference implementations:

### scx_simple — Minimal Global Scheduler

The starting point for understanding sched_ext. Implements a global FIFO
or weighted vtime scheduler with statistics tracking. Demonstrates the
basic `select_cpu → enqueue → dispatch` flow.

**Key lesson**: a working scheduler can be just ~50 lines of BPF code.

### scx_qmap — Feature Showcase

Five-level priority FIFO queues using `BPF_MAP_TYPE_QUEUE`. Demonstrates:
- BPF-side task queueing (tasks stored in BPF maps, not kernel DSQs)
- Per-task storage with sleepable allocation
- `ops.cpu_release()` for handling RT preemption
- Core-scheduling ordering
- CPU performance scaling

**Key lesson**: tasks can be queued entirely on the BPF side, not just in
kernel DSQs.

### scx_central — Central CPU Scheduler

All scheduling decisions made from a single designated CPU. Other CPUs
run with infinite slices (tickless/nohz). A BPF timer on the central CPU
periodically kicks other CPUs to preempt long-running tasks.

**Key lesson**: demonstrates `SCX_DSQ_LOCAL_ON | cpu` for cross-CPU
dispatch and `SCX_KICK_PREEMPT` for remote preemption.

### scx_pair — Core-Scheduling Demo

Ensures SMT sibling CPUs always run tasks from the same cgroup. Uses
`ops.cpu_acquire()` / `ops.cpu_release()` for CPU pair synchronization.

**Key lesson**: how to implement group-aware scheduling with CPU pairing.

### scx_flatcg — Flattened Cgroup Scheduler

Implements hierarchical cgroup weight control by flattening the cgroup
tree into compound weights. Claims ~10% throughput improvement over CFS
for nested cgroup workloads.

**Key lesson**: demonstrates all cgroup callbacks and hierarchical weight
computation.

### scx_userland — Kernel/Userspace Split

Routes affinity-constrained tasks through kernel-side direct dispatch,
while other tasks are queued to userspace via `BPF_MAP_TYPE_QUEUE` for
scheduling decisions.

**Key lesson**: scheduling decisions can be split between BPF and
userspace.

### scx_sdt — Arena Memory Demo

Uses BPF arena memory (shared BPF/userspace memory) for per-task data.
Demonstrates the newer arena-based allocator pattern.

### scx_cpu0 — Bypass Testing

Routes all tasks to CPU 0, useful for testing bypass load balancer
behavior under extreme queue concentration.

## Building and Running

### Kernel Configuration

Enable these options:

```
CONFIG_BPF=y
CONFIG_SCHED_CLASS_EXT=y
CONFIG_BPF_SYSCALL=y
CONFIG_BPF_JIT=y
CONFIG_DEBUG_INFO_BTF=y
CONFIG_BPF_JIT_ALWAYS_ON=y

```

### Building Example Schedulers

```bash
make -j$(nproc) -C tools/sched_ext
```

Binaries are placed in `tools/sched_ext/build/bin/`.

### Running

```bash
# Start a scheduler (runs as a daemon until Ctrl-C or error)
sudo tools/sched_ext/build/bin/scx_simple

# Check status
cat /sys/kernel/sched_ext/state          # "enabled"
cat /sys/kernel/sched_ext/root/ops       # "simple"
cat /sys/kernel/sched_ext/enable_seq     # monotonic counter

# Check if a task is using sched_ext
grep ext /proc/self/sched

# Force-stop the scheduler
# SysRq-S (or echo s > /proc/sysrq-trigger)

# Debug dump without stopping
# SysRq-D (or echo d > /proc/sysrq-trigger)
```

### Detailed State Inspection

```bash
# Using the included drgn script
sudo tools/sched_ext/scx_show_state.py
```

### ABI Stability

The sched_ext APIs (ops callbacks, kfuncs, constants) have **no stability
guarantees**. They may change between kernel versions. While the
maintainers aim for reasonable stability, BPF schedulers should be
prepared to adapt to interface changes.
