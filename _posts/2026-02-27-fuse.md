## 1. What is FUSE?

FUSE (Filesystem in Userspace) lets you implement a fully functional filesystem as
a regular userspace program. Instead of writing kernel code, you write a daemon that
responds to filesystem operations — and the kernel handles everything else.

### Why FUSE Exists

Writing kernel filesystems is hard: you need to understand VFS internals, handle
locking and memory management carefully, and a bug means a kernel panic. FUSE removes
all of that. Your filesystem runs as a normal process — it can be written in C, Python,
Go, Rust, or any language. If it crashes, only the mount fails, not the whole system.

Real-world FUSE filesystems include:
- **sshfs** — mount remote directories over SSH
- **s3fs** — mount Amazon S3 buckets as local directories
- **ntfs-3g** — the standard Linux NTFS driver
- **GlusterFS** — distributed storage
- **Android's sdcardfs replacement** — uses FUSE with passthrough for app storage

### The Three Components

```
┌─────────────────────────────────────────────────────────────────┐
│                        Userspace                                │
│                                                                 │
│  ┌──────────────┐     ┌──────────────────────────────────────┐  │
│  │  fusermount3  │     │        FUSE Daemon (your code)       │  │
│  │              │     │                                      │  │
│  │  • Mounts    │     │  • Implements filesystem callbacks   │  │
│  │  • Validates │     │  • Reads requests from /dev/fuse     │  │
│  │  • Handles   │     │  • Writes replies back               │  │
│  │    security  │     │  • Uses libfuse (high or low-level)  │  │
│  └──────┬───────┘     └──────────────┬───────────────────────┘  │
│         │                            │                          │
│         │ mount()                    │ read()/write()           │
├─────────┼────────────────────────────┼──────────────────────────┤
│         │            Kernel          │                          │
│         │                            ▼                          │
│  ┌──────▼──────────────────────────────────────────────────┐    │
│  │                    FUSE Kernel Module                    │    │
│  │                    (fs/fuse/*.c)                         │    │
│  │                                                         │    │
│  │  • Registers as a VFS filesystem                        │    │
│  │  • Translates VFS calls → FUSE protocol messages        │    │
│  │  • Manages /dev/fuse communication channel              │    │
│  │  • Handles caching, permissions, inode lifecycle         │    │
│  │  • Presents /dev/ublkb*-style block/file interfaces     │    │
│  └────────────────────────┬────────────────────────────────┘    │
│                           │                                     │
│                    ┌──────▼──────┐                               │
│                    │     VFS     │                               │
│                    └─────────────┘                               │
└─────────────────────────────────────────────────────────────────┘
```

1. **FUSE kernel module** (`fs/fuse/`): Registers with VFS, converts VFS operations
   into FUSE protocol messages, queues them for the daemon.

2. **libfuse** (userspace library): Provides two APIs (high-level and low-level) that
   handle mounting, reading requests from `/dev/fuse`, dispatching to your callbacks,
   and writing replies.

3. **fusermount3**: A setuid helper that performs the actual `mount()` system call.
   It enforces security (adding `nosuid`, `nodev`) and allows unprivileged users to
   mount FUSE filesystems.

## 2. Architecture Overview

### How the Pieces Connect

When a process accesses a FUSE-mounted path (e.g., `cat /mnt/fuse/hello`), the
request follows this journey:

```
Application          Kernel                    FUSE Daemon
    │                  │                          │
    │  read()          │                          │
    ├────────────────► │                          │
    │                  │ VFS dispatches to        │
    │                  │ fuse_file_read_iter()    │
    │                  │                          │
    │                  │ Build FUSE_READ request  │
    │                  │ Queue on fiq->pending    │
    │                  │ Wake fiq->waitq          │
    │                  │                          │
    │                  │  ◄────────────────────── │ read(/dev/fuse)
    │                  │  Copy request to daemon  │
    │                  │                          │
    │                  │                          │ Process request
    │                  │                          │ (read from backend,
    │                  │                          │  memory, network, etc.)
    │                  │                          │
    │                  │  write(/dev/fuse) ──────►│
    │                  │  Copy reply from daemon  │
    │                  │                          │
    │                  │ Complete fuse_req         │
    │                  │ Wake original caller      │
    │                  │                          │
    │  ◄───────────── │                          │
    │  data returned   │                          │
```

### The Request/Reply Protocol at a Glance

Every FUSE operation follows this pattern:

1. **Kernel builds a request**: A `fuse_in_header` (40 bytes) followed by
   operation-specific arguments.
2. **Request is queued**: Added to the input queue (`fiq->pending`), the daemon
   is woken up.
3. **Daemon reads the request**: Via `read()` on `/dev/fuse`, which dequeues and
   copies the request to userspace.
4. **Request moves to processing**: It's tracked in a hash table (`fpq->processing`)
   keyed by the unique ID.
5. **Daemon sends a reply**: Via `write()` on `/dev/fuse`, containing a
   `fuse_out_header` (16 bytes) followed by operation-specific output.
6. **Kernel completes the request**: The original caller (sleeping on
   `req->waitq`) is woken up.

### Connection vs. Mount

FUSE separates the concept of a **connection** (`struct fuse_conn`) from a
**mount** (`struct fuse_mount`). A single connection can serve multiple submounts
(e.g., virtiofs uses this for shared filesystems). The connection holds all
protocol state — negotiated capabilities, request queues, background limits. The
mount holds the superblock and VFS integration.

## 3. The FUSE Protocol

### Message Format

Every FUSE message (in both directions) starts with a fixed header:

**Kernel → Daemon (Request)**:
```c
struct fuse_in_header {       /* 40 bytes */
    uint32_t len;             /* Total message length (header + args) */
    uint32_t opcode;          /* enum fuse_opcode (FUSE_LOOKUP, FUSE_READ, etc.) */
    uint64_t unique;          /* Unique request ID (for matching replies) */
    uint64_t nodeid;          /* Target inode (FUSE_ROOT_ID=1 for root) */
    uint32_t uid;             /* Requesting process's UID */
    uint32_t gid;             /* Requesting process's GID */
    uint32_t pid;             /* Requesting process's PID */
    uint16_t total_extlen;    /* Length of request extensions (in 8-byte units) */
    uint16_t padding;
};
```

**Daemon → Kernel (Reply)**:
```c
struct fuse_out_header {      /* 16 bytes */
    uint32_t len;             /* Total reply length (header + args) */
    int32_t  error;           /* Negative errno, or 0 for success */
    uint64_t unique;          /* Must match the request's unique */
};
```

The `unique` field ties replies to requests. The kernel generates monotonically
increasing unique IDs (from `fiq->reqctr`, which lives inside the `fuse_iqueue`
embedded in `fuse_conn`). When the daemon writes a reply, the kernel looks up the
matching request in `fpq->processing[]` by hashing the unique ID.

### The FUSE_INIT Handshake

Before any filesystem operations can occur, the kernel and daemon must negotiate
protocol version and capabilities:

```
Kernel                              Daemon
  │                                   │
  │  FUSE_INIT                        │
  │  ┌─────────────────────────┐      │
  │  │ major = 7               │      │
  │  │ minor = 45              │      │
  │  │ max_readahead = 131072  │      │
  │  │ flags = (all we support)│      │
  │  └─────────────────────────┘      │
  ├──────────────────────────────────►│
  │                                   │
  │                                   │ Check major version
  │                                   │ Select desired capabilities
  │                                   │
  │  FUSE_INIT reply                  │
  │  ┌─────────────────────────┐      │
  │  │ major = 7               │      │
  │  │ minor = 38              │      │
  │  │ max_readahead = 131072  │      │
  │  │ flags = (what we want)  │      │
  │  │ max_write = 1048576     │      │
  │  │ max_background = 12     │      │
  │  │ congestion_threshold = 9│      │
  │  │ max_pages = 256         │      │
  │  │ time_gran = 1           │      │
  │  └─────────────────────────┘      │
  │◄──────────────────────────────────│
  │                                   │
  │  fc->initialized = 1             │
  │  (filesystem is now usable)       │
```

**Capability negotiation** works by intersection: the kernel offers all capabilities
it supports in `flags`/`flags2`, and the daemon replies with the subset it wants.
Some important capabilities:

| Flag | Effect when negotiated |
|------|----------------------|
| `FUSE_ASYNC_READ` | Kernel can send multiple concurrent reads |
| `FUSE_WRITEBACK_CACHE` | Kernel uses writeback caching (not writethrough) |
| `FUSE_DO_READDIRPLUS` | Enable READDIRPLUS (readdir + lookup combined) |
| `FUSE_PARALLEL_DIROPS` | Allow concurrent directory operations |
| `FUSE_NO_OPEN_SUPPORT` | Kernel supports zero-message opens (daemon returns ENOSYS to skip) |
| `FUSE_PASSTHROUGH` | Enable passthrough I/O to backing files |
| `FUSE_OVER_IO_URING` | Use io_uring transport instead of read/write |
| `FUSE_POSIX_ACL` | Kernel enforces POSIX ACLs |
| `FUSE_CACHE_SYMLINKS` | Cache READLINK results in page cache |
| `FUSE_ATOMIC_O_TRUNC` | Server handles O_TRUNC during OPEN (no separate SETATTR) |
| `FUSE_REQUEST_TIMEOUT` | Enable request timeout detection |

The kernel code for this is in `inode.c`:
- `fuse_new_init()` (~line 1485) builds the INIT request arguments
- `fuse_send_init()` (~line 1548) sends the INIT as a background request
- `process_init_reply()` (~line 1319) processes the reply and configures `fuse_conn`

### Opcodes

The protocol defines 53 opcodes (as of FUSE protocol 7.45). Every VFS operation maps
to one or more opcodes:

```c
enum fuse_opcode {
    FUSE_LOOKUP       =  1,    FUSE_FORGET       =  2,
    FUSE_GETATTR      =  3,    FUSE_SETATTR      =  4,
    FUSE_READLINK     =  5,    FUSE_SYMLINK      =  6,
    /* 7 is unused */
    FUSE_MKNOD        =  8,    FUSE_MKDIR        =  9,
    FUSE_UNLINK       = 10,    FUSE_RMDIR        = 11,
    FUSE_RENAME       = 12,    FUSE_LINK         = 13,
    FUSE_OPEN         = 14,    FUSE_READ         = 15,
    FUSE_WRITE        = 16,    FUSE_STATFS       = 17,
    FUSE_RELEASE      = 18,
    /* 19 is unused */
    FUSE_FSYNC        = 20,    FUSE_SETXATTR     = 21,
    FUSE_GETXATTR     = 22,    FUSE_LISTXATTR    = 23,
    FUSE_REMOVEXATTR  = 24,    FUSE_FLUSH        = 25,
    FUSE_INIT         = 26,    FUSE_OPENDIR      = 27,
    FUSE_READDIR      = 28,    FUSE_RELEASEDIR   = 29,
    FUSE_FSYNCDIR     = 30,    FUSE_GETLK        = 31,
    FUSE_SETLK        = 32,    FUSE_SETLKW       = 33,
    FUSE_ACCESS       = 34,    FUSE_CREATE       = 35,
    FUSE_INTERRUPT    = 36,    FUSE_BMAP         = 37,
    FUSE_DESTROY      = 38,    FUSE_IOCTL        = 39,
    FUSE_POLL         = 40,    FUSE_NOTIFY_REPLY = 41,
    FUSE_BATCH_FORGET = 42,    FUSE_FALLOCATE    = 43,
    FUSE_READDIRPLUS  = 44,    FUSE_RENAME2      = 45,
    FUSE_LSEEK        = 46,    FUSE_COPY_FILE_RANGE = 47,
    FUSE_SETUPMAPPING = 48,    FUSE_REMOVEMAPPING = 49,
    FUSE_SYNCFS       = 50,    FUSE_TMPFILE      = 51,
    FUSE_STATX        = 52,
    FUSE_COPY_FILE_RANGE_64 = 53,

    CUSE_INIT         = 4096,  /* Character device in Userspace */
};
```

### Notifications (Daemon → Kernel)

Unlike normal operations (kernel→daemon), **notifications** flow from the daemon
to the kernel. The daemon writes a message with `unique = 0` and the `error` field
repurposed as a notification code:

```c
enum fuse_notify_code {
    FUSE_NOTIFY_POLL         = 1,  /* Wake poll waiters */
    FUSE_NOTIFY_INVAL_INODE  = 2,  /* Invalidate inode cache */
    FUSE_NOTIFY_INVAL_ENTRY  = 3,  /* Invalidate dentry cache */
    FUSE_NOTIFY_STORE        = 4,  /* Store data in page cache */
    FUSE_NOTIFY_RETRIEVE     = 5,  /* Retrieve data from page cache */
    FUSE_NOTIFY_DELETE       = 6,  /* Delete a dentry */
    FUSE_NOTIFY_RESEND       = 7,  /* Resend pending requests */
    FUSE_NOTIFY_INC_EPOCH    = 8,  /* Increment inode epoch */
    FUSE_NOTIFY_PRUNE        = 9,  /* Prune inodes */
};
```

These are essential for cache coherency in multi-client scenarios (e.g., a
network filesystem where another client modifies a file).

## 4. Kernel Data Structures

The FUSE kernel module centers around five key structures. Understanding their
relationships is critical to understanding the implementation.

```
                    ┌───────────────────────┐
                    │    struct fuse_conn    │  One per filesystem connection
                    │                       │
                    │  • iq (input queue)    │─── requests waiting for daemon
                    │  • flags/capabilities  │
                    │  • max_read/max_write  │
                    │  • devices list        │─── multiple /dev/fuse fds
                    │  • mounts list         │─── multiple submounts
                    └───────────┬───────────┘
                                │ 1:N
                   ┌────────────┼────────────┐
                   ▼                         ▼
        ┌──────────────────┐      ┌──────────────────┐
        │ struct fuse_mount│      │  struct fuse_dev  │  One per open
        │                  │      │                   │  /dev/fuse fd
        │  • fc (conn)     │      │  • fc (conn)      │
        │  • sb (superblk) │      │  • pq (proc queue)│── requests being processed
        └────────┬─────────┘      └───────────────────┘
                 │
                 │ sb->s_fs_info
                 ▼
        ┌──────────────────┐           ┌──────────────────┐
        │  struct inode     │──────────►│ struct fuse_inode │
        │  (VFS inode)      │container  │                  │
        │                   │   of      │  • nodeid        │
        └──────────────────┘           │  • nlookup       │
                                       │  • attr_version  │
                                       │  • state flags   │
                                       └──────────────────┘
```

### `struct fuse_conn` — The Connection

Defined in `fs/fuse/fuse_i.h`. This is the heart of a FUSE filesystem instance.
One `fuse_conn` per mounted filesystem (or shared among submounts for virtiofs).

**Key fields**:

```c
struct fuse_conn {
    /* --- Protocol state --- */
    unsigned minor;                      /* Negotiated protocol minor version */
    unsigned connected;                  /* Connection is alive */
    bool aborted;                        /* Connection was aborted */
    int initialized;                     /* INIT handshake completed */

    /* --- Request queues --- */
    struct fuse_iqueue iq;               /* Input queue (pending requests) */
                                         /* iq.reqctr: request ID counter */
    atomic_t num_waiting;                /* Total waiting requests */

    /* --- Background request management --- */
    unsigned max_background;             /* Max concurrent background requests */
    unsigned congestion_threshold;       /* Writeback throttle threshold */
    unsigned num_background;             /* Current background request count */
    struct list_head bg_queue;           /* Overflow queue for background reqs */

    /* --- I/O limits --- */
    unsigned max_read;                   /* Max bytes per READ */
    unsigned max_write;                  /* Max bytes per WRITE */
    unsigned int max_pages;              /* Max pages per request */

    /* --- Capability flags (set during INIT) --- */
    unsigned async_read:1;
    unsigned writeback_cache:1;
    unsigned parallel_dirops:1;
    unsigned do_readdirplus:1;
    unsigned posix_acl:1;
    unsigned int passthrough:1;
    unsigned cache_symlinks:1;
    unsigned no_open:1;                  /* Server doesn't need OPEN (ENOSYS) */
    unsigned no_opendir:1;               /* Server doesn't need OPENDIR */
    /* ... many more no_* flags for ENOSYS fallback ... */

    /* --- Attribute cache --- */
    atomic64_t attr_version;             /* Incremented on attr changes */

    /* --- Device list --- */
    struct list_head devices;            /* All fuse_dev for this conn */
    struct list_head mounts;             /* All fuse_mount for this conn */
};
```

### `struct fuse_mount` — The Mount Point

```c
struct fuse_mount {
    struct fuse_conn *fc;           /* The backing connection */
    struct super_block *sb;         /* VFS superblock */
    struct list_head fc_entry;      /* Entry in fc->mounts */
};
```

Accessed via `get_fuse_mount(inode)` which does `inode->i_sb->s_fs_info`.

### `struct fuse_inode` — Per-Inode FUSE State

Embedded inside the VFS `struct inode` (allocated together via the inode slab cache).
Accessed via `get_fuse_inode(inode)` which uses `container_of`.

```c
struct fuse_inode {
    u64 nodeid;                     /* FUSE node ID (server's identifier) */
    u64 nlookup;                    /* Lookup reference count */
    u64 attr_version;               /* Tracks cache validity */
    unsigned long state;            /* Bitmask of FUSE_I_* flags (set_bit) */

    struct fuse_forget_link *forget; /* For batched FORGET messages */

    /* For regular files: */
    struct list_head write_files;    /* Open files with write access */
    int writectr;                    /* Write reference counter */
    wait_queue_head_t page_waitq;    /* Page invalidation waitqueue */

    /* For directories: */
    struct {
        bool cached;                 /* Readdir cache is valid */
        loff_t size;                 /* Cached directory size */
        u64 version;                 /* Cache version */
    } rdc;
};
```

**The `nodeid`**: This is how the kernel identifies inodes to the daemon. The root
inode is always `FUSE_ROOT_ID` (1). The daemon assigns nodeids in LOOKUP/CREATE
replies. The kernel uses `iget5_locked()` to find-or-create VFS inodes keyed by nodeid.

**The `nlookup` count**: Every successful LOOKUP increments `nlookup`. When the VFS
evicts the inode (`fuse_evict_inode()`), the kernel sends `FUSE_FORGET` with the
current nlookup count so the daemon can release its reference. `FUSE_FORGET` is
fire-and-forget — it never gets a reply.

### `struct fuse_req` — A Request In Flight

```c
struct fuse_req {
    struct list_head list;           /* Link in whatever queue it's on */
    struct {
        struct fuse_in_header h;     /* Request header (access via req->in.h) */
    } in;
    struct {
        struct fuse_out_header h;    /* Reply header (access via req->out.h) */
    } out;
    struct fuse_args *args;          /* Operation arguments (has end callback) */
    unsigned long flags;             /* enum fuse_req_flag bitmask */
    wait_queue_head_t waitq;         /* Synchronous callers wait here */
    refcount_t count;                /* Reference count */
    unsigned long create_time;       /* jiffies timestamp for timeout */
};
```

**Request flags** (`enum fuse_req_flag`):
- `FR_ISREPLY` — expects a reply (most operations; not FORGET)
- `FR_BACKGROUND` — asynchronous background request
- `FR_PENDING` — on the pending list (not yet read by daemon)
- `FR_SENT` — has been read by the daemon
- `FR_FINISHED` — reply received, request is done
- `FR_FORCE` — send even under memory pressure
- `FR_INTERRUPTED` — caller was interrupted (triggers FUSE_INTERRUPT)

### `struct fuse_args` — Operation Arguments

```c
struct fuse_args {
    uint64_t nodeid;                 /* Target inode */
    uint32_t opcode;                 /* FUSE opcode */
    uint8_t in_numargs;              /* Number of input argument segments */
    uint8_t out_numargs;             /* Number of output argument segments */
    bool force:1;                    /* Force send */
    bool noreply:1;                  /* No reply expected (FORGET) */
    bool nocreds:1;                  /* Don't fill credentials */
    struct fuse_in_arg in_args[4];   /* Up to 4 input segments */
    struct fuse_arg out_args[2];     /* Up to 2 output segments */
    void (*end)(...);                /* Async completion callback */
};
```

**The argument construction pattern** used throughout the kernel FUSE code:

```c
/* Every FUSE operation follows this pattern (example: GETATTR) */
FUSE_ARGS(args);                              /* Stack-allocate fuse_args */
args.opcode = FUSE_GETATTR;
args.nodeid = get_node_id(inode);
args.in_numargs = 1;
args.in_args[0].size = sizeof(inarg);
args.in_args[0].value = &inarg;
args.out_numargs = 1;
args.out_args[0].size = sizeof(outarg);
args.out_args[0].value = &outarg;
err = fuse_simple_request(fm, &args);         /* Send and wait for reply */
```

## 5. Interfaces & Examples

FUSE has five distinct interfaces. Understanding each one — and how they connect —
is key to understanding the system as a whole.

### Interface 1: `/dev/fuse` — The Communication Channel

`/dev/fuse` is a character device (misc device, minor 229) that serves as the
communication pipe between kernel and daemon. The daemon opens it, passes the fd
to `mount()`, and then uses it to read requests and write replies.

**Kernel implementation**: `fs/fuse/dev.c`

**File operations** (`fuse_dev_operations`):

```c
const struct file_operations fuse_dev_operations = {
    .open          = fuse_dev_open,
    .read_iter     = fuse_dev_read,       /* Daemon reads requests */
    .write_iter    = fuse_dev_write,      /* Daemon writes replies */
    .splice_read   = fuse_dev_splice_read,  /* Zero-copy request read */
    .splice_write  = fuse_dev_splice_write, /* Zero-copy reply write */
    .poll          = fuse_dev_poll,        /* Daemon polls for requests */
    .release       = fuse_dev_release,    /* Daemon closes the fd */
    .unlocked_ioctl = fuse_dev_ioctl,     /* CLONE, BACKING_OPEN, etc. */
    .uring_cmd     = fuse_uring_cmd,      /* io_uring transport */
};
```

**How reading works** (`fuse_dev_do_read` in `dev.c`):

1. The daemon calls `read()` on `/dev/fuse`
2. The kernel waits on `fiq->waitq` until a request is available
3. Requests are dequeued in priority order:
   - **Interrupt requests** first (from `fiq->interrupts`)
   - **Forget requests** second (from `fiq->forget_list`)
   - **Regular requests** last (from `fiq->pending`)
4. For regular requests: the request is moved to `fpq->processing[hash]` and
   marked `FR_SENT`
5. The `fuse_in_header` + operation arguments are copied to the daemon's buffer

**How writing works** (`fuse_dev_do_write` in `dev.c`):

1. The daemon calls `write()` on `/dev/fuse`
2. The kernel reads the `fuse_out_header` to get `unique` and `error`
3. If `unique == 0`: this is a **notification** — dispatched to `fuse_notify()`
4. Otherwise: the kernel looks up the matching request in `fpq->processing[]`
5. Output arguments are copied into the request's output buffers
6. `fuse_request_end()` completes the request, waking the original caller

**IOCTLs**:

| IOCTL | Purpose |
|-------|---------|
| `FUSE_DEV_IOC_CLONE` | Clone the fd for multi-threaded daemons |
| `FUSE_DEV_IOC_BACKING_OPEN` | Register a backing file for passthrough I/O |
| `FUSE_DEV_IOC_BACKING_CLOSE` | Unregister a backing file |
| `FUSE_DEV_IOC_SYNC_INIT` | Use synchronous INIT mode |

### Interface 2: VFS → FUSE Dispatch

This is where VFS operations become FUSE protocol messages. Let's trace two
real operations end-to-end.

#### Trace: `ls /mnt/fuse/` (readdir)

```
   ls calls getdents64()
        │
        ▼
   VFS calls fuse_readdir(file, ctx)              [readdir.c:587]
        │
        ├── Is FOPEN_CACHE_DIR set?
        │   YES: Try fuse_readdir_cached()         [readdir.c:445]
        │        If cache valid → parse from page cache, return
        │
        ├── fuse_readdir_uncached()                [readdir.c:334]
        │   │
        │   ├── Decide: READDIR vs READDIRPLUS?
        │   │   • Uses READDIRPLUS if fc->do_readdirplus
```

### Interface 3: High-Level libfuse API (`struct fuse_operations`)

The high-level API is the easiest way to write a FUSE filesystem. You implement
callbacks that receive **file paths** and return error codes. libfuse handles
inode management, path resolution, and all the protocol details internally.

**Complete hello.c example** (from `libfuse/example/hello.c`):

```c
#define FUSE_USE_VERSION 31
#include <fuse.h>
#include <string.h>
#include <errno.h>

/* A trivial FS: root directory containing one file "hello" */

static const char *hello_path = "/hello";
static const char *hello_str  = "Hello World!\n";

/*
 * getattr: called for stat(), lstat(), fstat().
 * Fill struct stat with file metadata. Return 0 or -errno.
 */
static int hello_getattr(const char *path, struct stat *stbuf,
                          struct fuse_file_info *fi)
{
    memset(stbuf, 0, sizeof(struct stat));
    if (strcmp(path, "/") == 0) {
        stbuf->st_mode = S_IFDIR | 0755;
        stbuf->st_nlink = 2;
    } else if (strcmp(path, hello_path) == 0) {
        stbuf->st_mode = S_IFREG | 0444;
        stbuf->st_nlink = 1;
        stbuf->st_size = strlen(hello_str);
    } else {
        return -ENOENT;
    }
    return 0;
}

/*
 * readdir: called for ls.
 * Use the filler() callback to add each entry.
 */
static int hello_readdir(const char *path, void *buf,
                          fuse_fill_dir_t filler, off_t offset,
                          struct fuse_file_info *fi,
                          enum fuse_readdir_flags flags)
{
    if (strcmp(path, "/") != 0)
        return -ENOENT;

    filler(buf, ".", NULL, 0, FUSE_FILL_DIR_DEFAULTS);
    filler(buf, "..", NULL, 0, FUSE_FILL_DIR_DEFAULTS);
    filler(buf, "hello", NULL, 0, FUSE_FILL_DIR_DEFAULTS);
    return 0;
}

/*
 * open: called when a file is opened.
 * Validate access; optionally store state in fi->fh.
 */
static int hello_open(const char *path, struct fuse_file_info *fi)
{
    if (strcmp(path, hello_path) != 0)
        return -ENOENT;
    if ((fi->flags & O_ACCMODE) != O_RDONLY)
        return -EACCES;
    return 0;
}

/*
 * read: called to read data from an open file.
 * Return number of bytes read, or 0 for EOF.
 */
static int hello_read(const char *path, char *buf, size_t size,
                       off_t offset, struct fuse_file_info *fi)
{
    size_t len = strlen(hello_str);
    if (offset >= len)
        return 0;
    if (offset + size > len)
        size = len - offset;
    memcpy(buf, hello_str + offset, size);
    return size;
}

static const struct fuse_operations hello_oper = {
    .getattr = hello_getattr,
    .readdir = hello_readdir,
    .open    = hello_open,
    .read    = hello_read,
};

int main(int argc, char *argv[])
{
    /* fuse_main handles everything: mount, event loop, unmount */
    return fuse_main(argc, argv, &hello_oper, NULL);
}
```

```bash

mkdir /tmp/fuse_mount
./hello /tmp/fuse_mount
ls /tmp/fuse_mount/           # Shows: hello
cat /tmp/fuse_mount/hello     # Shows: Hello World!
```

### Interface 4: Low-Level libfuse API (`struct fuse_lowlevel_ops`)

The low-level API gives you direct control over the FUSE protocol. Callbacks
receive **inode numbers** instead of paths, and you must explicitly reply to
each request. This is more work but more efficient — no path-to-inode translation.

**Annotated hello_ll.c** (from `libfuse/example/hello_ll.c`):

```c
#define FUSE_USE_VERSION FUSE_MAKE_VERSION(3, 12)
#include <fuse_lowlevel.h>
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <errno.h>

static const char *hello_str = "Hello World!\n";
static const char *hello_name = "hello";

/* Root inode is always 1, our file is inode 2 */
static int hello_stat(fuse_ino_t ino, struct stat *stbuf)
{
    stbuf->st_ino = ino;
    switch (ino) {
    case 1:  /* root directory */
        stbuf->st_mode = S_IFDIR | 0755;
        stbuf->st_nlink = 2;
        break;
    case 2:  /* "hello" file */
        stbuf->st_mode = S_IFREG | 0444;
        stbuf->st_nlink = 1;
        stbuf->st_size = strlen(hello_str);
        break;
    default:
        return -1;
    }
    return 0;
}

/*
 * lookup: resolve a name in a directory.
 * Must reply with fuse_entry_param containing the inode number,
 * attributes, and cache timeouts.
 * Each successful reply increments the kernel's nlookup count.
 */
static void hello_ll_lookup(fuse_req_t req, fuse_ino_t parent,
                             const char *name)
{
    struct fuse_entry_param e;

    if (parent != 1 || strcmp(name, hello_name) != 0) {
        fuse_reply_err(req, ENOENT);
        return;
    }

    memset(&e, 0, sizeof(e));
    e.ino = 2;                    /* Our file's inode number */
    e.attr_timeout = 1.0;        /* Cache attrs for 1 second */
    e.entry_timeout = 1.0;       /* Cache name→inode for 1 second */
    hello_stat(e.ino, &e.attr);
    fuse_reply_entry(req, &e);   /* Must reply! */
}

/*
 * getattr: get file attributes by inode number.
 * Reply with fuse_reply_attr() providing struct stat and timeout.
 */
static void hello_ll_getattr(fuse_req_t req, fuse_ino_t ino,
                              struct fuse_file_info *fi)
{
    struct stat stbuf = {0};
    if (hello_stat(ino, &stbuf) == -1)
        fuse_reply_err(req, ENOENT);
    else
        fuse_reply_attr(req, &stbuf, 1.0);  /* 1.0s cache timeout */
}

/*
 * readdir: read directory entries.
 * Build a buffer using fuse_add_direntry(), then reply with the
 * portion the caller requested (offset + size).
 */
static void hello_ll_readdir(fuse_req_t req, fuse_ino_t ino,
                              size_t size, off_t off,
                              struct fuse_file_info *fi)
{
    if (ino != 1) {
        fuse_reply_err(req, ENOTDIR);
        return;
    }

    /* Build complete directory listing */
    struct dirbuf { char *p; size_t size; } b = {0};
    /* ... add ".", "..", "hello" entries ... */
    /* Reply with the requested slice */
    reply_buf_limited(req, b.p, b.size, off, size);
    free(b.p);
}

/*
 * open: validate access, reply with fuse_reply_open().
 */
static void hello_ll_open(fuse_req_t req, fuse_ino_t ino,
                           struct fuse_file_info *fi)
{
    if (ino != 2)
        fuse_reply_err(req, EISDIR);
    else if ((fi->flags & O_ACCMODE) != O_RDONLY)
        fuse_reply_err(req, EACCES);
    else
        fuse_reply_open(req, fi);
}

/*
 * read: serve file data. Reply with fuse_reply_buf().
 */
static void hello_ll_read(fuse_req_t req, fuse_ino_t ino,
                           size_t size, off_t off,
                           struct fuse_file_info *fi)
{
    /* Serve the requested range of hello_str */
    reply_buf_limited(req, hello_str, strlen(hello_str), off, size);
}

static const struct fuse_lowlevel_ops hello_ll_oper = {
    .lookup  = hello_ll_lookup,
    .getattr = hello_ll_getattr,
    .readdir = hello_ll_readdir,
    .open    = hello_ll_open,
    .read    = hello_ll_read,
};

int main(int argc, char *argv[])
{
    struct fuse_args args = FUSE_ARGS_INIT(argc, argv);
    struct fuse_session *se;
    struct fuse_cmdline_opts opts;

    fuse_parse_cmdline(&args, &opts);

    /* Create session with our low-level operations */
    se = fuse_session_new(&args, &hello_ll_oper,
                          sizeof(hello_ll_oper), NULL);

    fuse_set_signal_handlers(se);
    fuse_session_mount(se, opts.mountpoint);
    fuse_daemonize(opts.foreground);

    /* Run the event loop (single or multi-threaded) */
    if (opts.singlethread)
        fuse_session_loop(se);
    else {
        struct fuse_loop_config *config = fuse_loop_cfg_create();
        fuse_loop_cfg_set_clone_fd(config, opts.clone_fd);
        fuse_session_loop_mt(se, config);
        fuse_loop_cfg_destroy(config);
    }

    /* Cleanup */
    fuse_session_unmount(se);
    fuse_remove_signal_handlers(se);
    fuse_session_destroy(se);
    free(opts.mountpoint);
    fuse_opt_free_args(&args);
    return 0;
}
```

**High-level vs Low-level API comparison**:

| Aspect | High-Level (`fuse_operations`) | Low-Level (`fuse_lowlevel_ops`) |
|--------|-------------------------------|--------------------------------|
| Addressing | File paths (`"/dir/file"`) | Inode numbers (`fuse_ino_t`) |
| Response | Return int (0 or -errno) | Call `fuse_reply_*()` explicitly |
| Path resolution | Automatic (libfuse internal) | You manage it |
| Inode lifecycle | Automatic | You handle `forget` callbacks |
| Setup | `fuse_main()` (one line) | Manual: session_new → mount → loop → teardown |
| Performance | Slower (path→inode lookup) | Faster (direct inode access) |
| setattr | Split into `chmod`/`chown`/`truncate`/`utimens` | Single `setattr` callback |
| readdirplus | Not exposed | Available via `readdirplus` callback |

### Interface 5: Notifications (Daemon → Kernel)

Normally, the kernel sends requests and the daemon replies. But the daemon can
also proactively send **notifications** to the kernel to manage the page cache
and dentry cache. This is critical for correctness in scenarios where data
changes externally (e.g., network filesystems, multi-client access).

The daemon writes a message with `unique = 0` to `/dev/fuse`:

```c
/* Low-level libfuse notification functions */

/* Invalidate inode attributes and/or page cache */
fuse_lowlevel_notify_inval_inode(se, ino, offset, len);

/* Invalidate a directory entry (cached name→inode mapping) */
fuse_lowlevel_notify_inval_entry(se, parent_ino, name, namelen);

/* Delete a directory entry (stronger than inval_entry) */
fuse_lowlevel_notify_delete(se, parent_ino, child_ino, name, namelen);

/* Store data directly into the kernel's page cache */
fuse_lowlevel_notify_store(se, ino, offset, bufvec, flags);

/* Retrieve data from the kernel's page cache */
fuse_lowlevel_notify_retrieve(se, ino, size, offset, cookie);

/* Wake up poll waiters */
fuse_lowlevel_notify_poll(ph);
```

**Kernel-side handling** (in `dev.c`, `fuse_notify()`):

When the kernel receives a notification (a write with `unique == 0`):
- `FUSE_NOTIFY_INVAL_INODE` → `fuse_notify_inval_inode()` — calls
  `invalidate_inode_pages2_range()` to evict page cache entries
- `FUSE_NOTIFY_INVAL_ENTRY` → `fuse_notify_inval_entry()` — calls
  `d_invalidate()` on the matching dentry
- `FUSE_NOTIFY_STORE` → `fuse_notify_store()` — directly inserts data
  into the inode's page cache (push model instead of pull)
- `FUSE_NOTIFY_DELETE` → `fuse_notify_delete()` — like INVAL_ENTRY but
  also calls `d_delete()` for immediate removal

### Readdir Implementation (`fs/fuse/readdir.c`)

This file has the full readdir pipeline:

```
fuse_readdir()
     │
     ├── Cached? → fuse_readdir_cached()
     │              Read from page cache
     │              (entries stored as fuse_dirent structs)
     │
     └── Uncached → fuse_readdir_uncached()
                     │
                     ├── READDIR mode:
                     │   parse_dirfile() → extract fuse_dirent entries
                     │
                     └── READDIRPLUS mode:
                         parse_dirplusfile() →
                           extract fuse_direntplus entries →
                             dir_emit() + fuse_direntplus_link()
                             (populates dcache with inodes!)
```

**READDIRPLUS** is particularly clever: it combines a directory listing with
attribute lookups for every entry. When you `ls -l` a directory, the kernel
would normally do READDIR + one LOOKUP per entry. With READDIRPLUS, it's a
single round trip. The kernel also pre-populates the dentry cache, so
subsequent `stat()` calls are cache hits.

Simple and direct — each xattr operation maps 1:1 to a FUSE opcode:

## 7. Advanced Features

### Writeback Cache

Normally, FUSE uses **writethrough** semantics: each `write()` immediately sends
a `FUSE_WRITE` to the daemon. With `FUSE_WRITEBACK_CACHE` negotiated, the kernel
caches writes in the page cache and flushes them asynchronously.

**Benefits**: Dramatically better write performance (batching, reordering).
**Trade-off**: Data may be lost if the daemon crashes before writeback completes.
**Kernel implementation**: `fuse_writepages()` in `file.c` uses the iomap
infrastructure to batch dirty folios into `FUSE_WRITE` requests with the
`FUSE_WRITE_CACHE` flag.

### Passthrough I/O (`fs/fuse/passthrough.c`, `fs/fuse/backing.c`)

Passthrough allows the kernel to directly forward read/write/mmap operations
to a **backing file** on a local filesystem, completely bypassing the daemon
for data operations.

**Setup flow**:
1. Daemon opens a local file and registers it via `FUSE_DEV_IOC_BACKING_OPEN`
   ioctl → returns a `backing_id`
2. When the daemon handles `FUSE_OPEN`, it sets `FOPEN_PASSTHROUGH` and
   includes the `backing_id` in the reply
3. All subsequent reads/writes go directly to the backing file

**Use case**: Android's FUSE layer uses this extensively. The daemon handles
permission checks and metadata, but data I/O achieves near-native performance
by going directly to the underlying ext4/f2fs file.

```c
/* In passthrough.c: read bypass path */
ssize_t fuse_passthrough_read_iter(struct kiocb *iocb, struct iov_iter *iter)
{
    /* Read directly from backing file — daemon is never contacted */
    return backing_file_read_iter(iocb, iter, ...);
}
```

**Safety**: The `max_stack_depth` check prevents recursive stacking (FUSE on
FUSE on FUSE...). `CAP_SYS_ADMIN` is required to register backing files.

### DAX (Direct Access) — `fs/fuse/dax.c`

DAX is designed for **virtiofs** (FUSE over virtio for VM guests). It allows
the guest to directly map host memory, completely eliminating data copies.

**How it works**:
1. The host exposes a DAX window (shared memory region) to the guest
2. The guest FUSE driver maps file regions into 2 MB slots within the window
3. File I/O uses `dax_iomap_rw()` for reads/writes and `dax_iomap_fault()`
   for page faults — all zero-copy

**Special opcodes**: `FUSE_SETUPMAPPING` (48) and `FUSE_REMOVEMAPPING` (49)
manage the mapping lifecycle. When the fixed-size window is full, the kernel
reclaims mappings from other inodes.

### io_uring Transport — `fs/fuse/dev_uring.c`

The traditional FUSE protocol requires two system calls per operation:
`read()` to get a request, `write()` to send a reply. The io_uring transport
eliminates both by using io_uring's submission/completion queues.

**Key innovation**: The `COMMIT_AND_FETCH` command atomically delivers a reply
AND picks up the next request in a single io_uring CQE. This halves the
system call overhead.

```
Traditional /dev/fuse:          io_uring transport:
  read()  → get request           ┐
  process                         │ COMMIT_AND_FETCH
  write() → send reply            │ (single CQE)
  read()  → get request           ┘
  process
  write() → send reply
  ...

  6 syscalls for 3 ops            3 CQEs for 3 ops
```

**Setup**: The daemon submits `FUSE_IO_URING_CMD_REGISTER` SQEs (one per
ring entry per queue). Each SQE provides header and payload buffers. Once all
queues are ready, the ring switches from `/dev/fuse` mode to io_uring mode.

**Per-CPU queues**: Requests are routed by `task_cpu(current)`, providing
CPU-local processing without cross-CPU contention.

### CUSE (Character Device in Userspace) — `fs/fuse/cuse.c`

CUSE extends FUSE from filesystems to **character devices**. The daemon opens
`/dev/cuse`, performs a `CUSE_INIT` handshake (providing a device name and
optional major/minor), and the kernel creates `/dev/<name>`. Applications
opening that device have their operations forwarded to the daemon.

```
Application      Kernel                    CUSE Daemon
    │               │                          │
    │ open(/dev/foo) │                          │
    ├───────────────►│                          │
    │               │ FUSE_OPEN ──────────────►│
    │               │◄──────────── reply ──────│
    │               │                          │
    │ write(fd,..)  │                          │
    ├───────────────►│                          │
    │               │ FUSE_WRITE ─────────────►│
    │               │◄──────────── reply ──────│
```

CUSE reuses almost all FUSE infrastructure — `fuse_conn`, `fuse_dev`,
`fuse_direct_io()`, `fuse_do_ioctl()`. The main difference is that I/O
always uses direct I/O (no page cache) and there's no real filesystem mount.

## 8. Mount & Unmount Flow

### Mount (End-to-End)

```
1. User runs:  ./hello /mnt/fuse
                  │
2. libfuse:       ├── Open /dev/fuse → get fd N
                  ├── fork+exec fusermount3:
                  │     fusermount3 calls mount("hello", "/mnt/fuse", "fuse.hello",
                  │                             MS_NOSUID|MS_NODEV,
                  │                             "fd=N,rootmode=40000,user_id=1000,...")
                  │     kernel creates superblock via fuse_fill_super()
                  │
3. Kernel:        ├── fuse_get_tree()                       [inode.c ~1975]
                  │   ├── fuse_conn_init()                 initialize connection
                  │   ├── fuse_fill_super()                [inode.c ~1930]
                  │   │   └── fuse_fill_super_common()     set up superblock
                  │   │       ├── Create root inode (nodeid=1)
                  │   │       └── Create root dentry
                  │   └── fuse_send_init()                 send FUSE_INIT
                  │       └── (background request with process_init_reply callback)
                  │
4. Daemon:        ├── fuse_session_loop() starts
                  │   ├── read(/dev/fuse) → gets FUSE_INIT request
                  │   ├── libfuse processes INIT internally:
                  │   │   - Negotiates capabilities
                  │   │   - Sets max_write, max_readahead
                  │   │   - Calls user's init() callback
                  │   └── write(/dev/fuse) → sends FUSE_INIT reply
                  │
5. Kernel:        └── process_init_reply()
                      ├── Configure fc->* flags from negotiated caps
                      ├── Set fc->max_read, fc->max_write, fc->max_pages
                      ├── fc->initialized = 1
                      └── Wake all waiters (filesystem is now usable!)
```

### Unmount (End-to-End)

```
1. User runs:  fusermount3 -u /mnt/fuse
               OR: daemon exits / is killed
                  │
2. Kernel:        ├── fuse_kill_sb_anon() / fuse_umount_begin()  [inode.c]
                  │   ├── fc->connected = 0
                  │   ├── fuse_abort_conn(fc):
                  │   │   ├── fiq->connected = 0
                  │   │   ├── For each fuse_dev: fpq->connected = 0
                  │   │   ├── Drain all queues (pending, processing, bg)
                  │   │   ├── End all pending requests with error
                  │   │   └── Wake all waiters
                  │   └── fuse_conn_put(fc) → may free connection
                  │
3. VFS:           ├── Evict all inodes:
                  │   └── fuse_evict_inode() for each inode:
                  │       └── Send FUSE_FORGET(nodeid, nlookup) to daemon
                  │           (only while SB_ACTIVE; fire-and-forget, no reply)
                  │
4. Daemon:        └── read() returns error or 0
                      └── fuse_session_loop exits
                      └── cleanup: fuse_session_unmount, fuse_session_destroy
```

### Connection Abort

A FUSE connection can be aborted (killing all pending operations) via:
- **Kill the daemon**: Triggers `fuse_dev_release()` → `fuse_abort_conn()`
- **Forced unmount**: `umount -f /mnt/fuse`
- **fusectl**: `echo 1 > /sys/fs/fuse/connections/<id>/abort`
- **Kill all users**: If all processes using the mount die

### Common Patterns

**-ENOSYS fallback**: When the daemon returns `-ENOSYS` for an opcode, the
kernel sets a `fc->no_*` flag and never sends that opcode again. This enables
graceful feature degradation — a daemon only needs to implement the operations
it actually supports.

**Attribute caching**: Every LOOKUP/CREATE/SETATTR reply includes attributes
and cache timeouts. The kernel caches these and uses `attr_version` (a
per-connection counter) to detect staleness. An `inval_mask` allows
fine-grained invalidation of specific attributes.

**Lookup reference counting**: Every successful LOOKUP increments
`fi->nlookup` on the inode. When the VFS evicts the inode, the kernel sends
`FUSE_FORGET(nlookup)` so the daemon can release its reference. The daemon
must not free its internal state for an inode until the matching FORGET
arrives.
